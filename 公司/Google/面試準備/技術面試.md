# Google SWE - 技術面試

## 💻 Coding 面試
- **形式**：線上輪用**共享編輯器**（Google Docs / 線上 IDE，通常**無自動執行/補全**，要能手寫出可執行程式）；onsite 可能在 Chromebook 或白板上寫。
- **輪數**：onsite 通常 coding × 3-4 輪，每輪 45 分鐘、1-2 題。

### 建議流程（每題）
1. **釐清題意**：問清輸入輸出、範圍、限制、edge cases（空輸入、重複、負數、超大）。
2. **舉例**：用具體例子確認理解，順便當測資。
3. **講想法**：先講高階思路與資料結構選擇；若有多解，比較取捨。
4. **講複雜度**：**動手寫之前**先說 time / space complexity，並說明能否優化。
5. **Coding**：寫**乾淨、可執行**的程式，命名清楚、拆函式，邊寫邊解釋。
6. **測試**：用步驟 2 的例子 + edge cases 走一遍（dry run），主動找 bug。

### 重點技巧
- **邊想邊講（think aloud）**：面試官評估的是思考過程，不是只看最終答案。
- **不要卡死沉默**：卡住時講出你在想什麼、嘗試的方向；面試官會給提示，能接住提示是加分。
- **先求正確可跑，再優化**：可先寫暴力解再講如何改進。
- **主動處理 edge cases** 與輸入驗證。
- **常見主題**：依 [學習方向](../學習資料/學習方向.md) 的 DSA 路線；以 **Medium** 為主、偶有 Hard，常見 Arrays/Hashing、Trees、Graphs、DP、Two Pointers。

## 🏗️ System Design 面試（資深 L5+ 必考）
- **形式**：45 分鐘，白板/共享文件畫架構並討論。

### 建議流程
1. **需求釐清**：功能需求 + 非功能需求（QPS、延遲、可用性、一致性）。
2. **容量估算**：使用者數、讀寫比、儲存量、頻寬（back-of-envelope）。
3. **API 設計**：定義核心端點與資料模型。
4. **高階架構**：client → LB → service → cache → DB，畫出資料流。
5. **細節深掘**：DB schema、分庫分表、快取策略、Queue、索引。
6. **瓶頸 / 取捨**：找單點故障、scale 瓶頸，說明 CAP 取捨與替代方案。

- **常見題**：URL Shortener、News Feed、Rate Limiter、Chat、Typeahead、YouTube、Google Drive（見 [練習題目](../學習資料/練習題目.md)）。
- **評分點**：scalability、清楚說明 **trade-off**、結構化溝通、能主動收斂範圍。

- **Pramp**（免費配對）、**interviewing.io**（匿名真人）、或找同伴互練。
- 每週至少 1 次，重點練「邊想邊講 + 計時」。

## 📌 面試官 / 流程小知識
- 每位面試官填**評分卡**，最終由 **Hiring Committee** 綜合判斷，因此每一輪都重要、表現要穩定。
- 通過後進入 **team match**，準備好簡述你的興趣方向與想做的領域。
