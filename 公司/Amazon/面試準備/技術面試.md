# Amazon - 技術面試

> Amazon 技術面試含 **Coding** 與 **System Design**。注意：技術輪也會穿插 **Leadership Principles** 問題（行為面試見 [行為面試.md](./行為面試.md)）。

## 💻 Coding 面試
- **形式**：線上協作編輯器（如 Amazon Chime / CoderPad 類工具）；virtual loop 為主，無 IDE 自動補全。
- **時間**：約 45~60 分鐘，常為 1~2 題（或 1 大題 + follow-up），需保留時間回答 LP。
- **難度**：以 LeetCode Easy~Medium 為主，偶見 Hard 或 Medium 的多重 follow-up。

### 解題流程（務必照做）
1. **釐清題意**：複述問題、確認輸入輸出、邊界（空輸入、重複、負數、極大值）、是否可改原陣列。
2. **舉例**：自己舉一個小例子走過一遍，確認理解一致。
3. **講想法**：先說 brute force，再說優化方向；**先講複雜度再動手寫**。
4. **分析複雜度**：時間/空間 Big-O，並說明取捨。
5. **Coding**：邊寫邊講（think aloud），命名清楚、模組化。
6. **測試**：用例子手動 trace、補 edge cases、討論可能的 bug。

### 重點技巧
- **邊想邊講（think aloud）**——溝通是評分核心，沉默是大忌。
- **先講複雜度再寫**，並主動比較不同解法取捨。
- **主動處理 edge cases**（空集合、單元素、溢位、cycle）。
- 寫**可執行、可讀**的程式碼，而非偽碼；變數命名能體現意圖。
- 卡關時說出思路、請面試官給 hint，展現 **Dive Deep** 與冷靜。
- **常見主題**：見 [學習方向 DSA 路線](../學習資料/學習方向.md) 與 [練習題目](../學習資料/練習題目.md)。

## 🏗️ System Design 面試（L5 / SDE II 以上必考）
- **流程**：
  1. **需求釐清**：功能性 vs 非功能性需求（規模、延遲、可用性、一致性要求）。
  2. **估算（Capacity Estimation）**：DAU、QPS（讀/寫）、儲存量、頻寬、cache 大小。
  3. **API 設計**：定義主要端點與資料模型。
  4. **高階架構**：畫出元件（client、LB、service、cache、DB、queue、CDN）。
  5. **細節深掘（Dive Deep）**：DB schema、sharding/partition key、cache 策略與失效、一致性、idempotency。
  6. **瓶頸 / 取捨**：找單點故障、熱點、擴展瓶頸，討論 trade-off。
- **常見題**：URL shortener、News Feed、Rate Limiter、Chat、Notification Service、Top K、設計分散式 KV store（DynamoDB 風格）、電商相關（購物車/訂單/庫存）。
- **評分點**：
  - **Scalability**：能否水平擴展、處理熱點與尖峰。
  - **Trade-off 說明**：SQL vs NoSQL、一致性 vs 可用性（CAP）、同步 vs 非同步。
  - **溝通與結構**：由淺入深、主動驅動討論、回應追問。
  - **Amazon 加分**：以實際運作角度談 on-call、監控/告警、failure handling（呼應 Ownership / Dive Deep）。
