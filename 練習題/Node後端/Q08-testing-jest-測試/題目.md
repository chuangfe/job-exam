# Q08 測試（Jest + Supertest）

前端你可能寫過 React Testing Library；後端用 **Jest** 跑邏輯單元測試、用 **Supertest** 對 Express app 發假請求做整合測試（不用真的開 server）。

## 要安裝的套件

```bash
npm install --save-dev jest supertest
```

設定 `package.json`：

```json
{
  "scripts": { "test": "NODE_ENV=test node --experimental-vm-modules node_modules/jest/bin/jest.js" }
}
```

> ES Module + Jest 需要 `--experimental-vm-modules`。若你用 CommonJS，直接 `"test": "jest"` 即可。

---

## 子題 8-1：單元測試（測純函式 / 驗證邏輯）

**目標**：為「驗證 title」的邏輯寫單元測試（不碰 HTTP、不碰 DB，最快最穩）。

**步驟**：
1. 建立 `validate.test.js`。
2. 寫純函式 `isValidTitle(title)`：是非空字串才回 `true`。
3. 用 `describe` / `test` / `expect` 測：正常字串通過、空字串不通過、非字串不通過。

---

## 子題 8-2：整合測試（對整個 API 發假請求）

**目標**：用 Supertest 測 CRUD 流程：新增 → 查詢 → 更新 → 刪除，驗證狀態碼與回傳內容。

**步驟**：
1. 建立 `app.test.js`，`import { createApp } from './app.js'`（Q05 有 `export`）。
2. `beforeEach` 重建 `app = createApp()`，讓每個測試狀態互不污染。
3. 測這些案例：
   - `GET /tasks` 回 200 且為陣列。
   - `POST /tasks` 帶 title 回 201、回傳含 id。
   - `POST /tasks` 缺 title 回 400 且有 `error`。
   - `GET` 不存在的 task 回 404。
   - 完整流程：建立 → 更新（200）→ 刪除（204）→ 再查回 404。

**提示**：`createApp()` 回傳 app，直接餵給 `request(app)`，**不需 `listen`**。

跑測試：

```bash
npm test
```

## 重點

- **單元測試**測純邏輯（快、穩、好定位）；**整合測試**測「請求進來到回應出去」整條路（更接近真實）。
- 用 `beforeEach` 重建狀態，確保測試彼此獨立（前端測試同理）。
- 測**失敗路徑**（400 / 404 / 401）和成功路徑一樣重要——這是後端工程師的基本功，面試常被問。
