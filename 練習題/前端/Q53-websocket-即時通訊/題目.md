# WebSocket 與即時通訊

「聊天室、即時通知、線上協作、股價/看板即時更新、多人遊戲」——這些都需要**伺服器主動把資料推給瀏覽器**。但 HTTP 天生是「client 問、server 才答」的請求／回應模型，要做到即時就得想辦法。請依序回答以下問題，先自己用白話講出來，再對照解析。以瀏覽器原生 `WebSocket` 與 socket.io、React 整合為主要示範環境。

> 提醒：本題與 **Q04 Event Loop**、**Q26 跨分頁溝通**、**Q19 自訂 hook** 互補，建議一起服用。

## Q1. 即時通訊四種方案比較？

- **短輪詢（short polling）**：瀏覽器每隔固定時間（例如 3 秒）發一次 HTTP 請求問「有沒有新資料」。它的優缺點與適用情境是什麼？延遲與伺服器負擔如何？
- **長輪詢（long polling）**：請求送出後伺服器**先不回應**，等到有資料或逾時才回，瀏覽器收到後立刻再發下一個請求。它比短輪詢好在哪？又有什麼限制？
- **SSE（Server-Sent Events）**：`EventSource` 是什麼？它是單向還是雙向？走的是什麼協定？相較 WebSocket 有什麼優點（自動重連、走 HTTP）、有什麼限制（單向、瀏覽器連線數、二進位）？
- **WebSocket**：全雙工、長連線。什麼情境**非它不可**、什麼情境其實用 SSE 或輪詢就夠了？
- 請用一句話總結四者的取捨光譜（實作複雜度 vs 即時性 vs 伺服器成本）。

## Q2. WebSocket 協定本身？

- WebSocket 連線是怎麼「從 HTTP 升級」上來的？請說明 **handshake（HTTP Upgrade）** 的流程：`Upgrade: websocket`、`Connection: Upgrade`、`Sec-WebSocket-Key` / `Sec-WebSocket-Accept`、狀態碼 `101 Switching Protocols`。
- 握手成功之後，這條連線還是 HTTP 嗎？「全雙工（full-duplex）」是什麼意思？跟 HTTP 的半雙工請求/回應差在哪？
- 什麼是 **frame（訊框）**？WebSocket 傳的是 frame 而非完整 HTTP 訊息，這帶來什麼好處（低額外負擔、可傳文字或二進位）？
- `ws://` 與 `wss://` 差在哪？正式環境為什麼一定要用 `wss://`？（提示：加密、避免 proxy/防火牆干擾）
- WebSocket 與 HTTP 共用 80/443 埠有什麼好處？

## Q3. 瀏覽器原生 `WebSocket` API？

- 如何建立連線？`new WebSocket(url)` 之後有哪四個主要事件：`onopen` / `onmessage` / `onclose` / `onerror`，各自何時觸發？
- `readyState` 有哪幾種狀態（`CONNECTING` / `OPEN` / `CLOSING` / `CLOSED`）？為什麼「送訊息前要先檢查 `readyState === OPEN`」？
- `send()` 可以送哪些型別（字串、`ArrayBuffer`、`Blob`）？實務上為什麼常用 JSON 字串？
- `close(code, reason)` 的 code 是什麼？`1000` 正常關閉、`1006` 異常斷線，這對重連邏輯有什麼意義？
- `onerror` 拿得到詳細錯誤原因嗎？瀏覽器基於安全考量對 error 事件做了什麼限制？

## Q4. 可靠性：斷線重連、心跳、訊息可靠性？

- 網路會斷、伺服器會重啟、手機會切換基地台。為什麼「一斷就無限、無延遲地狂重連」是災難？什麼是**指數退避（exponential backoff）**？為什麼還要加 **jitter（隨機抖動）**？
- 什麼是**心跳（heartbeat / ping-pong）**？TCP 都保證送達了，為什麼還需要應用層心跳來偵測斷線？（提示：連線「半開（half-open）」、NAT/proxy 閒置逾時、TCP 偵測太慢）
- 連線斷掉的期間使用者送出的訊息怎麼辦？**訊息佇列 / 補償（offline queue）** 的概念是什麼？重連後怎麼處理？
- 重連後可能收到重複訊息，或訊息順序錯亂。**冪等（idempotency）** 與**訊息序號 / ack** 如何解決？
- 「送出去的訊息對方到底收到沒」怎麼確認？ack（acknowledgement）機制是什麼？

## Q5. socket.io：在原生 WS 上加了什麼？

- socket.io **不是**單純的 WebSocket，它是一套協定 + 函式庫。它在原生 WebSocket 之上內建了哪些功能：**自動重連（含退避）**、**心跳**、**傳輸降級（fallback 到 HTTP long polling）**、**rooms / namespaces**、**ack 回呼**、**廣播（broadcast）**？
- namespace 與 room 差在哪？分別解決什麼問題？
- socket.io 的 client 一定要搭配 socket.io 的 server，能不能用原生 `WebSocket` 連 socket.io？為什麼？
- 什麼情況「用原生 WebSocket 就好、不必上 socket.io」？什麼情況「上 socket.io 明顯划算」？請列出取捨。

- 為什麼要用 `useEffect` 管理 WebSocket 連線？**清理函式（cleanup）** 為什麼一定要 `close()`？不清理會發生什麼？

## Q7. 安全與水平擴展？

- WebSocket 握手是 HTTP，能不能像一般 API 那樣做**驗證（authentication）**？token 要放哪裡（query string vs 連線後第一則訊息 vs cookie）？各有什麼優缺點？
- **授權（authorization）**：使用者連上來之後，怎麼確保他只能加入自己有權限的 room、只收得到該收的訊息？
- WebSocket 有沒有同源政策？CSRF 對它有沒有影響？後端該怎麼檢查 `Origin`？
- **水平擴展**：WebSocket 是有狀態的長連線。當你把服務擴成多台機器（多個 node），「A 使用者連在 server1、B 使用者連在 server2，A 要廣播訊息給 B」會出什麼問題？**Redis adapter / Pub-Sub** 是怎麼解決的？
- 為什麼 WebSocket 服務前面放的 Load Balancer 常需要注意 **sticky session** 或改用其他機制？

## Q8. 實作題：帶「自動重連 + 心跳」的 WebSocket 封裝？

- 請實作一個 `ReliableSocket` 類別（原生 WebSocket 之上），需具備：
  - 斷線後**指數退避 + jitter** 自動重連，並有次數上限。
  - **心跳**：定時送 ping，一段時間內沒收到 pong 就判定連線死掉、主動關閉並觸發重連。
  - **離線訊息佇列**：連線未開時 `send` 的訊息先暫存，連上後補送。
  - 簡單的事件訂閱（`on('message' | 'open' | 'close', cb)`）。
  - 手動 `close()` 後**不再重連**。
- 並示範一個對應的 React `useWebSocket` hook（正確處理 StrictMode 與清理）。
