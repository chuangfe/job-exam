# Q53 - WebSocket 與即時通訊 解析

一句話心智圖：**HTTP 是「client 問才答」，即時通訊就是想辦法讓 server 能主動推。** 手段從「一直問（輪詢）」到「掛著等（長輪詢）」到「單向長連線（SSE）」到「雙向長連線（WebSocket）」，即時性越來越好、實作也越來越重。

```
即時性 / 複雜度 低 ────────────────────────────▶ 高
短輪詢 ── 長輪詢 ── SSE(單向推) ── WebSocket(雙向) ── (+socket.io 補強)
每次都問   掛著等回    EventSource     全雙工長連線      重連/fallback/room/ack
```

---

## Q1. 即時通訊四種方案比較

### 短輪詢（short polling）

瀏覽器用 `setInterval` 每隔固定時間打一次 API 問「有沒有新資料」。

```js
setInterval(async () => {
  const res = await fetch('/api/messages?since=' + lastId);
  const data = await res.json();
  if (data.length) render(data);
}, 3000);
```

- 優點：**實作最簡單**，就是普通 HTTP，任何後端、任何 proxy 都支援，無狀態好擴展。
- 缺點：**延遲 = 輪詢間隔**（間隔越短越即時但請求越多）；大量「空回應」浪費頻寬與伺服器資源；使用者一多，QPS 爆炸。
- 適用：即時性要求低（分鐘級）、資料變動不頻繁，例如「每分鐘更新一次的儀表板」。

### 長輪詢（long polling）

瀏覽器發請求後，伺服器**若沒有新資料就 hold 住連線不回**，直到有資料或逾時（例如 30 秒）才回應；瀏覽器一收到就立刻再發下一個請求。

- 比短輪詢好在：**有新資料幾乎是立刻推**（沒有固定間隔的延遲），空回應也少很多。
- 限制：伺服器要維持大量「掛著」的連線（佔用資源）；每收到一次資料就要重新建立一次 HTTP 請求（有 header 額外負擔）；只是「模擬」推播，不是真雙向。是 WebSocket 普及前的主流方案（也是 socket.io 的 fallback）。

### SSE（Server-Sent Events）

瀏覽器用 `EventSource` 開一條 HTTP 長連線，伺服器持續往這條連線寫入 `text/event-stream` 格式的資料。**單向：只有 server → client。**

```js
const es = new EventSource('/api/stream');
es.onmessage = (e) => console.log('收到', e.data);
es.addEventListener('price', (e) => updatePrice(e.data)); // 具名事件
es.onerror = () => { /* EventSource 會自動重連 */ };
```

- 優點：**走純 HTTP**（不需協定升級，對 proxy/CDN/防火牆友善）；**瀏覽器內建自動重連**（還支援 `Last-Event-ID` 續傳）；用起來比 WebSocket 簡單。
- 限制：**單向**（client 要傳資料仍得另開 HTTP 請求）；預設只能傳 **UTF-8 文字**（不適合二進位）；HTTP/1.1 下**同網域連線數有上限（約 6 條）**，多分頁會卡（HTTP/2 多工可緩解）；IE 完全不支援。
- 適用：**只需要「伺服器單向推」** 的場景——即時通知、股價/看板、動態消息、AI 逐字串流（許多 LLM chat 就是用 SSE）。

### WebSocket

握手後升級成全雙工長連線，client 與 server 都能隨時互推。

- **非它不可**：需要**低延遲雙向**互動——聊天室（打字中提示、已讀）、多人協作（Google Docs 式）、多人遊戲、即時互動白板。
- **其實不必用它**：只要「單向推」→ SSE 更簡單；即時性要求低 → 短/長輪詢就夠。**不要為了「看起來高級」硬上 WebSocket**（它有狀態、難擴展、要自己處理重連心跳）。

### 取捨總結

| 方案 | 方向 | 即時性 | 伺服器成本 | 實作複雜度 |
| --- | --- | --- | --- | --- |
| 短輪詢 | 雙向(偽) | 差（=間隔） | 中（很多空請求） | 最低 |
| 長輪詢 | 雙向(偽) | 好 | 中高（掛連線） | 中 |
| SSE | 單向(server→client) | 好 | 低 | 低 |
| WebSocket | 全雙工 | 最好 | 高（有狀態長連線） | 高 |

## Q2. WebSocket 協定本身

### Handshake（HTTP Upgrade）

WebSocket 連線**是從一個普通 HTTP 請求「升級」上來的**。瀏覽器送出：

```http
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
```

伺服器同意就回 **`101 Switching Protocols`**：

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

- `Sec-WebSocket-Key` 是瀏覽器隨機產生的字串；伺服器把它接上一段固定的 GUID、做 SHA-1 再 base64，得出 `Sec-WebSocket-Accept`。這**不是安全機制**，而是確保對方「真的懂 WebSocket 協定」而非傻傻回應的快取或 proxy。
- 握手成功後，這條 TCP 連線**不再說 HTTP**，改用 WebSocket 的 frame 協定。

### 全雙工（full-duplex）

- HTTP 是**半雙工的請求/回應**：一定是 client 問、server 才答，server 不能主動開口。
- WebSocket 握手後是**全雙工**：同一條連線上，雙方都能**隨時、同時**送資料，不用等對方。

### Frame（訊框）

- WebSocket 傳輸的最小單位是 **frame**，不是完整 HTTP 訊息。frame header 極小（最少 2 bytes），沒有 HTTP 那一大包 header，所以**每則訊息的額外負擔非常低**，適合高頻小訊息。
- frame 有 opcode 區分型別：文字（text）、二進位（binary）、關閉（close）、ping、pong。所以**文字與二進位都能傳**。
- 瀏覽器送出的 frame 會做 masking（遮罩）以防某些 proxy 中毒攻擊，這由瀏覽器自動處理。

### `ws://` vs `wss://`

- `ws://` = 明文（對應 http）；`wss://` = **TLS 加密**（對應 https）。
- 正式環境**一律用 `wss://`**：除了加密保護資料，加密後中間的 proxy / 企業防火牆看不到內容，**比較不會亂改或擋掉升級請求**（明文 `ws://` 常被中間設備破壞握手）。而且在 https 頁面裡連 `ws://` 會被瀏覽器當混合內容擋掉。
- WebSocket 復用 **80 / 443 埠**（跟 HTTP 一樣），所以一般不需要為它另開防火牆埠，部署友善。

## Q3. 瀏覽器原生 `WebSocket` API

```js
const ws = new WebSocket('wss://example.com/chat');

ws.onopen = () => {
  console.log('連上了', ws.readyState); // 1 (OPEN)
  ws.send(JSON.stringify({ type: 'hello' }));
};
ws.onmessage = (event) => {
  const data = JSON.parse(event.data); // event.data 是字串 / Blob / ArrayBuffer
  console.log('收到', data);
};
ws.onclose = (event) => {
  console.log('關閉', event.code, event.reason, '正常關閉?', event.wasClean);
};
ws.onerror = (event) => {
  console.log('錯誤'); // 拿不到細節，緊接著通常會觸發 onclose
};
```

### 四個事件

- `onopen`：握手完成、連線建立時觸發一次。**此時才可以 `send`。**
- `onmessage`：每收到一則訊息觸發，資料在 `event.data`。
- `onclose`：連線關閉時觸發，`event.code` / `event.reason` / `event.wasClean` 說明原因。
- `onerror`：發生錯誤時觸發，**通常接著就 `onclose`**。

### `readyState`

| 值 | 常數 | 意義 |
| --- | --- | --- |
| 0 | `CONNECTING` | 握手中，還不能送 |
| 1 | `OPEN` | 已連線，可收發 |
| 2 | `CLOSING` | 關閉握手進行中 |
| 3 | `CLOSED` | 已關閉或連不上 |

- **送訊息前必須檢查 `ws.readyState === WebSocket.OPEN`**：在 `CONNECTING`（還沒開）或 `CLOSED`（已斷）時呼叫 `send()` 會丟例外或直接掉訊息。這正是後面實作「離線佇列」的動機。

### `send()` 可送型別

- 字串、`ArrayBuffer`、`ArrayBufferView`（TypedArray）、`Blob`。
- 實務上最常送 **JSON 字串**（`JSON.stringify`），因為好序列化、好除錯、前後端通用；只有傳檔案/音視訊/大量數值時才用二進位（省頻寬）。可設 `ws.binaryType = 'arraybuffer'` 決定收到二進位時的型別。

### `close(code, reason)` 與 code

- `close(1000, 'bye')`：`1000` 代表**正常關閉**（通常是我們主動關的）。
- **`1006`：異常斷線**（連線非正常中斷，例如網路掉、伺服器崩），這個 code 是瀏覽器產生的、不會出現在線上封包裡。
- 重連邏輯常據此判斷：**`wasClean === false` 或非 `1000`** → 是意外斷線，應該重連；**我們自己主動 `close(1000)`** → 不重連。

### `onerror` 的限制

- 基於安全與隱私，瀏覽器**不會把詳細錯誤原因（例如握手失敗的 HTTP 狀態）暴露給 JS**，`error` 事件幾乎沒有可用資訊。真正的斷線原因要看 `onclose` 的 `code`（且很多情況也只是 `1006`）。除錯時主要靠瀏覽器 Network 面板的 WS 分頁。

## Q4. 可靠性：斷線重連、心跳、訊息可靠性

### 指數退避（exponential backoff）+ jitter

「一斷就無延遲狂重連」的災難：伺服器剛重啟或短暫過載時，**成千上萬 client 同時、瘋狂重連**，會把剛恢復的伺服器再次打垮（thundering herd / 重連風暴），自己也白白耗電。

**指數退避**：每次失敗，等待時間翻倍（並設上限），例如 1s → 2s → 4s → 8s → …（capped 於 30s）。

**jitter（隨機抖動）**：在退避時間上加隨機量，**打散大家的重連時間點**，避免所有 client 卡在同一秒同時重連。

```js
// 第 attempt 次重連的延遲（毫秒）
function backoff(attempt, base = 1000, cap = 30000) {
  const exp = Math.min(cap, base * 2 ** attempt);
  return Math.random() * exp; // full jitter：0 ~ exp 之間取隨機
}
```

### 心跳（heartbeat / ping-pong）

「TCP 不是保證送達嗎，為什麼還要應用層心跳？」——因為**「連線壞掉」和「TCP 察覺連線壞掉」之間有很大時間差，有時甚至永遠不會察覺**：

- **半開連線（half-open）**：一端已消失（斷電、崩潰），另一端的作業系統毫不知情，還以為連著。
- **NAT / proxy 閒置逾時**：中間設備會把「太久沒流量」的連線悄悄砍掉，兩端都不知道。
- **TCP keepalive 太慢**：預設常常是 2 小時，對聊天室毫無意義。

所以在**應用層**定時互送 ping/pong：一段時間內沒收到對方的 pong，就判定連線已死、**主動 close 並觸發重連**。（WebSocket 協定本身有 ping/pong 控制 frame，但瀏覽器原生 API 沒開放讓 JS 發，所以前端常自己用一則 `{type:'ping'}` 訊息模擬。）

```js
// 每 25 秒送一次 ping，若 30 秒內沒任何訊息就當作斷線
```

### 訊息佇列 / 補償（offline queue）

斷線期間使用者送出的訊息不能直接丟。做法：**連線不是 OPEN 時，把要送的訊息先塞進一個陣列（queue）**，等 `onopen` 後再依序補送（flush）。這就是後面實作的核心之一。若要更強（跨重新整理都不丟），可暫存到 `localStorage`/IndexedDB。

### 順序與冪等（idempotency）

重連 + 補送可能造成**重複**或**亂序**：

- **冪等**：每則訊息帶唯一 `id`（例如 client 產生的 UUID），伺服器/接收端**看到收過的 id 就忽略**，這樣「重送同一則」不會產生兩筆資料。
- **序號（seq）**：訊息帶遞增序號，接收端據此排序、也能發現「中間漏了哪幾號」而要求補傳。

### ack（送達確認）

「送出去對方到底收到沒？」——**ack 機制**：送訊息時帶一個 id，接收端處理完回一則 `{type:'ack', id}`。送方在收到 ack 前把訊息標記為「未確認」，逾時未 ack 就重送（重送靠上面的冪等 id 去重）。socket.io 把這個包成好用的回呼（見下）。

## Q5. socket.io：在原生 WS 上加了什麼

socket.io **不是**「WebSocket 的另一個名字」，而是一套**自有協定 + client/server 函式庫**，底層可能用 WebSocket、也可能用 HTTP long polling。它幫你內建了原生 WebSocket 要自己刻的一堆東西：

- **自動重連**（含退避策略），開箱即用。
- **心跳**（內建 ping/pong 偵測斷線）。
- **傳輸降級（fallback）**：先用 long polling 建立連線確保能通，再嘗試「升級」到 WebSocket；若環境擋 WebSocket 就退回 polling。對付難搞的企業 proxy 很有用。
- **rooms / namespaces**：分組與邏輯分頻（見下）。
- **ack 回呼**：`socket.emit('msg', data, (reply) => {...})`，送達確認變成一個 callback。
- **廣播（broadcast）**：`io.emit(...)` 廣播給所有人、`socket.to(room).emit(...)` 廣播給某 room。
- **自訂事件**：`emit('chat', ...)` / `on('chat', ...)`，比原生「所有訊息都進同一個 onmessage 自己 switch」好用。

```js
// server（Node，socket.io）
io.on('connection', (socket) => {
  socket.join('room-42');                       // 加入 room
  socket.on('chat', (msg, ack) => {             // 收到帶 ack 的事件
    io.to('room-42').emit('chat', msg);         // 廣播給 room
    ack({ ok: true });                          // 回 ack
  });
});

// client
const socket = io('wss://example.com', { auth: { token } });
socket.emit('chat', { text: 'hi' }, (res) => console.log('server 已收到', res));
socket.on('chat', (msg) => render(msg));
```

### namespace vs room

- **namespace**（如 `/chat`、`/admin`）：**連線層級**的邏輯切分，不同 namespace 像不同「頻道端點」，可各自套不同中介層/授權。用 `io.of('/admin')`。
- **room**：namespace **底下的分組標籤**，一個 socket 可加入多個 room。room 是**伺服器端概念**（client 不能自己亂加），用來「只廣播給某群人」，例如每個聊天群一個 room。

### 能不能用原生 `WebSocket` 連 socket.io？

**不能直接連。** socket.io 有自己的握手/封包格式（Engine.IO 協定），原生 `WebSocket` 不認得。要用 socket.io server 就得搭 socket.io client（反之亦然）。若你想用原生 WebSocket，後端也要用原生的 WS 伺服器（如 `ws`）。

### 取捨

**用原生 WebSocket 就好**：協定單純、雙方都你可控、量體不大、想要最小依賴與最低額外負擔、或後端非 Node（socket.io 主要是 Node 生態）。

**上 socket.io 划算**：需要重連/心跳/fallback/rooms/ack/廣播這些「反正都要自己刻」的功能、要應付各種惡劣網路與 proxy、團隊想快速開發。代價是**多一層抽象與封包額外負擔、綁定 socket.io 生態、跨語言支援較弱**。

## Q6. React 整合

### 用 `useEffect` 管理生命週期 + 清理

WebSocket 是「副作用」且有生命週期（建立→使用→關閉），天生該放 `useEffect`，並在 **cleanup 關掉**：

```tsx
useEffect(() => {
  const ws = new WebSocket(url);
  ws.onmessage = (e) => setMessages((prev) => [...prev, JSON.parse(e.data)]);
  return () => ws.close(); // 元件卸載 / url 改變時關閉，避免連線洩漏
}, [url]);
```

不清理的後果：元件卸載後連線還開著（**記憶體/連線洩漏**）；`url` 改變時舊連線沒關就開新的，變成**多條殭屍連線**同時收訊息、重複更新 state。

### React 18 StrictMode 的坑

開發模式下 StrictMode 會**故意** mount → unmount → 再 mount，於是 effect 跑成「建連線 → 立刻 close → 再建連線」。若封裝寫得不好，會看到「連線剛開就被關」或短暫開了兩條。**正解就是好好寫 cleanup（一定要 `close`）**，讓每次 mount 都自成一組建立/清理即可（正式環境不會有雙重呼叫）。若用單例連線，也可用 ref 標記避免重複建立。

### 高頻訊息的效能優化

每秒數十筆訊息直接 `setState` 會觸發**大量 re-render**、拖垮畫面。優化：

- **批次（buffer + flush）**：把收到的訊息先塞進 `useRef` 的緩衝陣列，用 `requestAnimationFrame` 或每 100ms 才一次性 `setState`。
- 只保留**最近 N 筆**（環形緩衝），別讓陣列無限長。
- 純顯示、不需觸發 render 的資料放 `useRef`；把訊息列表元件 `memo` 化。

### 一條連線跨元件共享

不要每個元件各 `new WebSocket`。用 **Context 提供單一連線實例**（或用 Zustand/Redux 等狀態庫存放連線與訊息），元件透過 hook 取用：

```tsx
const WsContext = createContext<ReliableSocket | null>(null);
export const useWs = () => useContext(WsContext);
```

### 為什麼抽成 `useWebSocket` hook

把「建立/清理連線、重連、心跳、送訊息、暴露連線狀態」封裝進一個自訂 hook，**邏輯集中、可重用、元件端乾淨**（只拿 `messages` / `status` / `send`），也好測試。呼應 Q19 自訂 hook 的精神。

## Q7. 安全與水平擴展

### 驗證（authentication）：token 放哪？

WebSocket 握手是 HTTP，理論上可帶 header，但**瀏覽器原生 `WebSocket` API 不讓你自訂 header**（不能加 `Authorization`）。所以常見做法：

- **query string**：`wss://host/ws?token=xxx`。最簡單，但 **token 可能被記進伺服器/proxy 的 log**（有外洩風險），且會出現在網址。
- **cookie**：握手是 HTTP，瀏覽器會自動帶同源 cookie。方便，但要**防 CSRF**（見下），且跨網域 cookie 麻煩。
- **連線後第一則訊息傳 token**：先建立連線、馬上送一則 auth 訊息，後端驗證前不給訂閱任何資料、逾時未驗證就踢掉。較安全、可控，socket.io 的 `auth` 欄位就是走這條路。

不管放哪，**token 要能過期、能撤銷**，長連線期間 token 失效時要有重新驗證或斷線機制。

### 授權（authorization）

驗證只確認「你是誰」，授權要確認「你能做什麼」。**room 的加入與訊息的訂閱必須由伺服器把關**：使用者要求加入 `room:project-9` 時，後端要查他是否為該專案成員才 `join`；廣播前也只發給有權限的 room。**絕不能相信 client 自稱的身份或它想加入的房間**。

### 同源政策與 CSRF

- WebSocket **不受瀏覽器同源政策（CORS）約束**——任何網站的 JS 都能對你的 WS endpoint 發起連線。所以**後端必須自己檢查握手請求的 `Origin` header**，只允許白名單來源，否則會有 **Cross-Site WebSocket Hijacking（CSWSH）** 風險（惡意站台用受害者的 cookie 偷連你的服務）。
- 因此若用 cookie 驗證，一定要**同時檢查 `Origin` + 用 token/CSRF 機制**，別只靠 cookie。

### 水平擴展與 Redis adapter

WebSocket 是**有狀態的長連線**：連線資訊只存在「接到它的那一台」記憶體裡。擴成多台後問題來了——**A 連在 server1、B 連在 server2，A 送訊息要廣播給 B，但 server1 根本不知道 B 在 server2 上**。

解法：**Pub/Sub 訊息匯流排（backplane）**，最常見是 **Redis adapter**：

```
A ──▶ server1 ──publish──▶ Redis (channel) ──subscribe──▶ server2 ──▶ B
```

每台 server 都訂閱 Redis 的頻道；任一台要廣播時，先 publish 到 Redis，其他台收到後再各自推給「連在自己身上」的相關 client。socket.io 的 `@socket.io/redis-adapter` 就是做這件事。

### sticky session

- 用 socket.io 且啟用 long polling fallback 時，**同一個 client 的多個 HTTP 請求必須打到同一台 server**（握手狀態在該台記憶體），所以 Load Balancer 要開 **sticky session（依 IP/cookie 綁定）**，否則握手會失敗。
- 純 WebSocket（升級後就是單一長連線）比較沒這問題，但升級請求本身仍建議一致路由。真正的跨機廣播還是得靠上面的 Redis backplane，sticky 只解「同一 client 路由一致」。

## Q8. 實作：帶「自動重連 + 心跳」的 WebSocket 封裝

### `ReliableSocket`（原生 WebSocket 之上）

```js
class ReliableSocket {
  /**
   * @param {string} url wss:// 位址
   * @param {object} opts
   */
  constructor(url, opts = {}) {
    this.url = url;
    this.opts = {
      maxRetries: 10,        // 重連次數上限
      baseDelay: 1000,       // 退避基準（ms）
      capDelay: 30000,       // 退避上限（ms）
      heartbeatInterval: 25000, // 每 25s 送一次 ping
      pongTimeout: 10000,    // 送 ping 後 10s 內沒任何訊息 → 判定斷線
      ...opts,
    };
    this.ws = null;
    this.retries = 0;
    this.queue = [];          // 離線訊息佇列
    this.listeners = { open: [], close: [], message: [], error: [] };
    this.manualClose = false; // 手動關閉旗標：關了就不再重連
    this._heartbeatTimer = null;
    this._pongTimer = null;
    this._reconnectTimer = null;
    this.connect();
  }

  connect() {
    this.manualClose = false;
    this.ws = new WebSocket(this.url);

    this.ws.onopen = () => {
      this.retries = 0;                 // 連上就重置退避
      this._flushQueue();               // 補送離線期間累積的訊息
      this._startHeartbeat();
      this._emit('open');
    };

    this.ws.onmessage = (event) => {
      this._resetPongTimer();           // 收到任何訊息都代表連線活著
      if (event.data === 'pong') return; // 心跳回應，不往外丟
      this._emit('message', event.data);
    };

    this.ws.onerror = (event) => this._emit('error', event);

    this.ws.onclose = (event) => {
      this._stopHeartbeat();
      this._emit('close', event);
      if (!this.manualClose) this._scheduleReconnect(); // 非手動關才重連
    };
  }

  // 指數退避 + full jitter
  _scheduleReconnect() {
    if (this.retries >= this.opts.maxRetries) {
      this._emit('error', new Error('達到重連次數上限，放棄'));
      return;
    }
    const exp = Math.min(this.opts.capDelay, this.opts.baseDelay * 2 ** this.retries);
    const delay = Math.random() * exp; // 0 ~ exp 隨機，打散重連風暴
    this.retries++;
    this._reconnectTimer = setTimeout(() => this.connect(), delay);
  }

  _startHeartbeat() {
    this._stopHeartbeat();
    this._heartbeatTimer = setInterval(() => {
      if (this.ws.readyState === WebSocket.OPEN) {
        this.ws.send('ping');
        this._resetPongTimer();
      }
    }, this.opts.heartbeatInterval);
  }

  // 送出 ping 後開一個逾時；期間收到任何訊息會把它清掉並重設
  _resetPongTimer() {
    clearTimeout(this._pongTimer);
    this._pongTimer = setTimeout(() => {
      // 逾時沒任何訊息 → 連線可能已半開死掉，主動關閉觸發重連
      this.ws.close(); // 觸發 onclose → _scheduleReconnect
    }, this.opts.pongTimeout);
  }

  _stopHeartbeat() {
    clearInterval(this._heartbeatTimer);
    clearTimeout(this._pongTimer);
  }

  send(data) {
    const payload = typeof data === 'string' ? data : JSON.stringify(data);
    if (this.ws && this.ws.readyState === WebSocket.OPEN) {
      this.ws.send(payload);
    } else {
      this.queue.push(payload); // 還沒連上 → 先進佇列
    }
  }

  _flushQueue() {
    while (this.queue.length && this.ws.readyState === WebSocket.OPEN) {
      this.ws.send(this.queue.shift());
    }
  }

  on(type, cb) {
    (this.listeners[type] ||= []).push(cb);
    return () => { // 回傳取消訂閱函式
      this.listeners[type] = this.listeners[type].filter((f) => f !== cb);
    };
  }

  _emit(type, ...args) {
    (this.listeners[type] || []).forEach((cb) => cb(...args));
  }

  // 手動關閉：清乾淨、且不再重連
  close() {
    this.manualClose = true;
    clearTimeout(this._reconnectTimer);
    this._stopHeartbeat();
    if (this.ws) this.ws.close(1000, 'client closed');
  }
}
```

重點回顧：

- **重連**：`onclose` → `_scheduleReconnect`，指數退避 `base * 2^retries`（capped）再乘上隨機 jitter；連上後 `retries` 歸零；`manualClose` 為 true 時完全不重連。
- **心跳**：定時送 `ping`；每次 `send('ping')` 都開一個 `pongTimeout`；只要**收到任何訊息**就把逾時清掉（代表連線活著）；逾時到就 `ws.close()`，走一般重連流程。
- **離線佇列**：非 OPEN 時 `send` 進 `queue`，`onopen` 後 `_flushQueue` 補送。
- **事件訂閱**：`on()` 回傳取消訂閱函式，方便在 React cleanup 用。

### 對應的 React `useWebSocket` hook

```tsx
import { useEffect, useRef, useState, useCallback } from 'react';

type Status = 'connecting' | 'open' | 'closed';

export function useWebSocket(url: string) {
  const socketRef = useRef<ReliableSocket | null>(null);
  const [status, setStatus] = useState<Status>('connecting');
  const [messages, setMessages] = useState<any[]>([]);

  useEffect(() => {
    const socket = new ReliableSocket(url);
    socketRef.current = socket;

    const offOpen = socket.on('open', () => setStatus('open'));
    const offClose = socket.on('close', () => setStatus('closed'));
    const offMsg = socket.on('message', (raw: string) => {
      const data = JSON.parse(raw);
      setMessages((prev) => [...prev, data]);
    });

    // cleanup：取消訂閱 + 真正關閉連線（StrictMode 二次掛載也安全）
    return () => {
      offOpen();
      offClose();
      offMsg();
      socket.close();       // manualClose=true → 不會再重連
      socketRef.current = null;
    };
  }, [url]);

  // send 不放進依賴、用 ref 取當前 socket，避免因函式身份改變造成重連
  const send = useCallback((data: unknown) => {
    socketRef.current?.send(data as any);
  }, []);

  return { status, messages, send };
}
```

```tsx
// 使用端
function Chat() {
  const { status, messages, send } = useWebSocket('wss://example.com/chat');
  return (
    <div>
      <span>連線狀態：{status}</span>
      <ul>{messages.map((m, i) => <li key={m.id ?? i}>{m.text}</li>)}</ul>
      <button onClick={() => send({ id: crypto.randomUUID(), text: 'hi' })}>送出</button>
    </div>
  );
}
```

hook 重點：

- 連線建立/清理都在 `useEffect`，**cleanup 一定 `socket.close()`**，StrictMode 二次掛載也不會殘留連線。
- `send` 用 `useRef` 取當前 socket、`useCallback` 空依賴，**避免因為 `send` 身份改變導致 effect 重跑、連線被重建**。
- 若要跨元件共享同一條連線，把 `ReliableSocket` 提到 Context 或狀態庫，而非每個元件各建一條。
- 高頻訊息時，把 `setMessages` 改成「先進 ref buffer、用 rAF/setInterval 批次 flush」以減少 re-render（見 Q6）。
