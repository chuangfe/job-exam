# Q25 - CORS（跨來源資源共享）解答

## 什麼是 CORS

瀏覽器基於安全考量有「同源政策（Same-Origin Policy）」，預設禁止網頁讀取跨來源的回應。CORS 是一套機制，讓伺服器透過回應標頭「明確授權」特定來源可以存取資源，藉此**放寬**同源限制。

## 同源的定義

**同源 = 協定（protocol）+ 網域（host）+ 埠（port）三者皆相同。**

以 `https://example.com:443` 為基準：

| URL | 是否同源 | 原因 |
| --- | --- | --- |
| `https://example.com/api` | 是 | 三者相同 |
| `http://example.com` | 否 | 協定不同（http vs https） |
| `https://api.example.com` | 否 | 網域不同（子網域也算） |
| `https://example.com:8080` | 否 | 埠不同 |

## 簡單請求 vs 非簡單請求

### 簡單請求（simple request）

同時滿足以下條件才算簡單請求：

- 方法為 `GET` / `HEAD` / `POST`。
- 標頭只用安全清單內的（如 `Accept`、`Content-Language`、`Content-Type`）。
- `Content-Type` 只能是 `text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded`。

簡單請求**直接送出**並帶上 `Origin` 標頭，伺服器回應 `Access-Control-Allow-Origin` 對應的來源才放行。

```
請求： Origin: https://app.example.com
回應： Access-Control-Allow-Origin: https://app.example.com
```

### 非簡單請求 → 觸發預檢（preflight）

只要不符合簡單請求條件就會先發預檢，常見觸發原因：

- 使用 `PUT` / `DELETE` / `PATCH` 等方法。
- 帶自訂標頭（如 `Authorization`、`X-Token`）。
- `Content-Type: application/json`。

瀏覽器先自動發一個 `OPTIONS` 預檢請求問伺服器「我接下來這樣的請求可以嗎」：

```
預檢請求（OPTIONS）：
  Origin: https://app.example.com
  Access-Control-Request-Method: PUT
  Access-Control-Request-Headers: Content-Type, Authorization

預檢回應：
  Access-Control-Allow-Origin: https://app.example.com
  Access-Control-Allow-Methods: GET, POST, PUT, DELETE
  Access-Control-Allow-Headers: Content-Type, Authorization
  Access-Control-Max-Age: 86400   ← 預檢結果可快取的秒數，避免每次都預檢
```

預檢通過後，才會發出真正的請求。

## 帶 cookie 的跨源請求（credentialed request）

前端要顯式開啟、後端要明確允許，且不可用萬用字元。

```tsx
// 前端：fetch 要帶 credentials
await fetch("https://api.example.com/me", {
  method: "GET",
  credentials: "include", // 帶上跨源 cookie
});
```

```
後端回應必須：
  Access-Control-Allow-Origin: https://app.example.com   ← 不可為 *
  Access-Control-Allow-Credentials: true
```

- 帶 credentials 時，`Access-Control-Allow-Origin` **不能是 `*`**，必須回填具體來源（通常依請求的 `Origin` 動態回填）。
- `Access-Control-Allow-Headers` / `Access-Control-Allow-Methods` 同樣不能用 `*`。

## 重點釐清

- CORS 是**瀏覽器**的安全機制；用 Postman、curl 或後端對後端呼叫不受 CORS 限制。
- CORS 錯誤代表「瀏覽器擋下了你讀取回應」，但請求本身**可能已經送達伺服器**（尤其簡單請求），所以不能把 CORS 當成安全防護。
- 預檢失敗時主請求根本不會送出。
