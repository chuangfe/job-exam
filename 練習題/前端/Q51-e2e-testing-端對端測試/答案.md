# Q51 - 自動化測試與 E2E（Playwright / Cypress）解析

整體心智圖：

```
        /\
       /  \      E2E（少）  Playwright / Cypress
      /----\     真瀏覽器跑完整流程，最貼近使用者，最慢、最貴、最易 flaky
     /      \
    /--------\   整合（中）  Jest/Vitest + Testing Library
   /          \  多個元件/hook 一起測，含互動與狀態
  /------------\
 /______________\ 單元（多）  Jest/Vitest
                   純函式、工具、reducer，最快最穩最便宜
```

原則：**越往上越像真實使用者、信心越高，但越慢越脆弱越貴**；越往下越快越穩，但離「使用者真的能不能用」越遠。所以「大量放底層、少量放頂層」，並讓每一層只測它最擅長的事。

---

## Q1. 測試金字塔與工具選型

### 三層對照

| 層級 | 數量 | 速度 | 維護成本 | 回饋可信度 | 典型工具 |
| --- | --- | --- | --- | --- | --- |
| 單元 Unit | 多 | 毫秒級 | 低 | 中（邏輯對，不代表整合對） | Jest / Vitest |
| 整合 Integration | 中 | 快 | 中 | 高（元件真的能協作） | Jest/Vitest + Testing Library |
| E2E | 少 | 秒～分鐘 | 高 | 最高（真的能用） | Playwright / Cypress |

- **不要做成「冰淇淋甜筒」**（大量 E2E、少量單元）：E2E 慢、貴、易 flaky，一旦數量太多，CI 會又慢又常紅，開發者最後會習慣性忽略紅燈——反而失去測試意義。
- **測試獎盃（Testing Trophy，Kent C. Dodds 提出）** 修正了金字塔：它把**整合測試**當作 CP 值最高的一層而放最大，理由是「整合測試同時涵蓋多個單元的協作，卻又比 E2E 便宜穩定」，能用相對低的成本換到相對高的信心。底部另加一層「靜態檢查（TypeScript / ESLint）」，把一部分錯誤在編譯期就攔下來。

### 什麼放哪一層

- **適合單元**：純邏輯、沒有 DOM 或網路副作用的東西。例如金額格式化 `formatCurrency`、表單驗證規則 `validateEmail`、Redux reducer 的狀態轉移。
- **非 E2E 不可**：跨頁面、跨系統、依賴真實瀏覽器行為的流程。例如「登入 → 導頁 → 讀取受保護資源」、「加入購物車 → 結帳 → 金流跳轉」、「OAuth 第三方登入」。這些牽涉路由、cookie、重導向、多個服務，單元/整合測試模擬不完整。

## Q2. Testing Library 的核心原則

核心哲學：**測試「使用者看到與操作的東西」，不要測「元件內部怎麼實作」**。這樣重構內部（改 state 結構、換 class 名稱、拆元件）時，只要外顯行為不變，測試就不該壞——這正是測試該有的價值：讓你敢重構。

### 為什麼避免測實作細節

- 用 `getByTestId`、CSS class、或斷言 `state.xxx === true` 這類查詢，等於把測試綁死在「目前的寫法」上。改個 class 名、換個內部變數，測試就無故變紅（false negative）；而使用者其實完全沒受影響。
- 反過來，即使實作有 bug，只要 test-id 還在，測試可能仍然綠燈（false positive）。
- 用 `getByRole('button', { name: '登入' })` 查詢時，你同時驗證了「這東西是按鈕、有可及性名稱、螢幕報讀器讀得到」——查詢本身就順帶檢查了可及性。

### 查詢方法怎麼選

| 方法 | 找不到時 | 找到多個 | 是否等待（async） | 用途 |
| --- | --- | --- | --- | --- |
| `getBy*` | 直接丟錯 | 丟錯 | 否 | 元素「應該存在」時 |
| `queryBy*` | 回傳 `null`（不丟錯） | 丟錯 | 否 | 斷言元素「不存在」時 |
| `findBy*` | 逾時前反覆重試，逾時才丟錯 | 丟錯 | 是（回 Promise） | 元素「稍後才會出現」時（非同步） |

```tsx
// 存在 → getBy
expect(screen.getByRole('heading', { name: '歡迎' })).toBeInTheDocument();
// 不存在 → queryBy（getBy 會直接丟錯，無法斷言「不存在」）
expect(screen.queryByText('錯誤')).not.toBeInTheDocument();
// 非同步出現 → findBy（等 API 回來後才渲染）
expect(await screen.findByText('儲存成功')).toBeInTheDocument();
```

### 查詢優先序

依「越接近使用者感知」排序：**Role > Label/Placeholder/Text（可及性文字）> TestId（最後手段）**。`getByRole` 最推薦，因為它同時反映了可及性樹的樣貌。

### userEvent vs fireEvent

- `fireEvent.click(el)` 只派發**單一** DOM 事件，不還原真實互動。
- `userEvent.click(el)` 會模擬真人的**完整互動序列**（hover → mousedown → focus → mouseup → click），打字還會逐字觸發 `keydown/keyup`。更貼近真實，能抓到「按鈕其實被遮住點不到」「input 沒 focus 就不觸發」這類 bug。實務上一律優先用 `userEvent`（記得它是 async，要 `await`）。

## Q3. Playwright 核心概念

### 基本結構與 web-first assertion

```ts
import { test, expect } from '@playwright/test';

test('首頁標題正確', async ({ page }) => {
  await page.goto('/');
  await expect(page.getByRole('heading', { level: 1 })).toHaveText('歡迎');
});
```

- `expect(locator).toHaveText(...)` 是 **web-first assertion**：它**內建重試（auto-retry）**，會在逾時前反覆檢查條件是否成立。所以不需要自己寫「等一下再斷言」——這是 Playwright 穩定的關鍵。
- 對照：`expect(await locator.textContent()).toBe('歡迎')` 只抓「當下」的值、不重試，容易因為時間差而 flaky。**能用 web-first assertion 就別自己 await 取值再比對**。

### auto-waiting（自動等待）

Playwright 在執行 `click`、`fill`、以及 web-first assertion 前，會自動等待元素滿足一系列**可操作性（actionability）**條件，全部通過才動作，否則重試到逾時：

- 元素已 **attached** 到 DOM
- **visible**（可見、非 `display:none`）
- **stable**（動畫停止、位置不再移動）
- **enabled**（非 `disabled`）
- **receives events**（沒有被其他元素遮住，點得到）

這讓你幾乎不必手動寫等待——這正是它比舊工具（Selenium 需大量顯式等待）穩定的原因。

### locator 為什麼優於 CSS/XPath

- **user-facing locator**（`getByRole` / `getByLabel` / `getByText` / `getByPlaceholder`）用「使用者/報讀器看到的樣子」定位，畫面結構重構時不易壞，且順帶驗證可及性。
- **locator 是惰性（lazy）的**：`page.getByRole(...)` 只是「描述如何找到元素」，並不立即抓取；直到 `.click()` / 斷言時才即時解析當下 DOM，並自動重試。傳統 selector 常是「當下抓一個 handle」，DOM 一變就 stale。

```ts
// 推薦：語意化、抗重構
page.getByRole('button', { name: '送出' });
page.getByLabel('電子郵件');
// 不推薦：脆弱，class 一改就壞
page.locator('.btn-primary.submit-btn');
```

### fixture

fixture 是「測試執行前後自動幫你準備 / 清理環境」的機制，取代 `beforeEach/afterEach` 的重複樣板。

- **內建 fixture 的層級**：
  - `browser`：整個 worker 共用的瀏覽器實例（最重，少開）。
  - `context`：`BrowserContext`，等於一個乾淨、獨立的「瀏覽器設定檔」（獨立 cookie / storage）——**每個測試預設拿到全新 context，這就是測試隔離的基礎**。
  - `page`：context 裡的一個分頁，最常用。
- **自訂 fixture**：把「每個測試都要做的前置」封裝成可注入的相依，例如「已登入的 page」「建立好的測試使用者」。見 Q8 範例。

## Q4. 網路攔截與測試隔離

### page.route()

`page.route(url, handler)` 攔截符合條件的網路請求，由你決定怎麼處理：

```ts
// 攔截並回傳假資料（mock）
await page.route('**/api/user', async (route) => {
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify({ name: '小明', role: 'admin' }),
  });
});

// 放行真實請求（可先改寫再送出）
await page.route('**/api/**', (route) => route.continue());

// 模擬失敗，測前端錯誤處理
await page.route('**/api/user', (route) => route.abort());
```

- **mock 假資料**：適合測「前端拿到某種回應時的呈現」（空狀態、錯誤、極端資料），快又穩，不依賴後端狀態。缺點是「後端契約若改了測不到」。
- **放行真請求**：更接近真實，但慢、需要可用的後端與可控的測試資料。

### 打真後端 vs mock

| | 打真後端 | mock 網路 |
| --- | --- | --- |
| 真實度 | 高（連契約都測到） | 中（只測前端這半邊） |
| 速度 / 穩定 | 慢、易受後端影響 | 快、穩 |
| 資料準備 | 需 seed / 清理 | 不需要 |
| 適用 | 關鍵路徑的「冒煙測試」 | 邊界情境、錯誤處理 |

實務常混用：**關鍵主流程打真後端**（少量、確保端到端真的通），**大量情境用 mock**。

### storageState

`storageState` 把某個 context 的 **cookie + localStorage** 序列化存成 JSON。「先登入一次、把登入態存檔，之後所有測試直接載入這份狀態」就不必每個測試都重跑登入 UI——登入流程慢又重複，這招能省下大量時間（見 Q8 的 setup project）。

## Q5. 平行、跨瀏覽器、除錯產物

### 平行執行

- Playwright 預設用多個 **worker 進程**平行跑測試檔。每個 worker 有自己的瀏覽器，且**每個測試拿到獨立的 `context`**，天然隔離、互不干擾。
- worker 數由 `workers` 設定（CI 上常設較保守以免資源不足）。要注意：平行的前提是「測試彼此獨立」——若共用同一筆後端資料就會互相踩到。

### 跨瀏覽器（projects）

用 `projects` 對三大引擎跑同一份測試：

```ts
// playwright.config.ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './e2e',
  use: { baseURL: 'http://localhost:3000', trace: 'on-first-retry' },
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox',  use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit',   use: { ...devices['Desktop Safari'] } },
  ],
  // 自動幫測試啟動開發伺服器
  webServer: {
    command: 'npm run start',
    url: 'http://localhost:3000',
    reuseExistingServer: !process.env.CI,
  },
});
```

### 除錯產物

| 產物 | 內容 | 建議設定 |
| --- | --- | --- |
| `trace` | 完整時間軸：每步 DOM 快照、網路、console、動作前後畫面，可用 `npx playwright show-trace` 逐格回放 | `trace: 'on-first-retry'`（只在重試時錄，省成本） |
| `screenshot` | 失敗當下的截圖 | `screenshot: 'only-on-failure'` |
| `video` | 整段執行錄影 | `video: 'retain-on-failure'` |

原則：**平常不錄、失敗（或重試）才錄**，兼顧速度與可追查性。trace 是最強的除錯工具，能像「時光機」一樣重播當時整個頁面。

## Q6. 撰寫穩定測試、避免 flaky

### 什麼是 flaky

flaky test =「同樣的程式碼，時綠時紅」的測試。它比「一直失敗」更危險，因為它會**侵蝕團隊對測試的信任**：大家開始把紅燈當雜訊、習慣性 re-run 或忽略，於是真正的 bug 也被一起忽略掉。

### 常見原因與解法

| 原因 | 解法 |
| --- | --- |
| 用 `sleep` / `waitForTimeout(3000)` 硬等 | 改成**等狀態**：web-first assertion（`await expect(...).toBeVisible()`）或 `waitForResponse`。等太短會 flaky、等太長會拖慢 |
| 測試共用狀態、有先後依賴 | 每個測試**自備資料、獨立可單跑**；善用獨立 `context`；避免「測試 B 依賴測試 A 建好的資料」 |
| 動態 / 隨機資料（時間、亂數、排序不定） | 固定亂數種子、用 `page.clock` 或 mock 凍結時間、斷言時放寬（比對集合而非固定順序） |
| 依賴不穩定的第三方服務 | 用 `page.route()` mock 掉外部 API，把不可控因素隔離在測試之外 |

### 為什麼「等狀態」優於「等秒數」

固定秒數是在賭「這段時間內一定完成」——機器慢一點就爆掉（flaky），快的時候又白等（浪費時間）。web-first assertion 是「**條件成立就立刻往下走，逾時前持續重試**」，既穩又快，這就是為什麼 Playwright 幾乎不需要手動 sleep。

## Q7. E2E 在 CI 的執行

- **headless（無頭）模式**：CI 環境沒有顯示器，且不開 GUI 可省資源、跑更快。Playwright 預設 headless；本機除錯才用 `--headed` / `--ui`。
- **典型 CI 流程**：
  1. checkout 程式碼、安裝依賴（`npm ci`）
  2. **安裝瀏覽器**（`npx playwright install --with-deps`）——CI 上常漏這步
  3. build 並啟動應用（或交給 config 的 `webServer` 自動啟）
  4. 跑測試（`npx playwright test`）
  5. **上傳 artifacts**（HTML report、trace、截圖、影片）

```yaml
# .github/workflows/e2e.yml（節錄）
- run: npm ci
- run: npx playwright install --with-deps
- run: npx playwright test
- uses: actions/upload-artifact@v4
  if: ${{ !cancelled() }}   # 即使測試失敗也要上傳
  with:
    name: playwright-report
    path: playwright-report/
    retention-days: 7
```

- **要保留的 artifacts**：`playwright-report/`（HTML 報告）、`test-results/`（失敗的 trace / screenshot / video）。有了 trace，就能在本機 `show-trace` 回放 CI 上那次失敗的完整現場。

## Q8. 實作：完整登入流程 E2E

先用一個 **setup project** 登入一次、存下 `storageState`，需登入的測試再重用它。

```ts
// playwright.config.ts（節錄，重點在 projects 依賴）
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './e2e',
  use: { baseURL: 'http://localhost:3000', trace: 'on-first-retry' },
  projects: [
    // 1) 先跑登入 setup，產出 storageState 檔
    { name: 'setup', testMatch: /auth\.setup\.ts/ },
    // 2) 需登入的測試：載入 setup 存下的登入態，且依賴 setup 先完成
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'], storageState: 'e2e/.auth/user.json' },
      dependencies: ['setup'],
    },
  ],
});
```

```ts
// e2e/auth.setup.ts —— 只登入一次，把登入態存檔
import { test as setup, expect } from '@playwright/test';

const authFile = 'e2e/.auth/user.json';

setup('登入並保存狀態', async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('電子郵件').fill('user@example.com');
  await page.getByLabel('密碼').fill('correct-password');
  await page.getByRole('button', { name: '登入' }).click();

  // 等「登入成功」的可觀察結果，而非 sleep
  await expect(page).toHaveURL('/dashboard');
  await expect(page.getByRole('heading', { name: /歡迎/ })).toBeVisible();

  // 把 cookie + localStorage 存起來供其他測試重用
  await page.context().storageState({ path: authFile });
});
```

```ts
// e2e/login.spec.ts —— 登入流程的各種情境
import { test, expect } from '@playwright/test';

// 這組測試專測登入頁本身，需要「未登入」的乾淨狀態
test.use({ storageState: { cookies: [], origins: [] } });

test.describe('登入流程', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('/login');
  });

  test('成功登入後導向 dashboard 並顯示歡迎訊息', async ({ page }) => {
    await page.getByLabel('電子郵件').fill('user@example.com');
    await page.getByLabel('密碼').fill('correct-password');
    await page.getByRole('button', { name: '登入' }).click();

    await expect(page).toHaveURL('/dashboard');
    await expect(page.getByRole('heading', { name: /歡迎回來/ })).toBeVisible();
  });

  test('帳密錯誤時顯示錯誤訊息且停留在登入頁', async ({ page }) => {
    await page.getByLabel('電子郵件').fill('user@example.com');
    await page.getByLabel('密碼').fill('wrong-password');
    await page.getByRole('button', { name: '登入' }).click();

    // 用 role=alert 斷言錯誤訊息（同時驗證可及性）
    await expect(page.getByRole('alert')).toHaveText('帳號或密碼錯誤');
    await expect(page).toHaveURL('/login');
  });

  test('未填欄位就送出，顯示必填提示', async ({ page }) => {
    await page.getByRole('button', { name: '登入' }).click();

    await expect(page.getByText('請輸入電子郵件')).toBeVisible();
    await expect(page.getByText('請輸入密碼')).toBeVisible();
    await expect(page).toHaveURL('/login'); // 沒有送出、沒導頁
  });

  test('後端回 500 時，前端顯示系統錯誤（用 route mock）', async ({ page }) => {
    // 攔截登入 API，強制回傳伺服器錯誤
    await page.route('**/api/login', async (route) => {
      await route.fulfill({
        status: 500,
        contentType: 'application/json',
        body: JSON.stringify({ message: '系統忙碌中，請稍後再試' }),
      });
    });

    await page.getByLabel('電子郵件').fill('user@example.com');
    await page.getByLabel('密碼').fill('correct-password');
    await page.getByRole('button', { name: '登入' }).click();

    await expect(page.getByRole('alert')).toHaveText('系統忙碌中，請稍後再試');
  });
});

// 這個測試靠 config 的 storageState（已登入），直接進受保護頁面
test('已登入者可直接進入 dashboard 的個人資料', async ({ page }) => {
  await page.goto('/dashboard/profile');
  // 沒有被踢回 /login，代表登入態成功重用
  await expect(page.getByRole('heading', { name: '個人資料' })).toBeVisible();
});
```

重點回顧：

- 全程用 **user-facing locator**（`getByLabel` / `getByRole`），可讀、抗重構、順帶驗可及性。
- 全程用 **web-first assertion**（`await expect(...).toBeVisible()` / `toHaveURL()`）取代 sleep，靠 auto-waiting 保持穩定。
- 用 **route mock** 造出真實後端難以穩定重現的錯誤情境（500）。
- 用 **storageState + setup project** 讓「需登入」的測試不必每次重登，大幅加速。
- 登入頁測試用 `test.use({ storageState: { cookies: [], origins: [] } })` 明確清空登入態，確保測試隔離。

## Cypress 對照（差異與取捨）

| 面向 | Playwright | Cypress |
| --- | --- | --- |
| 架構 | Node 進程用 CDP/協定驅動瀏覽器（out-of-process） | 測試碼跑在**瀏覽器內**、與應用同一個事件迴圈 |
| 瀏覽器引擎 | Chromium / Firefox / **WebKit（Safari 核心）** | Chromium 系 + Firefox（無真正 WebKit） |
| 語法風格 | `async/await`，接近一般 JS | 鏈式 `cy.get().click()`，指令進佇列自動等待 |
| 平行執行 | 內建免費多 worker | 需付費 Dashboard 或自行切分 |
| 多分頁 / 多來源 | 原生支援多 tab、多 origin | 傳統上受限（同源限制較多） |
| 網路攔截 | `page.route()` | `cy.intercept()` |
| 除錯體驗 | Trace Viewer（時光機回放）、`--ui` 模式 | 互動式 Test Runner、可即時看每步 DOM，體驗直覺 |
| 上手 | 略陡（需懂 async / fixture） | 較平易近人，文件與生態成熟 |

**取捨**：需要**跨瀏覽器（尤其 Safari/WebKit）、免費平行、多分頁 / 多來源**的複雜流程 → Playwright 較合適，也是近年新專案的主流選擇。看重**互動式除錯體驗、團隊上手速度、成熟社群外掛** → Cypress 仍是很好的選擇。兩者哲學相通（真瀏覽器、自動等待、user-facing 查詢），觀念可互通，選型看團隊與需求即可。
