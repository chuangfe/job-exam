# Q51 - 自動化測試與 E2E（Playwright / Cypress）

## 問題

一個穩健的前端專案，通常會用「單元測試」守住邏輯、用「整合測試」守住元件互動、用「E2E（端對端）測試」守住真實使用者流程。本題請你說明測試金字塔的取捨、Testing Library 的核心哲學、Playwright / Cypress 的運作方式與差異，最後實作一個完整的登入流程 E2E 測試。

## 引導思考

### Q1. 測試金字塔與工具選型

- 「測試金字塔」由下而上分成哪三層？各層的**數量比例**、**執行速度**、**維護成本**、**回饋可信度**大致如何？
- 為什麼不建議做成「測試冰淇淋甜筒（大量 E2E、少量單元）」？「測試獎盃（Testing Trophy）」又修正了金字塔的什麼觀點？
- 各層通常用什麼工具？
  - 單元 / 整合：Jest 或 Vitest 搭配 Testing Library
  - E2E：Playwright 或 Cypress
- 什麼樣的邏輯適合放單元測試？什麼樣的流程「非 E2E 不可」？請各舉一例。

### Q2. Testing Library 的核心原則

- Testing Library 的口號是「The more your tests resemble the way your software is used, the more confidence they can give you」。這句話在實務上代表什麼取捨？
- 為什麼優先用 `getByRole` / `getByLabelText`，而**避免**用 `getByTestId`、CSS class、DOM 結構來查詢元素？「測實作細節（implementation details）」會造成什麼問題？
- 查詢方法 `getBy*` / `queryBy*` / `findBy*` 差在哪？何時該用哪一個？
- 為什麼測試裡應該用 `userEvent` 而不是直接 `fireEvent`？

### Q3. Playwright 核心概念

- `test` 與 `expect` 的基本結構長怎樣？什麼是 web-first assertion？
- Playwright 的 **auto-waiting（自動等待）** 解決了什麼問題？它在點擊 / 斷言前會自動檢查哪些「可操作性（actionability）」條件？
- 為什麼官方推薦用 `getByRole` / `getByLabel` / `getByText` 這類 **user-facing locator**，而非 CSS / XPath？locator 與傳統 selector 的差異（惰性、可重試）是什麼？
- 什麼是 **fixture**？`page`、`context`、`browser` 這些內建 fixture 各是什麼層級？自訂 fixture 能解決什麼重複問題？

### Q4. 網路攔截與測試隔離

- Playwright 的 `page.route()` 能做什麼？請說明「攔截並回傳假資料（mock）」與「放行真實請求」的差別與適用時機。
- E2E 測試該不該打真的後端？打真後端與 mock 網路，各有什麼優缺點？
- 什麼是 `storageState`？為什麼「先登入一次、把登入狀態存起來重用」能大幅加速測試？

### Q5. 平行、跨瀏覽器、除錯產物

- Playwright 如何做平行執行？worker 與測試隔離的關係是什麼？
- 如何一次對 Chromium / Firefox / WebKit 三種引擎跑同一份測試（projects 設定）？
- 測試失敗時，`trace`、`screenshot`、`video` 各能提供什麼線索？在 CI 上通常怎麼設定才不會拖慢速度又留得住證據？

### Q6. 撰寫穩定測試、避免 flaky（不穩定）

- 什麼是 flaky test？它為什麼比「一直失敗的測試」更危險？
- 列出至少四種造成 flaky 的常見原因，並給出對應解法：
  - 用 `sleep` / 固定 `waitForTimeout` 等待
  - 測試之間共用狀態、有先後依賴
  - 動態 / 隨機資料（時間、亂數、排序不定）
  - 依賴不穩定的第三方服務
- 為什麼「明確等待某個狀態出現」優於「等固定秒數」？web-first assertion 如何天然避免這個問題？

### Q7. E2E 在 CI 的執行

- 為什麼 CI 上要用 headless 模式？
- 一條完整的 CI E2E 流程大致有哪些步驟（安裝依賴、安裝瀏覽器、啟動應用、跑測試、上傳產物）？
- 測試失敗後，要保留哪些 **artifacts** 才方便事後追查？

### Q8. 實作：完整登入流程 E2E

- 請用 Playwright 寫一份「登入流程」的 E2E 測試，需涵蓋：
  - 成功登入 → 導向 `/dashboard` 並看到歡迎訊息
  - 帳密錯誤 → 顯示錯誤訊息且停留在登入頁
  - 表單驗證 → 未填欄位時送出，顯示必填提示
  - 至少一個情境用 `page.route()` mock 後端 API 回應
  - 用 `storageState` 或 fixture 讓「需登入才可看的頁面」測試不必每次重登
- 全程使用 user-facing locator（`getByRole` / `getByLabel`），不要用脆弱的 CSS selector。
