# Q45 - 網頁可及性 a11y（Accessibility / WCAG）解析

## Q1. 為何重要與 WCAG 四原則

**為什麼要做可及性：**

- **法規責任**：許多國家有法律要求。美國有 ADA（身心障礙者法）與 Section 508、歐盟有 EAA（European Accessibility Act，2025 年生效），台灣政府網站則須符合「網站無障礙規範」。做不好可能面臨訴訟或無法承接標案。
- **使用者涵蓋範圍**：全球約有 15%~20% 的人口有某種形式的障礙。而且可及性不只服務「永久障礙者」，也涵蓋**情境障礙**——例如手受傷只能單手操作、在強光下看不清螢幕、在吵雜環境無法聽聲音。做好可及性等於提升所有人的體驗。
- **SEO 與商業價值**：語意化標籤、正確的 `alt`、清楚的標題結構同時有利於搜尋引擎理解頁面，且更擴大潛在客群。可及性與良好工程實踐高度重疊，通常「順手就做了」。

**WCAG 四原則（POUR）：**

| 原則 | 英文 | 意思 |
| --- | --- | --- |
| 可感知 | **P**erceivable | 資訊要能被「感知」到——圖片有替代文字、影片有字幕、文字對比足夠。 |
| 可操作 | **O**perable | 介面要能被「操作」——所有功能都能只用鍵盤完成、不設定會誘發癲癇的閃爍。 |
| 可理解 | **U**nderstandable | 內容與操作要能「理解」——文字清楚、行為可預期、錯誤有明確提示。 |
| 穩健性 | **R**obust | 要能被各種輔助技術「穩健」解析——用標準語意，讓報讀器等工具正確運作。 |

**等級**：WCAG 分 A（最低）、AA、AAA（最高）三級。**業界與法規普遍以 AA 為目標**（AAA 常有現實上難以全達成的要求，如更嚴格的對比）。目前主流版本為 WCAG 2.1 / 2.2。

## Q2. 語意化 HTML 與 ARIA 的關係

**語意化 HTML 是地基。** 原生元素「免費」附帶了大量可及性能力，用 `<div>` 自己拼會全部遺失。以 `<button>` 為例，換成 `<div onClick>` 會失去：

- **可聚焦**：`<button>` 天生可被 Tab 聚焦；`<div>` 不行（要手動加 `tabindex="0"`）。
- **鍵盤觸發**：按 Enter / Space 會觸發 `<button>`；`<div>` 的 `onClick` 不會回應鍵盤。
- **語意角色**：報讀器會唸「按鈕」；`<div>` 什麼都不唸，使用者不知道那能點。
- **表單行為**：`<button type="submit">` 能送出表單。

所以正確順序是：**先選對的原生標籤**（`button`、`a`、`nav`、`main`、`header`、`ul`、`label`、`input`…），不夠時**才**用 ARIA 補強。

**ARIA 第一守則（No ARIA is better than bad ARIA）**：意思是「錯誤的 ARIA 比沒有 ARIA 更糟」。ARIA 只會改變報讀器「聽到的資訊」，**不會**幫你加上任何行為（不會讓元素可聚焦、不會處理鍵盤）。若你寫了 `role="button"` 卻沒補鍵盤與焦點，反而是騙報讀器說「這是按鈕」，使用者卻按不動——比誠實的 `<div>` 更糟。

**什麼時候才需要 ARIA：**

- 原生 HTML 沒有對應元件時（例如 tabs、tree、真正的自訂下拉、carousel）。
- 需要描述**動態狀態**時（`aria-expanded`、`aria-selected`、`aria-live`）。
- 需要補充原生元素沒有的說明時（`aria-label`、`aria-describedby`）。

反之，能用 `<button>`、`<nav>`、`<details>/<summary>` 解決的，就不要用 `<div role="...">` 重造。

## Q3. ARIA 的三類屬性

- **role（角色）**：告訴輔助技術「這是什麼」。例如 `role="dialog"`、`role="alert"`、`role="listbox"`。優先用原生標籤取得角色，別亂加。
- **state（狀態）**：描述**會變動**的當前狀態，例如 `aria-expanded`（展開/收合）、`aria-checked`、`aria-disabled`、`aria-selected`、`aria-hidden`。
- **property（屬性）**：描述相對固定的特性/關係，例如 `aria-label`、`aria-labelledby`、`aria-describedby`、`aria-required`、`aria-controls`。

（state 與 property 界線有時模糊，重點是分清「會隨互動改變」vs「相對固定」。）

**常見屬性說明：**

| 屬性 | 用途 |
| --- | --- |
| `aria-label` | 直接給元素一個「文字名稱」（本身無可見文字時用，如只有圖示的按鈕）。 |
| `aria-labelledby` | 用**頁面上其他元素的 id** 來當名稱（避免重複文字，可指向多個 id）。 |
| `aria-describedby` | 補充**說明/描述**（如輸入框的格式提示、錯誤訊息），報讀器會在名稱後補讀。 |
| `aria-expanded` | 布林狀態，表示可展開元素目前是展開（`true`）或收合（`false`）。 |
| `aria-hidden` | `true` 時對報讀器**隱藏**該元素（用於純裝飾圖示；注意別藏到可聚焦元素）。 |
| `aria-live` | 標記一個「即時區域」，內容變動時報讀器會自動朗讀（見 Q5）。 |

`label` vs `labelledby` 差別：`aria-label` 直接寫字串；`aria-labelledby` 指向現有可見文字的 id（維護單一來源，且會朗讀該可見文字）。

## Q4. 鍵盤操作與焦點管理

**Tab 順序**：使用者按 Tab 依序在可聚焦元素間移動。預設順序**跟隨 DOM 的先後順序**，所以「讓 DOM 順序符合視覺閱讀順序」很重要（別用 CSS 把後面的元素搬到前面）。

**`tabindex` 三種值：**

| 值 | 效果 | 使用時機 |
| --- | --- | --- |
| `0` | 加入自然 Tab 順序（依 DOM 位置） | 讓一個原生不可聚焦的自訂互動元素可被 Tab 到 |
| `-1` | **不在** Tab 順序中，但可用 JS `.focus()` 聚焦 | 需要程式控制焦點的目標（如 Modal 開啟時聚焦標題） |
| 正整數 | 強制排到最前面（`1`、`2`… 先於 `0`） | **避免使用**——會打亂自然順序、極難維護 |

**絕不要移除 focus 樣式**：`outline: none` 會讓鍵盤使用者完全看不到「焦點現在在哪」，等於無法操作。若預設外框不好看，應**替換**而非移除，並優先用 `:focus-visible`（只在鍵盤操作時顯示，滑鼠點擊不顯示）：

```css
/* 錯誤：直接拿掉，鍵盤使用者迷路 */
button:focus { outline: none; }

/* 正確：自訂但保留清楚可見的焦點指示，且用 :focus-visible */
button:focus-visible {
  outline: 2px solid #2563eb;
  outline-offset: 2px;
}
```

**Focus 管理**：在 SPA / 動態 UI 中，焦點不會自己跑到對的地方，需手動安排。例如：Modal 開啟時把焦點移進去、關閉時**還回**觸發它的按鈕；路由切換後把焦點移到新頁標題。

**Focus trap（焦點陷阱）**：Modal 開啟時，Tab 應**被困在 Modal 內循環**，不能跑到後面被遮住的頁面。實作重點：攔截 Tab / Shift+Tab，在第一個與最後一個可聚焦元素間循環（見 Q8 範例）。

## Q5. 螢幕報讀器與動態通知

**運作方式**：螢幕報讀器（如 NVDA、JAWS、VoiceOver）讀的**不是螢幕畫素**，而是瀏覽器建構的**可及性樹（accessibility tree）**——由 DOM 的語意、角色、名稱、狀態組成。所以「畫面上看起來像按鈕」沒用，要「在可及性樹裡是按鈕」才行。使用者透過語音逐一瀏覽、跳標題、列出連結等方式導覽。

**為什麼動態內容讀不到**：報讀器通常聚焦在使用者「當前所在」的元素。表單送出後在別處冒出的錯誤、右下角的 toast，因為焦點沒過去、DOM 又是「悄悄」變的，報讀器不會主動朗讀。解法是用 **`aria-live` 即時區域**：

- `aria-live="polite"`：**等使用者手邊唸完**再播報。適合非緊急通知（存檔成功、一般 toast）。
- `aria-live="assertive"`：**立刻打斷**目前朗讀。只用於緊急/重要（如送出失敗的錯誤）。
- `role="alert"` 等同 `aria-live="assertive"`（且 `aria-atomic="true"`）；`role="status"` 約等於 `polite`。

關鍵：live region 的**容器要先存在於 DOM**，之後再把文字塞進去，報讀器才會偵測到變化。

```tsx
// toast 容器一開始就在頁面上（空的），之後更新內容才會被朗讀
function Toast({ message }: { message: string }) {
  return (
    <div aria-live="polite" role="status" className="toast-region">
      {message}
    </div>
  );
}

// 表單錯誤：用 assertive / role="alert" 立即通知
function FieldError({ id, error }: { id: string; error?: string }) {
  return (
    <p id={id} role="alert" className="error">
      {error}
    </p>
  );
}
```

## Q6. 色彩、表單與其他常見要點

**色彩對比（WCAG AA）：**

- 一般文字（小於 18pt 或 14pt 粗體）：文字與背景對比至少 **4.5:1**。
- 大文字（≥ 18pt 或 14pt 粗體）：至少 **3:1**。
- 非文字（如按鈕邊框、圖示、表單框線）：至少 **3:1**。

**不能只用顏色傳達資訊**：約 8% 男性有色覺辨識障礙。「紅色代表錯誤、綠色代表成功」若**只靠顏色**，他們分不出來。必須**同時**加上文字、圖示或形狀。例如錯誤欄位除了變紅框，還要加驚嘆號圖示與文字訊息。

**可及的表單：**

- 每個輸入框都要有綁定的 `<label>`（`<label htmlFor="email">` 對應 `id="email"`），點 label 也能聚焦，報讀器會唸出名稱。
- placeholder **不能當 label**（一輸入就消失、對比常不足）。
- 錯誤訊息用 `aria-describedby` 綁到輸入框，並用 `aria-invalid="true"` 標記，讓報讀器唸出。

**按鈕 vs 連結**：判準是「行為」而非外觀。**觸發動作**（送出、開 Modal、切換）用 `<button>`；**導向另一個網址 / 位置**用 `<a href>`。連結可被複製、開新分頁、被搜尋引擎爬；按鈕會回應 Space 鍵。不要用連結假裝按鈕（`<a href="#" onClick>`），也不要用按鈕做導覽。

**圖片 `alt`：**

- 有意義的圖片：`alt` 寫出它傳達的資訊（不是寫「圖片」）。
- 純裝飾圖片：用**空的** `alt=""`（讓報讀器略過）；若是 CSS 背景圖則本就不會被讀。
- 資訊型圖片（圖表）：`alt` 給摘要，或提供旁邊的文字說明。

**跳過導覽連結（skip link）**：頁面最上方放一個平常隱藏、聚焦時出現的「跳到主要內容」連結，讓鍵盤 / 報讀器使用者不必每次都 Tab 過一長串導覽列。

```html
<a href="#main" class="skip-link">跳到主要內容</a>
<!-- ...導覽列... -->
<main id="main" tabindex="-1"> ... </main>
```

```css
.skip-link {
  position: absolute;
  left: -9999px; /* 平常移出畫面 */
}
.skip-link:focus {
  left: 8px;
  top: 8px; /* 聚焦時才顯示 */
}
```

## Q7. 測試工具

**沒有任何單一工具能保證可及性**——自動化工具只能抓約 30%~40% 的問題（多為機械式可檢查項，如缺 alt、對比不足），語意是否正確、鍵盤流程是否順、報讀器唸起來是否合理都得靠人工。

| 方式 | 能抓到 | 抓不到 |
| --- | --- | --- |
| **axe**（axe DevTools / `@axe-core`，可整合到測試） | 缺 label、對比、無效 ARIA、重複 id 等規則化問題 | 邏輯性、操作流暢度、內容是否合理 |
| **Lighthouse**（Chrome 內建） | 快速的可及性總分與常見問題清單 | 同上，只是 axe 規則的子集，較粗略 |
| **鍵盤手測** | 只用 Tab / Shift+Tab / Enter / Space / Esc / 方向鍵是否能完成所有操作、焦點是否可見、有無陷阱 | 需要人親自操作，無法完全自動化 |
| **螢幕報讀器**（VoiceOver / NVDA） | 唸出來的名稱、角色、狀態、順序是否正確且合理 | 需要人聽與判斷 |

建議：CI 中放 axe 自動檢查擋掉低級錯誤，開發時**每個互動元件都用鍵盤走一遍**，重要流程再用報讀器實測。

## Q8. 重構實作

### （A）可及的 Modal

問題點：`<div>` 沒有語意與角色、不能鍵盤操作、按 Esc 無法關閉、開啟後焦點沒進去、沒有焦點陷阱、關閉後焦點沒還回去、標題與 dialog 沒有關聯。

```tsx
import { useEffect, useRef } from "react";

function AccessibleModal({
  open,
  onClose,
  onConfirm,
}: {
  open: boolean;
  onClose: () => void;
  onConfirm: () => void;
}) {
  const dialogRef = useRef<HTMLDivElement>(null);
  const previouslyFocused = useRef<HTMLElement | null>(null);

  useEffect(() => {
    if (!open) return;

    // 1) 記住開啟前的焦點，關閉後好還回去
    previouslyFocused.current = document.activeElement as HTMLElement;

    // 2) 開啟後把焦點移進對話框
    const dialog = dialogRef.current;
    const focusables = dialog?.querySelectorAll<HTMLElement>(
      'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
    );
    focusables?.[0]?.focus();

    function handleKeyDown(e: KeyboardEvent) {
      // 3) Esc 關閉
      if (e.key === "Escape") {
        onClose();
        return;
      }
      // 4) Focus trap：Tab 在對話框內循環
      if (e.key === "Tab" && focusables && focusables.length > 0) {
        const first = focusables[0];
        const last = focusables[focusables.length - 1];
        if (e.shiftKey && document.activeElement === first) {
          e.preventDefault();
          last.focus();
        } else if (!e.shiftKey && document.activeElement === last) {
          e.preventDefault();
          first.focus();
        }
      }
    }

    document.addEventListener("keydown", handleKeyDown);
    return () => {
      document.removeEventListener("keydown", handleKeyDown);
      // 5) 關閉後把焦點還給原本的觸發元素
      previouslyFocused.current?.focus();
    };
  }, [open, onClose]);

  if (!open) return null;

  return (
    // 遮罩點擊關閉；用 role="presentation" 表示它只是背景
    <div className="overlay" onClick={onClose} role="presentation">
      <div
        ref={dialogRef}
        className="modal"
        role="dialog"
        aria-modal="true"
        aria-labelledby="modal-title"
        aria-describedby="modal-desc"
        onClick={(e) => e.stopPropagation()} // 點對話框內部不關閉
      >
        <h2 id="modal-title">確認刪除</h2>
        <p id="modal-desc">此操作無法復原，確定要刪除嗎？</p>

        {/* 真正的 button，天生可聚焦、可鍵盤觸發、報讀器會唸「按鈕」 */}
        <button type="button" onClick={onClose}>
          取消
        </button>
        <button type="button" onClick={onConfirm}>
          刪除
        </button>

        {/* 只有圖示的關閉鈕，用 aria-label 補上可讀名稱 */}
        <button type="button" className="close" aria-label="關閉" onClick={onClose}>
          ✕
        </button>
      </div>
    </div>
  );
}
```

重點：`role="dialog"` + `aria-modal="true"` 告訴報讀器這是對話框並會遮蔽背景；`aria-labelledby` / `aria-describedby` 把標題與說明綁上；用真正的 `<button>`；補齊 Esc、焦點移入、focus trap、關閉還原焦點。

（實務上建議直接用原生 `<dialog>` 元素或成熟的無障礙元件庫如 Radix UI / React Aria，它們已處理好上述所有細節。）

### （B）可及的自訂下拉（combobox / listbox 模式）

問題點：`<div>` 無法聚焦、鍵盤無法開合與選取、報讀器不知道這是選單、不知道展開狀態、不知道目前選了哪個。

若能用原生 `<select>` 就**優先用 `<select>`**（它自帶所有可及性）。以下示範真的需要自訂樣式時的最小可及版本：

```tsx
import { useRef, useState } from "react";

function AccessibleSelect({
  label,
  options,
  value,
  onChange,
}: {
  label: string;
  options: string[];
  value: string;
  onChange: (v: string) => void;
}) {
  const [open, setOpen] = useState(false);
  const [activeIndex, setActiveIndex] = useState(0);
  const listRef = useRef<HTMLUListElement>(null);

  function handleKeyDown(e: React.KeyboardEvent) {
    switch (e.key) {
      case "ArrowDown":
        e.preventDefault();
        if (!open) setOpen(true);
        else setActiveIndex((i) => Math.min(i + 1, options.length - 1));
        break;
      case "ArrowUp":
        e.preventDefault();
        setActiveIndex((i) => Math.max(i - 1, 0));
        break;
      case "Enter":
      case " ":
        e.preventDefault();
        if (open) {
          onChange(options[activeIndex]);
          setOpen(false);
        } else {
          setOpen(true);
        }
        break;
      case "Escape":
        setOpen(false);
        break;
    }
  }

  return (
    <div className="select">
      {/* label 用 id 綁到觸發鈕 */}
      <span id="select-label">{label}</span>

      {/* 觸發鈕：真正的 button，宣告展開狀態與它控制的清單 */}
      <button
        type="button"
        aria-haspopup="listbox"
        aria-expanded={open}
        aria-labelledby="select-label"
        onClick={() => setOpen((o) => !o)}
        onKeyDown={handleKeyDown}
      >
        {value || "請選擇"}
      </button>

      {open && (
        <ul
          ref={listRef}
          role="listbox"
          aria-labelledby="select-label"
          aria-activedescendant={`opt-${activeIndex}`}
          tabIndex={-1}
        >
          {options.map((opt, i) => (
            <li
              key={opt}
              id={`opt-${i}`}
              role="option"
              aria-selected={value === opt}
              className={i === activeIndex ? "active" : ""}
              onClick={() => {
                onChange(opt);
                setOpen(false);
              }}
            >
              {opt}
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}
```

重點：觸發鈕用 `<button>` 並以 `aria-haspopup="listbox"`、`aria-expanded` 宣告狀態；清單用 `role="listbox"`、選項用 `role="option"` + `aria-selected`；`aria-activedescendant` 讓報讀器知道目前「虛擬聚焦」在哪一項；補齊方向鍵、Enter/Space、Esc 鍵盤操作。**這類元件細節繁多，強烈建議直接採用 React Aria、Radix UI、Headless UI 等已通過驗證的無障礙元件庫。**

## 重點回顧

- 可及性同時是法規、道德與商業需求；以 WCAG **AA** 為務實目標，記住四原則 **POUR**（可感知、可操作、可理解、穩健）。
- **先用語意化 HTML**，不夠再補 ARIA；「壞的 ARIA 比沒有更糟」——ARIA 只改語意、不加行為。
- 所有互動都必須能只用鍵盤完成，焦點樣式**絕不能拿掉**，Modal 要做焦點移入、focus trap、關閉還原。
- 動態訊息用 `aria-live`（polite / assertive）讓報讀器唸得到；色彩對比 AA 需 **4.5:1**，且**不能只靠顏色**傳達資訊。
- 自動化工具（axe / Lighthouse）只能抓一部分，務必**鍵盤手測 + 報讀器實測**。
