# MUI（v5+）主題與樣式系統 解析

## Q1. `createTheme` 與主題結構

`createTheme` 產生一個「深度合併」預設主題後的 theme 物件，成為整套 UI 的設計 token 來源。主要區塊：

| 區塊 | 負責什麼 |
|------|----------|
| `palette` | 顏色系統：`primary`/`secondary`/`error`/`warning`/`info`/`success`、`text`、`background`、`divider`、`mode`（light/dark） |
| `typography` | 字體家族、各層級字級（`h1`~`h6`、`body1`、`button`…）、字重、行高 |
| `spacing` | 間距的基準函式，預設 `8`（px）。`theme.spacing(2)` = `'16px'` |
| `breakpoints` | 響應式斷點與其查詢字串產生器 |
| `shape` | 形狀 token，最常用 `borderRadius`（預設 `4`） |
| `shadows` | 長度 25 的陰影陣列（index 0 = `'none'`），對應 elevation 0~24 |
| `zIndex` | 各種疊層元件的 z-index（`appBar`、`drawer`、`modal`、`tooltip`…），避免打架 |
| `transitions` | 動畫 easing 與 duration token |

**spacing**：預設基準 8，`theme.spacing(2)` → `'16px'`；也可傳多值 `theme.spacing(1, 2)` → `'8px 16px'`。在 `sx` 裡 `p: 2` 等同 `padding: theme.spacing(2)`。

**breakpoints**：預設 `xs:0, sm:600, md:900, lg:1200, xl:1536`。`theme.breakpoints.up('md')` → `'@media (min-width:900px)'`；另有 `down` / `between` / `only`。

**shadows 與 elevation**：`<Paper elevation={3} />` 實際上是套用 `theme.shadows[3]`。elevation 值就是 shadows 陣列的索引，數字越大陰影越深、視覺上「浮得越高」。

```tsx
import { createTheme } from '@mui/material/styles';

const theme = createTheme({
  palette: {
    primary: { main: '#1976d2' },
    secondary: { main: '#9c27b0' },
    // 自訂 palette 色（TS 需搭配 Q7 的 module augmentation）
    brand: {
      main: '#ff5722',
      light: '#ff8a65',
      dark: '#e64a19',
      contrastText: '#fff',
    },
  },
  typography: {
    fontFamily: '"Noto Sans TC", Roboto, sans-serif',
    button: { textTransform: 'none' }, // 關掉按鈕預設全大寫
  },
  shape: { borderRadius: 12 },
});
```

白話：theme 就是一份「設計規格書」，元件與 `sx`/`styled` 都從這裡取值，改一處全站同步。

## Q2. `components` 覆寫：`styleOverrides`、`defaultProps`、自訂 `variants`

`theme.components.<MuiXxx>` 可對「某個 MUI 元件」做三件事：

- **`defaultProps`**：改該元件的預設 props（例如全站 Button 預設 `disableElevation`、`size="small"`）。
- **`styleOverrides`**：覆寫該元件各「插槽（slot）」的樣式。`root` 是最外層根插槽，其他如 `Button` 有 `contained`、`sizeLarge` 等。
- **`variants`**：依「props 條件」新增一組樣式規則，是做自訂變體的官方推薦作法（v5.4+）。

```tsx
const theme = createTheme({
  components: {
    MuiButton: {
      // 1) 預設 props
      defaultProps: {
        disableElevation: true,
        size: 'medium',
      },
      // 2) styleOverrides，可用函式拿到 theme 與 ownerState
      styleOverrides: {
        root: ({ ownerState, theme }) => ({
          borderRadius: theme.shape.borderRadius,
          ...(ownerState.size === 'large' && {
            paddingInline: theme.spacing(4),
          }),
        }),
      },
      // 3) 自訂 variants：props 比對成立就套 style
      variants: [
        {
          props: { variant: 'dashed' },
          style: {
            border: '2px dashed currentColor',
            backgroundColor: 'transparent',
          },
        },
        {
          props: { variant: 'dashed', color: 'secondary' },
          style: { color: '#9c27b0' },
        },
      ],
    },
  },
});

// 使用（TS 需搭配 Q7 擴充 variant 型別）
// <Button variant="dashed">虛線按鈕</Button>
```

**`ownerState`** 是 MUI 傳進 `styleOverrides` 函式的物件，內含該元件「當前的 props + 內部狀態」，讓你能依 `variant`/`size`/`color` 做條件樣式。

**取捨**：全域 `components` 覆寫適合「跨全站一致的規則」（設計系統層級）；`sx` 適合「單一使用點的一次性微調」。設計系統應把可複用規則收斂到 theme，避免每個使用點各寫一份 `sx`。

## Q3. `sx` prop：用法與底層原理

`sx` 是 MUI 提供的「樣式捷徑 prop」，底層由 `@mui/system` 把它轉成 Emotion 樣式。與 inline `style` 的差異：

- `style` 只接受純 CSS 字串值、不吃 theme、不能寫偽類/media query。
- `sx` 是 **theme-aware**：可用 token 捷徑（`p`/`m`/`color: 'primary.main'`）、支援偽類、子選擇器、responsive 語法，最終產生 class（非 inline style）。

**theme-aware**：`sx={{ p: 2 }}` 中的 `p` 是 padding 捷徑，`2` 會過 `theme.spacing(2)` → 16px；`color: 'primary.main'` 會沿 theme.palette 的路徑取值。也就是這些「值」是相對 theme 解析的，不是死值。

**responsive 兩種語法**：

```tsx
// array 語法：值依序對應 [xs, sm, md, lg, xl]
<Box sx={{ width: [100, 200, 300] }} />
// xs → 100px、sm → 200px、md 以上 → 300px

// object 語法：明確指定斷點（可跳過中間斷點，較好讀）
<Box sx={{ width: { xs: 100, md: 300 }, p: { xs: 1, md: 4 } }} />
```

兩者最終都會展開成對應斷點的 `@media (min-width: …)` 規則。

**偽類 / 子選擇器 / media query**：用巢狀物件，`&` 代表自己：

```tsx
<Box
  sx={{
    color: 'text.primary',
    '&:hover': { color: 'primary.main' },
    '& .icon': { mr: 1 },
    '@media (min-width:900px)': { fontSize: 20 }, // 也可直接寫 media
  }}
/>
```

**效能考量**：`sx` 每次渲染都要「序列化物件 → 產生/查找 class」，有 runtime 成本。單點使用無感，但在「大量 list item、每格都帶 `sx`」這種熱路徑會累積。優化方式：把穩定樣式抽成 `styled()` 元件（樣式在建立時就處理好），只把真正會變的部分留給 `sx`，或把 `sx` 物件提到元件外避免每次新建。

## Q4. `styled()`（Emotion）建立樣式化元件

`styled()` 有兩個來源，設計系統請用 **MUI 版**（`@mui/material/styles`），因為它的樣式回呼會自動注入 theme、支援 `ownerState`、預設處理好 MUI 元件的 prop 轉發：

```tsx
import { styled } from '@mui/material/styles'; // ✅ MUI 版（吃 theme）
// import { styled } from '@emotion/styled';   // ⚠️ Emotion 版，theme 需自行取
```

樣式回呼的第一參數是 `{ theme }`：

```tsx
const StyledBox = styled('div')(({ theme }) => ({
  padding: theme.spacing(2),
  color: theme.palette.text.primary,
  borderRadius: theme.shape.borderRadius,
}));
```

**`shouldForwardProp`**：`styled` 預設會把「不認得的 prop」透傳到底層 DOM。若你用一個「只是拿來決定樣式」的自訂 prop（如 `highlighted`），不擋掉它就會變成 `<div highlighted="true">` 出現在 DOM——React 會噴 unknown attribute warning、還可能污染 HTML。`shouldForwardProp` 就是決定「哪些 prop 該真的傳給 DOM」的過濾器。

```tsx
import { styled } from '@mui/material/styles';

const StyledCard = styled('div', {
  // highlighted 只用來算樣式，不要傳到 DOM
  shouldForwardProp: (prop) => prop !== 'highlighted',
})<{ highlighted?: boolean }>(({ theme, highlighted }) => ({
  padding: theme.spacing(2),
  borderRadius: theme.shape.borderRadius,
  border: `2px solid ${
    highlighted ? theme.palette.primary.main : theme.palette.divider
  }`,
  transition: theme.transitions.create('border-color'),
}));

// <StyledCard highlighted>內容</StyledCard>  → DOM 上不會有 highlighted 屬性
```

（慣例：也可用 `$` 前綴命名 transient prop，如 `$highlighted`，再用 `shouldForwardProp` 擋掉所有 `$` 開頭的 prop。）

**`styled('div')` vs `styled(MuiButton)`**：前者從原生標籤起手；後者包裝既有 MUI 元件、繼承它全部行為與 props。包裝 MUI 元件時，記得該元件的 props 型別要一起帶上（用泛型），且避免與它既有 prop 撞名。

## Q5. 打造可複用元件 API：`forwardRef`、透傳 props、`ownerState`

**為什麼要 `forwardRef`**：MUI 很多功能靠「拿到子元件的真實 DOM ref」運作——`Tooltip`/`Popover` 要定位到觸發元素、`focus()`/`scrollIntoView()` 要操作節點、transition 元件要量測尺寸。若你的元件沒轉發 ref，把它塞進 `<Tooltip>` 就會失效或報 warning。設計系統元件應「一律 `forwardRef`」。

**props 透傳**：把使用者傳入、你沒特別處理的 props 用 `...rest` 全部交給底層元件/原生標籤，讓元件保有原生彈性（`onClick`、`aria-*`、`data-*`…）。型別上用 `React.ComponentPropsWithRef<'button'>` 或 `ComponentProps<typeof Base>` 延用而不用手打。

**`ownerState` 模式**：不要把「純樣式用」的 prop 直接 forward 到 DOM，而是收進一個 `ownerState` 物件，交給 `styled` 的回呼用。好處：(1) 樣式邏輯集中、(2) DOM 乾淨（不會漏出自訂屬性）、(3) 與 MUI 內部寫法一致，`styleOverrides` 也能拿到同一份 `ownerState`。

```tsx
import * as React from 'react';
import { styled } from '@mui/material/styles';

type TagColor = 'default' | 'success' | 'danger';

interface TagProps extends React.ComponentPropsWithoutRef<'span'> {
  color?: TagColor;
}

// ownerState 專供樣式使用，用 shouldForwardProp 擋住不進 DOM
interface TagOwnerState {
  color: TagColor;
}

const TagRoot = styled('span', {
  shouldForwardProp: (prop) => prop !== 'ownerState',
})<{ ownerState: TagOwnerState }>(({ theme, ownerState }) => {
  const map = {
    default: theme.palette.grey[500],
    success: theme.palette.success.main,
    danger: theme.palette.error.main,
  };
  return {
    display: 'inline-block',
    padding: theme.spacing(0.25, 1),
    borderRadius: 999,
    fontSize: theme.typography.caption.fontSize,
    color: '#fff',
    backgroundColor: map[ownerState.color],
  };
});

export const Tag = React.forwardRef<HTMLSpanElement, TagProps>(
  function Tag({ color = 'default', ...rest }, ref) {
    const ownerState: TagOwnerState = { color };
    return <TagRoot ref={ref} ownerState={ownerState} {...rest} />;
  }
);

// <Tag color="success" onClick={...}>上線</Tag>  → DOM 只有 span + onClick，無 color/ownerState 漏出
```

## Q6. `sx` vs `styled` vs 傳統 CSS / CSS Modules

| 方式 | 優點 | 缺點 | 適用 |
|------|------|------|------|
| `sx` | 就地寫、theme-aware、支援響應式/偽類、零命名成本 | 每次渲染有序列化成本、樣式散落、不利複用 | 單點一次性微調、原型 |
| `styled()` | 可複用、變體集中、樣式在建立時處理、`ownerState` 乾淨 | 需額外抽元件、CSS-in-JS runtime 成本、SSR 需設定 | 設計系統元件、有多變體的可複用元件 |
| 傳統 CSS / CSS Modules | 零 runtime、易被工具鏈快取/壓縮、CSS scope（Modules）、可用 CSS 變數 | 不吃 theme（需自接 CSS 變數）、動態值較麻煩、跨檔追蹤 | 全域 reset、大量靜態樣式、極致效能頁面 |

- **一次性局部微調** → `sx`。
- **可複用、多變體元件** → `styled()`（或 theme 的 `components.variants`）。
- **CSS-in-JS 的 runtime 成本**：Emotion 要在瀏覽器序列化樣式物件、插入 `<style>`、管理 cache；量大時有 CPU 成本。
- **SSR（Next.js）**：需正確設定 Emotion cache 與樣式抽取（App Router 用 Emotion cache provider / `useServerInsertedHTML`），否則會出現「首屏樣式閃爍（FOUC）」或 hydration 不一致。也可評估改用零 runtime 方案（如 Pigment CSS / CSS 變數）。
- **刻意不用 CSS-in-JS**：追求零 runtime、需要極致 LCP/INP、或大量純靜態樣式時，改用 CSS 變數 + 傳統樣式（MUI v6 亦支援 CSS variables 主題，讓深淺色切換不必重建整棵 theme）。

## Q7. TypeScript 主題型別擴充（module augmentation）

MUI 的 theme 與元件 props 型別是「寫死的介面」。你在 runtime 加了 `palette.brand` 或 `variant="dashed"`，但**型別定義裡沒有**，所以 TS 會報錯。解法是 **module augmentation**：對 MUI 的模組「補宣告」，把你的自訂項合併進官方介面。

**(a) 擴充 palette 自訂色**：需同時擴充「取值用的 `Palette`」與「傳入用的 `PaletteOptions`」。

```ts
// theme.d.ts（確保被 tsconfig include）
import '@mui/material/styles';

declare module '@mui/material/styles' {
  interface Palette {
    brand: Palette['primary'];
  }
  interface PaletteOptions {
    brand?: PaletteOptions['primary'];
  }
}
// 之後 theme.palette.brand.main 有型別；createTheme({ palette:{ brand:{...} }}) 也通過
```

若還想讓 `<Button color="brand" />` 這種「把自訂色當 color prop」通過，要再擴充該元件的 color overrides：

```ts
declare module '@mui/material/Button' {
  interface ButtonPropsColorOverrides {
    brand: true;
  }
}
```

**(b) 擴充自訂 variant**：擴充對應元件的 `...PropsVariantOverrides`。

```ts
declare module '@mui/material/Button' {
  interface ButtonPropsVariantOverrides {
    dashed: true;
  }
}
// 之後 <Button variant="dashed" /> 通過型別檢查
```

**(c) 擴充 Theme 自訂欄位**：擴充 `Theme` 與 `ThemeOptions`。

```ts
declare module '@mui/material/styles' {
  interface Theme {
    status: { danger: string };
  }
  interface ThemeOptions {
    status?: { danger?: string };
  }
}

const theme = createTheme({ status: { danger: '#e53935' } });
// 在 styled/sx 裡：theme.status.danger 有型別
```

重點：augmentation 檔案要被 tsconfig 掃到（放進 `include`），且 `interface` 名稱與模組路徑要完全對上，TS 才會做宣告合併。

## Q8. `ThemeProvider`、巢狀主題與 `useTheme`

**`ThemeProvider`** 用 React Context 把 theme 往下供給整棵樹，讓 `sx`/`styled`/`useTheme` 都能取到。通常搭配 `<CssBaseline />`（套用一致的跨瀏覽器 reset 與 theme 背景/文字色）放在應用最外層：

```tsx
import { ThemeProvider, CssBaseline } from '@mui/material';

<ThemeProvider theme={theme}>
  <CssBaseline />
  <App />
</ThemeProvider>
```

**巢狀主題**：內層再包一個 `ThemeProvider`，只覆寫局部區塊。想「基於外層主題」做調整，把 theme 傳成「函式」，MUI 會把外層 theme 當參數傳進來：

```tsx
<ThemeProvider theme={theme}>
  <Header />
  {/* 內層：拿到 outer，只改 primary，其餘沿用 */}
  <ThemeProvider theme={(outer) => ({ ...outer, palette: { ...outer.palette, primary: outer.palette.secondary } })}>
    <Sidebar />
  </ThemeProvider>
</ThemeProvider>
```

**`useTheme()`**：當你需要「在 JS 邏輯裡」讀 theme 值時用它——`sx`/`styled` 已能拿到 theme，所以純樣式不必；但像「傳陰影值給非 MUI 元件」「用 `useMediaQuery(theme.breakpoints.up('md'))` 做條件渲染」「把 spacing 算成數字傳給圖表庫」等就需要：

```tsx
import { useTheme, useMediaQuery } from '@mui/material';

function Toolbar() {
  const theme = useTheme();
  const isDesktop = useMediaQuery(theme.breakpoints.up('md'));
  return isDesktop ? <FullMenu /> : <BurgerMenu />;
}
```

**深/淺色切換**：用 `palette.mode` 建兩套，並以 `useMemo` 快取，避免每次 render 重建整棵 theme：

```tsx
const [mode, setMode] = React.useState<'light' | 'dark'>('light');
const theme = React.useMemo(
  () => createTheme({ palette: { mode } }),
  [mode]
);
// <ThemeProvider theme={theme}> ... setMode 切換即可
```

（MUI v6 另提供 CSS variables 主題 `createTheme({ cssVariables: true })` + `colorSchemes`，讓深淺色靠 CSS 變數切換、不需重建 theme，也更利於 SSR 避免閃爍。）
