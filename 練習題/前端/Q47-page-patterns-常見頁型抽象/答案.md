# Q47 常見頁型抽象與可組合元件 解析

核心觀念：**交付速度來自「不必每次重造輪子」**。把重複出現的頁面骨架與狀態邏輯抽象成設計系統層級的樣板與元件，讓每個新頁面只需要「填內容」與「接資料」，設計輸入不完整時也能穩定、快速、一致地上線。但抽象是有成本的，抽錯比不抽更貴，所以還要懂得「何時不抽」。

---

## Q1. 為何要抽象頁型與樣板來加速交付

**低設計輸入下的價值**：當只有 PRD、沒有完整視覺稿時，如果團隊有一套現成頁型樣板（Layout、Form、List、Detail），工程師可以直接套用，間距、字級、斷點、按鈕位置全部由設計系統決定，不需要每次跟設計師來回確認像素。結果是「不用等完整設計稿就能先做出符合規範的版本」，把設計決策前置到系統層一次做好。

**重複成本問題**：若一致的間距 / 排版 / RWD 斷點每頁重寫，會有：
- 不一致（A 頁 gap 16、B 頁 gap 20），使用者感覺產品「不像同一個」。
- 難維護（改一個斷點要全站找）。
- Bug 分散（每頁各自處理 loading，總有幾頁忘了處理錯誤）。

**好處**：
- 可維護性：邏輯集中，一處修正全站受益。
- 設計系統對齊：元件即規範的落地，避免「規範寫在 Figma、程式碼各做各的」。

**代價**：抽象層本身要設計與維護；抽象若不貼合真實需求，反而綁手綁腳（見 Q5）。抽象也會增加初學者理解專案的門檻。

## Q2. 常見頁型盤點

| 頁型 | 職責 | 典型組成 |
|------|------|---------|
| **Layout** | 全站外框，放置導覽與內容區 | AppBar（頂欄）、Sidebar/Drawer（側欄）、主內容區、Footer；RWD 行為 |
| **Form 頁** | 新增 / 編輯單一資源 | 欄位群組、驗證訊息、送出 / 取消、送出中 disabled、成功導回 |
| **Table/List 頁** | 瀏覽多筆資料 | 工具列（搜尋 / 篩選）、表頭排序、資料列、分頁器、批次操作 |
| **Dialog/Drawer** | 覆蓋層互動（確認 / 快速編輯） | 遮罩、標題、內容、行動按鈕、焦點鎖定與 Esc 關閉 |
| **Detail 頁** | 檢視單一資源 | 標題區、屬性區塊、關聯資料分頁、操作按鈕 |

**Layout 的 RWD**：桌機常用 `permanent` drawer（側欄常駐，內容向右推）；手機用 `temporary` drawer（漢堡鈕點開，覆蓋在內容上、含遮罩），關閉後不佔空間。MUI 的 `Drawer` 用 `variant="permanent" / "temporary"` 搭配斷點切換即可。

**Table 的分頁 / 排序 / 篩選——伺服器端 vs 前端端**：
- **伺服器端**：把 page、sort、filter 當查詢參數送給 API，只取當頁資料。適合資料量大（上千筆以上），但每次操作都要往返 API、需處理 loading。
- **前端端**：一次抓全部資料，排序 / 篩選 / 分頁都在瀏覽器算。適合資料量小、操作要即時，但資料一大就吃記憶體且首次載入慢。
- 判準：資料量與成長性。會持續成長 → 伺服器端；固定小量 → 前端端。

## Q3. 三態（四態）的元件化

**四種狀態**：
- **loading**：資料請求中。
- **error**：請求失敗 / 例外。
- **empty**：請求成功但沒有資料（要與 loading 區分！）。
- **success**：有資料，正常渲染。

**為何標準化**：每個資料驅動的頁面都會遇到這四態。若每頁各寫一套 `if (loading) ... if (error) ...`，會漏、會不一致（有的頁面沒空狀態、有的錯誤沒重試）。標準化成一個抽象後，新頁面只要「給資料狀態 + 成功時要渲染什麼」，四態的 UX 自動一致，交付更快、體驗更統一。

**三種非成功狀態的做法**：
- **載入**：
  - **Skeleton**（骨架屏）：畫出內容的輪廓灰塊，適合「已知版面結構」的內容（列表、卡片、表格），能降低跳動感、感知更快。
  - **Spinner**：單純轉圈，適合小區塊、短等待、或版面未知時。
- **錯誤**：
  - **Error Boundary** 能攔「子元件 render / lifecycle 期間拋出的例外」，避免整頁白屏；**攔不到**事件處理器、非同步（setTimeout / Promise）、以及 SSR 期間的錯誤——這些要自己 try/catch 或用資料層（React Query 的 error 狀態）處理。
  - 「重試」：Error Boundary 提供 `resetErrorBoundary`，或資料層提供 `refetch()`，錯誤畫面放一顆「重試」按鈕呼叫它。
- **空狀態**：好的 Empty state 應包含——圖示 / 插圖、一句說明「為什麼是空的」、以及一個「下一步 CTA」（例如「新增第一筆」）。空狀態是引導使用者的機會，不該只顯示「無資料」。

**可重用抽象介面**（見下方 `<AsyncBoundary>` 範例）：把「isLoading / isError / isEmpty / 成功 render」收斂成單一元件的 props，讓呼叫端只描述資料與內容。

## Q4. 可組合性模式

| 模式 | 說明 | 適用 | 取捨 |
|------|------|------|------|
| **Compound Components** | 一組元件共享隱性狀態（透過 Context），如 `<Tabs>` + `<Tabs.Tab>` | 元件有多個相關子部件、要給使用者排列自由度 | API 直覺、彈性高；實作較複雜、需 Context |
| **Slots / render props** | 把「渲染什麼」以 prop（常為函式）交給呼叫端 | 需要注入自訂內容或把內部狀態暴露出去 | 極彈性；巢狀多會變「callback 金字塔」 |
| **`children` 組合** | 直接把內容當 children 塞入 | 容器類（Card、Panel、Layout） | 最簡單直覺；無法傳結構化資料 |
| **Polymorphic (`as` prop)** | 同一元件可渲染成不同標籤 / 元件 | 樣式相同語意不同（`<Button as="a">`） | 重用樣式；TS 型別較難寫對 |

**組合優於配置**：當一個元件的 boolean props 越加越多（`isPrimary`、`hasIcon`、`withHeader`、`compact`…），代表它試圖用「開關」涵蓋所有變體，會產生組合爆炸（2^n 種組合、多數沒測過）與難懂的 API。改用組合——讓呼叫端用 children / 子元件自己拼出想要的樣子——更彈性也更好維護。boolean props 暴增是「該改用組合」的警訊。

## Q5. 何時抽象、何時別過早抽象

**過早 / 錯誤抽象的風險**：抽象是對「未來的變化」下賭注。太早抽（只出現一兩次就抽），你其實還不知道真正的變異點在哪，抽出來的 API 常常猜錯。之後每個使用端的新需求都要塞進這個錯的抽象裡，加一堆 flag、特例，最後比重複還難改。Sandi Metz 的名言：「**重複遠比錯誤的抽象便宜（duplication is far cheaper than the wrong abstraction）**」。

**判準（AHA / Rule of Three）**：
- **Rule of Three**：同樣的東西出現到第三次，才考慮抽象。前兩次先容忍重複。
- **變異點是否穩定**：如果每個使用端「不同的地方」都不一樣且還在變，代表需求未定型，先別抽。
- **抽出的 API 是否好懂**：如果要一長串 props / flag 才能描述差異，這個抽象大概是錯的。
- **AHA（Avoid Hasty Abstractions）**：寧可先重複，等 pattern 清楚了再抽，也不要為了 DRY 硬抽。

**良好元件 API 設計原則**：
- **命名一致且語意化**：跟 HTML / 生態慣例對齊（`onChange`、`value`、`disabled`）。
- **合理預設值**：常見情境零設定就能用。
- **受控 vs 非受控**：提供 `value`+`onChange`（受控）與 `defaultValue`（非受控）兩種，別強迫使用者管狀態。
- **可組合**：優先用 children / compound，而非無限增加 props。
- **逃生艙（escape hatch）**：允許傳 `className` / `sx` / `style` / `slotProps`，讓使用者在抽象不夠用時仍能微調，而不必 fork 整個元件。
- **最小驚訝原則**：行為符合直覺，不要偷偷有副作用。

## Q6. 大量列表效能：MUI DataGrid 與虛擬化

**虛擬化（windowing）原理**：只渲染「目前視窗（viewport）可見的那幾列」加上少量緩衝，其餘用上下 padding 撐出正確的捲動高度。捲動時動態換掉 DOM 中的列。這樣就算資料有一萬筆，DOM 裡永遠只有數十個節點，解決「一次渲染大量 DOM 造成的卡頓與記憶體暴增」。

**MUI `DataGrid`**：內建排序、篩選、分頁，且**列的虛擬化預設開啟**（還有欄虛擬化）。版本差異概觀：
- **社群版（`@mui/x-data-grid`，MIT 免費）**：單欄排序、基本篩選、分頁、列虛擬化——中小型表格夠用。
- **Pro / Premium（商業授權）**：多欄排序、進階篩選、欄釘選、樹狀 / 分組、伺服器端整合強化，以及對超大資料集更完整的支援。
- 大量資料時，若需要伺服器端分頁 + 進階功能，通常走 Pro；純前端小量用社群版即可。

**虛擬化的取捨 / 注意事項**：
- 瀏覽器 **Ctrl+F 找不到**未渲染的列（內容不在 DOM）。
- **SEO**：爬蟲看不到未渲染內容（對後台工具通常無所謂）。
- **可及性（a11y）**：螢幕報讀器與鍵盤導覽需要正確的 ARIA（DataGrid 已處理，自刻要小心）。
- **動態 / 不定列高**需要量測，較複雜；固定列高最順。
- 錨點跳轉、列展開等要特別處理。

**何時用 DataGrid vs 自刻 table**：
- 需要排序 / 篩選 / 分頁 / 虛擬化 / 大量資料 → 用 DataGrid，省下大量輪子。
- 只是少量、版面高度客製的靜態表格 → 用原生 `<table>` 或 MUI `Table` 自刻，更輕、更好控樣式，不必背 DataGrid 的 API 與體積。

## 程式碼示範

### 1) 四態抽象：`<AsyncBoundary>`

把 loading / error / empty / success 收斂成單一元件，新頁面只描述資料與成功內容。

```tsx
import * as React from 'react';
import { CircularProgress, Box, Button, Typography } from '@mui/material';

type AsyncBoundaryProps<T> = {
  /** 是否載入中 */
  isLoading: boolean;
  /** 錯誤物件（null 表示無錯誤） */
  error?: unknown;
  /** 資料（可能為 undefined / 空陣列） */
  data: T | undefined;
  /** 判斷是否為空狀態，預設：undefined / null / 空陣列 視為空 */
  isEmpty?: (data: T) => boolean;
  /** 重試 callback（錯誤畫面的按鈕會呼叫它） */
  onRetry?: () => void;
  /** 自訂各狀態畫面（皆有預設） */
  loadingFallback?: React.ReactNode;
  emptyFallback?: React.ReactNode;
  /** 成功時如何渲染（render prop，拿到非空 data） */
  children: (data: T) => React.ReactNode;
};

function defaultIsEmpty(data: unknown): boolean {
  if (data == null) return true;
  if (Array.isArray(data)) return data.length === 0;
  return false;
}

export function AsyncBoundary<T>({
  isLoading,
  error,
  data,
  isEmpty = defaultIsEmpty,
  onRetry,
  loadingFallback,
  emptyFallback,
  children,
}: AsyncBoundaryProps<T>) {
  if (isLoading) {
    return (
      <>{loadingFallback ?? (
        <Box sx={{ display: 'flex', justifyContent: 'center', py: 6 }}>
          <CircularProgress aria-label="載入中" />
        </Box>
      )}</>
    );
  }

  if (error) {
    return (
      <Box sx={{ textAlign: 'center', py: 6 }}>
        <Typography color="error" gutterBottom>
          載入失敗，請稍後再試。
        </Typography>
        {onRetry && (
          <Button variant="outlined" onClick={onRetry}>重試</Button>
        )}
      </Box>
    );
  }

  if (data === undefined || isEmpty(data)) {
    return (
      <>{emptyFallback ?? (
        <Box sx={{ textAlign: 'center', py: 6, color: 'text.secondary' }}>
          <Typography gutterBottom>目前沒有資料</Typography>
        </Box>
      )}</>
    );
  }

  return <>{children(data)}</>;
}
```

呼叫端（搭配 React Query，把四態一次處理掉）：

```tsx
function UserListPage() {
  const { data, isLoading, error, refetch } = useQuery({
    queryKey: ['users'],
    queryFn: fetchUsers, // 回傳 User[]
  });

  return (
    <AsyncBoundary
      isLoading={isLoading}
      error={error}
      data={data}
      onRetry={() => refetch()}
      emptyFallback={<EmptyState title="還沒有使用者" cta="新增第一位" />}
    >
      {(users) => (
        <ul>
          {users.map((u) => <li key={u.id}>{u.name}</li>)}
        </ul>
      )}
    </AsyncBoundary>
  );
}
```

### 2) Compound Component：`<Tabs>`

以 Context 共享隱性狀態，呼叫端可自由排列子元件——組合優於一堆 props。

```tsx
import * as React from 'react';

type TabsContextValue = {
  active: string;
  setActive: (id: string) => void;
};
const TabsContext = React.createContext<TabsContextValue | null>(null);

function useTabsContext() {
  const ctx = React.useContext(TabsContext);
  if (!ctx) throw new Error('Tabs.* 必須放在 <Tabs> 內使用');
  return ctx;
}

type TabsProps = {
  /** 非受控預設值 */
  defaultValue: string;
  children: React.ReactNode;
};

export function Tabs({ defaultValue, children }: TabsProps) {
  const [active, setActive] = React.useState(defaultValue);
  const value = React.useMemo(() => ({ active, setActive }), [active]);
  return <TabsContext.Provider value={value}>{children}</TabsContext.Provider>;
}

function TabList({ children }: { children: React.ReactNode }) {
  return <div role="tablist" style={{ display: 'flex', gap: 8 }}>{children}</div>;
}

function Tab({ id, children }: { id: string; children: React.ReactNode }) {
  const { active, setActive } = useTabsContext();
  const selected = active === id;
  return (
    <button
      role="tab"
      aria-selected={selected}
      onClick={() => setActive(id)}
      style={{ fontWeight: selected ? 700 : 400 }}
    >
      {children}
    </button>
  );
}

function TabPanel({ id, children }: { id: string; children: React.ReactNode }) {
  const { active } = useTabsContext();
  if (active !== id) return null;
  return <div role="tabpanel">{children}</div>;
}

// 掛成 compound API
Tabs.List = TabList;
Tabs.Tab = Tab;
Tabs.Panel = TabPanel;
```

使用：

```tsx
<Tabs defaultValue="profile">
  <Tabs.List>
    <Tabs.Tab id="profile">個人資料</Tabs.Tab>
    <Tabs.Tab id="security">安全性</Tabs.Tab>
  </Tabs.List>
  <Tabs.Panel id="profile">…個人資料表單…</Tabs.Panel>
  <Tabs.Panel id="security">…密碼設定…</Tabs.Panel>
</Tabs>
```

### 3) Polymorphic component：`as` prop

同一套樣式，語意可切換（`<Text as="h1">` / `<Text as="span">`）。

```tsx
import * as React from 'react';

type TextOwnProps<E extends React.ElementType> = {
  as?: E;
  children: React.ReactNode;
};

type TextProps<E extends React.ElementType> = TextOwnProps<E> &
  Omit<React.ComponentPropsWithoutRef<E>, keyof TextOwnProps<E>>;

export function Text<E extends React.ElementType = 'span'>({
  as,
  children,
  ...rest
}: TextProps<E>) {
  const Component = as ?? 'span';
  return <Component {...rest}>{children}</Component>;
}

// <Text as="h1">標題</Text>  → 渲染成 <h1>
// <Text as="a" href="/x">連結</Text> → 型別自動帶入 href
```

### 4) Layout 樣板（RWD 側欄）

```tsx
import { AppBar, Toolbar, Drawer, Box, IconButton, useMediaQuery, useTheme } from '@mui/material';
import MenuIcon from '@mui/icons-material/Menu';
import * as React from 'react';

const DRAWER_WIDTH = 240;

export function AppLayout({ nav, children }: { nav: React.ReactNode; children: React.ReactNode }) {
  const theme = useTheme();
  const isDesktop = useMediaQuery(theme.breakpoints.up('md'));
  const [open, setOpen] = React.useState(false);

  return (
    <Box sx={{ display: 'flex' }}>
      <AppBar position="fixed" sx={{ zIndex: (t) => t.zIndex.drawer + 1 }}>
        <Toolbar>
          {!isDesktop && (
            <IconButton color="inherit" edge="start" onClick={() => setOpen(true)} aria-label="開啟選單">
              <MenuIcon />
            </IconButton>
          )}
        </Toolbar>
      </AppBar>

      {/* 桌機：常駐側欄；手機：暫時性 drawer */}
      <Drawer
        variant={isDesktop ? 'permanent' : 'temporary'}
        open={isDesktop ? true : open}
        onClose={() => setOpen(false)}
        sx={{ width: DRAWER_WIDTH, '& .MuiDrawer-paper': { width: DRAWER_WIDTH } }}
      >
        <Toolbar />
        {nav}
      </Drawer>

      <Box component="main" sx={{ flexGrow: 1, p: 3 }}>
        <Toolbar />
        {children}
      </Box>
    </Box>
  );
}
```

## 一句話總結

**先抽穩定的頁型骨架與四態邏輯，讓交付變成填內容；用組合（compound / children / slots / `as`）而非爆炸的 props 保留彈性；抽象等出現第三次、變異點清楚了再動手——錯的抽象比重複更貴。**
