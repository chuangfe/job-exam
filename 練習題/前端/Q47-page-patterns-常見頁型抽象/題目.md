# Q47 常見頁型抽象與可組合元件（設計系統加速交付：從 PRD 到上線）

在真實產品團隊中，設計輸入往往不完整：可能只有一頁粗略的 PRD、少量 Figma 稿，或只有口頭需求，就要求你在短時間內把功能上線。此時「重新為每個頁面手刻」是最慢也最不穩定的做法。有經驗的前端會把**常見頁型（page patterns）抽象成可組合的樣板與元件**，讓交付從「畫像素」變成「填內容」，同時把載入 / 錯誤 / 空 / 成功四種狀態標準化，避免每頁各寫一套。

本題探討：為何要抽象頁型、有哪些常見頁型、三態（四態）如何元件化、可組合性模式、以及「何時該抽象、何時別過早抽象」的判斷。請以繁體中文（台灣用語）作答，並在需要處佐以 React + TypeScript（MUI 生態）的思路。

---

## Q1. 為何要抽象頁型與樣板來加速交付

- 說明在「低設計輸入」情境下（只有 PRD、沒有完整視覺稿），抽象頁型如何讓團隊快速產出且結果一致。
- 從「重複成本」角度說明：一致的間距 / 排版 / RWD 斷點若每頁重寫，會產生哪些問題？
- 抽象樣板對「可維護性」與「設計系統對齊」的好處各是什麼？
- 抽象頁型有沒有代價？請至少列出一項。

## Q2. 盤點常見頁型

- 請列出並簡述以下常見頁型的職責與典型組成：
  - Layout 頁（含 AppBar、側欄 Sidebar / Drawer、內容區、RWD 行為）。
  - Form 頁（新增 / 編輯）。
  - Table / List 頁（含篩選 filter、排序 sort、分頁 pagination）。
  - Dialog / Drawer（覆蓋層互動）。
  - Detail 頁（單一資源的檢視）。
- 針對 Layout，說明桌機與手機在側欄呈現上的常見差異（permanent vs temporary drawer）。
- 針對 Table 頁，說明「伺服器端分頁 / 排序 / 篩選」與「前端端」兩種做法的取捨。

## Q3. 三態（四態）處理的元件化

- 何謂 UI 的四種狀態：loading（載入）/ error（錯誤）/ empty（空）/ success（成功）？
- 為何要把這四態「標準化」成統一元件或統一 render 模式？對交付速度與 UX 一致性的價值為何？
- 分別說明三種非成功狀態的元件化做法：
  - 載入：Skeleton 與 Spinner 各適合什麼場景？
  - 錯誤：Error Boundary 能攔什麼、不能攔什麼？如何提供「重試」？
  - 空狀態：一個好的 Empty state 應包含哪些元素？
- 請設計一個能同時處理四態的可重用抽象（例如 `<AsyncBoundary>` 或 `renderState` helper）的介面。

## Q4. 可組合性模式

- 請比較以下四種組合模式，各自的適用時機與優缺點：
  - Compound Components（複合元件，如 `<Tabs><Tabs.List/><Tabs.Panel/></Tabs>`）。
  - Slots / render props。
  - 以 `children` 做組合（composition over configuration）。
  - Polymorphic component（`as` prop，讓同一元件能渲染成不同標籤 / 元件）。
- 為何「組合優於配置（composition over props explosion）」？當一個元件的 boolean props 越加越多時代表什麼警訊？

## Q5. 何時抽象、何時別過早抽象

- 說明「過早抽象 / 錯誤抽象」的風險，以及為何「重複有時比錯誤的抽象便宜」（可引用 AHA / Rule of Three 觀念）。
- 提出一套判斷準則：出現幾次重複、變異點是否穩定、抽象後的 API 是否好懂，才值得抽象？
- 列出良好元件 API 設計原則（命名、預設值、受控 vs 非受控、可組合、逃生艙 escape hatch 等）。

## Q6. 大量列表效能：MUI DataGrid 與虛擬化

- 說明「虛擬化（virtualization / windowing）」的原理，以及它解決了什麼效能問題。
- MUI `DataGrid` 提供了哪些能力（排序 / 篩選 / 分頁 / 虛擬化）？社群版與 Pro/Premium 版在大量資料上的差異概觀為何？
- 虛擬化有哪些取捨與注意事項（例如 Ctrl+F 找不到、SEO、可及性 accessibility、動態列高）？
- 何時該用 DataGrid、何時自己刻 table 就好？
