# React Hook Form 與複雜表單體驗

請依序回答以下 React Hook Form（RHF）相關問題。設計系統／表單重度職缺（如後台、金融、B2B SaaS）很看重這塊——重點不在背 API，而在能不能講清楚「受控 vs 非受控」的取捨、RHF 為什麼快、以及複雜表單（動態欄位、相依驗證、非同步驗證、a11y）該怎麼設計。建議先自己白話講一遍，再對照解析。

## Q1. 受控 vs 非受控表單，以及 RHF 為何快？

- React 裡「受控元件（controlled）」與「非受控元件（uncontrolled）」分別是什麼？各自誰是資料的來源（single source of truth）？
- 純用 `useState` 做受控表單，當表單有 30 個欄位時會有什麼效能問題？為什麼每次打字都可能觸發整張表單重渲染？
- RHF 主要走非受控 + `ref` 的路線，這如何「大幅減少重渲染」？「訂閱式（subscription）更新」是什麼意思？
- RHF 完全不重渲染嗎？什麼情況它還是需要重渲染（例如 `formState.errors`、`watch`）？

## Q2. 核心 API：`useForm` 家族？

- `useForm` 常用的初始化選項有哪些（`defaultValues`、`mode`、`resolver`）？`mode` 的 `onSubmit` / `onBlur` / `onChange` / `all` 差在哪？
- `register` 做了什麼？它回傳哪些東西掛到 input 上（`name`、`onChange`、`onBlur`、`ref`）？為什麼要用展開（`{...register('email')}`）？
- `handleSubmit` 的職責是什麼？它會在什麼時候呼叫你的 `onValid`、什麼時候呼叫 `onInvalid`？
- `formState` 常用的旗標：`errors`、`isDirty`、`isValid`、`isSubmitting`、`dirtyFields`、`touchedFields`、`isSubmitSuccessful` 各代表什麼？使用 `formState` 有什麼「Proxy 訂閱」的注意事項？
- `watch`、`getValues`、`setValue`、`reset` 分別在什麼情境用？`watch('x')` 與 `getValues('x')` 最大差別是什麼（會不會觸發重渲染）？
- `setValue` 的 `shouldValidate` / `shouldDirty` / `shouldTouch` 選項在做什麼？

## Q3. `Controller`：整合受控 UI 套件？

- 為什麼像 MUI 的 `TextField` / `Select`、Ant Design、react-select 這類元件不能直接用 `register`？
- `Controller` 解決什麼問題？`render` prop 裡的 `field`（`value`、`onChange`、`onBlur`、`ref`、`name`）與 `fieldState`（`error`、`isDirty` …）怎麼用？
- `Controller` 與 `register` 在效能／重渲染上的差別？為什麼包一層 `Controller` 通常只讓「那一個欄位」重渲染？
- 請用 MUI 的 `TextField`（或任一受控 Select）寫一個 `Controller` 範例。

## Q4. 驗證：內建 rules vs schema（zod / yup）？

- `register` 內建的驗證 rules（`required`、`min`、`max`、`minLength`、`pattern`、`validate`）怎麼寫？`validate` 可以做到什麼內建 rule 做不到的事？
- 為什麼實務上大型表單偏好用 schema 驗證（`zodResolver` / `yupResolver`）？相較於把 rules 散在各個 `register`，schema 有什麼好處（單一來源、型別推導、前後端共用）？
- 請寫一個 zod schema + `zodResolver` 的完整範例，並示範如何用 `z.infer` 推導出表單型別、餵給 `useForm<T>`。

## Q5. 複雜場景：動態欄位、相依驗證、非同步、巢狀？

- `useFieldArray` 解決什麼問題？`fields`、`append`、`remove`、`insert`、`move` 怎麼用？為什麼 render 時 key 要用 `field.id` 而不是陣列 index？
- 「跨欄位／相依驗證」怎麼做（例如「確認密碼」要等於「密碼」、「結束日期」要晚於「開始日期」）？用內建 `validate` 與用 zod `refine` / `superRefine` 各怎麼寫？
- 「非同步驗證」怎麼做（例如打 API 檢查帳號是否已被註冊）？搭配 debounce 要注意什麼？
- 巢狀物件（`address.city`）與陣列欄位的 `name` 命名規則是什麼？

## Q6. 錯誤呈現與送出中 UX？

- 錯誤訊息通常放哪、何時顯示（`mode` 與 `reValidateMode` 的影響）？
- 送出時要「focus 到第一個錯誤欄位」，RHF 預設會做嗎？`shouldFocusError` 是什麼？若用 `Controller` 要 focus 需要注意什麼？
- 送出中（`isSubmitting`）該怎麼處理按鈕與重複送出？送出成功／失敗後 `reset` 的時機？
- 「伺服器端驗證錯誤」（例如後端回傳某欄位不合法）怎麼塞回表單？`setError` 怎麼用？
- 樂觀更新 / 送出失敗重試在表單情境要注意什麼（不要清掉使用者輸入、保留 dirty 狀態）？

## Q7. 表單無障礙（a11y）？

- `<label>` 要怎麼跟 input 正確關聯（`htmlFor` / `id`，或包起來）？為什麼 placeholder 不能取代 label？
- 欄位有錯誤時，`aria-invalid` 與 `aria-describedby` 分別扮演什麼角色？怎麼把錯誤訊息元素的 `id` 接到 input 上讓螢幕報讀器唸出來？
- 錯誤訊息容器適合用什麼 `role` / `aria-live`，讓動態出現的錯誤能被報讀？
- 必填欄位除了紅色星號，還該補什麼給輔助科技？
