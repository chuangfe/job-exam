# React Hook Form 與複雜表單體驗 解析

## Q1. 受控 vs 非受控表單，以及 RHF 為何快？

- **受控元件（controlled）**：input 的 `value` 由 React state 決定，每次變動都經 `onChange` 更新 state，再由 state 回灌 `value`。React state 是唯一資料來源（single source of truth）。
- **非受控元件（uncontrolled）**：input 自己管理內部值，React 只在需要時透過 `ref`（或 `defaultValue`）讀取。DOM 才是資料來源。
- **純 `useState` 的效能問題**：受控表單每打一個字就 `setState`，觸發元件重渲染。若整張表單的 state 放在同一個父元件，30 個欄位任何一個變動都會重渲染整棵表單樹（含所有欄位、驗證邏輯）。欄位越多、每次輸入越卡。要避免就得把 state 下放、拆元件、`memo`，成本很高。
- **RHF 為何快**：RHF 預設走**非受控 + `ref`**。輸入值由 DOM 保存，RHF 只在必要時（送出、驗證）從 `ref` 讀值，所以**打字本身不觸發 React 重渲染**。它採**訂閱式（subscription）更新**——只有真正「訂閱」了某段狀態的元件才會在該狀態變動時重渲染。例如某欄位的錯誤訊息元件只訂閱自己的 error，別的欄位變動不會連動。
- **不是完全不重渲染**：當你用 `watch()` 觀察某欄位、或 `formState.errors` 變化、或 `formState.isValid` 改變時，訂閱到的元件仍會重渲染——這是必要的。RHF 的價值是把重渲染**最小化到訂閱範圍**，而不是「零重渲染」。用 `watch` 觀察整張表單，或在最上層讀很多 `formState` 旗標，就會把重渲染範圍放大，要留意。

## Q2. 核心 API：`useForm` 家族？

- **`useForm` 初始化選項**：
  - `defaultValues`：每個欄位初始值（強烈建議提供，尤其非受控 input，避免 controlled/uncontrolled 警告；也讓 `reset` 有基準）。
  - `mode`：**第一次觸發驗證的時機**——`onSubmit`（預設，送出才驗）/ `onBlur`（失焦驗）/ `onChange`（每次變動驗，較耗效能）/ `onTouched`（第一次 blur 後才開始）/ `all`（blur + change 都驗）。
  - `reValidateMode`：**第一次出錯後**的重新驗證時機（預設 `onChange`）。
  - `resolver`：接 schema 驗證（zod/yup）。
- **`register`**：把一個欄位「登記」進表單，回傳 `{ name, onChange, onBlur, ref }` 一組 props。用展開 `{...register('email')}` 是為了一次把這幾個 handler / ref 掛到 input 上，RHF 靠 `ref` 讀值、靠 `onChange`/`onBlur` 觸發驗證。
- **`handleSubmit(onValid, onInvalid?)`**：包住原生 submit，先 `preventDefault`、跑完整驗證；**全部通過**才呼叫 `onValid(data)`，**有錯**則呼叫 `onInvalid(errors)`（並依 `shouldFocusError` focus 第一個錯誤）。
- **`formState` 旗標**：
  - `errors`：各欄位錯誤物件。
  - `isDirty`：使用者是否改過（與 `defaultValues` 相比）。
  - `dirtyFields` / `touchedFields`：哪些欄位被改過 / 被碰過（blur 過）。
  - `isValid`：目前是否全部通過（需要非 `onSubmit` 的 mode，或 schema，才會即時更新）。
  - `isSubmitting`：`onValid` 若是 async，執行期間為 `true`。
  - `isSubmitSuccessful`：上次送出是否成功。
  - **Proxy 訂閱注意**：`formState` 是 Proxy，**你讀了哪個旗標，才會訂閱那個旗標**。所以要「解構出來讀」才有效（`const { isDirty } = formState`），不要用條件式動態存取而漏掉訂閱，否則畫面不更新。
- **`watch` vs `getValues`**：`watch('x')` 會**訂閱**該欄位，值變時觸發元件重渲染（適合要即時反映到 UI）；`getValues('x')` 只是**當下讀一次快照，不訂閱、不觸發重渲染**（適合事件 handler 裡臨時讀值）。
- **`setValue` / `reset`**：`setValue('x', v, opts)` 程式化設值；`reset(values)` 把整張表單重設（常用於載入既有資料、送出成功後清空）。
- **`setValue` 選項**：`shouldValidate`（設完馬上驗這欄）、`shouldDirty`（標記為 dirty）、`shouldTouch`（標記為 touched）。預設都不做，需要就明確開。

## Q3. `Controller`：整合受控 UI 套件？

- **為什麼不能直接 `register`**：`register` 靠把 `ref` 掛到「真正的 DOM input」並讀 `event.target.value`。MUI `TextField`、Ant Design、react-select 這類元件是**受控**的——它們只吃 `value` / `onChange`，且 `onChange` 的簽名往往不是標準 DOM event（可能是 `(value)` 或 `(option)`），`ref` 也不一定轉發到底層 input。直接 `register` 拿不到值。
- **`Controller` 做的事**：它在 RHF 與受控元件之間當橋樑。透過 `render` prop 給你：
  - `field`：`{ value, onChange, onBlur, ref, name }`——把這組直接餵給受控元件，等於「RHF 幫你做受控」。
  - `fieldState`：`{ error, isDirty, isTouched, invalid }`——用來顯示錯誤。
- **效能**：`Controller` 內部是受控的，所以該欄位輸入時**那一個 `Controller` 會重渲染**，但因為它是獨立元件、隔離了狀態訂閱，**不會連累整張表單**。等於「局部受控、整體非受控」，兼顧相容性與效能。
- **MUI 範例**：

```tsx
import { useForm, Controller } from 'react-hook-form'
import { TextField, MenuItem } from '@mui/material'

type FormValues = { email: string; role: string }

function ProfileForm() {
  const { control, handleSubmit } = useForm<FormValues>({
    defaultValues: { email: '', role: 'user' },
  })

  return (
    <form onSubmit={handleSubmit((data) => console.log(data))}>
      <Controller
        name="email"
        control={control}
        rules={{ required: 'Email 必填' }}
        render={({ field, fieldState }) => (
          <TextField
            {...field}                       // value / onChange / onBlur / ref / name
            label="Email"
            error={!!fieldState.error}       // MUI 的錯誤外觀
            helperText={fieldState.error?.message}
          />
        )}
      />

      <Controller
        name="role"
        control={control}
        render={({ field }) => (
          <TextField {...field} select label="角色">
            <MenuItem value="user">一般使用者</MenuItem>
            <MenuItem value="admin">管理員</MenuItem>
          </TextField>
        )}
      />

      <button type="submit">送出</button>
    </form>
  )
}
```

## Q4. 驗證：內建 rules vs schema（zod / yup）？

- **內建 rules**（寫在 `register` 第二參數）：

```tsx
<input
  {...register('username', {
    required: '必填',
    minLength: { value: 3, message: '至少 3 個字' },
    maxLength: { value: 20, message: '最多 20 個字' },
    pattern: { value: /^[a-zA-Z0-9_]+$/, message: '只能用英數與底線' },
    validate: (v) => v !== 'admin' || '這個名稱被保留',   // 自訂邏輯
  })}
/>
```

  `validate` 可以是函式或物件（多條規則），能做內建 rule 做不到的**任意邏輯**（跨欄位、查表、非同步）。
- **為什麼大型表單偏好 schema**：
  - **單一來源**：所有規則集中在一份 schema，不散落在數十個 `register`，好讀好維護。
  - **型別推導**：zod 可用 `z.infer` 直接推出 TypeScript 型別，表單型別與驗證規則保證同步。
  - **前後端共用**：同一份 zod schema 可在後端 API 再驗一次，避免規則兩邊各寫一份而不一致。
  - **可組合**：`refine` / `superRefine` / `merge` / `extend` 表達複雜與相依規則比散裝 rules 清楚。
- **zod + `zodResolver` 完整範例**：

```tsx
import { useForm } from 'react-hook-form'
import { zodResolver } from '@hookform/resolvers/zod'
import { z } from 'zod'

const schema = z
  .object({
    email: z.string().email('Email 格式不正確'),
    password: z.string().min(8, '密碼至少 8 碼'),
    confirm: z.string(),
    age: z.coerce.number().int().min(18, '需年滿 18 歲'), // coerce：input 是字串
  })
  .refine((d) => d.password === d.confirm, {
    message: '兩次密碼不一致',
    path: ['confirm'], // 錯誤掛到 confirm 欄位
  })

type FormValues = z.infer<typeof schema> // 型別由 schema 推導

function SignUpForm() {
  const {
    register,
    handleSubmit,
    formState: { errors, isSubmitting },
  } = useForm<FormValues>({
    resolver: zodResolver(schema),
    mode: 'onTouched',
    defaultValues: { email: '', password: '', confirm: '', age: 18 },
  })

  const onSubmit = async (data: FormValues) => {
    await api.signUp(data)
  }

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('email')} />
      {errors.email && <p>{errors.email.message}</p>}
      <input type="password" {...register('password')} />
      {errors.password && <p>{errors.password.message}</p>}
      <input type="password" {...register('confirm')} />
      {errors.confirm && <p>{errors.confirm.message}</p>}
      <button disabled={isSubmitting}>註冊</button>
    </form>
  )
}
```

## Q5. 複雜場景：動態欄位、相依驗證、非同步、巢狀？

- **`useFieldArray`（動態欄位）**：處理「一組可增刪的欄位」（如多筆聯絡人、訂單明細）。回傳 `fields`（目前列陣列）、`append` / `remove` / `insert` / `move` / `update` / `swap` 等操作：

```tsx
import { useForm, useFieldArray } from 'react-hook-form'

type FormValues = { items: { name: string; qty: number }[] }

function OrderForm() {
  const { register, control, handleSubmit } = useForm<FormValues>({
    defaultValues: { items: [{ name: '', qty: 1 }] },
  })
  const { fields, append, remove } = useFieldArray({ control, name: 'items' })

  return (
    <form onSubmit={handleSubmit((d) => console.log(d))}>
      {fields.map((field, i) => (
        <div key={field.id}>{/* 用 field.id，不要用 index 當 key */}
          <input {...register(`items.${i}.name` as const)} />
          <input type="number" {...register(`items.${i}.qty` as const, { valueAsNumber: true })} />
          <button type="button" onClick={() => remove(i)}>刪除</button>
        </div>
      ))}
      <button type="button" onClick={() => append({ name: '', qty: 1 })}>新增一列</button>
      <button>送出</button>
    </form>
  )
}
```

  **為什麼 key 用 `field.id`**：RHF 給每列一個穩定的 `id`。用陣列 `index` 當 key，刪除／插入中間項時，React 會誤判哪列是哪列，導致輸入值錯位、DOM 狀態亂掉。`field.id` 才能正確對應。

- **跨欄位／相依驗證**：
  - 內建 `validate` 讀其他欄位（用 `getValues`）：

```tsx
<input
  type="password"
  {...register('confirm', {
    validate: (v, formValues) => v === formValues.password || '兩次密碼不一致',
  })}
/>
```

  - zod 用 `refine`（單一條件）或 `superRefine`（多條件、可各自指定 path）：

```ts
const schema = z
  .object({ start: z.string(), end: z.string() })
  .superRefine((d, ctx) => {
    if (new Date(d.end) < new Date(d.start)) {
      ctx.addIssue({ code: 'custom', message: '結束日不能早於開始日', path: ['end'] })
    }
  })
```

- **非同步驗證**（查帳號是否重複）：`validate` 可回傳 Promise：

```tsx
{...register('account', {
  validate: async (v) => {
    const taken = await api.checkAccountExists(v)
    return !taken || '此帳號已被註冊'
  },
})}
```

  搭配 debounce 注意：頻繁打字會狂打 API，實務上用 `mode: 'onBlur'`（失焦才驗）或把檢查函式 debounce、並處理**競態（race condition）**——晚回來的舊請求不要覆蓋新結果（可用 AbortController 取消上一個請求）。

- **巢狀 / 陣列命名**：用點與中括號路徑字串——巢狀物件 `register('address.city')`、陣列 `register('items.0.name')`。RHF 會依此自動組成巢狀的 `data` 結構。`defaultValues` 也照同樣結構給。

## Q6. 錯誤呈現與送出中 UX？

- **錯誤何時顯示**：由 `mode`（首次驗證時機）與 `reValidateMode`（出錯後重驗時機）決定。常見體驗是 `mode: 'onTouched'` + `reValidateMode: 'onChange'`——**碰過欄位才開始驗，一旦出錯就邊改邊即時消除**，避免一進頁面就滿江紅。錯誤訊息通常緊貼欄位下方。
- **focus 到第一個錯誤**：RHF **預設會做**（`useForm` 的 `shouldFocusError` 預設 `true`）。送出驗證失敗時，會 focus 到第一個有錯的欄位。用 `register` 的原生 input 自動支援；用 `Controller` 包第三方元件時，要確保 `field.ref` 有被轉發到可 focus 的底層 DOM 節點，否則 focus 不到。
- **送出中處理**：`onValid` 設成 async，`formState.isSubmitting` 會在期間為 `true`。用它 `disabled` 送出鈕、顯示 loading，天然防重複送出：

```tsx
<button type="submit" disabled={isSubmitting}>
  {isSubmitting ? '送出中…' : '送出'}
</button>
```

  送出成功後常搭配 `reset()`（清空或以回傳資料為新基準）。可監看 `isSubmitSuccessful` 在 `useEffect` 裡 `reset`。
- **伺服器端驗證錯誤塞回表單**：用 `setError` 把後端回傳的欄位錯誤對應回去：

```tsx
const onSubmit = async (data: FormValues) => {
  try {
    await api.signUp(data)
  } catch (e) {
    if (e.field) {
      setError(e.field, { type: 'server', message: e.message })
    } else {
      setError('root.serverError', { message: '系統忙碌，請稍後再試' }) // 表單層級錯誤
    }
  }
}
```

- **失敗重試 / 保留輸入**：送出失敗時**不要 `reset`**——保留使用者輸入與 dirty 狀態，只顯示錯誤讓他修改重送。樂觀更新若最終失敗，要能回滾 UI 但同樣保住表單內容。

## Q7. 表單無障礙（a11y）？

- **label 關聯**：每個 input 要有對應 `<label>`。兩種正確做法：`<label htmlFor="email">` + `<input id="email">`，或把 input 包在 `<label>` 裡。**placeholder 不能取代 label**——它是灰字提示、輸入後消失、對比度差、螢幕報讀器不一定唸，且無法作為永久欄位名稱。
- **`aria-invalid` 與 `aria-describedby`**：
  - `aria-invalid={!!error}`：告訴輔助科技「這欄目前無效」。
  - `aria-describedby="email-error"`：把**錯誤訊息元素的 `id`** 接到 input，報讀器聚焦該欄時會連錯誤訊息一起唸出來。

```tsx
<label htmlFor="email">Email</label>
<input
  id="email"
  {...register('email')}
  aria-invalid={!!errors.email}
  aria-describedby={errors.email ? 'email-error' : undefined}
/>
{errors.email && (
  <p id="email-error" role="alert">
    {errors.email.message}
  </p>
)}
```

- **錯誤容器的 role / aria-live**：動態出現的錯誤訊息用 `role="alert"`（等同 `aria-live="assertive"`）或 `aria-live="polite"`，讓報讀器在錯誤出現時主動朗讀，而不必等使用者移動焦點。表單層級錯誤（如「送出失敗」）尤其需要。
- **必填的表達**：除了視覺上的紅色星號，要在 input 加 `aria-required="true"`（或原生 `required`），並讓星號本身別只靠顏色傳達（可加 `aria-hidden` 的星號 + label 文字含「必填」，或用可讀說明），確保色盲與報讀器使用者都能得知。
