# Q57 參考解：從零到一交付 Mini E-commerce

> 這是一份**參考解**，不是唯一答案。面試時最有價值的是「拆解問題 → 分析利弊 → 做出決策 → 說出取捨」的過程。以下每個決定都會標明**放棄了什麼**，因為「只講優點」不叫取捨。核心心法：**在速度、可擴展性、工程成本之間做平衡**——一人 4 週的專案，預設偏向「速度」，但要在關鍵處預留擴充。

---

## 開場心法：先問，再做

拿到需求第一件事不是開 IDE，而是**釐清假設**。工程的成本主要來自「做錯方向後重做」，不是打字慢。因此先列出會影響架構的關鍵問題（見 §1.3），確認後再定案。

---

## 1. 需求拆解與範圍界定

### 1.1 功能拆解與 MVP 劃分

把功能拆到「可獨立驗收」的顆粒度，並用**價值 / 成本**兩軸決定進 MVP 與否。原則：**MVP = 讓使用者能「逛 → 選 → 買」的最短閉環**，其餘都往後放。

| 功能 | 顆粒 | MVP？ | 理由 |
|------|------|-------|------|
| 商品列表 + 卡片 | 必要 | ✅ MVP | 沒有列表就無法逛，核心閉環起點 |
| 分類篩選 / 排序 | 增強 | ✅ MVP（基本款） | 電商基本體驗；但進階多條件篩選延後 |
| 關鍵字搜尋 | 增強 | 🔶 v2 | 若後端搜尋 API 未就緒，先用前端簡單過濾，完整搜尋延後 |
| 分頁 / 無限捲動 | 必要 | ✅ MVP（先分頁） | 分頁比無限捲動好做且對 SEO/可及性友善，無限捲動列 v2 |
| 商品詳情 | 必要 | ✅ MVP | 購買決策頁 |
| 多圖切換 / 相關商品 | 增強 | 🔶 v2 | 詳情先單圖 + 描述即可下單；相關商品需推薦資料，延後 |
| 購物車（增刪改 + 持久化） | 必要 | ✅ MVP | 閉環核心 |
| 結帳表單 + 驗證 | 必要 | ✅ MVP | 沒有結帳就不是電商 |
| 金流串接 | — | ❌ 延後 | 題目允許 mock；真金流是獨立大工程，v2 再談 |
| 會員 / 登入 / 收藏 / 優惠券 | — | ❌ v2+ | 訪客即可完成購買，登入不阻擋核心閉環 |
| i18n 多語系 | — | 🔶 預留 | 先做繁中，但文案抽離成 key，不寫死在 JSX |

### 1.2 我砍掉 / 延後了什麼，為什麼

- **砍「登入 / 會員」**：訪客結帳（guest checkout）就能完成購買閉環，登入是留存功能不是購買功能，4 週要先保住「能賣東西」。
- **延「完整搜尋 / 推薦」**：這兩者高度依賴後端能力，前端先做退化版（前端過濾、隱藏推薦區），不讓自己被後端進度綁死。
- **金流用 mock**：真串金流涉及安全、對帳、回調，風險高、不確定性大，放進 4 週會壓垮其他功能。

**取捨的本質**：時間固定時，範圍是唯一能調的變數。我選擇**縮小範圍、保住品質**，而不是全做但每個都半成品。

### 1.3 我會向 PM / 後端釐清的問題

這些答案會**直接改變架構決定**，所以要先問：

- **SEO 要求多強？** 商品頁要進 Google 嗎 → 決定 CSR vs SSR/SSG（最關鍵的一題）。
- **API 分頁是 offset 還是 cursor？** 影響列表分頁 / 無限捲動實作與快取 key。
- **庫存即時性？** 允許有延遲，還是加入購物車 / 結帳要即時鎖庫存 → 影響是否需要輪詢 / 樂觀更新。
- **是否一定要訪客購物？** 若強制登入，購物車就綁後端而非只存本地。
- **商品數量級？** 幾百還是幾十萬 → 影響是否需要虛擬化、搜尋是否得走後端。
- **設計稿完整度 / 是否有設計系統？** 影響選「元件庫（MUI）」還是「自刻 + Tailwind」。

## 2. 技術選型與取捨（決策表）

> 貫穿原則：**一人 4 週，能用成熟方案就不自己造輪子**；把工程成本花在「業務邏輯」而非「基礎設施」上。

### 2.1 框架 / 渲染策略

| 選項 | 取捨 | 決定 |
|------|------|------|
| Vite + React（純 CSR） | 起步最快、心智負擔低；但**SEO 差**、首屏白畫面、社群分享無 meta | ❌ |
| **Next.js（App Router）** | SSR/SSG/ISR 皆可、內建圖片優化 / 路由 / metadata；學習與部署成本略高 | ✅ **採用** |
| Remix / Astro | 也很適合電商；但團隊熟悉度與生態考量下選更主流的 Next.js | ❌ |

**理由**：電商靠**自然搜尋流量**，商品頁必須被索引，這一條就足以否決純 CSR。用 Next.js：**商品列表 / 詳情用 SSR 或 ISR**（SEO + 首屏快），**購物車 / 結帳用 CSR**（互動為主、不需索引）。ISR 讓商品頁能靜態快取又能定期更新，兼顧效能與新鮮度。
**放棄的**：純 CSR 的極簡起步；換來 SEO 與首屏，值得。

### 2.2 狀態管理（關鍵：Server state 與 Client state 要分開）

這是最容易做錯的地方。**把「伺服器資料」和「使用者端狀態」當成兩種東西管理。**

| 狀態種類 | 例子 | 方案 | 理由 |
|----------|------|------|------|
| Server state | 商品列表、詳情、庫存 | **React Query（TanStack Query）** | 內建快取、去重、重試、背景更新、loading/error 狀態，省下大量樣板 |
| Client state | **購物車**、UI 開關 | **Zustand**（輕量）+ persist | 購物車是純前端狀態、需持久化；Zustand API 簡單、無 boilerplate |
| Server cache 也能兼 | — | 不用 Redux | Redux 對這規模是過度工程，樣板多 |

**理由**：很多人把 API 資料塞進 Redux/Context 再手寫 loading/error/快取，這是重複造 React Query 已解決的問題。購物車則是道地的 client state，用 Zustand + `persist` middleware 存 localStorage，一次解決「跨頁保存 + 重新整理不遺失」。
**放棄的**：Redux 的生態與 devtools 強大，但這規模不需要，換取更低的工程成本。

### 2.3 資料抓取

- 用 **React Query** 統一處理：`useQuery` 抓列表 / 詳情，`useMutation` 做加入購物車 / 送出訂單。
- **loading / error / empty** 三態用統一的 `<AsyncBoundary>` 或 render helper 包裝，避免每頁各寫一套。
- **快取**：列表用 `staleTime` 減少重抓；分頁用 `keepPreviousData` 避免翻頁閃爍。
- **樂觀更新**：加入購物車立即反映在 UI，失敗再回滾（購物車主要在本地，這點成本低、體驗好）。
- **重試**：查詢預設重試，變更（下單）**不自動重試**避免重複下單。

### 2.4 樣式方案

| 選項 | 取捨 | 決定 |
|------|------|------|
| **MUI（元件庫）** | 元件齊全、a11y 有基礎、交付快；客製化重設計時較綁手、bundle 較大 | ✅ 若無完整設計稿、要衝速度 |
| Tailwind CSS | 開發快、體積可控、彈性高；需自己拼元件（按鈕 / 對話框） | ✅ 備選（有設計系統時更好） |
| CSS Modules | 零依賴、scoped；但要自己刻所有元件 | 🔶 小專案可 |
| CSS-in-JS（Emotion） | DX 好；有 runtime 成本、與 RSC 相容性要注意 | ❌ 非首選 |

**決定**：一人衝刺、設計稿不完整 → **MUI**，用它的表單 / 對話框 / RWD 斷點 / a11y 基礎快速交付；主題（theme）集中管理色彩與間距，預留設計系統對齊。
**放棄的**：極致客製化與最小 bundle，換取交付速度。

### 2.5 表單

- **React Hook Form + Zod**。RHF 非受控、效能好、樣板少；Zod 定義 schema **同時**得到執行期驗證與 TypeScript 型別（一份 schema 兩用），結帳表單驗證直接受惠。
- **放棄**：手刻 useState 表單，狀態一多就難維護且易漏驗證。

### 2.6 測試策略（一人專案下，把錢花在刀口上）

投資報酬率排序：**E2E 關鍵路徑 > 核心邏輯單元測試 > 元件測試 > 覆蓋率數字**。

| 層級 | 測什麼 | 工具 |
|------|--------|------|
| 單元 | 購物車計算（小計 / 運費 / 總計）、格式化、Zod schema | Vitest |
| 元件 | 關鍵可重用元件（AsyncBoundary、Cart item）行為 | Testing Library |
| E2E | **一條主流程**：逛列表 → 進詳情 → 加購物車 → 結帳成功 | Playwright |

**理由**：一人時間有限，先用**一條 E2E** 守住「錢流得動」的核心閉環，回歸時最安心；購物車金額計算是最容易出錯又最不能錯的地方，一定要單元測試。**不追高覆蓋率數字**——那是虛榮指標。

### 2.7 部署 / CI

- **部署 Vercel**：與 Next.js 原生整合，PR 自動 preview deployment，方便給 PM 看。
- **CI（GitHub Actions）**：PR 觸發 `lint + typecheck + unit test + build`；主線 merge 後自動部署。
- **放棄**：自架 Docker / K8s——對一人專案是過度工程。

## 3. 架構與資料夾結構

### 3.1 資料夾結構（by feature，功能導向分層）

採 **feature-based**：相關的元件 / hook / API / 型別放一起，比 by-type（把所有 components 堆一資料夾）更好定位與擴充。

```
src/
├─ app/                      # Next.js App Router
│  ├─ (shop)/
│  │  ├─ page.tsx            # 首頁 / 商品列表（SSR/ISR）
│  │  ├─ products/[id]/      # 商品詳情（SSR/ISR + generateMetadata 做 SEO）
│  │  ├─ cart/               # 購物車（CSR）
│  │  └─ checkout/           # 結帳（CSR）
│  └─ layout.tsx             # 全域 Layout、Providers
├─ features/                 # 業務功能（核心）
│  ├─ products/
│  │  ├─ api.ts              # 商品 API 呼叫
│  │  ├─ queries.ts          # useProducts / useProduct（React Query）
│  │  ├─ components/         # ProductCard、ProductList、FilterBar
│  │  └─ types.ts
│  ├─ cart/
│  │  ├─ store.ts            # Zustand + persist（購物車核心）
│  │  ├─ selectors.ts        # 小計 / 運費 / 總計計算（可單元測試）
│  │  └─ components/         # CartItem、CartSummary
│  └─ checkout/
│     ├─ schema.ts           # Zod schema（驗證 + 型別）
│     └─ components/         # CheckoutForm、OrderSummary
├─ components/               # 跨功能可重用 UI（Button、AsyncBoundary、EmptyState）
├─ lib/                      # 底層工具（apiClient、queryClient、formatters）
├─ hooks/                    # 跨功能共用 hook
├─ styles/ theme.ts          # MUI theme / design tokens
└─ i18n/                     # 文案 key（預留多語系）
```

### 3.2 資料流

```
[後端 REST API]
      │  fetch (lib/apiClient)
      ▼
[React Query 快取]  ←─ 伺服器狀態：商品、庫存（去重 / 快取 / 重試）
      │  useQuery
      ▼
[頁面 / 容器元件] ──→ [可重用 UI 元件] ──→ 畫面

[使用者操作：加入購物車]
      │
      ▼
[Zustand cart store] ──persist──▶ localStorage（重新整理不遺失）
      │  selector（小計/運費/總計）
      ▼
[Cart / Checkout UI]
```

**兩條線分開**：讀商品走 React Query（伺服器真相），購物車走 Zustand（前端真相 + 本地持久化）。結帳送單時把購物車內容打包成 `useMutation` 送給後端。

### 3.3 元件切分（三類）

- **可重用 UI 元件**（`components/`）：`Button`、`AsyncBoundary`（統一 loading/error/empty）、`EmptyState`、`Price`、`QuantityStepper`。
- **功能 / 容器元件**（`features/*/components`）：`ProductList`、`ProductCard`、`CartSummary`、`CheckoutForm`——含業務邏輯，組合可重用元件。
- **頁面**（`app/`）：負責路由、資料抓取入口、渲染策略（SSR/ISR/CSR）與 SEO metadata。
- **共用 hook**：`useDebouncedSearch`、`useCartTotal`（包 selector）。

## 4. 里程碑切分（4 週）

原則：**每個里程碑都要有「跑得起來、看得到」的成果**，避免最後一週才整合的地獄。走「垂直切片」——每段都貫穿 API→狀態→UI，而非先做完所有 API 再做 UI。

| 里程碑 | 期程 | 交付（可驗收成果） |
|--------|------|-------------------|
| **M0 骨架** | 第 1 週前半 | Next.js 專案、CI、部署管線通、Providers（React Query/Theme）、apiClient、AsyncBoundary、mock server 就緒 → **能在 Vercel 看到空殼首頁** |
| **M1 逛** | 第 1 週後半~第 2 週 | 商品列表（SSR + SEO）、卡片、基本篩選/排序、分頁、圖片懶載入、RWD → **能逛商品** |
| **M2 選** | 第 3 週前半 | 商品詳情（SSR + metadata）、數量選擇、加入購物車、購物車增刪改 + 持久化 → **能把東西放進購物車** |
| **M3 買** | 第 3 週後半 | 結帳表單 + Zod 驗證、訂單摘要、送出（mock）、成功頁 → **完成購買閉環** |
| **M4 收尾** | 第 4 週 | E2E 主流程測試、a11y / Core Web Vitals 檢查與優化、錯誤/空狀態打磨、README、上線 |

**交付順序理由**：先把「骨架 + 部署」打通（M0），讓每次進度都能立刻 demo；接著沿使用者旅程「逛→選→買」逐段交付，任何一段做完都是可展示的價值；把測試與非功能優化收在最後但**不省略**，因為 a11y / 效能是驗收硬指標。

## 5. 風險與取捨

| 風險 | 影響 | 因應 |
|------|------|------|
| 後端 API 延遲交付 | 卡住前端開發 | **先訂 API 契約 + MSW mock**，前端照契約平行開發，後端好了再切換 |
| 需求膨脹（PM 一直加） | 4 週做不完 | 守住 MVP 範圍，新需求進 backlog、標 v2 |
| SEO 沒做好 | 電商沒流量 | Next.js SSR/ISR + `generateMetadata` + 結構化資料（JSON-LD） |
| 圖片拖慢首屏 / CLS | LCP、CLS 變差 | `next/image`（自動尺寸 / 懶載入 / 現代格式），固定圖片容器尺寸防位移 |
| 一人專案品質失控 | 上線後爆 bug | CI 擋 lint/type/test，E2E 守核心路徑 |

**技術債的誠實揭露**：搜尋用前端過濾（資料量大會撐不住，v2 改後端）、金流是 mock、無登入。這些都是**有意識的取捨**，並在 README 標記，而非假裝完美。

## 6. 驗收清單（Definition of Done）

**單一功能 PR 的 DoD**：
- [ ] 功能符合需求、手動走過 happy path 與主要錯誤情境
- [ ] TypeScript 無 error、ESLint 無 error
- [ ] 三態（loading / error / empty）都有處理
- [ ] RWD 三斷點檢查過
- [ ] 關鍵邏輯有測試（如金額計算）
- [ ] CI 綠燈、preview deployment 可點

**專案整體 DoD**：
- [ ] 功能：逛→選→買閉環完整可用（訪客可完成購買）
- [ ] 效能：LCP < 2.5s、CLS < 0.1、INP 良好（Lighthouse 行動版達標）
- [ ] a11y：鍵盤可完成整個購買流程、表單有 label、對比達 WCAG AA、focus 可見
- [ ] SEO：商品頁可被索引、有正確 title/description/OG、JSON-LD
- [ ] 品質：核心邏輯單元測試 + 一條 E2E 主流程綠燈
- [ ] 上線：部署成功、環境變數管理妥當、README 說明啟動 / 架構 / 已知技術債

## 7. 加分題參考方向

- **流量 10 倍**：靜態頁（ISR）由 CDN 扛、影響不大；瓶頸在「即時庫存 API」→ 加快取 / 降低輪詢頻率 / 用 SWR 背景更新。前端先確認沒有 N+1 請求、圖片有 CDN。
- **API 未就緒**：先定契約 + MSW / mock server，前端照假資料開發，切換只改 baseURL。
- **三個月後加會員 / 收藏 / 優惠券**：feature-based 結構好加 `features/auth`、`features/coupon`；購物車 store 已抽象，接後端同步只需在 store 加一層 sync；優惠券影響金額計算——幸好計算集中在 `cart/selectors.ts`，改一處即可。**預留點**：金額計算集中、文案抽 key、狀態分層清楚。**技術債**：訪客購物車與登入後購物車的合併邏輯屆時要補。
- **向 PM 解釋 Next.js**：「純 React 做出來的頁面，Google 看到的是空白，商品搜不到就沒自然流量；Next.js 讓伺服器先把商品內容產好給搜尋引擎和使用者，首頁也開得更快。多花的一點設定成本，換來『客人找得到我們的商品』，對電商是划算的投資。」（用**業務語言**講技術決定。）

## 總結：面試官在看什麼

這題沒有標準答案，評分看的是：**你能不能把一個模糊的大需求，拆成有優先序的小塊；每個技術決定都講得出取捨；規劃出每段都能 demo 的里程碑；並誠實面對風險與技術債。** 這正是「能獨立從無到有 ship 一個專案」的人的思考方式——不是會用最多工具的人，而是**會在限制下做出好權衡**的人。
