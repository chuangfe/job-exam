# Q57 從零到一：獨立交付一個完整前端專案（Mini E-commerce）

這是一題**專案規劃練習**，而不是單一手寫題。目標是模擬面試裡常見的「請你從無到有 ship 一個完整前端專案」的情境，訓練你在**沒有人幫你切好票、沒有現成架構**的情況下，如何**拆解問題、做技術取捨、規劃里程碑、定義完成標準**。

很多工程師會寫功能，但被問到「這個專案你會怎麼開？為什麼這樣選？MVP 先做什麼？」時就答得零散。這一題就是要你把整個思路走一遍——面試官真正想看的，不是你會用哪個框架，而是你**在速度、可擴展性、工程成本之間怎麼平衡**。

請以繁體中文（台灣用語）作答，並在需要處佐以 React + TypeScript 的思路。**不要只寫「我會用 React」——每個決定都要說明「為什麼」與「放棄了什麼」。**

---

## 1. 需求（產品情境）

你被指派**獨立**交付一個小型但完整的電商前端：**Mini E-commerce**。後端 API 由他人提供（假設是 REST，回傳 JSON），你只負責前端。上線期限抓 **4 週**，你一個人做。

### 主線功能（Functional Requirements）

1. **商品列表頁（Product List）**
   - 顯示商品卡片（圖片、名稱、價格、庫存狀態）。
   - 支援分類篩選（filter）、關鍵字搜尋（search）、排序（價格 / 上架時間）。
   - 分頁或無限捲動（infinite scroll）。
2. **商品詳情頁（Product Detail）**
   - 商品大圖、多圖切換、描述、規格、價格、庫存。
   - 選擇數量、加入購物車。
   - 「相關商品」推薦區塊。
3. **購物車（Cart）**
   - 加入 / 移除 / 修改數量。
   - 顯示小計、運費、總計。
   - 購物車內容需**跨頁面保存、重新整理後不遺失**（未登入也能用）。
4. **結帳流程（Checkout，簡化版）**
   - 填寫收件資訊表單（含驗證）。
   - 顯示訂單摘要 → 送出 → 成功頁。（金流可 mock，不需真的串。）

### 非功能需求（Non-Functional Requirements）

- **RWD**：手機 / 平板 / 桌機三斷點都要好用（電商流量以手機為主）。
- **a11y（可及性）**：鍵盤可操作、語意化標籤、表單有 label、對比足夠、focus 可見。
- **效能**：列表頁首屏快（圖片懶載入）、避免 CLS、互動流暢；關注 Core Web Vitals（LCP / CLS / INP）。
- **SEO**：商品列表與詳情頁需能被搜尋引擎索引（電商靠自然流量）→ 影響渲染策略選擇。
- **i18n（選配）**：先做繁中，但文案抽離、預留多語系擴充。
- **可維護性**：資料夾結構清楚、型別安全、狀態管理不打結、可測試。

> 若你想換題練習，也可用第二選項：**Admin 儀表板**（登入 + 資料表格 CRUD + 圖表 + 篩選匯出）。它的取捨重點會不同（SEO 不重要、但表格效能 / 權限 / 大量資料虛擬化更重要），可用同一套框架回答，體會「情境不同、取捨不同」。

## 2. 你要交付什麼（作答項目）

請針對上面的 Mini E-commerce，依序產出以下五個部分。**重點在思路與取捨，不是把 code 全部寫出來。**

### 2.1 需求拆解與範圍界定（MVP vs 後續）

- 把功能拆成明確的清單，並標記 **MVP（第一版一定要上）** vs **後續迭代（v2+）**。
- 說明你**砍掉 / 延後**了什麼，以及**為什麼**（時間有限，取捨的理由是什麼？）。
- 列出你會向 PM / 後端**釐清的問題**（哪些需求不明確、哪些會影響架構？例如：API 分頁是 offset 還是 cursor？庫存即時性要求？是否要登入？）。

### 2.2 技術選型與取捨（Technical Decisions）

針對以下面向各自**做出決定並說明理由**，重點是「為什麼選它、放棄了什麼替代方案」：

- **框架 / 渲染策略**：CSR（Vite + React）vs SSR/SSG（Next.js）——考量 SEO 與首屏。
- **狀態管理**：Server state（React Query / SWR）vs Client state（購物車）——兩者要分開嗎？購物車用 Context / Zustand / Redux？
- **資料抓取**：如何處理 loading / error / 快取 / 重試 / 樂觀更新？
- **樣式方案**：CSS Modules / Tailwind / CSS-in-JS / MUI 元件庫——速度 vs 客製化。
- **表單**：手刻 vs React Hook Form + 驗證（Zod / Yup）。
- **測試策略**：單元 / 元件 / E2E 各測什麼？一人專案下測試投資到哪裡划算？
- **部署 / CI**：部署到哪（Vercel / Netlify / 靜態主機）？CI 要跑什麼？

請用**決策表**呈現（選項 / 取捨 / 最終決定 / 理由）。

### 2.3 架構與資料夾結構設計

- 畫出（用文字樹狀）你的資料夾結構，並說明分層邏輯（by feature 還是 by type？）。
- 說明**資料流**：從 API → 快取 → 元件 → UI，購物車狀態怎麼流動與持久化。
- 列出核心**元件切分**（哪些是可重用 UI 元件、哪些是頁面 / 容器、哪些是共用 hook）。

### 2.4 里程碑切分與交付順序

- 把 4 週拆成里程碑（例如 M1~M4），每個里程碑**交付什麼可被驗收的成果**。
- 說明交付順序的理由（為何先做這個？如何盡早讓東西「跑得起來、看得到」？）。

### 2.5 驗收標準（Definition of Done）

- 定義「這個專案算完成」的清單：功能面、非功能面（效能 / a11y / RWD）、品質面（測試 / 型別 / lint）、上線面（部署 / 監控）。
- 一個功能 PR 要通過哪些檢查才算 Done？

## 3. 加分思考（Bonus）

- 如果**流量突然變 10 倍**，你的前端哪裡會先出問題？怎麼因應？
- 如果**後端 API 還沒好**，你如何不被卡住、平行開發？（mock 策略）
- 如果**三個月後要加「會員 / 收藏 / 優惠券」**，你目前的架構撐得住嗎？哪裡預留了擴充點、哪裡是技術債？
- 你如何向**非技術的 PM** 解釋「為什麼選 Next.js 而不是純 React」這個決定？

## 4. 如何自我檢核

寫完後，用下面問題檢查你的答案是否「像一個能獨立 ship 專案的人」：

- 每個技術決定都有講「**放棄了什麼**」嗎？（只講優點不算取捨）
- MVP 範圍是否真的能在時間內做完？有沒有偷偷把整包都當 MVP？
- 有沒有從**使用者情境**（電商以手機、靠 SEO 自然流量）反推技術決定？
- 里程碑是否**每一段都有可 demo 的成果**，而不是最後一週才整合？
- 有沒有誠實標出**風險與技術債**，而不是假裝完美？
