# Q48 - PWA / Service Worker / Push Notification

PWA（Progressive Web App，漸進式網頁應用）讓網頁能像原生 App 一樣被安裝到桌面、離線可用、還能收推播通知。而這一切的核心引擎就是 **Service Worker**——一支跑在背景、獨立於網頁主執行緒的腳本，能攔截網路請求、管理快取、接收推播。這題請你把 PWA 的三大支柱、Service Worker 生命週期、快取策略、Background Sync 與 Push Notification 完整串起來。

> 本題與 Q41（多層快取策略）互補：Q41 從整體視角比較各層快取，本題聚焦「在 Service Worker 裡怎麼實作」，請對照閱讀。

## Q1. PWA 是什麼、三大支柱與 Web App Manifest

- PWA 是什麼？它和一般網頁、和原生 App 的差異在哪？「Progressive（漸進式增強）」這個詞代表什麼精神？
- PWA 三大支柱：**可安裝（Installable）**、**離線可用（Offline / Reliable）**、**類原生體驗（App-like / Engaging）**，各自靠什麼技術達成？
- Web App Manifest（`manifest.json`）的用途是什麼？請列出常見欄位：`name` / `short_name`、`start_url`、`scope`、`display`、`theme_color` / `background_color`、`icons`。
- `display` 有哪些值（`browser` / `standalone` / `fullscreen` / `minimal-ui`）？差別是什麼？
- `icons` 為什麼建議提供多種尺寸？什麼是 maskable icon（`purpose: "maskable"`）？
- 瀏覽器安裝提示怎麼客製？請說明 `beforeinstallprompt` 事件的攔截與延後觸發流程，以及為什麼要自己做安裝按鈕。

## Q2. Service Worker 生命週期與更新流程

- Service Worker 的完整生命週期：`register` → `install` → `activate` → `fetch` / idle → `terminated`，各階段做什麼？
- `install` 階段通常做什麼事（預快取 App Shell）？`event.waitUntil()` 的作用是什麼？
- `activate` 階段通常做什麼事（清除舊版快取）？
- 為什麼新版 Service Worker 預設會進入「waiting」狀態而不是立刻接管？`skipWaiting()` 與 `clients.claim()` 各自解決什麼問題？各有什麼副作用要注意？
- 使用者「卡在舊版」的問題怎麼發生、怎麼解？請說明版本控制（快取名稱加版本號）與更新提示 UX（偵測到新版 SW 時提示使用者重新整理）的做法。

## Q3. 快取策略與離線 fallback

- 在 `fetch` 事件裡用 Cache API 攔截請求，常見策略：**cache-first**、**network-first**、**stale-while-revalidate**，分別適合什麼資源？（可與 Q41 對照）
- 請寫出攔截 `fetch` 事件、對靜態資源用 cache-first、對 API 用 network-first 的程式碼。
- 什麼是「離線 fallback 頁」？當使用者離線且快取也沒有時，如何回傳一個預先快取好的 `offline.html`？
- App Shell 模型是什麼？為什麼要把外殼（HTML/CSS/JS 框架）與內容分開快取？

## Q5. Push Notification（推播通知）

- 推播牽涉三方：**瀏覽器 / 使用者裝置**、**推播服務（Push Service，如 FCM/Mozilla）**、**你的應用伺服器**。請說明一次推播的完整流程。
- Push API 與 Notification API 的分工是什麼？
- 訂閱流程：`Notification.requestPermission()` 取得權限 → `pushManager.subscribe()` 取得 `PushSubscription` → 傳給後端儲存。請說明每一步。
- 什麼是 **VAPID**？`applicationServerKey` 是做什麼用的？後端推播時為什麼要簽章？
- Service Worker 裡怎麼接收推播並顯示通知（`push` 事件 → `showNotification`）？點擊通知怎麼處理（`notificationclick` 事件 → 開啟/聚焦分頁）？
- 權限請求的 UX 有哪些原則？為什麼不該一進站就跳權限請求？

## Q6. Service Worker 的限制

- 為什麼 Service Worker 只能在 **HTTPS**（或 `localhost`）下運作？
- 什麼是 **scope**？註冊路徑如何決定 SW 能攔截哪些請求？為什麼放在根目錄 `/sw.js` 很重要？
- 為什麼 Service Worker **不能直接存取 DOM**？它要跟頁面溝通得靠什麼（`postMessage` / Clients API）？
- Service Worker 的生命週期與一般 JS 有何不同（隨時可能被瀏覽器終止、事件驅動、無持久記憶體狀態）？這對寫 SW 有什麼影響？
