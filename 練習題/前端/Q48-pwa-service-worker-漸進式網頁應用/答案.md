# Q48 - PWA / Service Worker / Push Notification 解析

---

## Q1. PWA 是什麼、三大支柱與 Web App Manifest

### PWA 是什麼

**PWA（Progressive Web App）** 是用一般 Web 技術（HTML/CSS/JS）打造、但具備「原生 App 特質」的網頁應用：可以被安裝到主畫面、離線也能開、能收推播、有全螢幕沉浸體驗。它不是單一 API，而是一組能力的集合（Service Worker + Manifest + HTTPS）。

- **和一般網頁的差異**：一般網頁離線就白畫面、不能安裝、不能推播；PWA 透過 Service Worker 快取與 Manifest 補足這些。
- **和原生 App 的差異**：不用上架商店、不用下載安裝包、一份程式碼跨平台、可被搜尋引擎索引；但對系統 API 的存取（藍牙、深層檔案系統、背景常駐）仍不如原生。
- **「Progressive（漸進式增強）」的精神**：核心功能在任何瀏覽器都能用，進階能力（安裝、離線、推播）在支援的瀏覽器上「錦上添花」。不支援的環境要能優雅降級，而不是整個壞掉。

### 三大支柱

| 支柱 | 意義 | 靠什麼技術 |
|------|------|-----------|
| 可安裝（Installable） | 加到主畫面、桌面圖示、獨立視窗開啟 | Web App Manifest + `beforeinstallprompt` |
| 離線可用（Reliable） | 弱網/離線也能開、載入快 | Service Worker + Cache API |
| 類原生（Engaging） | 全螢幕、推播、背景同步 | Manifest `display` + Push API + Background Sync |

### Web App Manifest（`manifest.json`）

一份 JSON，描述 App 被安裝時的外觀與行為。用 `<link rel="manifest" href="/manifest.json">` 掛到 HTML。

```json
{
  "name": "求職追蹤器",
  "short_name": "求職追蹤",
  "description": "集中管理多家公司求職進度",
  "start_url": "/?source=pwa",
  "scope": "/",
  "display": "standalone",
  "orientation": "portrait",
  "theme_color": "#0d47a1",
  "background_color": "#ffffff",
  "icons": [
    { "src": "/icons/icon-192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "/icons/icon-512.png", "sizes": "512x512", "type": "image/png" },
    {
      "src": "/icons/icon-maskable-512.png",
      "sizes": "512x512",
      "type": "image/png",
      "purpose": "maskable"
    }
  ]
}
```

**常見欄位**：

- `name` / `short_name`：完整名稱 / 圖示下方的短名稱（空間有限時顯示）。
- `start_url`：從圖示啟動時的進入頁（常帶追蹤參數區分來源）。
- `scope`：App 的「勢力範圍」，超出 scope 的連結會用瀏覽器分頁開。
- `display`：見下表。
- `theme_color`：狀態列/工具列顏色；`background_color`：啟動畫面（splash）背景色。
- `icons`：不同尺寸的圖示。

**`display` 四種值**：

| 值 | 行為 |
|----|------|
| `browser` | 就是一般分頁，有完整網址列（幾乎沒安裝感） |
| `minimal-ui` | 獨立視窗，但保留最基本的導覽控制（如返回、重整） |
| `standalone` | 像原生 App，無網址列（最常用） |
| `fullscreen` | 完全全螢幕，連系統狀態列都藏（遊戲/多媒體） |

**icons 多尺寸與 maskable**：

- 提供多尺寸（192、512…）讓不同裝置/密度挑最合適的，避免糊掉。512 通常用於啟動畫面。
- **maskable icon**：Android 會把圖示裁成圓形/圓角/水滴等各種形狀。一般圖示可能被裁掉邊角或四周留白難看。maskable 圖示要求把重要內容放在中央「安全區」（約中心 80% 的圓內），四周留出可被裁切的 padding，`purpose: "maskable"` 告訴系統這張圖可以安全地被遮罩。

### 安裝提示：`beforeinstallprompt`

瀏覽器判斷符合安裝條件（有 manifest、有 SW、HTTPS…）時會觸發 `beforeinstallprompt`。預設有些瀏覽器會自己跳「安裝」小橫幅，但體驗與時機不受控。實務做法是**攔截它、`preventDefault()` 阻止預設提示、把事件存起來**，等使用者在你設計好的時機（例如用了一陣子後）點自訂按鈕，再呼叫 `prompt()`。

```js
let deferredPrompt = null;
const installBtn = document.querySelector('#install-btn');
installBtn.hidden = true;

window.addEventListener('beforeinstallprompt', (e) => {
  e.preventDefault();       // 阻止瀏覽器自動跳的預設提示
  deferredPrompt = e;       // 存起來，稍後自己觸發
  installBtn.hidden = false; // 顯示自訂的「安裝」按鈕
});

installBtn.addEventListener('click', async () => {
  if (!deferredPrompt) return;
  deferredPrompt.prompt();                        // 跳出安裝對話框
  const { outcome } = await deferredPrompt.userChoice;
  console.log(outcome === 'accepted' ? '使用者已安裝' : '使用者拒絕');
  deferredPrompt = null;    // 只能用一次，用完清掉
  installBtn.hidden = true;
});

// 安裝完成後可埋點
window.addEventListener('appinstalled', () => {
  console.log('PWA 已安裝');
  deferredPrompt = null;
});
```

> 注意：`prompt()` 必須由使用者手勢（點擊）觸發；`deferredPrompt` 只能用一次。iOS Safari 不支援 `beforeinstallprompt`，需引導使用者手動「加入主畫面」。

## Q2. Service Worker 生命週期與更新流程

### 生命週期

```
頁面 register()
      │
      ▼
   install  ── event.waitUntil(預快取 App Shell)
      │  （成功後）
      ▼
   waiting  ── 若已有舊 SW 在控制頁面，新 SW 在此等待
      │  （舊 SW 的所有分頁關閉，或呼叫 skipWaiting）
      ▼
  activate  ── event.waitUntil(清除舊版快取)
      │
      ▼
  activated ── 開始攔截 fetch；閒置時被瀏覽器 terminated，
               有事件（fetch/push/sync）時再喚醒
```

- **register**：在頁面主執行緒呼叫 `navigator.serviceWorker.register('/sw.js')`。這只是「登記」，瀏覽器會在背景下載並比對 SW 檔案。
- **install**：SW 第一次安裝或檔案有更新時觸發。典型工作是**預快取 App Shell**（外殼 HTML、CSS、JS、離線頁）。用 `event.waitUntil(promise)` 告訴瀏覽器「等我這個 Promise 完成再算安裝成功」，否則快取還沒存完 SW 就被當成裝好了。
- **waiting**：install 成功後，若目前已有一個舊 SW 正在控制頁面，新 SW 不會馬上接管，而是進入 waiting，避免新舊邏輯在同一批頁面混用。
- **activate**：新 SW 開始接管時觸發。典型工作是**清除舊版快取**。同樣用 `event.waitUntil()` 確保清完。
- **idle / terminated**：SW 是事件驅動的，閒置會被瀏覽器終止以省資源；有 `fetch`/`push`/`sync` 事件時再喚醒。**因此不能依賴 SW 的全域變數長期保存狀態**。

### install / activate 範例

```js
// sw.js
const CACHE_VERSION = 'v3';
const CACHE_NAME = `app-shell-${CACHE_VERSION}`;
const APP_SHELL = [
  '/',
  '/index.html',
  '/styles.css',
  '/app.js',
  '/offline.html',
  '/icons/icon-192.png',
];

self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) => cache.addAll(APP_SHELL))
  );
});

self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then((keys) =>
      Promise.all(
        keys
          .filter((key) => key.startsWith('app-shell-') && key !== CACHE_NAME)
          .map((key) => caches.delete(key)) // 刪掉舊版本快取
      )
    )
  );
});
```

### skipWaiting 與 clients.claim

- **`self.skipWaiting()`**：讓新 SW **不等 waiting，安裝完立刻進入 activate**。解決「使用者要關掉所有分頁才會更新」的痛點。副作用：頁面可能同時被舊 SW 載入、之後被新 SW 接管，若新舊快取結構不相容可能出錯，因此常搭配「提示使用者重新整理」使用，而不是無聲切換。
- **`self.clients.claim()`**：讓 activate 後的 SW **立刻控制當前已開啟、原本沒被任何 SW 控制的頁面**（例如首次安裝那一次載入的頁面）。沒有它，首次安裝的那個分頁要重新整理才會被 SW 接管。

```js
self.addEventListener('install', () => self.skipWaiting());
self.addEventListener('activate', (event) => {
  event.waitUntil(self.clients.claim());
});
```

### 「卡在舊版」與更新 UX

問題成因：SW 與快取會讓瀏覽器優先吃本地快取，若沒有版本控制與更新提示，使用者可能一直看到舊版。

正解兩件事：

1. **版本控制**：快取名稱帶版本號（如 `app-shell-v3`），發版時改版本號，`activate` 時刪掉舊的。靜態資源檔名帶 hash（`app.abc123.js`）+ `immutable`，內容一變檔名就變，天然避開快取。
2. **更新提示 UX**：偵測到新 SW 進入 waiting 時，跳一個「有新版本，點此更新」的提示，使用者點了才 `postMessage` 叫 SW `skipWaiting()`，再 `reload`。這樣使用者才不會在操作到一半被無預警換版。

```js
// 頁面端：偵測新版
navigator.serviceWorker.register('/sw.js').then((reg) => {
  reg.addEventListener('updatefound', () => {
    const newSW = reg.installing;
    newSW.addEventListener('statechange', () => {
      // 已裝好且目前有舊 SW 在控制 → 代表是「更新」而非首次安裝
      if (newSW.state === 'installed' && navigator.serviceWorker.controller) {
        showUpdateToast(() => {
          newSW.postMessage({ type: 'SKIP_WAITING' });
        });
      }
    });
  });
});

// 控制權換手後重新整理一次，載入新版資源
let refreshing = false;
navigator.serviceWorker.addEventListener('controllerchange', () => {
  if (refreshing) return;
  refreshing = true;
  window.location.reload();
});

// sw.js：收到訊息才 skipWaiting
self.addEventListener('message', (event) => {
  if (event.data?.type === 'SKIP_WAITING') self.skipWaiting();
});
```

## Q3. 快取策略與離線 fallback

### 三種策略速記（詳細比較見 Q41）

| 策略 | 流程 | 適合 |
|------|------|------|
| **cache-first** | 先查快取，命中就回；沒有才打網路並存起來 | 不常變動的靜態資源（JS/CSS/字型/圖片） |
| **network-first** | 先打網路，成功回並更新快取；失敗才回快取 | 要即時、但也想要離線 fallback 的 API/HTML |
| **stale-while-revalidate** | 立刻回快取（快），同時背景更新快取供下次用 | 可容忍短暫舊資料、又想要快的內容（頭像、列表） |

### fetch 攔截：靜態 cache-first + API network-first

```js
// sw.js
const STATIC_CACHE = 'app-shell-v3';
const RUNTIME_CACHE = 'runtime-v3';

self.addEventListener('fetch', (event) => {
  const { request } = event;

  // 只處理 GET，其他（POST 等）交給網路
  if (request.method !== 'GET') return;

  const url = new URL(request.url);

  // 1) API 請求 → network-first
  if (url.pathname.startsWith('/api/')) {
    event.respondWith(networkFirst(request));
    return;
  }

  // 2) 導覽請求（HTML）→ network-first，離線退回 offline.html
  if (request.mode === 'navigate') {
    event.respondWith(
      networkFirst(request).catch(() => caches.match('/offline.html'))
    );
    return;
  }

  // 3) 其餘靜態資源 → cache-first
  event.respondWith(cacheFirst(request));
});

async function cacheFirst(request) {
  const cached = await caches.match(request);
  if (cached) return cached;
  const response = await fetch(request);
  const cache = await caches.open(STATIC_CACHE);
  cache.put(request, response.clone()); // Response 是串流，只能讀一次，先 clone
  return response;
}

async function networkFirst(request) {
  const cache = await caches.open(RUNTIME_CACHE);
  try {
    const response = await fetch(request);
    cache.put(request, response.clone());
    return response;
  } catch (err) {
    const cached = await cache.match(request);
    if (cached) return cached;
    throw err; // 網路掛了又沒快取 → 交給上層做 fallback
  }
}
```

> 重點細節：`Response` body 是一次性串流，要「回給頁面」又要「存進快取」就得 `response.clone()`。

### 離線 fallback 頁

`offline.html` 在 `install` 時就預快取好（見 Q2）。當導覽請求離線、快取又沒有對應頁時，回傳這張預備頁，避免瀏覽器的「無網路連線」恐龍畫面。也可為圖片準備占位圖 fallback。

### App Shell 模型

把「外殼」（頁面骨架 HTML、CSS、JS 框架、導覽列）與「內容」（動態資料）分開：

- **外殼**用 cache-first 預快取 → 第二次開啟秒開、離線也能顯示框架。
- **內容**用 network-first / SWR → 保持新鮮。

這樣既有原生 App 般的即時外框，又能載入最新資料，是 PWA 感覺「快」的關鍵。

## Q4. Background Sync（背景同步）

### 解決的問題

使用者在**離線或弱網**下做了「送出」動作（送表單、發留言、按讚），一般 fetch 會直接失敗。Background Sync 讓你把這個動作「排隊」，等瀏覽器偵測到**網路恢復**時，由 Service Worker 在背景自動重送——即使使用者已經關掉分頁也能完成。

### 一次性同步流程

```js
// 頁面端：送出失敗時登記一次背景同步
async function submitComment(data) {
  try {
    await fetch('/api/comments', { method: 'POST', body: JSON.stringify(data) });
  } catch (err) {
    await saveToIndexedDB('pending-comments', data); // 先存本地佇列
    const reg = await navigator.serviceWorker.ready;
    await reg.sync.register('sync-comments');         // 登記 tag
  }
}

// sw.js：網路恢復時觸發
self.addEventListener('sync', (event) => {
  if (event.tag === 'sync-comments') {
    event.waitUntil(flushPendingComments()); // 讀 IndexedDB 佇列逐筆重送
  }
});
```

### 和「SW 裡自己重試 fetch」的差別

- 自己重試：只有在 SW 還活著、頁面還開著時才可能跑，網路一直不好就一直失敗，且 SW 閒置會被終止。
- Background Sync：**由瀏覽器保證**在網路恢復時喚醒 SW 執行，並在失敗時自動再排程重試，跨分頁關閉也有效——可靠性來自作業系統/瀏覽器層級的排程，而不是你的頁面。

> 支援度：Background Sync 主要在 Chromium 系；Safari 不支援，需以「頁面重連時手動 flush 佇列」降級。

## Q5. Push Notification（推播通知）

### 三方與完整流程

```
① 你的網頁 → 向瀏覽器要通知權限、subscribe → 拿到 PushSubscription
② PushSubscription（含 endpoint）→ 傳給「你的應用伺服器」儲存
③ 事件發生時：你的伺服器 → 用 VAPID 簽章，把加密訊息 POST 到
   PushSubscription.endpoint（那是「推播服務」，如 FCM / Mozilla autopush）
④ 推播服務 → 把訊息送到使用者裝置，喚醒 Service Worker 的 push 事件
⑤ Service Worker → self.registration.showNotification(...) 顯示通知
⑥ 使用者點擊 → notificationclick 事件 → 開啟/聚焦分頁
```

### Push API vs Notification API 的分工

- **Push API**：負責「從伺服器**接收**訊息」的機制（訂閱、endpoint、push 事件）。屬於 Service Worker。
- **Notification API**：負責「**顯示**一則系統通知」（`showNotification` / `Notification`）。可獨立使用（例如純本地提醒），也被 push 事件呼叫。
- 兩者組合：Push 把訊息送進來，Notification 把它變成螢幕上的通知。

### 訂閱流程（頁面端）

```js
async function subscribeToPush() {
  // 1) 要權限（必須由使用者手勢觸發）
  const permission = await Notification.requestPermission();
  if (permission !== 'granted') return;

  const reg = await navigator.serviceWorker.ready;

  // 2) 用 VAPID 公鑰訂閱，拿到 PushSubscription
  const subscription = await reg.pushManager.subscribe({
    userVisibleOnly: true, // 承諾每次推播都會顯示通知（不做「靜默推播」）
    applicationServerKey: urlBase64ToUint8Array(VAPID_PUBLIC_KEY),
  });

  // 3) 把訂閱資訊送到後端儲存
  await fetch('/api/push/subscribe', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(subscription),
  });
}

// VAPID 公鑰是 base64url 字串，subscribe 需要 Uint8Array
function urlBase64ToUint8Array(base64) {
  const padding = '='.repeat((4 - (base64.length % 4)) % 4);
  const b64 = (base64 + padding).replace(/-/g, '+').replace(/_/g, '/');
  const raw = atob(b64);
  return Uint8Array.from([...raw].map((c) => c.charCodeAt(0)));
}
```

`PushSubscription` 大致長這樣（後端用它來推播）：

```json
{
  "endpoint": "https://fcm.googleapis.com/fcm/send/xxxxx",
  "keys": {
    "p256dh": "使用者的公鑰，用於加密訊息內容",
    "auth": "驗證密鑰"
  }
}
```

### VAPID 是什麼

**VAPID（Voluntary Application Server Identification）** 是「應用伺服器向推播服務證明自己身分」的機制。你產生一對金鑰：

- **公鑰**：放前端，`subscribe` 時當 `applicationServerKey`。
- **私鑰**：放後端保密，推播時用它對請求做 JWT 簽章。

推播服務（FCM 等）收到請求時，用公鑰驗簽，確認「這則推播真的來自當初這個 subscription 綁定的伺服器」，防止別人拿到你的 endpoint 就亂發。此外，訊息內容還會用 subscription 的 `p256dh`/`auth` 做端到端加密，連推播服務也讀不到明文。

### 後端推播（Node，用 `web-push`）

```js
const webpush = require('web-push');

webpush.setVapidDetails(
  'mailto:you@example.com',
  process.env.VAPID_PUBLIC_KEY,
  process.env.VAPID_PRIVATE_KEY
);

// subscription 是先前存下來的那份
async function sendPush(subscription, payload) {
  await webpush.sendNotification(
    subscription,
    JSON.stringify({ title: '新職缺', body: '台積電開了前端缺！', url: '/jobs/1' })
  );
}
```

### Service Worker 端：接收與點擊

```js
// sw.js
self.addEventListener('push', (event) => {
  const data = event.data ? event.data.json() : {};
  event.waitUntil(
    self.registration.showNotification(data.title || '通知', {
      body: data.body,
      icon: '/icons/icon-192.png',
      badge: '/icons/badge-72.png',
      data: { url: data.url || '/' }, // 帶給 click 用
    })
  );
});

self.addEventListener('notificationclick', (event) => {
  event.notification.close();
  const targetUrl = event.notification.data.url;
  event.waitUntil(
    clients.matchAll({ type: 'window', includeUncontrolled: true }).then((wins) => {
      // 若已有分頁開著就聚焦，否則開新分頁
      const existing = wins.find((w) => w.url.includes(targetUrl));
      if (existing) return existing.focus();
      return clients.openWindow(targetUrl);
    })
  );
});
```

### 權限 UX 原則

- **不要一進站就跳權限請求**：使用者還不了解價值就被彈窗，多半直接拒絕，而且瀏覽器層級的拒絕很難挽回（之後要手動去設定改）。
- **先做「軟提問（soft ask）」**：用自訂的頁面 UI 說明「開啟通知能收到職缺更新」，使用者點「好」再觸發真正的 `requestPermission()`。
- **有明確價值、對的時機才問**（例如剛追蹤一家公司後）。
- 尊重拒絕，提供關閉/管理入口，避免反覆騷擾。

## Q6. Service Worker 的限制

- **必須 HTTPS（或 localhost）**：SW 能攔截、竄改所有網路請求，威力極大。若允許在 HTTP 下運作，中間人（MITM）就能植入惡意 SW 長期挾持站台。`localhost` 因為是本機開發環境，例外放行。
- **scope（作用範圍）**：SW 只能控制其**註冊路徑所在目錄及其子路徑**。`/sw.js` → scope 為 `/`（整站）；`/app/sw.js` → 只能控制 `/app/...`。所以要控制整站就要把 SW 放在**根目錄**。（可用回應標頭 `Service-Worker-Allowed` 放寬，或 register 時指定較窄的 `scope`。）
- **不能存取 DOM**：SW 跑在**獨立於頁面的 worker 執行緒**，沒有 `window`、`document`。這是刻意設計——它可能同時服務多個分頁、也可能在沒有任何頁面開啟時被喚醒（推播/同步），綁死某個 DOM 沒有意義。要操作畫面得透過 `postMessage` 與 Clients API 跟頁面溝通，由頁面改 DOM。
- **事件驅動、無持久狀態**：SW 閒置會被瀏覽器**隨時終止**，下次事件再重啟，全域變數不保證存活。需要持久化就用 **IndexedDB / Cache API**，別把狀態放記憶體。所有非同步工作要用 `event.waitUntil()` 包住，否則 SW 可能在你做完前就被殺掉。

## 面試速記

- PWA = Manifest（可安裝）+ Service Worker（離線）+ HTTPS，三大支柱：可安裝 / 離線 / 類原生。
- SW 生命週期：register → install（預快取，`waitUntil`）→ waiting →（`skipWaiting`）activate（清舊快取，`clients.claim`）→ fetch/idle。
- 更新別無聲切換，偵測到新版提示使用者 reload；快取名稱帶版本號、靜態資源 hash 檔名 + `immutable`。
- 快取策略：靜態 cache-first、API/HTML network-first、可容忍舊資料用 SWR；備 `offline.html` 做離線 fallback。
- Push 三方：網頁訂閱（`pushManager.subscribe` + VAPID 公鑰）→ 後端存 subscription → 後端用 VAPID 私鑰簽章推到 endpoint → SW `push` 事件 `showNotification`。
- SW 限制：HTTPS、受 scope 約束、無 DOM（靠 postMessage）、事件驅動無持久記憶體。
