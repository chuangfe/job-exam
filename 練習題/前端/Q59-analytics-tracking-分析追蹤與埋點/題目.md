# 分析追蹤與事件埋點（Analytics / Tracking）

請依序回答以下前端分析追蹤的核心問題。「埋點」是把使用者在網站上的行為變成**可分析的資料**——從最基本的頁面瀏覽，到按鈕點擊、加入購物車、完成結帳。做得好，產品與行銷團隊才能靠數據決策；做不好，資料髒亂、指標對不上、甚至違反隱私法規。前端工程師是埋點的第一線實作者，必須懂得「送什麼、怎麼送、送到哪、能不能送」。

建議先自己用白話講出來，再對照解析。以 **GA4 + Google Tag Manager（GTM）+ React** 為主要示範環境。

> 提醒：本題與 **Q45 網頁可及性**、**Q49 SEO** 一樣屬於「前端周邊但很常被面試問到」的實務題，建議一起服用。

## Q1. 為何要埋點？常見指標有哪些？

- 為什麼要花力氣做分析追蹤？埋點的資料最終服務於哪些決策（產品、行銷、營運）？
- 請解釋以下常見指標各自的意思：**PV（Page View）**、**UV（Unique Visitor）**、**Session（工作階段）**。UV 跟 PV 有什麼差別？
- 什麼是**轉換（Conversion）**？什麼是**轉換率**？舉一個電商的例子。
- 什麼是**留存（Retention）**？為什麼「留存」常被視為比「拉新」更重要的指標？
- 什麼是**漏斗（Funnel）**分析？以「瀏覽商品 → 加入購物車 → 結帳 → 付款成功」為例，漏斗能告訴你什麼？

## Q2. GA4 與 GTM：差異與分工

- GA4（Google Analytics 4）是什麼？GTM（Google Tag Manager）又是什麼？兩者是「二選一」還是「搭配使用」？各自負責什麼？
- 為什麼很多團隊選擇「透過 GTM 佈署 GA4」而不是把 GA4 的程式碼直接寫死在網站裡？（提示：不必每次改追蹤都動到程式碼、發版）
- GA4 相較於舊版 Universal Analytics（UA）最大的觀念轉變是什麼？（提示：從「工作階段 / 頁面」為中心，改為**以事件（event）為中心**的資料模型）
- GTM 裡的三個核心概念 **Tag（代碼）**、**Trigger（觸發條件）**、**Variable（變數）** 各是什麼？請用「當使用者點擊購買按鈕時，送出一個 GA4 事件」串起來說明。

## Q3. `dataLayer` 的角色與事件驅動模型

- `dataLayer` 是什麼？它在網站與 GTM 之間扮演什麼角色？（提示：一個中介的資料緩衝區 / 傳話筒）
- 為什麼說 `dataLayer` 讓「前端工程師」與「數據 / 行銷團隊」得以**解耦**？
- 什麼是「事件驅動（event-driven）」模型？前端 `window.dataLayer.push({ event: '...' })` 之後，GTM 端會發生什麼事？
- 為什麼實務上建議「前端只負責 push 乾淨的資料到 `dataLayer`，由 GTM 決定要送到哪些平台」？

## Q4. 事件設計：page_view、自訂事件、電商事件

- GA4 的事件大致可分成哪幾類？（提示：自動收集、加強型評估、推薦事件、自訂事件）
- `page_view` 事件的意義是什麼？在傳統多頁式網站（MPA）它通常自動送出，那 SPA 為什麼是個問題？（Q5 會深入）
- 什麼是**自訂事件（custom event）**？請設計一個「使用者點擊訂閱電子報按鈕」的事件，包含事件名稱與參數。
- GA4 的**電商（e-commerce）事件**有一套推薦規格，請說明 `view_item`、`add_to_cart`、`begin_checkout`、`purchase` 各自代表什麼、大概帶哪些參數（如 `items`、`value`、`currency`、`transaction_id`）。
- **事件命名慣例**該怎麼訂？（提示：`snake_case`、動詞_名詞、大小寫一致、避免臨時起名）命名混亂會造成什麼後果？
- 什麼是 **Tracking Plan（追蹤計畫文件）**？它通常記錄哪些欄位？為什麼團隊一定要有一份、而且要當成「單一事實來源」維護？

## Q5. SPA（單頁應用）的追蹤

- 為什麼 React / Vue 這類 SPA 的頁面切換，GA4 預設抓不到後續的 `page_view`？（提示：沒有真正的整頁重新載入，`gtag` 的自動 page_view 只在第一次載入觸發）
- 正確做法是什麼？（提示：監聽路由變更，在每次換頁時**手動**送出 `page_view`）
- 在 React Router 裡，你會用哪個 hook 來偵測路由變更並送出頁面瀏覽事件？
- 為什麼建議把追蹤邏輯**封裝成一個 `trackEvent` 函式 / 自訂 hook**，而不是在每個元件裡直接呼叫 `window.dataLayer.push`？列出至少三個好處。

## Q6. 隱私與合規

- 什麼是 **PII（個人可識別資訊）**？請舉例。為什麼**絕對不可以**把 PII（如 email、姓名、身分證字號、完整地址）送進 GA4？
- 什麼是 **Cookie 同意（consent）**？GDPR（歐盟）與 PDPA（台灣個資法）對「在取得同意前就追蹤使用者」有什麼要求？
- 什麼是 GA4 / GTM 的 **Consent Mode（同意模式）**？「使用者尚未同意 cookie」時，Consent Mode 會怎麼調整追蹤行為？（提示：不寫 cookie、送出無 cookie 的模型化訊號）
- **First-party cookie** 與 **third-party cookie** 有什麼差別？為什麼瀏覽器逐步封鎖 third-party cookie 對廣告追蹤衝擊很大？
- 什麼是**伺服器端追蹤（server-side tagging）**？它相較純瀏覽器端追蹤有哪些好處（資料品質、隱私控制、避免被廣告攔截器擋掉）？

## Q7. 除錯與驗證

- GA4 的 **DebugView** 是什麼？怎麼進入 debug 模式讓事件即時顯示在 DebugView？
- GTM 的 **Preview（預覽 / Tag Assistant）** 模式能幫你看到什麼？（提示：哪些 Tag 觸發了、`dataLayer` 內容、Trigger 有沒有符合）
- 不開任何工具，你要怎麼在瀏覽器 Console 快速檢查目前 `dataLayer` 的內容？
- 上線前你會用什麼 checklist 驗證埋點正確？（提示：事件名稱與 Tracking Plan 一致、參數齊全、沒有重複送、沒有夾帶 PII、consent 未同意時不送）
