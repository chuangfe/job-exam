# Q59 - 分析追蹤與事件埋點（Analytics / Tracking）解析

## Q1. 為何要埋點？常見指標

**為什麼要埋點：** 埋點是把「使用者做了什麼」變成**可量化、可分析的資料**。沒有資料就只能靠感覺做決策；有了資料才能回答「哪個功能沒人用」「使用者卡在結帳的哪一步」「這波廣告帶來的流量有沒有轉換」。資料最終服務三種決策：

- **產品**：功能有沒有被使用、使用者在哪一步流失、A/B 測試哪個版本好。
- **行銷**：哪個管道（Google 搜尋、FB 廣告、EDM）帶來的流量最有價值、廣告投報率（ROAS）。
- **營運 / 商業**：營收、客單價、回購率。

**常見指標：**

| 指標 | 意思 |
| --- | --- |
| **PV（Page View）** | 頁面被瀏覽的**次數**。同一人重整 5 次算 5 次 PV。 |
| **UV（Unique Visitor）** | 不重複的**訪客數**。同一人一天內看 5 頁、重整 10 次，仍只算 1 個 UV（GA4 以 device / user 維度去重）。 |
| **Session（工作階段）** | 一個訪客在一段連續時間內的一「次」造訪。GA4 預設**閒置 30 分鐘**無互動即結束，之後再互動算新的 session。 |

- **PV vs UV**：PV 看「總流量」，UV 看「有多少人」。一個 UV 可以貢獻多個 PV。單看 PV 高不代表受眾廣，可能是少數人狂重整。
- **轉換（Conversion）**：使用者完成一個你**定義為有價值**的目標動作。電商例子：加入購物車、完成結帳、註冊會員都可設為轉換。**轉換率 = 完成轉換的人數 / 總人數**。例如 1000 人瀏覽商品、50 人購買，購買轉換率 5%。
- **留存（Retention）**：使用者在首次使用後，隔一段時間**是否回來**（次日留存、7 日留存、30 日留存）。為什麼常比「拉新」重要？因為拉新的成本高，留存代表產品真的有價值、能持續變現；一個「漏水的桶子」（留存差）再怎麼拉新都留不住人。常用「同類群組分析（cohort analysis）」觀察。
- **漏斗（Funnel）**：把一連串有先後順序的步驟畫成逐層縮小的漏斗，看**每一步流失多少**。以「瀏覽商品 → 加入購物車 → 開始結帳 → 付款成功」為例：若「加入購物車 → 開始結帳」掉了 70%，就知道結帳入口有問題（可能運費太晚揭露、要強制註冊）。漏斗能精準定位**優化的著力點**，而不是空泛地說「轉換率要提高」。

## Q2. GA4 與 GTM：差異與分工

**兩者是「搭配使用」，不是二選一：**

- **GA4（Google Analytics 4）** 是**分析平台 / 資料的目的地**。它負責**收集、儲存、報表化**使用者行為資料，讓你看 PV、UV、轉換、漏斗、受眾等報表。
- **GTM（Google Tag Manager）** 是**代碼管理工具 / 派送中心**。它本身不分析資料，而是幫你**統一管理與佈署各種追蹤代碼**（GA4、Meta Pixel、Google Ads、LINE Tag…），決定「什麼時機、把什麼資料、送到哪個平台」。

**為什麼透過 GTM 佈署 GA4，而不是把程式碼寫死？**

- **不必每次改追蹤都動程式碼、重新發版**：行銷 / 數據人員可以在 GTM 網頁介面自行新增或修改代碼與觸發條件，前端只要把資料 push 到 `dataLayer` 一次即可。
- **集中管理**：所有第三方追蹤都在 GTM 一處管理，避免程式碼裡散落十幾段追蹤 script。
- **版本控制與回滾**：GTM 有版本紀錄，出錯能一鍵回到前一版。
- **預覽除錯**：GTM Preview 模式可在上線前驗證。

**GA4 相較 UA 最大的觀念轉變：以事件為中心。** 舊版 Universal Analytics 以「工作階段 / 頁面瀏覽」為核心資料模型，事件是附屬的（有 category / action / label 三段式結構）。GA4 則**所有東西都是事件（event）**——連 `page_view` 都只是一個事件，每個事件可帶任意參數（parameters）。這讓資料模型更彈性、更適合追蹤 App 與網站的複雜互動。

**GTM 三核心（用「點擊購買按鈕送 GA4 事件」串起來）：**

- **Trigger（觸發條件）**：定義「**什麼時候**要動作」。例：偵測到 `dataLayer` 出現 `event: 'add_to_cart'`。
- **Tag（代碼）**：定義「**要做什麼**」。例：一個 GA4 事件代碼，把 `add_to_cart` 事件送往 GA4。
- **Variable（變數）**：定義「**要帶什麼資料**」。例：從 `dataLayer` 取出商品名稱、價格、幣別，填進 Tag 的參數。

串起來：使用者點購買 → 前端 push `add_to_cart` 到 `dataLayer` → **Trigger** 偵測到 → 觸發 GA4 **Tag** → Tag 透過 **Variable** 讀取商品資料 → 送往 GA4。

## Q3. `dataLayer` 的角色與事件驅動模型

**`dataLayer` 是什麼：** 它就是掛在 `window` 上的一個**全域陣列**（`window.dataLayer`），扮演網站與 GTM 之間的**中介資料緩衝區 / 傳話筒**。前端把「發生了什麼事、相關資料是什麼」push 進去，GTM 監聽這個陣列，據此觸發對應代碼。

```html
<!-- GTM snippet 之前要先初始化 dataLayer -->
<script>
  window.dataLayer = window.dataLayer || [];
</script>
```

**為什麼能讓前端與數據團隊解耦：** 前端的責任邊界很清楚——**只負責 push 乾淨、結構一致的資料到 `dataLayer`**。至於這些資料要送去 GA4、Meta、Google Ads 還是全部都送、要不要加工，是 GTM 端（數據 / 行銷團隊）的事。雙方以「`dataLayer` 的資料結構（也就是 Tracking Plan）」為契約，各改各的，不必互相等待發版。

**事件驅動模型：** 當前端執行：

```js
window.dataLayer.push({
  event: 'add_to_cart',
  ecommerce: { currency: 'TWD', value: 590, items: [/* ... */] },
});
```

`event` 這個特殊鍵會觸發 GTM 內部的 **Custom Event 機制**：GTM 監聽到 `dataLayer` 被 push 了一筆帶 `event` 的資料，就去比對所有 Trigger，符合條件的 Tag 隨即被觸發、讀取變數、送出資料。整個流程是「**push 一筆事件 → GTM 反應**」，故稱事件驅動。

**為什麼建議前端只 push、由 GTM 決定去向：** 追蹤需求變動非常頻繁（今天多送 Meta、明天加 TikTok Pixel）。若前端把「送到哪個平台」的邏輯寫死在程式裡，每次都要改碼發版；改由 GTM 決定，前端的埋點就穩定不動，維護成本大幅降低。

## Q4. 事件設計

**GA4 事件四大類：**

1. **自動收集事件（Automatically collected）**：GA4 內建自動送，如 `first_visit`、`session_start`、`user_engagement`。
2. **加強型評估事件（Enhanced measurement）**：在 GA4 後台一鍵開啟即可自動追蹤，如 `scroll`（捲動 90%）、`click`（外部連結）、`file_download`、`video_start`。
3. **推薦事件（Recommended）**：Google 有**預先定義好名稱與參數規格**、但需要你自己實作的事件，最典型的就是電商事件（`add_to_cart`、`purchase`…）。用推薦事件才能吃到 GA4 對應的專屬報表。
4. **自訂事件（Custom）**：以上都不涵蓋、你自己命名的事件，如 `newsletter_subscribe`。

**`page_view`：** 代表「一次頁面瀏覽」。傳統多頁式網站（MPA）每次換頁都是整頁重新載入，`gtag`/GTM 會自動送出 `page_view`。但 SPA 換頁不重載整頁，自動送出的機制就失效了（詳見 Q5）。

**自訂事件範例（點擊訂閱電子報）：**

```js
window.dataLayer.push({
  event: 'newsletter_subscribe',
  method: 'footer_form',   // 從哪個位置訂閱
  plan: 'weekly',          // 訂閱頻率
});
```

**電商事件（GA4 推薦規格）：**

| 事件 | 何時送 | 主要參數 |
| --- | --- | --- |
| `view_item` | 檢視單一商品頁 | `currency`、`value`、`items[]` |
| `add_to_cart` | 加入購物車 | `currency`、`value`、`items[]` |
| `begin_checkout` | 進入結帳流程 | `currency`、`value`、`items[]` |
| `purchase` | 付款成功 | `transaction_id`（必填、去重用）、`currency`、`value`、`tax`、`shipping`、`items[]` |

`items` 是商品陣列，每個項目常帶 `item_id`、`item_name`、`price`、`quantity`、`item_category`、`item_brand`。`value` 是這筆事件的金額（通常等於商品小計），`currency` 用 ISO 4217 代碼（台幣 `TWD`）。**`purchase` 一定要帶 `transaction_id`**，GA4 靠它去除重複計算（使用者重整成功頁不會重複算一筆營收）。

**事件命名慣例：**

- 統一用 **`snake_case`**（GA4 官方推薦事件就是這種風格，如 `add_to_cart`）。
- 用「**動詞_名詞**」描述動作，如 `select_promotion`、`share_content`。
- **大小寫敏感**：`Add_To_Cart`、`add_to_cart`、`addToCart` 在 GA4 會被當成**三個不同事件**，資料就此分裂。
- 事件名與參數名都以 Tracking Plan 為準，不臨時起名。

命名混亂的後果：報表裡出現一堆意思相同但名稱不同的事件，資料無法彙總、漏斗接不起來、分析師花大量時間清資料，甚至做出錯誤結論。而且**歷史資料無法回溯修正**（送錯就是錯了）。

**Tracking Plan（追蹤計畫文件）：** 一份記錄「這個網站要追蹤哪些事件」的**單一事實來源（single source of truth）**，通常是一張表，欄位包含：

| 欄位 | 說明 |
| --- | --- |
| 事件名稱 | 如 `add_to_cart` |
| 觸發時機 | 使用者何時、在哪個頁面 / 元件觸發 |
| 參數 | 參數名、型別、是否必填、範例值 |
| 對應指標 / 用途 | 這個事件要回答什麼商業問題 |
| 負責人 / 狀態 | 誰實作、已上線 / 開發中 |

為什麼一定要有：它是前端、數據、行銷三方的**共同契約**。前端照它埋、分析師照它讀、行銷照它設轉換。沒有它，每個人對「這個事件代表什麼」的理解會漂移，資料品質崩壞。維護它比寫程式更重要。

## Q5. SPA 的追蹤

**為什麼 SPA 抓不到後續 page_view：** GA4 / `gtag` 的自動 `page_view` 只在**頁面首次載入、script 執行時**送出一次。SPA（React Router、Vue Router）換頁是靠 History API（`pushState`）在前端切換畫面，**沒有真正的整頁重新載入**，所以第一頁之後的所有「換頁」GA4 完全無感，導致只記錄到一個 PV。

**正確做法：監聽路由變更，每次換頁手動送 `page_view`。** 在 React Router 中用 `useLocation` hook 偵測 `pathname` 變化：

```tsx
import { useEffect } from 'react';
import { useLocation } from 'react-router-dom';
import { trackPageView } from './analytics';

export function usePageTracking() {
  const location = useLocation();

  useEffect(() => {
    // 每次路徑（含 query）改變就送一次 page_view
    trackPageView(location.pathname + location.search);
  }, [location.pathname, location.search]);
}
```

把 `usePageTracking()` 放在 Router 內、只掛載一次的最上層元件（如 `<App>`）即可。

> 注意：手動送 `page_view` 時，通常會在 GA4 設定裡**關閉加強型評估的「以瀏覽記錄變更為基礎的網頁瀏覽」**，避免和手動送出的重複計算。

**為什麼要封裝成 `trackEvent` / 自訂 hook：**

1. **單一入口、好維護**：所有埋點集中在一個模組，要換平台、加防呆、統一格式，只改一處。
2. **一致性**：強制所有事件走同一套結構與命名，避免每個元件各自 `window.dataLayer.push` 導致格式亂七八糟。
3. **防呆與環境隔離**：可在封裝層統一處理「開發環境不送 / 只 console.log」「`dataLayer` 尚未初始化的保護」「過濾 PII」「未同意 consent 就不送」。
4. **易測試**：元件只依賴 `trackEvent`，測試時 mock 一個函式即可，不必動 `window`。
5. **型別安全**：用 TypeScript 定義事件名與參數型別，打錯名字或漏參數編譯就報錯。

## Q6. 隱私與合規

**PII（個人可識別資訊）：** 能直接或間接識別特定個人的資料——**姓名、email、電話、身分證字號、完整地址、信用卡號、精確 GPS 位置**等。

**為什麼絕不可送進 GA4：** (1) **違反 Google 政策**：GA4 服務條款明文禁止上傳 PII，違反可能導致帳號被停用、資料被刪除。(2) **違法**：GDPR / 台灣個資法都對 PII 的蒐集處理有嚴格規範。**正確做法**是用**匿名化的識別碼**（如雜湊過的 user_id、內部會員編號），絕不送原始 email / 姓名。若真的要送 email 給廣告平台做比對，也必須先做 SHA-256 雜湊，且屬於另一套 Enhanced Conversions 機制。

**Cookie 同意（consent）與法規：**

- **GDPR（歐盟）** 採「選擇加入（opt-in）」：**在取得使用者明確同意前，不得寫入非必要 cookie、不得追蹤**。所以要先跳同意橫幅（consent banner），使用者點同意後才啟動追蹤。
- **台灣 PDPA（個人資料保護法）** 要求蒐集個資須告知目的並取得同意，實務上網站也應提供隱私權政策與 cookie 說明。整體趨勢是往「先同意、後追蹤」靠攏。

**Consent Mode（同意模式）：** Google 提供的機制，讓網站把使用者的同意狀態（`ad_storage`、`analytics_storage` 等是 `granted` 還是 `denied`）傳給 GA4 / Ads。當使用者**尚未同意**時：

- Consent Mode 會**不寫入 cookie、不儲存可識別的資料**。
- GA4 仍會送出**不含 cookie、去識別化的訊號**，Google 用**模型化（modeling）**估算這部分流量的行為，讓報表不至於完全失真。
- 使用者一旦同意，才切換為完整追蹤。

這讓網站能**同時**兼顧法規（未同意就不追蹤個人）與資料完整性（不至於整片空白）。

**First-party vs third-party cookie：**

- **First-party cookie**：由**使用者正在造訪的網域**自己寫入（如 `example.com` 寫 `example.com` 的 cookie）。用於記住登入、購物車、GA4 的訪客識別。瀏覽器信任度高。
- **Third-party cookie**：由**目前網域以外的第三方**（常是廣告商）透過嵌入的資源寫入，能**跨網站追蹤**同一使用者（你在 A 網站看的商品，B 網站出現該廣告）。

**為什麼封鎖 third-party cookie 衝擊大：** Safari（ITP）、Firefox 早已預設封鎖，Chrome 也在逐步限縮。跨站追蹤與再行銷（retargeting）高度依賴 third-party cookie，被封鎖後廣告的精準投放與轉換歸因大幅失準——這正是業界轉向 first-party 資料、Consent Mode、伺服器端追蹤的主因。

**伺服器端追蹤（server-side tagging）：** 把原本在瀏覽器執行的追蹤代碼，改到**自己的伺服器**（如 GTM Server Container）處理：瀏覽器先把資料送到你自己的伺服器端點，再由伺服器轉送給 GA4 / 廣告平台。好處：

- **資料品質**：不受瀏覽器 cookie 封鎖、廣告攔截器（AdBlocker）影響（因為請求打到自家網域，看起來是 first-party）。
- **隱私控制**：可在伺服器端**過濾 / 遮罩 PII** 再轉送，資料出網站前先清洗。
- **效能與安全**：減少瀏覽器載入的第三方 script，也不把各平台的金鑰暴露在前端。

代價是要維護伺服器基礎設施、成本較高、設定較複雜。

## Q7. 除錯與驗證

**GA4 DebugView：** GA4 後台的即時除錯面板，能看到**當前 debug 裝置**送出的每一個事件與其參數，逐筆即時顯示。進入 debug 模式的方式：

- 安裝 Chrome 擴充「**Google Analytics Debugger**」並開啟；或
- 用 **GTM Preview 模式**（Preview 中的流量自動進 DebugView）；或
- 事件帶上 `debug_mode: true` 參數。

用途：確認事件名稱、參數值是否正確、有沒有真的送出去。

**GTM Preview（Tag Assistant）：** 在 GTM 點「預覽」會開啟 Tag Assistant，把你的網站連上除錯視窗。它能顯示：

- 每個動作**觸發了哪些 Tag**（Tags Fired / Not Fired）以及為什麼。
- 當下 **`dataLayer` 的完整內容**（每一步 push 了什麼）。
- 每個 **Trigger 的條件有沒有符合**、**Variable 解析出什麼值**。

這是排查「Tag 沒觸發」「參數抓錯」最直接的工具。

**用 Console 快速檢查 `dataLayer`：**

```js
// 在瀏覽器 Console 直接印出目前累積的 dataLayer
console.table(window.dataLayer);
// 或看最後一筆 push
console.log(window.dataLayer[window.dataLayer.length - 1]);
```

**上線前 checklist：**

- 事件名稱、參數名與 **Tracking Plan 完全一致**（大小寫、`snake_case`）。
- 必填參數齊全（尤其 `purchase` 的 `transaction_id`、`value`、`currency`）。
- **沒有重複送出**（例如 `useEffect` 依賴寫錯導致重送、或自動 + 手動 page_view 疊加）。
- **沒有夾帶 PII**。
- **consent 未同意時確實不送**（或走 Consent Mode 的去識別化）。
- `purchase` 的 `transaction_id` 能正確去重（重整成功頁不重複計營收）。
- 用 DebugView / GTM Preview 逐一驗證關鍵事件都有正確送達。

## Q8. 實作

**`analytics.ts`——追蹤封裝模組：**

```ts
// analytics.ts
// 統一的追蹤入口：所有埋點都走這裡，方便維護、防呆、過濾與測試。

type EcommerceItem = {
  item_id: string;
  item_name: string;
  price: number;
  quantity: number;
  item_category?: string;
};

type TrackParams = Record<string, unknown>;

const isProd = process.env.NODE_ENV === 'production';

// 確保 dataLayer 存在
function getDataLayer(): Record<string, unknown>[] {
  window.dataLayer = window.dataLayer || [];
  return window.dataLayer;
}

// 通用事件追蹤
export function trackEvent(event: string, params: TrackParams = {}): void {
  // 非正式環境只印出來，不真的送
  if (!isProd) {
    console.log('[analytics]', event, params);
    return;
  }
  getDataLayer().push({ event, ...params });
}

// 頁面瀏覽（SPA 手動送）
export function trackPageView(path: string): void {
  trackEvent('page_view', { page_path: path });
}

// 電商：加入購物車
export function trackAddToCart(item: EcommerceItem, currency = 'TWD'): void {
  trackEvent('add_to_cart', {
    ecommerce: {
      currency,
      value: item.price * item.quantity,
      items: [item],
    },
  });
}

// 讓 TypeScript 認得 window.dataLayer
declare global {
  interface Window {
    dataLayer: Record<string, unknown>[];
  }
}
```

**`usePageTracking.ts`——路由變更自動送 page_view：**

```tsx
import { useEffect } from 'react';
import { useLocation } from 'react-router-dom';
import { trackPageView } from './analytics';

export function usePageTracking() {
  const location = useLocation();

  useEffect(() => {
    trackPageView(location.pathname + location.search);
  }, [location.pathname, location.search]);
}
```

在最上層掛一次：

```tsx
function App() {
  usePageTracking(); // 必須放在 <BrowserRouter> 內
  return <Routes>{/* ... */}</Routes>;
}
```

**「加入購物車」按鈕呼叫 `trackEvent`：**

```tsx
import { trackAddToCart } from './analytics';

function AddToCartButton({ product }: { product: Product }) {
  const handleClick = () => {
    addToCart(product); // 你的購物車邏輯

    // 埋點：元件只呼叫封裝好的函式，不直接碰 window.dataLayer
    trackAddToCart({
      item_id: product.id,
      item_name: product.name,
      price: product.price,
      quantity: 1,
      item_category: product.category,
    });
  };

  return <button onClick={handleClick}>加入購物車</button>;
}
```

**對應的 `dataLayer.push` 範例（`add_to_cart` 完整結構）：**

```js
// 上面 trackAddToCart 最終等同於 push 這一筆到 dataLayer：
window.dataLayer.push({
  event: 'add_to_cart',
  ecommerce: {
    currency: 'TWD',
    value: 590,
    items: [
      {
        item_id: 'SKU_12345',
        item_name: '無線藍牙耳機',
        price: 590,
        quantity: 1,
        item_category: '3C 配件',
        item_brand: 'Acme',
      },
    ],
  },
});
```

GTM 端接手：一個 Custom Event Trigger 監聽 `event = 'add_to_cart'`，觸發 GA4 事件 Tag，透過 Data Layer Variable 讀出 `ecommerce` 內容，送往 GA4，即完成整條追蹤鏈路。
