# Code Review 與重構 解析

Code Review 的重點不是「挑錯給人看」，而是**用具體理由讓程式碼變得更正確、更好維護**。下面逐段列出問題、分類、為什麼是問題，以及重構後的版本；最後補一節 **Code Review 心法**。

分類縮寫：**正確性**（會壞 / 行為錯）、**效能**、**可維護性**、**a11y**（可及性）、**型別**。

## 情境 1. React 商品清單與購物車

### 發現的問題

| # | 分類 | 問題 |
|---|------|------|
| 1 | 可維護性 / 效能 | **用 `useEffect` + `useState` 計算衍生狀態**。`filtered` 完全由 `products` 和 `keyword` 推導而來，不該存成 state。這是最常見的 React 反模式：多一次 render、狀態容易不同步、程式更難懂。 |
| 2 | 正確性 | **直接 mutate state**：`cart.push(product); setCart(cart)`。`push` 改的是同一個陣列參考，`setCart` 拿到的是「同一個參考」，React 用 `Object.is` 比對會認為「沒變」，可能不重繪；就算重繪也破壞了「state 不可變」的假設（會影響 `React.memo`、`useMemo`、時光旅行除錯等）。 |
| 3 | 正確性 / 效能 | **用陣列 index 當 `key`**。清單會過濾、之後也可能排序或刪除，index 會錯位，導致 React 重用錯的節點（輸入框內容跑掉、動畫錯亂）。應該用穩定且唯一的 `p.id`。 |
| 4 | 效能 | **render 內建立 inline `style` 物件**：每次 render 都產生新物件，讓 `<li>` 無法有效被記憶化。抽成常數或用 CSS class。 |
| 5 | 效能（次要） | 每個 `<button>` 都用 inline arrow function。多數情況可接受，但若 `<li>` 有做 `memo`，這會讓 memo 失效。 |

### 為什麼「衍生狀態」是問題

「能從既有 props/state 算出來的東西，就不要另外存成 state」。存了之後你得負責「同步」它（所以才需要那個 `useEffect`），一旦忘記同步就會出現「畫面與資料不一致」的 bug。直接在 render 期間算即可；若計算很貴才用 `useMemo`。

### 重構後

```tsx
import { useMemo, useState } from 'react';

type Product = { id: number; name: string; price: number };

const itemStyle = { padding: 8, borderBottom: '1px solid #ccc' }; // 抽到元件外，參考穩定

function ProductList({ products }: { products: Product[] }) {
  const [keyword, setKeyword] = useState('');
  const [cart, setCart] = useState<Product[]>([]);

  // 衍生資料在 render 時算；量大才用 useMemo
  const filtered = useMemo(
    () => products.filter((p) => p.name.includes(keyword)),
    [products, keyword],
  );

  const addToCart = (product: Product) => {
    setCart((prev) => [...prev, product]); // 不可變更新 + 函式式更新
  };

  return (
    <div>
      <input value={keyword} onChange={(e) => setKeyword(e.target.value)} />

      <ul>
        {filtered.map((p) => (
          <li key={p.id} style={itemStyle}>
            {p.name} - ${p.price}
            <button onClick={() => addToCart(p)}>加入購物車</button>
          </li>
        ))}
      </ul>

      <p>購物車共 {cart.length} 件</p>
    </div>
  );
}
```

## 情境 2. React 計時器與視窗監聽

### 發現的問題

| # | 分類 | 問題 |
|---|------|------|
| 1 | 正確性 | **過期閉包（stale closure）**：`setInterval` 的 callback 只在掛載時建立一次，永遠捕捉到 `count = 0`，所以每秒都執行 `setCount(0 + 1)`，畫面卡在 1。 |
| 2 | 正確性 | **副作用未清理**：`useEffect` 沒有回傳 cleanup，`setInterval` 不會被清掉。元件卸載後 timer 仍在跑並呼叫 `setCount`，造成記憶體洩漏與潛在警告。 |
| 3 | 正確性 / 效能 | 第二個 `useEffect` **沒有依賴陣列**，每次 render 都重新 `addEventListener`，而且**從不移除**，監聽器越疊越多。 |

### 為什麼會卡在 1

`count + 1` 讀的是「effect 建立當下」被閉包捕捉的 `count`（=0）。因為 effect 只跑一次，這個值不會更新，於是每秒都是 `setCount(1)`。解法是用**函式式更新** `setCount((c) => c + 1)`，讓 React 給你最新值，就不依賴閉包裡的舊 `count`。

### 重構後

```tsx
import { useState, useEffect } from 'react';

function Timer() {
  const [count, setCount] = useState(0);
  const [width, setWidth] = useState(window.innerWidth);

  useEffect(() => {
    const timer = setInterval(() => {
      setCount((c) => c + 1); // 函式式更新，避開 stale closure
    }, 1000);
    return () => clearInterval(timer); // 清理
  }, []);

  useEffect(() => {
    const handleResize = () => setWidth(window.innerWidth);
    window.addEventListener('resize', handleResize);
    return () => window.removeEventListener('resize', handleResize); // 對稱移除
  }, []); // 只需綁一次

  return (
    <div>
      <p>已經過 {count} 秒</p>
      <p>視窗寬度：{width}px</p>
    </div>
  );
}
```

> 口訣：**凡是在 effect 裡「開了什麼」（interval、listener、subscription、連線），cleanup 就要「關掉它」**——保持對稱。

## 情境 3. TypeScript 訂單資料處理

### 發現的問題

| # | 分類 | 問題 |
|---|------|------|
| 1 | 型別 | 參數用 `any`，完全放棄型別檢查；拼錯欄位名、少欄位都不會被抓到。應定義明確的資料型別。 |
| 2 | 可維護性 | **箭頭形巢狀（arrow anti-pattern）**：層層 `if` 往右縮排，難讀。應用**早退（guard clause）**與陣列方法攤平。 |
| 3 | 可維護性 | 用傳統 `for` 迴圈 + index 累加，冗長；`reduce` / `filter` 更能表達意圖。 |
| 4 | 正確性 / 可維護性 | `status == 'paid'` 用鬆散相等 `==`。這裡兩邊雖都是字串，但團隊規範應一律用 `===`，避免隱式轉型。 |
| 5 | 可維護性 | 命名可讀性差（`i`、`j`、`data`），且金額計算 `price * qty` 是「重複可抽出」的概念。 |

### 重構後

```ts
type OrderItem = { price: number; qty: number };
type Order = { status: 'paid' | 'pending' | 'cancelled'; items?: OrderItem[] };
type OrderData = { orders?: Order[] };

const itemAmount = (item: OrderItem) => item.price * item.qty;

function getPaidTotal(data: OrderData): number {
  const orders = data.orders ?? []; // 早退式的防呆，攤平巢狀

  return orders
    .filter((order) => order.status === 'paid')
    .flatMap((order) => order.items ?? [])
    .reduce((sum, item) => sum + itemAmount(item), 0);
}
```

**改了什麼**：`any` → 明確型別（呼叫端與維護者都受益）；巢狀 `if` → `?? []` 防呆 + `filter/flatMap/reduce` 的資料流；`==` → `===`；`i/j` → 有意義的名字；把 `price * qty` 抽成 `itemAmount`（單一職責、好測試、好複用）。

## 情境 4. React 登入表單

### 發現的問題

| # | 分類 | 問題 |
|---|------|------|
| 1 | a11y / 正確性 | **用 `<div onClick>` 當按鈕**。無法用鍵盤（Tab/Enter/Space）操作、螢幕報讀器不會念成「按鈕」、且**不在 `<form>` 內**所以按 Enter 不能送出。應該用 `<button type="submit">` 搭配 `<form onSubmit>`。 |
| 2 | a11y | **input 沒有關聯 `<label>`**。用純文字 `<div>Email</div>` 不算 label；報讀器聚焦到欄位時不知道這是什麼。要用 `<label htmlFor>` + input `id`。 |
| 3 | a11y | **密碼欄沒有 `type="password"`**、Email 欄沒有 `type="email"`，也缺 `autoComplete`。 |
| 4 | a11y | **錯誤訊息只用紅色文字**，且沒有 `role="alert"` / `aria-live`，報讀器不會主動念出；也沒把錯誤與欄位用 `aria-describedby` / `aria-invalid` 關聯。 |
| 5 | 可維護性 | 標題用 `<div style={{fontSize:20}}>` 而非語意標籤（如 `<h2>`）；行內樣式散落，難維護。 |

### 為什麼「div 當按鈕」是問題

`<button>` 原生就有：可被 Tab 聚焦、Enter/Space 觸發、報讀器念作「按鈕」、`disabled` 支援、在表單內能 submit。用 `<div>` 全都得自己補（`role="button"`、`tabIndex={0}`、鍵盤事件…），既繁瑣又容易漏——**能用原生元素就別自己造**。

### 重構後

```tsx
import { useState } from 'react';

function LoginForm({ onSubmit }: { onSubmit: (email: string, pwd: string) => void }) {
  const [email, setEmail] = useState('');
  const [pwd, setPwd] = useState('');
  const [error, setError] = useState('');

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault();
    if (email === '') return setError('請輸入 Email');
    if (pwd.length < 6) return setError('密碼至少 6 碼');
    setError('');
    onSubmit(email, pwd);
  };

  return (
    <form onSubmit={handleSubmit}>
      <h2>登入</h2>

      <label htmlFor="email">Email</label>
      <input
        id="email"
        type="email"
        autoComplete="email"
        value={email}
        onChange={(e) => setEmail(e.target.value)}
        aria-invalid={error !== ''}
        aria-describedby={error !== '' ? 'login-error' : undefined}
      />

      <label htmlFor="pwd">密碼</label>
      <input
        id="pwd"
        type="password"
        autoComplete="current-password"
        value={pwd}
        onChange={(e) => setPwd(e.target.value)}
      />

      {error !== '' && (
        <p id="login-error" role="alert" className="error">
          {error}
        </p>
      )}

      <button type="submit">送出</button>
    </form>
  );
}
```

> 這裡把 `submit` 改成 `handleSubmit(e)` 並 `e.preventDefault()`，讓 Enter 送出與點擊都走同一條路；顏色改用 `className` 交給 CSS，錯誤訊息用 `role="alert"` 讓報讀器主動播報。

## Code Review 心法

### 1. 分清 blocker、nit 與 suggestion

| 等級 | 意義 | 例子 |
|------|------|------|
| **Blocker（擋合併）** | 會壞、會有安全 / 資料問題、破壞正確性 | 情境 2 的 stale closure、情境 1 的 mutate state |
| **Should fix** | 明顯 code smell、效能、a11y | 情境 3 的 `any`、情境 4 的鍵盤操作 |
| **Nit（提醒即可）** | 命名、排版、個人偏好 | 變數叫 `i` 而非 `index` |

留言時**標明等級**（例如 `nit:`、`blocker:`），對方才知道哪些必須改、哪些可自行取捨。純風格問題交給 formatter / linter，別在 review 裡吵。

### 2. Review 的優先順序（由重到輕）

1. **正確性**：這段會不會壞？邊界條件、錯誤處理、競態。
2. **安全性**：XSS、注入、機敏資料外洩。
3. **架構 / 可維護性**：職責是否單一、抽象是否合理、有無重複。
4. **效能**：有無明顯的不必要重繪 / N+1 / 大量計算。
5. **可讀性 / 命名 / 風格**：最後才看，且多半可自動化。

先抓大問題，別一開頭就糾結分號。

### 3. 給有建設性的回饋

- **對事不對人**：說「這個函式做了三件事」，別說「你怎麼寫成這樣」。
- **說明為什麼**：附上理由或連結，讓對方學到而不只是照改。
- **給方向甚至範例**：「可用 `flatMap` 攤平」比「這裡很醜」有用。
- **用提問代替命令**：「這裡沒清 interval，卸載後會不會洩漏？」
- **也要稱讚好的地方**：正向回饋讓 review 是協作而非審判。

### 4. 讓自動化擋掉「不需要人看的問題」

人力應該花在邏輯與設計上；以下交給工具：

- **ESLint**：`react-hooks/exhaustive-deps`（抓依賴陣列漏 / stale closure）、`no-unused-vars`、a11y 規則（`eslint-plugin-jsx-a11y` 可抓「div 當按鈕」「input 缺 label」）。
- **Prettier**：排版、縮排、引號——完全不該進 review 討論。
- **TypeScript（`strict`）**：把情境 3 的 `any` 與空值問題在編譯期擋掉。
- **測試 + CI**：正確性回歸；PR 沒過 CI 就別開始 review。

> 一句話總結：**能被工具擋的別用人腦擋，人腦專心看正確性與設計。**
