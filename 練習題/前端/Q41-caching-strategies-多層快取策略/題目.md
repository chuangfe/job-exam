# Q41 - 前端多層快取策略（從瀏覽器到伺服器）

現代前端應用的一次請求，可能會被好幾層快取攔截：瀏覽器 HTTP 快取、Service Worker、CDN/Edge、伺服器端資料快取（Next.js Data Cache / Redis）。這題請你把「從瀏覽器到伺服器」的各層快取串起來，說明各層快取什麼、由誰控制 TTL、如何失效，以及對應的實務策略。

> 本題與 Q24（HTTP 快取）互補：Q24 聚焦瀏覽器 HTTP 快取的細節，本題把視野拉高到「多層」，請對照閱讀。

## Q1. 瀏覽器 HTTP 快取（強快取 vs 協商快取）

- 說明 `Cache-Control` 常見指令的意義與差異：`max-age`、`s-maxage`、`no-cache`、`no-store`、`must-revalidate`、`private` / `public`、`immutable`。
- 強快取（strong cache）與協商快取（negotiated / revalidation cache）分別在什麼時機作用？命中時瀏覽器的行為有何不同？
- `ETag` / `If-None-Match` 與 `Last-Modified` / `If-Modified-Since` 這兩組驗證機制如何配對運作？回 `304` 代表什麼？

## Q2. Service Worker Cache API 的四種策略

- 用 Service Worker 的 Cache API 攔截 `fetch` 事件時，常見有四種策略：**cache-first**、**network-first**、**stale-while-revalidate**、**cache-then-network**。分別畫出流程並說明適用情境與風險。
- 哪一種最適合「不常變動的靜態資源」？哪一種最適合「要即時但也想要離線 fallback 的 API」？
- Service Worker 快取的版本控制怎麼做？舊快取何時清除（`activate` 事件）？快取失效與「使用者卡在舊版」的風險如何避免？
- 請寫一段 `fetch` 事件攔截、實作 stale-while-revalidate 的程式碼。

## Q3. CDN / Edge 快取

- `s-maxage` 與 `max-age` 的差別是什麼？為什麼共享快取（CDN）要看 `s-maxage`？
- `Cache-Control: stale-while-revalidate` 指令在 CDN 上如何運作？對使用者感受到的延遲有何幫助？
- 什麼是「快取鍵（cache key）」？哪些因素會影響 cache key（URL、query、`Vary` 標頭）？
- CDN 的 purge / invalidation 有哪些方式？為什麼靜態資源建議「hash 檔名 + `immutable`」而不是靠 purge？

## Q4. 伺服器端資料快取（Next.js / Redis）

- Next.js（App Router）的 Data Cache / `fetch` 快取如何運作？`fetch(url, { next: { revalidate, tags } })` 各參數的意義？`revalidateTag` 與 `revalidatePath` 如何做事件驅動失效？
- 一般後端用 Redis 做應用層快取的 **cache-aside（旁路快取）** 模式流程為何？TTL 該怎麼設？
- 說明**快取雪崩（avalanche）**、**快取擊穿（breakdown / hotspot）**、**快取穿透（penetration）** 三者的差別與各自對策。

## Q5. 快取失效策略總整理

- 快取失效的三大手段：**時間過期（TTL）**、**事件驅動（tag / path 失效）**、**版本化 key（hash 檔名 / 版本前綴）**，各自的優缺點與適用層級。
