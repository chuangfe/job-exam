# Q41 - 前端多層快取策略（從瀏覽器到伺服器）解析

一次請求會依序經過多層快取。整體心智圖如下：

```
瀏覽器
  │  ① Service Worker（Cache API，可攔截 fetch）
  │  ② 瀏覽器 HTTP 快取（memory / disk cache，看 Cache-Control）
  ▼
CDN / Edge（③ 共享快取，看 s-maxage / stale-while-revalidate）
  ▼
Origin 伺服器
  │  ④ Next.js Data Cache / fetch 快取（框架層）
  │  ⑤ Redis 應用層快取（cache-aside）
  ▼
資料庫 / 上游 API
```

原則：**越靠近使用者的快取越快、越省成本，但越難即時失效**；越靠近資料源越好失效、越即時，但成本越高。實務上「靜態資源往前放（瀏覽器/CDN）、動態資料往後放（Server/Redis）」。

---

## Q1. 瀏覽器 HTTP 快取

### Cache-Control 指令對照

| 指令 | 意義 |
| --- | --- |
| `max-age=N` | 私有/共享快取皆適用，副本在 N 秒內視為新鮮（強快取） |
| `s-maxage=N` | **只給共享快取（CDN/proxy）** 用的新鮮期，優先權高於 `max-age` |
| `no-cache` | 可以存，但**每次使用前都要向伺服器協商驗證**（不是「不快取」） |
| `no-store` | 完全不存（連磁碟都不寫），用於敏感資料 |
| `must-revalidate` | 一旦過期就**必須**成功協商，不可用過期副本（不能在離線時 fallback 舊資料） |
| `private` | 只有終端瀏覽器可快取，CDN/proxy 不可（含使用者個人化內容時用） |
| `public` | 任何快取（含共享快取）都可存 |
| `immutable` | 資源永不改變，即使使用者按重新整理也不要發協商請求 |

### 強快取 vs 協商快取

```
發出請求前
  │
  ▼
強快取仍新鮮？（Cache-Control: max-age / Expires 未過期）
  ├── 是 ──▶ 直接用本地副本，「完全不發請求」（devtools 顯示 from disk/memory cache，狀態碼 200）
  └── 否 ──▶ 帶驗證標頭發請求（協商快取）
                │
                ▼
        伺服器判斷資源變了沒
          ├── 沒變 ──▶ 304 Not Modified（不傳 body，續用本地副本）
          └── 變了 ──▶ 200 OK（傳新 body + 新的快取標頭）
```

- **強快取命中 = 零網路往返**，最快；協商快取命中仍有一次往返，但 `304` 不傳 body、省頻寬。
- 兩組協商機制：
  - **ETag（內容指紋，較精確）**：回應帶 `ETag: "abc"`，下次請求帶 `If-None-Match: "abc"`。
  - **Last-Modified（時間戳，精度到秒）**：回應帶 `Last-Modified: <日期>`，下次帶 `If-Modified-Since: <日期>`。
  - 兩者並存時 **ETag 優先**（能避免「一秒內多次修改」「內容沒變但 mtime 變」的誤判）。

```http
# 首次回應
HTTP/1.1 200 OK
Cache-Control: no-cache
ETag: "v1-abc123"

# 後續請求
GET /index.html HTTP/1.1
If-None-Match: "v1-abc123"

# 未變更回應（無 body）
HTTP/1.1 304 Not Modified
ETag: "v1-abc123"
```

> 細節（`Expires`、HTML vs hash 靜態資源的策略）請見 Q24。

## Q2. Service Worker Cache API 的四種策略

Service Worker 攔在瀏覽器 HTTP 快取「之前」，可用 JS 完全掌控每個請求要走快取還是網路，也是實現 **離線（offline）** 與 PWA 的核心。

### 四種策略流程與取捨

| 策略 | 流程 | 適用 | 風險 |
| --- | --- | --- | --- |
| **cache-first** | 先查快取，命中就用；沒命中才打網路並寫入快取 | 版本化靜態資源（帶 hash 的 JS/CSS/字型/圖片） | 快取一旦寫入就「黏住」，內容更新不易被看到，需靠版本化 key |
| **network-first** | 先打網路，成功就用（並更新快取）；失敗才回退快取 | 需即時、但也想離線 fallback 的 API/HTML | 網路慢時 TTFB 差；需設 timeout |
| **stale-while-revalidate** | 立刻回快取（快），同時背景抓新版寫回快取（下次才生效） | 可容忍「稍舊一版」的內容（頭像、清單、非關鍵 API） | 使用者這次看到的是舊資料，下次才更新 |
| **cache-then-network** | 先回快取先渲染，網路回來後再更新畫面（UI 需處理二次更新） | 需要「秒開又要最新」的頁面 | UI 要能處理內容跳動（layout shift） |

### 版本控制與快取失效

- 用**版本化快取名稱**（如 `static-v3`），在 `activate` 事件刪掉舊版快取，避免使用者卡在舊資源。
- SW 本身的更新：新的 `sw.js` 會進入 `installing → waiting`，可用 `skipWaiting()` + `clients.claim()` 立即接管（但要注意頁面資源版本一致性）。

### 程式碼示範（stale-while-revalidate + 版本清理）

```js
const CACHE = 'assets-v3';

// 安裝：預先快取核心資源
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE).then((cache) =>
      cache.addAll(['/', '/offline.html', '/app.4f3a.js'])
    )
  );
  self.skipWaiting();
});

// 啟用：清掉舊版快取
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then((keys) =>
      Promise.all(keys.filter((k) => k !== CACHE).map((k) => caches.delete(k)))
    )
  );
  self.clients.claim();
});

// 攔截 fetch：stale-while-revalidate
self.addEventListener('fetch', (event) => {
  if (event.request.method !== 'GET') return; // 只快取 GET

  event.respondWith(
    caches.open(CACHE).then(async (cache) => {
      const cached = await cache.match(event.request);

      const network = fetch(event.request)
        .then((response) => {
          // 只快取成功且非 opaque 以外的正常回應
          if (response && response.status === 200) {
            cache.put(event.request, response.clone());
          }
          return response;
        })
        .catch(() => cached); // 離線時退回快取

      // 有快取就先回快取（快），沒有就等網路
      return cached || network;
    })
  );
});
```

## Q3. CDN / Edge 快取

CDN 是**共享快取**：一份副本服務眾多使用者，因此看的是 `s-maxage`。

### s-maxage vs max-age

```http
Cache-Control: public, max-age=0, s-maxage=600, stale-while-revalidate=86400
```

- `max-age=0`：瀏覽器每次都要回源/協商（不信任瀏覽器本地副本）。
- `s-maxage=600`：**CDN** 上這份副本 600 秒內視為新鮮，直接由邊緣節點回應，不回源。
- `stale-while-revalidate=86400`：新鮮期過後的 86400 秒內，CDN 可**先回舊副本給使用者（零延遲）**，同時背景回源抓新版更新快取。使用者永遠不必等回源，只有背景那一次會慢。

### 快取鍵（cache key）

- CDN 用「什麼 key 命中同一份副本」，預設是 **完整 URL（含 query string）**。
- `Vary` 標頭會把指定的 request header 納入 key，例如 `Vary: Accept-Encoding`（gzip/br 分開存）、`Vary: Accept-Language`。
- 陷阱：把個人化內容（帶 cookie / 使用者資訊）放到共享快取會**跨使用者洩漏**——這類請務必 `private` 或不快取。

### Purge/Invalidation 與「hash 檔名 + immutable」

- Purge 方式：依 URL 清除、依 tag（surrogate key）批次清除、全站 purge。缺點是**有延遲、有配額成本、且各節點清除不保證同步**。
- 更好的做法：**靜態資源用內容 hash 命名**（`app.4f3a.js`），配 `Cache-Control: public, max-age=31536000, immutable`。
  - 內容變了 → 檔名就變 → 是「新 URL」，天然拿到新版，**根本不需要 purge**。
  - 舊檔名可永久快取，零協商成本；只要更新 HTML 裡引用的檔名即可（HTML 本身用 `no-cache`/短 `s-maxage`）。

## Q4. 伺服器端資料快取

### Next.js（App Router）Data Cache / fetch 快取

Next.js 會自動快取 `fetch()` 的結果到 **Data Cache**（持久、跨請求、跨部署可共用）：

```js
// 時間過期（ISR 風格）：每 3600 秒最多重新驗證一次
const res = await fetch('https://api.example.com/posts', {
  next: { revalidate: 3600 },
});

// 事件驅動：打標籤，之後可精準失效
const res = await fetch('https://api.example.com/posts', {
  next: { tags: ['posts'] },
});

// 不快取（等同 cache: 'no-store'）
const res = await fetch(url, { cache: 'no-store' });
```

事件驅動失效（通常放在 Server Action 或 Route Handler，資料變動後呼叫）：

```js
'use server';
import { revalidateTag, revalidatePath } from 'next/cache';

export async function createPost(data) {
  await db.post.create({ data });
  revalidateTag('posts');     // 失效所有帶 'posts' tag 的 fetch 快取
  revalidatePath('/blog');    // 或依路徑失效
}
```

> 注意 Next.js 各版本預設值不同（例如 15 起 `fetch` 與 Route Handler 預設不快取），實作前請查對應版本文件。

### Redis 應用層快取：cache-aside（旁路快取）

最常見的後端快取模式：

```
讀取：
  1. 先查 Redis
  2. 命中 → 直接回傳（cache hit）
  3. 未命中 → 查 DB → 寫回 Redis（設 TTL）→ 回傳

寫入：
  1. 更新 DB
  2. 刪除（而非更新）對應 Redis key，讓下次讀取重新載入
```

```js
async function getUser(id) {
  const key = `user:${id}`;
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);          // 命中

  const user = await db.user.findById(id);         // 未命中，回源
  await redis.set(key, JSON.stringify(user), 'EX', 300); // TTL 300 秒
  return user;
}

async function updateUser(id, data) {
  await db.user.update(id, data);
  await redis.del(`user:${id}`);                   // 失效，避免髒讀
}
```

- 寫入時**刪除 key 優於更新 key**，可避免併發下寫入順序造成的髒資料。

### 雪崩 / 擊穿 / 穿透

| 問題 | 成因 | 對策 |
| --- | --- | --- |
| **快取雪崩（avalanche）** | 大量 key **同時過期**，瞬間全部打到 DB | TTL 加隨機抖動（如 `300 + rand(0,60)`）；多級快取；熱點永不過期 + 背景更新 |
| **快取擊穿（breakdown / hotspot）** | 某個**熱點 key** 剛好過期，瞬間大量併發回源同一 key | 互斥鎖（mutex / `SET NX`）只讓一個請求回源、其他等待；或熱點邏輯永不過期 |
| **快取穿透（penetration）** | 查詢**根本不存在的資料**（如惡意打不存在 id），每次都繞過快取打 DB | 快取空值（存 `null` 短 TTL）；布隆過濾器（Bloom filter）先擋掉不存在的 key；參數校驗 |

## Q5. 快取失效策略總整理

### 三大失效手段

| 手段 | 做法 | 優點 | 缺點 / 適用 |
| --- | --- | --- | --- |
| **時間過期（TTL）** | 設 `max-age` / `revalidate` / Redis `EX` | 簡單、自動 | 有「最長 TTL 的資料延遲」，且可能雪崩。適合可容忍稍舊的資料 |
| **事件驅動（tag/path 失效）** | 資料變動時主動 `revalidateTag` / purge / `del` | 即時、精準 | 需在寫入路徑埋點，漏埋就髒。適合關聯明確的資料 |
| **版本化 key** | 內容 hash 檔名 + `immutable`；Redis key 加版本前綴 | 天然免失效、可永久快取、可瞬間全量切換 | 需能改引用（HTML/設定）。最適合靜態資源 |

實務常**組合使用**：靜態資源用版本化 key，動態資料用 TTL 打底 + 事件驅動精準失效。

### 各層快取總比較表

| 層級 | 快取什麼 | TTL 由誰控制 | 如何失效 | 適用情境 |
| --- | --- | --- | --- | --- |
| **Browser HTTP 快取** | 靜態資源、可協商的 HTML/API 回應 | 伺服器回應的 `Cache-Control`（`max-age`），瀏覽器執行 | 過期後協商（ETag/Last-Modified）；換 URL（hash） | 減少重複下載，強快取零往返 |
| **Service Worker** | 由 JS 決定的任何 GET 資源、離線資源 | 開發者在 SW 程式中自訂 | `activate` 清舊版快取；版本化 cache 名稱；程式邏輯 | PWA / 離線 / 自訂策略（SWR） |
| **CDN / Edge** | 靜態資源、可共享的頁面/API | 伺服器回應的 `s-maxage` / `stale-while-revalidate` | purge/invalidate；tag 失效；換 hash URL | 就近服務、擋掉大量回源流量 |
| **Server fetch cache（Next.js Data Cache）** | `fetch()` 的資料結果 | `next.revalidate`（時間）+ tag | `revalidateTag` / `revalidatePath`（事件驅動） | SSR/ISR 資料層，跨請求共用 |
| **Redis 應用層** | DB 查詢結果、運算結果、Session | 程式設定的 `EX`（TTL） | 寫入時 `del` key（事件驅動）；TTL 到期 | 降 DB 負載、熱點資料、cache-aside |

**核心心法**：資料越靜態、越能容忍延遲 → 往前層放（瀏覽器/CDN）並用版本化 key；資料越動態、越要即時 → 往後層放（Server/Redis）並用事件驅動失效。
