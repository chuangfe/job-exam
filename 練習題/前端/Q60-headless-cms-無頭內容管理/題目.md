# Headless CMS：無頭內容管理系統

現代前端（尤其 Jamstack / Next.js）常把「內容」與「呈現」分離，交由 Headless CMS 管理。這題聚焦在你能不能講清楚「為什麼要用 Headless、怎麼建模內容、怎麼跟 Next.js 整合、以及選型的取捨」，而不是背某個產品的 API。建議先用白話講出來，再對照解析。

## Q1. 傳統 CMS vs Headless CMS？

- 傳統 CMS（如 WordPress 傳統用法）的架構長怎樣？「內容、後台、前台模板」是怎麼綁在一起的？
- Headless CMS 把什麼「砍掉」了（head 指的是什麼）？它只負責什麼、不負責什麼？
- 為什麼前端 / Jamstack 特別偏好 Headless？（多前端消費、前端框架自由、CDN 靜態化、效能與安全）
- 什麼叫 API-first？REST 與 GraphQL 兩種內容 API 各有什麼取捨？
- Headless 有什麼代價 / 缺點？（要自己做前端、預覽、路由；非技術人員的體驗）

## Q2. 常見產品定位與取捨？

- 請說明以下產品各自的定位、代管 vs 自架、以及適合誰：
  - **Builder.io**（視覺化拖拉、所見即所得）
  - **Contentful**（老牌 SaaS、企業級、結構化內容）
  - **Sanity**（可程式化的 Studio、即時協作、GROQ 查詢）
  - **Strapi**（開源、可自架、Node.js）
  - **Git-based / 檔案型**（如 Contentlayer、以 MDX + Git 為內容源）
- 「代管 SaaS」與「開源自架」在成本、維運、鎖定（lock-in）上怎麼取捨？
- Git-based CMS 的內容存在哪裡？它適合什麼團隊、不適合什麼團隊？

## Q3. 內容建模（content modeling）？

- 什麼是 content type（模型）與 field（欄位）？請舉一個「部落格文章」的模型該有哪些欄位。
- 什麼是 reference（關聯）？舉例文章關聯到作者、分類。one-to-many 跟 many-to-many 怎麼處理？
- 常見欄位型別有哪些？（text、rich text / portable text、media、boolean、date、reference、slug…）
- 什麼是 localization（在地化 / i18n）？CMS 通常用哪兩種策略存多語系內容？
- 內容建模時常見的坑是什麼？（把樣式塞進內容、模型過度扁平或過度巢狀、缺少 slug）

## Q4. 與 Next.js 整合（抓取與更新）？

- 在 App Router 用哪種渲染策略配 CMS 最常見？SSG / ISR / SSR 各對應什麼內容更新需求？
- 內容在 CMS 改了之後，靜態頁怎麼「即時生效」？請說明兩種做法：時間型 `revalidate` 與事件型 **on-demand revalidation**（webhook + `revalidateTag` / `revalidatePath`）。
- `revalidateTag` 為什麼比 `revalidatePath` 適合 CMS 情境？tag 要怎麼在 `fetch` 上標？
- CMS 通常怎麼觸發重新驗證？（發佈時打你的 webhook route handler）
- 動態路由（`[slug]`）要怎麼從 CMS 撈全部 slug 來預先產生頁面？

## Q5. 預覽模式與視覺化即時預覽？

- 為什麼需要「預覽模式（draft / preview mode）」？發佈前的草稿怎麼在正式站看到、又不外流給一般訪客？
- Next.js 的 `draftMode()` 大致怎麼運作？它如何讓 `fetch` 改抓草稿內容？
- 什麼是視覺化即時預覽（live preview / visual editing）？這是 Builder.io、Sanity 主打的賣點——編輯者在 CMS 裡改，右邊的真實網站即時更新，這通常靠什麼技術達成？
- Builder.io 的「視覺化拖拉建站」跟一般結構化 CMS 差在哪？各自適合誰用、風險是什麼？

## Q6. 選型考量？

- 針對這些面向，你會怎麼幫團隊評估要選哪個 CMS：
  - 非技術人員能不能自助改內容 / 自己排版？
  - 開發導入成本與長期維運成本？
  - 廠商鎖定（lock-in）與內容可攜性（能不能匯出、搬家）？
  - 價格模型（席次、API 呼叫量、record 數）？
  - 在地化 / 多語系支援是否原生？
- 舉三種團隊情境，各推薦一個方向並說明理由：
  - 一人 side project / 技術部落格
  - 行銷團隊要能自己改活動頁、不想煩工程師
  - 中大型產品、多國語系、多前端（web + app）

## Q7. 實作：從 CMS 抓資料渲染一頁（Next.js 示意）？

- 請寫出一個 App Router 的文章頁：`generateStaticParams` 撈 slug、page 用帶 tag 的 `fetch` 抓單篇、再寫一個 webhook route handler 讓 CMS 發佈時 `revalidateTag`。
- 說明這段程式碼裡，哪一行決定了「內容更新即時生效」。
