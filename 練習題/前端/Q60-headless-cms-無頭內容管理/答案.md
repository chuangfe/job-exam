# Headless CMS：無頭內容管理系統 解析

## Q1. 傳統 CMS vs Headless CMS？

- **傳統 CMS（monolithic / coupled）**：以 WordPress 傳統用法為代表，把三件事綁在同一套系統：**內容資料庫、編輯後台、以及前台呈現的模板（theme / template）**。後端直接產出 HTML 給瀏覽器，內容與樣式是耦合的——換前端等於動整套系統。
- **Headless CMS**：把「head」（前台呈現層 / 模板）**砍掉**，只保留「body」——**內容儲存 + 編輯後台 + 一組內容 API**。它只負責「管理與吐出結構化內容（通常是 JSON）」，**完全不管內容長怎樣、用什麼框架渲染**。呈現交給你自己的前端。
- **為什麼前端 / Jamstack 偏好 Headless**：
  - **一份內容、多處消費（omnichannel）**：同一份內容可餵給 web、iOS/Android app、電子看板、電子報，不必各做一套後台。
  - **前端框架自由**：可用 Next.js / Astro / Nuxt 等任意框架，不被 PHP 模板綁死。
  - **效能與安全**：內容可在 build 時抓下來做成靜態頁（SSG）放 CDN，首屏快、又沒有傳統 CMS 那種「每次請求都打 DB + 渲染」的攻擊面（後台與正式站分離）。
- **API-first**：CMS 的第一等公民是 API 而非模板。兩種主流內容 API：
  - **REST**：直覺、快取友善（可用 HTTP 快取 / CDN），但常有 **over-fetching / under-fetching**（撈到多餘欄位，或要多打幾支 API 才湊齊關聯資料）。
  - **GraphQL**：一次查詢就能精準拿到需要的欄位與關聯（解決 N+1 / over-fetching），對巢狀關聯特別方便；代價是快取較複雜、學習曲線較高。（Sanity 另有自家的 GROQ 查詢語言，概念類似）
- **代價 / 缺點**：前端要**自己做**呈現、路由、SEO meta、預覽、以及讓非技術人員「所見即所得」的體驗——這些傳統 CMS 開箱即有。對純內容站、又沒有工程資源的團隊，Headless 反而是負擔。

## Q2. 常見產品定位與取捨？

| 產品 | 類型 | 定位 / 賣點 | 適合誰 |
|------|------|------------|--------|
| **Builder.io** | 代管 SaaS | **視覺化拖拉建站 + 所見即所得**，行銷可自己排版、A/B 測試，可綁定你的元件 | 行銷 / 成長團隊要自助改頁面、少依賴工程師 |
| **Contentful** | 代管 SaaS | 老牌**企業級**、結構化內容、穩定成熟、REST + GraphQL、多環境（環境分支） | 中大型企業、重視穩定與治理 |
| **Sanity** | 代管（內容） + 可自架 Studio | **可程式化的 Studio（React 寫成）**、即時協作、GROQ 查詢、Portable Text、強即時預覽 | 想高度客製編輯體驗、工程主導的團隊 |
| **Strapi** | **開源、自架**（Node.js） | 完全掌控、資料存自己 DB、無 API 用量費、可深度客製 | 有維運能力、要避免鎖定 / 資料主權 / 成本考量 |
| **Contentlayer / Git-based** | 檔案型 / Git-based | 內容是 repo 裡的 **MDX / Markdown**，用 Git 版本控制，build 時轉成型別安全的資料 | 技術部落格、文件站、內容作者就是開發者 |

- **代管 SaaS vs 開源自架**：
  - SaaS（Builder / Contentful / Sanity）：**導入快、免維運、免自己扛可用性**；但有**訂閱費（常按席次 / API 呼叫量 / record 數計價）**、資料在別人手上、有一定鎖定風險。
  - 開源自架（Strapi）：**無授權費、資料自持、可任意改**；但要**自己顧伺服器、資料庫、備份、升級、擴充與安全**，長期人力成本不一定比較低。
- **Git-based 的內容存在哪**：直接是 **Git repo 裡的檔案（Markdown / MDX / JSON）**。優點是版本控制、PR review、免額外服務、完全免費；**不適合**需要非技術人員頻繁改內容的團隊（他們不會用 Git），也不適合內容量極大或要複雜查詢的情境。

## Q3. 內容建模（content modeling）？

- **content type（模型）**：一種內容的結構定義，像資料庫的 table schema。**field（欄位）**：模型裡的一個屬性。以「部落格文章 `Post`」為例：

  - `title`（text）
  - `slug`（slug，網址用、唯一）
  - `excerpt`（text，摘要）
  - `body`（rich text / portable text，內文）
  - `coverImage`（media）
  - `author`（reference → `Author`）
  - `categories`（reference 陣列 → `Category`，many-to-many）
  - `publishedAt`（date）
  - `isFeatured`（boolean）

- **reference（關聯）**：欄位指向「另一筆內容」而不是把資料複製進來。
  - **one-to-many**：一位作者有多篇文章——在 `Post` 上放單一 `author` reference；作者換名字只改一處。
  - **many-to-many**：一篇文章多個分類、一個分類多篇文章——用 reference 陣列（或雙向關聯）。
  - 好處是**內容正規化、單一事實來源**，避免重複與不一致。
- **常見欄位型別**：text / number / boolean / date / **rich text 或 portable text**（結構化的富文本，能存標題、清單、內嵌圖片與元件，而非一坨 HTML）/ media（圖片影片）/ reference / slug / geo / array / object。
- **localization（在地化）**：讓同一筆內容有多語系版本。兩種主流策略：
  - **field-level（欄位層）**：同一筆 record，`title` 欄位本身存 `{ en, zh-TW }`。共用結構、關聯只維護一份，適合欄位對齊的翻譯。
  - **entry/document-level（文件層）**：每個語系是**獨立一筆 record**，用 locale 標記並互相關聯。適合各語系內容差異大、可獨立發佈的情況。
- **建模常見的坑**：
  - **把「樣式 / 排版」塞進內容**（例如在 rich text 裡硬寫顏色、寬度）——內容應該與呈現分離，樣式屬於前端。
  - **模型過度扁平**（所有東西塞一個大模型）或**過度巢狀**（層層 reference 難維護）。
  - **忘了 slug / 唯一鍵**，導致無法做穩定的動態路由。

## Q4. 與 Next.js 整合（抓取與更新）？

- **渲染策略對應**：
  - **SSG**：內容幾乎不變（文件、已定稿文章）→ build 時抓下來變靜態頁。
  - **ISR**：內容會變但可容忍些許延遲、且量大（部落格、商品頁）→ 靜態 + 定時 / 事件重新驗證。**這是 CMS 最常見的組合。**
  - **SSR（動態）**：需要即時、個人化（依登入者、即時庫存）→ 每次請求抓。
- **內容改了怎麼即時生效**，兩種做法：
  1. **時間型 `revalidate`（stale-while-revalidate）**：設一個秒數，過期後下次請求先給舊的、背景重抓。簡單，但有「最長 N 秒才更新」的延遲。
  2. **事件型 on-demand revalidation**：CMS **發佈時打你的 webhook**，你在 route handler 內呼叫 `revalidateTag` / `revalidatePath` **精準失效**對應快取——**發佈幾乎即時上線**，且不需要短 `revalidate` 一直重抓。這是理想解。
- **`revalidateTag` 為何更適合 CMS**：`revalidatePath` 要你**事先知道確切路徑**；但 CMS 一筆內容可能出現在很多頁（首頁列表、分類頁、文章頁）。用 **tag** 幫相關 `fetch` 標記（如 `['posts']`），發佈時 `revalidateTag('posts')` 就能**一次失效所有用到這份資料的頁面**，不必逐一列路徑。標法：`fetch(url, { next: { tags: ['posts'] } })`。
- **觸發方式**：在 CMS 後台設 webhook 指向你的 `app/api/revalidate/route.ts`，發佈 / 更新時自動 POST 過來（通常帶一個 secret 驗證）。
- **動態路由預產生**：用 `generateStaticParams` 從 CMS 撈全部 slug（見 Q7）。

## Q5. 預覽模式與視覺化即時預覽？

- **為什麼要預覽模式**：編輯者想在**發佈前**就看到草稿在正式站長怎樣，但**草稿不能讓一般訪客看到**（還沒定稿、可能是機密）。所以需要一個「只有帶特定憑證的人，才會看到草稿版內容」的機制。
- **Next.js `draftMode()`**：呼叫後會在瀏覽器種一個 **cookie**，開啟後該使用者的請求會被視為草稿模式；你的資料抓取程式碼可以據此**改抓 CMS 的草稿 API（含未發佈內容）並關閉快取**（強制動態、`cache: 'no-store'`），一般訪客沒有這個 cookie 就照常看正式內容。通常流程是：CMS 的「預覽」按鈕連到你的 `/api/draft` route → 驗 secret → `draftMode().enable()` → 導回文章頁。

```ts
// app/api/draft/route.ts
import { draftMode } from 'next/headers'
import { redirect } from 'next/navigation'

export async function GET(request: Request) {
  const { searchParams } = new URL(request.url)
  const secret = searchParams.get('secret')
  const slug = searchParams.get('slug')

  if (secret !== process.env.CMS_PREVIEW_SECRET) {
    return new Response('Invalid token', { status: 401 })
  }

  ;(await draftMode()).enable() // 種下草稿 cookie
  redirect(`/blog/${slug}`)      // 導回文章頁，此時會抓草稿內容
}
```

- **視覺化即時預覽（live / visual editing）**：編輯者在 CMS 左邊改欄位，**右邊嵌入的真實網站即時反映變化**，不用重新整理。常見達成方式：
  - CMS 把你的網站用 **iframe 嵌在編輯畫面**，兩邊透過 **`postMessage`** 溝通，編輯時把草稿資料即時推給 iframe 內的頁面重新渲染。
  - 或在頁面內埋 SDK（Sanity Presentation / Builder SDK），做欄位層級的「點哪個文字就跳到對應欄位」的 overlay 編輯。
- **Builder.io 的視覺化拖拉 vs 一般結構化 CMS**：
  - **Builder（視覺化建站）**：行銷可**自由拖拉排版、組合區塊、綁你註冊過的 React 元件**做出整個頁面，工程師只需提供元件庫。**賣點是行銷自助、上線快**。
  - **結構化 CMS（Contentful 等）**：編輯只填**預先定義好的欄位**，排版由前端固定決定。**一致性高、可控**。
  - **風險**：視覺化建站給編輯太多自由，容易做出**不一致、破版、難維護、SEO / a11y 失控**的頁面；治理不好會變技術債。要靠「限制可用元件與版位」來收斂。

## Q6. 選型考量？

- 評估面向：
  - **非技術人員自助程度**：Builder 拖拉最自由 → Contentful / Sanity 填結構化欄位 → Git-based（要會 Git / Markdown，基本上只有開發者能改）。
  - **開發導入 vs 長期維運成本**：SaaS 導入快但持續付訂閱；Strapi 自架初期省錢但要長期扛維運人力。
  - **鎖定與可攜性**：問「內容能不能整包匯出、搬到別家？」。Git-based / 開源可攜性最好（內容是你的檔案 / DB）；SaaS 依賴其 API 與資料格式，鎖定較深（Portable Text、GROQ 等專有格式要留意）。
  - **價格模型**：常見按 **席次（editor 數）／API 呼叫量／record（內容筆數）／locale 數** 計價；用量成長後可能陡增，選型時要估未來量。
  - **在地化**：多語系是否**原生支援**（field-level / document-level 內建），還是要自己拼。多國站務必列為硬需求。
- 三種情境建議：
  - **一人 side project / 技術部落格** → **Git-based（Contentlayer / MDX）**。免費、版本控制、內容即程式碼，作者本來就是開發者，不需要花俏後台。
  - **行銷團隊要自己改活動頁、不煩工程師** → **Builder.io**。視覺化拖拉 + A/B 測試 + 綁自家元件，行銷自助、工程只維護元件庫。
  - **中大型產品、多語系、多前端（web + app）** → **Contentful 或 Sanity**。結構化內容 + 原生 i18n + 穩定 API + omnichannel，治理與協作成熟；要高度客製編輯體驗選 Sanity，要企業級穩定與生態選 Contentful。若有資料主權 / 成本 / 客製後端需求且有維運能力，則考慮自架 **Strapi**。

## Q7. 實作：從 CMS 抓資料渲染一頁（Next.js 示意）

以下為 App Router 的部落格文章頁，示意「SSG 預產生 + tag 快取 + webhook on-demand 失效」的完整鏈路（CMS API 為示意）。

```tsx
// app/blog/[slug]/page.tsx  —— Server Component
import { draftMode } from 'next/headers'
import { notFound } from 'next/navigation'

const CMS_API = 'https://cms.example.com/api'

type Post = { slug: string; title: string; body: string; publishedAt: string }

// 1) build 時從 CMS 撈全部 slug，預先產生每篇文章的靜態頁
export async function generateStaticParams() {
  const posts: Post[] = await fetch(`${CMS_API}/posts?fields=slug`).then((r) => r.json())
  return posts.map((p) => ({ slug: p.slug }))
}

async function getPost(slug: string): Promise<Post | null> {
  const { isEnabled } = await draftMode() // 預覽模式？改抓草稿、關快取

  const res = await fetch(`${CMS_API}/posts/${slug}${isEnabled ? '?preview=1' : ''}`, {
    // 關鍵：一般模式標上 tag，讓 CMS 發佈時能精準失效這份資料
    next: isEnabled ? { revalidate: 0 } : { tags: ['posts', `post:${slug}`] },
  })
  if (!res.ok) return null
  return res.json()
}

export default async function BlogPost({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params
  const post = await getPost(slug)
  if (!post) notFound()

  return (
    <article>
      <h1>{post.title}</h1>
      <time dateTime={post.publishedAt}>{post.publishedAt}</time>
      <div dangerouslySetInnerHTML={{ __html: post.body }} />
    </article>
  )
}
```

```ts
// app/api/revalidate/route.ts —— CMS 發佈時打這支 webhook
import { revalidateTag } from 'next/cache'
import { NextResponse } from 'next/server'

export async function POST(request: Request) {
  const secret = new URL(request.url).searchParams.get('secret')
  if (secret !== process.env.CMS_REVALIDATE_SECRET) {
    return NextResponse.json({ message: 'Invalid secret' }, { status: 401 })
  }

  const body = await request.json() // CMS webhook payload，通常帶被改動的 slug
  revalidateTag('posts')            // 失效列表等用到 posts 的頁面
  if (body?.slug) revalidateTag(`post:${body.slug}`) // 精準失效單篇

  return NextResponse.json({ revalidated: true, now: Date.now() })
}
```

- **哪一行決定「內容更新即時生效」**：文章頁那支 `fetch` 上的 **`next: { tags: [...] }`** 幫快取貼了標籤，而 webhook 內的 **`revalidateTag('posts')` / `revalidateTag('post:slug')`** 在 CMS 發佈當下把對應快取失效——下一個訪客就會拿到新內容。兩者是一組：**有標 tag，才有辦法被精準 revalidate**。這比單靠時間型 `revalidate = 60` 更即時，也比 `revalidatePath` 更不必事先窮舉路徑。
