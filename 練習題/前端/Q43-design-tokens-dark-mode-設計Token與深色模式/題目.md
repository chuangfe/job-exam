# Q43 - Design Token 流程與 Dark Mode / 多品牌治理

設計系統（Design System）的「單一事實來源」不是元件，而是 **Design Token**——把顏色、間距、字級、圓角、陰影等設計決策抽成可命名、可轉譯的變數。有了 token，才能一次維護、多平台輸出（Web / iOS / Android）、一鍵切換品牌與深色模式。這題請你把「token 從定義到落地」的整條 pipeline 串起來：分層設計、CSS Variables 載體、Style Dictionary 產出、多品牌治理，以及 Dark Mode 的實作眉角（含避免切換閃爍）。

> 本題與 Q40（瀏覽器儲存）互補：Dark Mode 記憶偏好會用到 `localStorage` / cookie；本題聚焦「設計系統與主題治理」的視角。

## Q1. 什麼是 Design Token，以及三層分級

- 用一句話定義 Design Token，並說明「為什麼要有 token，而不是直接在元件裡寫死 `#1976d2`」。
- 說明三層分級：**primitive（原始值 / global / core）→ semantic（語意 / alias）→ component（元件）** 各層負責什麼、彼此如何引用。
- 舉例：`color.blue.500` → `color.action.primary` → `button.primary.background` 這條引用鏈，並說明命名慣例（category-type-item-state 的思路）。
- 為什麼「元件不應該直接引用 primitive，而要透過 semantic」？這對切換品牌 / 深色模式有什麼好處？

## Q2. CSS Variables 作為 token 載體

- 為什麼用 CSS custom properties（`--token`）當 token 的執行期載體，比 Sass 變數（編譯期）更適合做「主題切換」？
- 說明「切換主題只換變數值、不換 class、不重寫 DOM」為什麼能避免重繪整棵樹、且能讓 CSS 保持 DRY。
- CSS 變數的**繼承（cascade / inheritance）** 特性如何讓你只在 `:root` 或某個容器上覆寫變數，就能讓底下所有元件跟著變？
- 寫一段用 `:root` 定義 semantic 變數、`[data-theme="dark"]` 覆寫、元件端只引用 `var(--color-bg)` 的範例。

## Q3. Style Dictionary：單一來源、多平台輸出

- 說明 Style Dictionary 的核心概念：**單一 token 來源（JSON）→ transform → format → 多平台輸出**（CSS 變數 / JS 常數 / iOS `.swift` / Android `.xml`）。
- 為什麼「同一份 token 產出多平台」比「各平台各自維護一份顏色」重要？
- 給一份 token 來源 JSON（含 primitive 與用 `{...}` 引用 alias 的 semantic token）與一份 `config.json`（platforms / transformGroup / files）的範例。
- token 引用（reference / alias）在 build 時如何被解析成最終值？

## Q4. 多品牌（multi-brand）主題治理

- 多品牌的核心原則：**semantic 層固定、primitive 層抽換**。用這個原則說明「品牌 A / 品牌 B」如何共用同一組元件與 semantic token，只換底層 primitive 值。
- 為什麼「元件只認 semantic token」是多品牌能規模化的關鍵？如果元件到處引用 primitive 會發生什麼災難？
- 用 Style Dictionary 的角度說明多品牌怎麼組 build（共用 core + 各品牌 override 檔），或在 runtime 用不同 `[data-brand]` 覆寫 CSS 變數。

## Q5. Dark Mode 實作與避免閃爍

- `prefers-color-scheme` 媒體查詢如何偵測使用者的系統偏好？「跟隨系統 / 手動指定亮 / 手動指定暗」三種狀態該怎麼設計？
- 手動切換 + 記憶偏好：用 `localStorage` 記住選擇，並讓「跟隨系統」時仍能響應系統變化，資料要怎麼存？
- **避免切換閃爍（FOUC / flash of wrong theme）**：SSR / 靜態輸出時，為什麼會先閃一下亮色再變暗色？請說明用 `<head>` 內 **inline script**（或 cookie + SSR）在頁面繪製前就把 `data-theme` 設好的做法。
- 為什麼這段防閃爍 script 必須是**同步、inline、放在 body 繪製之前**，不能用一般的 React effect 處理？
