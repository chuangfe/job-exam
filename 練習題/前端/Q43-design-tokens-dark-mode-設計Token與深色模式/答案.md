# Q43 - Design Token 流程與 Dark Mode / 多品牌治理 解析

整條 pipeline 的心智圖：

```
① 設計決策
   └─ Design Token 來源（單一事實來源，JSON）
        primitive（原始值） → semantic（語意） → component（元件）
   ▼
② Style Dictionary（transform + format）
   ├─ Web  → CSS variables（--token）/ JS 常數
   ├─ iOS  → .swift
   └─ Android → .xml
   ▼
③ Web 執行期：CSS custom properties 當載體
   └─ 切主題 / 切品牌 = 只換 :root 上的變數值
   ▼
④ 框架對接：MUI colorSchemes / CssVarsProvider
   └─ 防閃爍：<head> inline script 先設好 data-theme
```

核心口訣：**「單一來源、分層引用、語意固定、原始可換、執行期只換變數」**。

---

## Q1. 什麼是 Design Token 與三層分級

**定義**：Design Token 是把設計決策（顏色、間距、字級、圓角、陰影、動畫時長…）抽出來、賦予**有意義名字**的最小單位變數。它是設計與工程之間的「合約」，也是設計系統的單一事實來源（single source of truth）。

**為什麼不寫死 `#1976d2`**：寫死的值散落各處、無法一次改、無法切主題/品牌、設計與程式對不上。token 把「值」與「用途」解耦——改一個地方，全站跟著變。

### 三層分級

| 層級 | 別名 | 內容 | 誰引用它 | 範例 |
| --- | --- | --- | --- | --- |
| **primitive** | global / core / option | 純粹的原始值，無語意 | 只被 semantic 引用 | `color.blue.500 = #1976d2` |
| **semantic** | alias / system | 描述「用途 / 意圖」，指向 primitive | 被 component 與元件引用 | `color.action.primary = {color.blue.500}` |
| **component** | — | 綁定到特定元件的細節 | 被該元件引用 | `button.primary.background = {color.action.primary}` |

引用鏈：

```
color.blue.500 (#1976d2)          ← primitive：這是什麼顏色
   ▲
color.action.primary              ← semantic：這個顏色拿來做「主要動作」
   ▲
button.primary.background         ← component：主要按鈕的背景用它
```

**命名慣例**：常見思路是 `category-type-item-variant-state`，例如 `color-text-primary`、`color-bg-surface-hover`、`space-inset-md`。重點是**語意層命名要描述用途而非外觀**（用 `color.text.primary` 而不是 `color.dark-grey`），這樣深色模式下把它指到淺色，名字仍然合理。

**為什麼元件要透過 semantic、不直接引 primitive**：
- 切深色模式時，只要把 `color.bg.surface` 從 `grey.50` 重新指到 `grey.900`，所有引用它的元件自動翻轉，不用動元件。
- 切品牌時，只換 primitive 的實際值，semantic 名字與元件完全不動。
- semantic 是一個**穩定的抽象層 / 收斂點**——這是整個治理策略的地基。

## Q2. CSS Variables 作為 token 載體

**為什麼比 Sass 變數適合切主題**：Sass 變數是**編譯期**常數，build 完就寫死進 CSS，無法在瀏覽器裡改；要做亮/暗兩套就得產兩份 CSS 或大量 override class。CSS custom properties 是**執行期**的、會參與 cascade、可被 JS 讀寫、且會**繼承**——天生適合「執行期切換」。

**只換變數、不換 DOM**：主題切換時我們只改 `:root`（或某容器）上少數變數值。瀏覽器重新計算用到這些變數的樣式即可，**不需要換 class、不需要重建 DOM、React 也不用重新 render 整棵樹**。這比「切一個 `.dark` class 讓大量選擇器重算」更輕、也讓 CSS 保持 DRY（顏色只寫一次 `var(--color-bg)`，不用寫 `.dark .foo { ... }` 一堆對照）。

**繼承特性**：custom property 會沿 DOM 繼承。只要在 `:root` 定義，底下任何元件都能 `var(--color-bg)` 取用；只要在某個容器覆寫，就能做「局部主題」（例如某區塊永遠深色）。

```css
/* semantic 變數在 :root（亮色為預設） */
:root {
  --color-bg: #ffffff;
  --color-surface: #f5f5f5;
  --color-text: #1a1a1a;
  --color-action-primary: #1976d2;
  --radius-md: 8px;
  --space-md: 16px;
}

/* 深色只覆寫「值」，變數名稱不變 */
:root[data-theme="dark"] {
  --color-bg: #121212;
  --color-surface: #1e1e1e;
  --color-text: #e0e0e0;
  --color-action-primary: #90caf9;
}

/* 元件端永遠只認 semantic 變數，對亮/暗一無所知 */
.card {
  background: var(--color-surface);
  color: var(--color-text);
  border-radius: var(--radius-md);
  padding: var(--space-md);
}
.button-primary {
  background: var(--color-action-primary);
}
```

切換就是一行：`document.documentElement.dataset.theme = 'dark'`。

## Q3. Style Dictionary：單一來源、多平台輸出

**核心概念**：Style Dictionary（Amazon 開源）吃一份**平台無關的 token JSON**，經過 **transform**（轉單位、轉色彩格式、駝峰/短橫線命名等）與 **format**（產出對應檔案格式），輸出到各平台。一份來源，多平台落地。

```
tokens/*.json ──▶ [transform] ──▶ [format] ──┬─▶ build/css/variables.css   (--color-...)
                                             ├─▶ build/js/tokens.js        (export const ...)
                                             ├─▶ build/ios/Colors.swift
                                             └─▶ build/android/colors.xml
```

**為什麼重要**：若各平台各自維護顏色，遲早會漂移（Web 的藍跟 iOS 的藍不一樣）。單一來源保證所有平台的設計決策一致，設計師改一次、全平台同步。

**token 來源 JSON**（primitive + 用 `{...}` 引用的 semantic）：

```json
{
  "color": {
    "blue":  { "500": { "value": "#1976d2" } },
    "grey":  { "50": { "value": "#f5f5f5" }, "900": { "value": "#121212" } }
  },
  "semantic": {
    "action": {
      "primary": { "value": "{color.blue.500.value}" }
    },
    "bg": {
      "surface": { "value": "{color.grey.50.value}" }
    }
  }
}
```

**`config.json`**（多平台輸出設定）：

```json
{
  "source": ["tokens/**/*.json"],
  "platforms": {
    "css": {
      "transformGroup": "css",
      "buildPath": "build/css/",
      "files": [
        { "destination": "variables.css", "format": "css/variables" }
      ]
    },
    "js": {
      "transformGroup": "js",
      "buildPath": "build/js/",
      "files": [
        { "destination": "tokens.js", "format": "javascript/es6" }
      ]
    },
    "ios": {
      "transformGroup": "ios",
      "buildPath": "build/ios/",
      "files": [
        { "destination": "Colors.swift", "format": "ios-swift/class.swift", "className": "Colors" }
      ]
    },
    "android": {
      "transformGroup": "android",
      "buildPath": "build/android/",
      "files": [
        { "destination": "colors.xml", "format": "android/colors" }
      ]
    }
  }
}
```

CSS 輸出會長這樣：

```css
:root {
  --color-blue-500: #1976d2;
  --color-grey-50: #f5f5f5;
  --semantic-action-primary: #1976d2;   /* alias 被解析成最終值 */
  --semantic-bg-surface: #f5f5f5;
}
```

**引用解析**：build 時 Style Dictionary 會遞迴解析 `{color.blue.500.value}` 這類 reference，把 alias 攤平成最終值再輸出（也可保留成 `var()` 引用，視 format 而定）。這讓你在來源裡維持「語意指向 primitive」的關係，輸出時各平台各取所需。

## Q4. 多品牌主題治理

**核心原則：semantic 固定、primitive 抽換。** 元件與 semantic token 是共用資產（品牌無關），品牌差異只落在 primitive 這一層。

```
        共用元件（button, card…）
                 │  只引用
                 ▼
        semantic token（固定名字：action.primary, bg.surface…）
                 │  指向
        ┌────────┴────────┐
   品牌 A primitive     品牌 B primitive
   blue.500=#1976d2     brand.500=#e91e63
```

**為什麼「元件只認 semantic」是規模化關鍵**：新增一個品牌 = 新增一份 primitive override，元件零改動。反之若元件到處寫 `color.blue.500`，換品牌就得逐一改元件——N 個品牌 × M 個元件的維護災難，且必漏改、必漂移。

**做法一：build 期分品牌**（Style Dictionary）

共用 `core/` + 各品牌 override，跑多次 build 產出多份 CSS：

```json
// brand-a：core + semantic + brand-a 的 primitive override
{
  "source": ["tokens/core/**/*.json", "tokens/brands/brand-a/**/*.json"],
  "platforms": { "css": { "transformGroup": "css", "buildPath": "build/brand-a/",
    "files": [{ "destination": "variables.css", "format": "css/variables" }] } }
}
```

品牌 B 只換 `tokens/brands/brand-b/`，semantic 與元件完全不動。

**做法二：runtime 覆寫**（單一 bundle 切品牌）

semantic 變數指向 primitive 變數，用 `[data-brand]` 換 primitive：

```css
:root {                              /* 預設品牌 A */
  --brand-500: #1976d2;
  --color-action-primary: var(--brand-500);   /* semantic 指向 primitive */
}
:root[data-brand="b"] {
  --brand-500: #e91e63;             /* 只換 primitive，semantic 自動跟著變 */
}
```

品牌與深色模式可正交組合：`[data-brand="b"][data-theme="dark"]`。

## Q5. Dark Mode 實作與避免閃爍

### 偵測系統偏好

```css
@media (prefers-color-scheme: dark) {
  :root { /* 系統偏好深色時的變數 */ }
}
```

JS 端：`window.matchMedia('(prefers-color-scheme: dark)')`，可 `.addEventListener('change', ...)` 監聽系統切換。

**三態設計**：偏好值存 `'light' | 'dark' | 'system'` 三種。`system` 時才去讀 `matchMedia` 並跟隨系統變化；`light`/`dark` 則忽略系統、用使用者指定值。實際套用到 DOM 的永遠是解析後的 `'light' | 'dark'`（寫到 `data-theme`）。

### 手動切換 + 記憶偏好

```ts
type ThemePref = 'light' | 'dark' | 'system';

function resolveTheme(pref: ThemePref): 'light' | 'dark' {
  if (pref === 'system') {
    return window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
  }
  return pref;
}

function applyTheme(pref: ThemePref) {
  localStorage.setItem('theme', pref);              // 記住「使用者的選擇」（含 system）
  document.documentElement.dataset.theme = resolveTheme(pref);
}

// 若使用者選 system，仍要響應系統變化
window.matchMedia('(prefers-color-scheme: dark)').addEventListener('change', () => {
  if ((localStorage.getItem('theme') ?? 'system') === 'system') {
    document.documentElement.dataset.theme =
      window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
  }
});
```

要點：存的是**偏好（含 `system`）**，不是解析後的值——否則使用者選了「跟隨系統」卻無法再跟隨。

### 避免切換閃爍（FOUC / flash of wrong theme）

**問題**：SSR / 靜態輸出的 HTML 在伺服器端不知道使用者的 `localStorage`，只能先送出「預設亮色」。等 JS 載入、React hydrate 後才讀 `localStorage` 改成暗色——中間這段使用者會看到**亮色閃一下再變暗**。

**解法：在 `<head>` 放同步 inline script**，在頁面內容繪製之前就把 `data-theme` 設好：

```html
<head>
  <script>
    (function () {
      try {
        var pref = localStorage.getItem('theme') || 'system';
        var dark = pref === 'dark' ||
          (pref === 'system' &&
           window.matchMedia('(prefers-color-scheme: dark)').matches);
        document.documentElement.setAttribute('data-theme', dark ? 'dark' : 'light');
      } catch (e) {}
    })();
  </script>
</head>
```

**為什麼必須同步、inline、在 body 繪製前**：
- 瀏覽器解析 HTML 是**由上而下**的。放在 `<head>` 的同步 script 會**阻塞後續 DOM 的繪製**，因此能保證「`data-theme` 已正確」才畫出 body——沒有錯誤主題可閃。
- 若改用 React `useEffect`：effect 在 hydration **之後**才跑，那時第一幀（亮色）早已上畫面，閃爍已發生。inline script 是唯一能「在第一幀之前」介入的時機。
- 不能用外部 `<script src>`（非同步載入、時機不保證）；必須是 inline 同步。
- **SSR + cookie 變體**：把偏好也存一份到 cookie，伺服器端就能在 render 時直接把 `data-theme` 寫進 `<html>`，連 inline script 都省了（Next.js App Router 可在 `layout` 讀 cookie）。cookie 方案的好處是伺服器輸出的 HTML 一開始就正確。

## Q6. token 與 MUI theme 對接

**傳統 `createTheme` 的落差**：舊做法把顏色放在 `palette`，深色/亮色是**兩個不同的 theme 物件**，切換靠 `ThemeProvider` 換 theme → 觸發 context 變動 → **整棵樹 re-render**，且 SSR 下無法在繪製前決定主題（會閃爍）。

**MUI CSS variables 方案（v6/v7）**：MUI 改用 CSS custom properties 承載主題，切換只換 `data-mui-color-scheme` 屬性、由 CSS 變數翻轉，**不再 re-render 整棵樹**，也能配合防閃爍 script。

```ts
import { createTheme } from '@mui/material/styles';

const theme = createTheme({
  cssVariables: {
    colorSchemeSelector: 'data',   // 用 [data-mui-color-scheme="dark"] 切換
  },
  colorSchemes: {
    light: {
      palette: {
        // 把 Style Dictionary 產出的 semantic token 餵進來
        primary: { main: tokens.semantic.action.primary },        // e.g. #1976d2
        background: { default: tokens.semantic.bg.default,
                      paper: tokens.semantic.bg.surface },
      },
    },
    dark: {
      palette: {
        primary: { main: tokensDark.semantic.action.primary },    // e.g. #90caf9
        background: { default: '#121212', paper: '#1e1e1e' },
      },
    },
  },
});
```

各角色的職責：

| API | 做什麼 |
| --- | --- |
| `cssVariables: true` | 讓 MUI 以 CSS 變數輸出主題（`--mui-palette-primary-main` 等） |
| `colorSchemes: { light, dark }` | 一次定義亮/暗兩套 palette，MUI 生成兩組變數 |
| `CssVarsProvider`（或 v6 起的 `ThemeProvider` + `cssVariables`）| 注入 CSS 變數、提供 `useColorScheme()` 供手動切換 |
| `useColorScheme()` | 讀寫 `mode`（`'light' \| 'dark' \| 'system'`），自動記憶到 localStorage |
| `InitColorSchemeScript` | **就是 Q5 的防閃爍 inline script**，MUI 官方版本 |

**token → MUI**：把 Style Dictionary 輸出的 JS token（`build/js/tokens.js`）import 進來，對映到 `palette` 的 `primary/secondary/background/text` 等角色，讓「設計系統 semantic token」成為 MUI 主題的唯一顏色來源——設計改 token、重新 build，MUI 主題自動更新。

**`InitColorSchemeScript` 如何解決閃爍**（Next.js App Router）：把它放進 root layout 的 `<body>` 最前面，它會在 hydration 前**同步**讀 localStorage 並設好 `data-mui-color-scheme`，原理與 Q5 的 inline script 完全相同，只是由 MUI 幫你產生：

```tsx
// app/layout.tsx
import { InitColorSchemeScript } from '@mui/material/InitColorSchemeScript';

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html suppressHydrationWarning>
      <body>
        <InitColorSchemeScript attribute="data" />  {/* 繪製前先設好色彩模式 */}
        {children}
      </body>
    </html>
  );
}
```

（`suppressHydrationWarning` 是因為 script 會在 client 端修改 `<html>` 屬性，需告訴 React 這裡的差異是預期的。）

## 一句話總結

Design Token 用 **primitive → semantic → component** 三層把設計決策收斂成單一來源；用 **Style Dictionary** 一份輸出多平台；在 Web 端以 **CSS Variables** 當執行期載體，讓切主題/切品牌只換變數值；**多品牌**靠「semantic 固定、primitive 抽換」規模化；**Dark Mode** 靠 `prefers-color-scheme` + 記憶偏好，並用 `<head>` inline script（或 cookie）在繪製前設好 `data-theme` 消除閃爍；最後透過 MUI 的 `colorSchemes` + `cssVariables` + `InitColorSchemeScript` 把 token 落地成可切換、不閃爍、不重繪整棵樹的主題系統。
