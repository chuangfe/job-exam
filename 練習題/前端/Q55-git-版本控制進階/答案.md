# Q55 - Git 版本控制（進階／實戰）解析

Git 的核心心智模型只有一句話：**一連串「快照」，靠指標（HEAD、branch、tag）指來指去**。把這句話吃透，後面所有「怎麼救」都只是「移動指標」或「產生新快照」而已。

```
blob（檔案內容）
  └─ 被 tree（目錄結構）引用
        └─ 被 commit（一次快照 + 作者 + 訊息 + parent）引用
              └─ 被 branch / tag / HEAD（指標）指向
```

---

## Q1. Git 物件模型

- **blob**：存「檔案的內容」本身（不含檔名、不含路徑），以內容做 SHA-1/SHA-256 雜湊，內容相同的檔案只存一份。
- **tree**：存「一層目錄的結構」——一份清單，記錄這層有哪些檔名對應哪個 blob、哪些子目錄對應哪個 tree。
- **commit**：一次完整快照。它指向「最頂層的 tree（代表整個專案當下的樣子）」＋ 作者/committer/時間/訊息 ＋ 一個或多個 **parent commit**。

**「快照而非差異」**：每次 commit 都記錄「整個專案在那一刻的完整樣貌」（透過指向一棵 tree）。未變動的檔案不會重存，而是重複指向同一個 blob，所以並不浪費空間。這跟 SVN 那種「存每次的 delta，要重建某版本得層層套用差異」不同——Git 要看任一版本都是 O(1) 直接取那個快照。（`git diff` 的差異是「即時算出來的」，不是儲存形式。）

- **HEAD**：一個指向「你目前所在位置」的指標。正常情況下 HEAD 指向某個 branch（例如 `ref: refs/heads/main`），branch 再指向某個 commit。
- **detached HEAD**：當你 `git checkout <某個 commit hash 或 tag>`，HEAD 直接指向一個 commit 而非某條 branch。此時的新 commit 不屬於任何分支，切走後容易「遺失」（要靠 reflog 救）。
- **branch 只是指標**：一條分支本質上就是一個 41 bytes 的檔案，裡面存一個 commit hash。所以「開分支」只是寫一個新指標、「切分支」只是移動 HEAD，因此極快，跟專案大小無關。
- **parent**：一般 commit 有 1 個 parent（前一個 commit）；**merge commit 有 2 個（或以上）parent**；第一個 commit（root commit）沒有 parent。

```bash
# 直接窺看物件模型
git cat-file -t HEAD        # commit
git cat-file -p HEAD        # 看這個 commit 指向的 tree、parent、作者、訊息
git cat-file -p HEAD^{tree} # 看該 tree 底下的 blob/子 tree 清單
git log --oneline --graph --all   # 用圖形看指標與 parent 關係
```

## Q2. 分支策略與 PR 流程

| 策略 | 分支結構 | 適合 | 缺點 |
| --- | --- | --- | --- |
| **Git Flow** | `main` + `develop` + 多條 `feature`/`release`/`hotfix` | 有明確版本、定期發佈（如套裝軟體、App 商店版本） | 分支多、流程重，對持續部署的 Web 服務過度複雜 |
| **GitHub Flow** | 只有 `main` + 短命 `feature`，PR 合進 main 就部署 | 持續部署的 Web 產品、中小團隊 | 缺乏正式的 release/hotfix 隔離，靠 CI 與 revert 補 |
| **Trunk-Based** | 大家幾乎都往 `main`（trunk）提交，分支存活數小時～1、2 天 | 高頻整合、成熟 CI/CD、大團隊 | 需要強測試自動化與 **feature flag**，紀律要求高 |

- **Git Flow 各分支角色**：`main` 放正式上線版本（每個點都可發佈）、`develop` 是整合中的下一版、`feature/*` 從 develop 開出開發新功能、`release/*` 為發佈前的凍結測試、`hotfix/*` 從 main 開出緊急修線上 bug。缺點是分支多、合併方向複雜，對「一天上線多次」的服務太重。
- **PR 標準流程**：從最新的 base 開 feature 分支 → 開發並推上遠端 → 開 Pull Request → CI 跑測試 + 同事 code review → 修正 → 核准後合併 → 刪除 feature 分支。**PR 是「程式碼進主幹前的品質關卡與討論場」**：審查、討論、留下決策紀錄、跑自動化檢查。
- **PR 合併三方式**：
  - **Merge commit（`--no-ff`）**：保留 feature 的所有 commit ＋ 一個 merge commit。歷史完整但較雜。
  - **Squash and merge**：把整條 feature 壓成「一個 commit」進 main。main 歷史乾淨、一個 PR 一個 commit；缺點是遺失分支內的細粒度歷史。
  - **Rebase and merge**：把 feature 的 commit 逐一接到 main 尾端，線性、無 merge commit；缺點是改寫了這些 commit 的 hash。
- **為何轉向 trunk-based / 短命分支**：分支活越久 → 跟 main 分歧越大 → 合併越痛（大衝突、整合地獄）。短命分支＋頻繁整合能降低衝突，但「還沒做完的功能也在 main 上」得用 **feature flag** 把未完成功能藏起來，並靠 CI/CD 保證 main 隨時可部署。

## Q3. merge vs rebase

- **歷史長相**：`merge` 保留真實的分叉與合併，產生一個 merge commit，歷史呈「有分岔的圖」；`rebase` 把你的 commit「搬去接在目標分支的最新 commit 之後」，歷史變成一條**直線**。
- **適用**：
  - `merge`：整合他人/共享分支的變更、想忠實保留「什麼時候從哪分岔、又合回來」的紀錄。
  - `rebase`：整理**自己本地、還沒推送**的 commit，讓歷史乾淨線性後再開 PR。
- **黃金守則**：**不要 rebase 已經 push / 被別人共享的 commit。** 因為 rebase 會**改寫 commit（產生新的 hash）**。若別人已經基於舊 commit 工作，你改寫後歷史對不上，push 得用 `--force`，會覆蓋遠端、害協作者的分支分裂、被迫做痛苦的手動修復。原則：**rebase 只動「只有你自己有」的提交。**
- **fast-forward 家族**：
  - `--ff`（預設）：若 main 沒有新 commit、只是落後於 feature，直接把指標往前挪，**不產生 merge commit**。
  - `--no-ff`：即使可以 ff 也**強制產生 merge commit**，保留「這裡曾有一條分支」的紀錄（很多團隊主分支合併統一用它）。
  - `--ff-only`：只允許 ff，否則直接失敗（用於「我要確保是線性、不要意外 merge commit」）。

```bash
# 情境：feature 開發時 main 已被推進，想基於最新 main
# 作法 A：merge（保留分叉，安全，適合已共享的分支）
git checkout feature
git merge main            # feature 上多一個 merge commit

# 作法 B：rebase（線性乾淨，僅限 feature 只有你自己在動）
git checkout feature
git rebase main           # feature 的 commit 被搬到 main 尾端之後
# rebase 後若之前 push 過，需要（安全的）強推
git push --force-with-lease
```

> `--force-with-lease` 比 `--force` 安全：只有在「遠端還是你上次看到的樣子」時才覆蓋，若有人偷偷推了新東西會擋下，避免蓋掉別人的工作。

## Q4. 互動式 rebase 整理提交

`git rebase -i <base>` 會打開一個「待辦清單」，由上而下列出 base 之後的每個 commit，你逐行改動作：

| 指令 | 作用 |
| --- | --- |
| `pick` | 保留這個 commit（預設） |
| `reword` | 保留變更，只**改 commit 訊息** |
| `edit` | 停下來讓你**修改內容**（可改檔案、拆分 commit） |
| `squash` | 併入上一個 commit，並**合併兩者的訊息**（會讓你編輯） |
| `fixup` | 併入上一個 commit，但**丟棄這個 commit 的訊息** |
| `drop` | **刪掉**這個 commit（等同刪那一行） |
| `reorder` | 直接調整清單行的順序即可重排提交 |

- **squash vs fixup**：兩者都是「合併進上一個 commit」，差別只在**訊息**——squash 會把兩則訊息都帶出來給你編輯合併，fixup 直接沿用上一個 commit 的訊息、丟棄自己的。「我只是補修上一個 commit」通常用 fixup。

```bash
# 情境：整理 feature 上最近 5 個雜亂 commit
git rebase -i HEAD~5
# 編輯器內範例：
#   pick   a1b2c3 feat: 新增登入表單
#   fixup  d4e5f6 fix typo         <- 併入上一個、丟訊息
#   fixup  g7h8i9 wip              <- 併入上一個
#   pick   j0k1l2 feat: 串接登入 API
#   fixup  m3n4o5 oops             <- 併入上一個
# 存檔離開 -> 得到 2 個乾淨的原子提交
```

- **`--fixup` + `--autosquash` 工作流**：發現「某個舊 commit `a1b2c3` 有小錯要補」時，不必手動排順序：

```bash
git add .
git commit --fixup=a1b2c3        # 產生一個標記為 "fixup! ..." 的 commit
git rebase -i --autosquash a1b2c3~1   # Git 自動把 fixup commit 排到正確位置並設為 fixup
```

它解決了「手動在 rebase 清單裡搬行、對應 hash」的麻煩，讓補修可以自動歸位。

- **rebase 途中控制**：

```bash
git rebase --continue   # 解完衝突後繼續
git rebase --skip       # 跳過當前這個 commit
git rebase --abort      # 完全放棄，回到 rebase 前的狀態（最安全的後路）
```

## Q5. cherry-pick、revert vs reset

- **`git cherry-pick <hash>`**：把「某個特定 commit 的變更」複製一份、套用到目前分支（產生新 hash）。情境：hotfix 分支上修好的一個 commit，要單獨挑進 `main` 與 `develop`；或某個 commit 誤入某分支，想把它「撿」到正確分支。

```bash
git cherry-pick a1b2c3            # 套用單一 commit
git cherry-pick a1b2c3^..d4e5f6   # 套用一段範圍
```

- **revert vs reset 本質差別**：
  - **`git revert <hash>`**：**新增一個「反向操作」的 commit** 來抵銷指定 commit 的效果。歷史往前走、不改寫既有 commit → **安全，可用於公開/共享分支。**
  - **`git reset <hash>`**：把分支指標「往回搬」，讓後面的 commit 從歷史上「消失」→ **改寫歷史，危險，別用在已共享的分支。**
- **為何撤銷「已推到共享分支」的提交要用 revert**：reset 會讓遠端歷史與同事本地分裂，得強推、害人重建；revert 只是往前加一個新 commit，人人 `pull` 即可，安全無痛。
- **reset 三模式**（差在「動到哪幾層」）：

| 模式 | 移動 HEAD/branch | 暫存區(index) | 工作目錄(檔案) | 用途 |
| --- | --- | --- | --- | --- |
| `--soft` | ✅ | 不動（變更留在暫存） | 不動 | 撤銷 commit 但保留全部變更、準備重新 commit |
| `--mixed`（預設） | ✅ | ✅ 重設 | 不動 | 撤銷 commit 與 add，變更退回工作目錄 |
| `--hard` | ✅ | ✅ 重設 | ✅ **覆蓋** | 全部丟棄，回到某 commit 的乾淨狀態 |

**`--hard` 最危險**：它會直接覆蓋工作目錄裡「還沒 commit」的修改，這些改動沒有進過 Git 物件庫，reflog 也救不回來（reflog 只救 commit，不救未提交的工作目錄變更）。動手前務必確認沒有想留的未提交內容。

```bash
# 情境：剛 commit 完，發現漏檔＋想改訊息（不要多一個 commit）
git add 漏掉的檔案.js
git commit --amend                # 把新變更併入上一個 commit，並可改訊息
git commit --amend --no-edit      # 只補檔案、訊息不變
# 注意：amend 也是改寫歷史，若上一個 commit 已 push 就等同前述禁忌
```

## Q6. 衝突解決流程與心法

- **衝突標記**：

```
<<<<<<< HEAD
目前這一側（你所在分支）的內容
=======
另一側（要併進來/被 rebase 到的）內容
>>>>>>> other-branch
```

`=======` 是分隔線；上半 `<<<<<<<` 到 `=======` 是一側，下半到 `>>>>>>>` 是另一側。**解決 = 自己決定最終該長怎樣（可能兩邊都要、或重寫），把標記整段刪掉。**

- **衝突解決 SOP**：

```bash
git merge main            # 或 git rebase main，跳出 CONFLICT
git status                # 看哪些檔案 "both modified"（衝突檔）
# 打開衝突檔逐一處理，刪掉 <<< === >>> 標記，留下正確內容
git add 已解決的檔案       # 標記「這個衝突解好了」
# merge 的情況：
git commit                # 完成 merge commit（訊息通常已預填）
# rebase 的情況：
git rebase --continue     # 繼續套用剩下的 commit
```

- **rebase 時方向會反過來**：merge 時 `HEAD` 是「你的分支」很直覺；但 **rebase 時 Git 是「拿你的 commit 一個個套到目標分支上」，所以 `<<<<<<< HEAD` 那側其實是「目標分支（例如 main）」，下面才是「你正在套用的那個 commit」**。這常讓人 ours/theirs 搞反，務必看清楚。
- **輔助工具**：

```bash
git mergetool                 # 開圖形化三方比對工具
git checkout --ours 檔案       # 整檔採用「我方」（merge 語意）
git checkout --theirs 檔案     # 整檔採用「對方」
git config --global rerere.enabled true   # 記住你解過的衝突，下次同樣衝突自動套用
```

- **減少衝突心法**：小步、頻繁地把主幹合/rebase 進來（別讓分支活太久）；一次只改一件事；團隊約好格式化規則（避免大規模 reformat 洗掉整檔）；模組邊界清楚、避免多人狂改同一檔同一段。

## Q7. 救援工具：stash、reflog、bisect

- **`git stash`**：把目前「未 commit 的修改」暫時收進一個堆疊、讓工作目錄回到乾淨狀態，事後再取回。
  - `stash pop`＝套用最新一筆並**從堆疊移除**；`stash apply`＝套用但**保留**在堆疊（想套到多個分支時用）。
  - 預設不含未追蹤檔，要一起收得加 `-u`（untracked）或 `-a`（含 ignored）。

```bash
# 情境：手上一堆未 commit 的改動，要緊急切去修 bug
git stash -u                 # 連未追蹤檔一起收好，工作目錄變乾淨
git switch hotfix            # 去修 bug、commit、push
git switch feature
git stash pop                # 取回剛剛的工作
git stash list               # 查看所有暫存
```

- **`git reflog`**：記錄「HEAD（和分支）每一次移動」的本地日誌——commit、reset、rebase、checkout、amend 全都留痕。即使某 commit 已經不被任何分支指到，只要還在 reflog 裡（預設保留約 90 天、GC 前）就能救回。**它是 Git 的後悔藥。**

```bash
# 情境：不小心 git reset --hard 到很前面 / 刪了分支，commit 好像不見了
git reflog                        # 找出「災難發生前」那個 commit 的 hash，例 a1b2c3
git reset --hard a1b2c3           # 把目前分支拉回那個狀態
# 或救回被刪的分支：
git branch 救回的分支 a1b2c3
```

- **`git bisect`**：用**二分搜尋**在一段歷史中定位「哪一個 commit 開始出現 bug」。給它一個「好」與一個「壞」的端點，它每次 checkout 中間點讓你測，log₂ 次就能鎖定兇手。

```bash
git bisect start
git bisect bad                 # 目前是壞的
git bisect good v1.2.0         # 這個舊版本是好的
# Git 自動 checkout 中間 commit -> 你測試 -> 回報：
git bisect good   # 或 git bisect bad
# ...重複幾次，Git 印出「第一個壞掉的 commit」
git bisect reset               # 結束，回到原本位置
# 進階：可自動化
git bisect run npm test        # 用測試指令的退出碼自動判定 good/bad
```

## Q8. 好的 commit 訊息與原子提交

- **原子提交**：一個 commit 只完成「一件邏輯上完整、獨立」的事。好處：容易 review、容易 `revert`（撤一件事不會牽連別的）、容易 `cherry-pick`、`git bisect` 能精準定位、歷史可讀。
- **Conventional Commits 格式**：

```
<type>(<scope>): <subject>

<body：說明「為什麼」與背景，非必填>

<footer：BREAKING CHANGE、關聯 issue，如 Closes #123>
```

常見 type：`feat`（新功能）、`fix`（修 bug）、`docs`（文件）、`style`（格式，不影響邏輯）、`refactor`（重構）、`perf`（效能）、`test`（測試）、`build`、`ci`、`chore`（雜項）。範例：`feat(auth): 新增 Google 登入`、`fix(cart): 修正數量為 0 時仍可結帳`。

- **好訊息的規則**：subject 用**祈使句**（"add" 而非 "added"）、約 50 字內、句尾不加句號；body 說明**「為什麼要這樣改、解決什麼問題」**（做了什麼看 diff 就知道，動機才是 diff 看不出來的）；一行空行分隔 subject 與 body。
- **自動化好處**：type 對應**語意化版本（SemVer）**——`fix`→patch、`feat`→minor、`BREAKING CHANGE`→major；工具（semantic-release、standard-version）可據此**自動 bump 版本號、自動產生 CHANGELOG**。
- **`git add -p`（patch 模式）**：讓你在同一個檔案裡「挑選要提交哪幾個 hunk」，把混在一起的修改拆成多個原子提交。是實現原子提交的利器。

## Q9. `.gitignore`、tag、`git blame`

- **`.gitignore`**：列出「不要被 Git 追蹤」的檔案/樣式（`node_modules/`、`.env`、`dist/`、`*.log`…）。**關鍵坑**：`.gitignore` 只對「尚未被追蹤」的檔案有效；**已經被追蹤的檔案加進去沒用**，得先把它從追蹤中移除：

```bash
git rm --cached .env          # 停止追蹤但保留本地檔案
echo ".env" >> .gitignore
git commit -m "chore: 停止追蹤 .env"
```

- **tag 兩種**：
  - **輕量標籤**：只是一個指向 commit 的名字（`git tag v1.0.0`），無額外資訊。
  - **附註標籤**：完整物件，含**標籤者、日期、訊息、可簽章**（`git tag -a v1.0.0 -m "..."`）。**發佈版本一律用附註標籤。**
  - tag **預設不會**被 `git push` 推上去，要 `git push origin v1.0.0` 或 `git push --tags`。
- **`git blame`**：顯示某檔案「每一行最後是由哪個 commit／作者改的」。正經用途：追一段可疑邏輯「當初為何這樣寫」→ 找到 commit → 看訊息與 PR 脈絡；不是為了找戰犯，是為了找**上下文**。

```bash
git blame -L 40,60 檔案.js          # 只看第 40~60 行
# 略過「大型格式化」那次雜訊 commit：
git blame --ignore-rev <格式化的hash> 檔案.js
# 或建 .git-blame-ignore-revs 檔案，長期忽略多個雜訊 commit
git config blame.ignoreRevsFile .git-blame-ignore-revs
```

## Q10. 綜合情境題：常見「怎麼救」對照

- **commit 到了錯的分支（該在 feature 卻 commit 到 main，尚未 push）**：

```bash
# 假設有 3 個 commit 誤入 main
git branch feature            # 先在目前位置(含這3個commit)標一條 feature
git reset --hard HEAD~3       # 讓 main 退回乾淨狀態（那3個commit仍在 feature 上）
git switch feature            # 切過去繼續開發
# （若 feature 早已存在：用 cherry-pick 把 commit 撿過去，再 reset main）
```

- **把最後一次 commit 拆成兩個**：

```bash
git reset HEAD~1              # 用預設 --mixed 撤銷該 commit，變更退回工作目錄
git add 第一部分檔案
git commit -m "第一個原子提交"
git add 第二部分檔案
git commit -m "第二個原子提交"
```

- **把分散的兩三個 commit 合併成一個**：

```bash
git rebase -i HEAD~3
# 把後面幾行的 pick 改成 squash（或 fixup），存檔即可
```

- **誤 push 含密碼/token 的檔案（觀念）**：先假設「該 secret 已外洩，立刻到平台把它撤銷/輪替」——這比清歷史更重要。接著用 `git filter-repo`（或 BFG Repo-Cleaner）把該檔案從**整段歷史**移除、force push，並通知協作者重新 clone。單純 `git rm` 只是刪未來版本，歷史裡還在。

```bash
git rm --cached secrets.env && git commit -m "chore: 移除密鑰檔"  # 只擋未來，不夠
# 徹底清歷史（會改寫全歷史，需協調團隊）：
git filter-repo --path secrets.env --invert-paths
```

- **暫時看三個 commit 前的程式碼、看完回來（不改動）**：

```bash
git switch --detach HEAD~3    # 進 detached HEAD 看舊版
# 看完
git switch -                  # 回到剛剛的分支
```

- **抓遠端最新變更但不要一堆 merge commit**：

```bash
git pull --rebase             # 用 rebase 取代 merge，保持線性歷史
# 想永久預設：
git config --global pull.rebase true
```

---

**核心心法**：Git 一切都是「快照 + 指標」。改壞了先別慌——只要 commit 曾經存在，`git reflog` 幾乎都救得回；真正救不回的只有「從沒 commit 過就被 `--hard` / 清掉的工作目錄變更」。對**共享分支**永遠用「往前加」的 `revert`／`merge`，把改寫歷史的 `reset`／`rebase`／`amend` 留給「只有自己有」的本地提交，就不會害到隊友。
