# Git 版本控制（進階／實戰）

請依序回答以下 Git 進階問題。Git 幾乎是所有工程師每天都在用、卻很少「真的搞懂」的工具——大多數人只會 `add` / `commit` / `push`，一遇到「改到錯分支」「提交訊息打錯」「不小心 reset 掉東西」就慌了。這一題的目標是把 Git 從「魔法咒語」變成「有心智模型、能自救」的技能。建議先自己用白話講出來、甚至打開 terminal 實測，再對照解析。

> 提醒：本題偏概念＋實戰指令，情境題請試著自己寫出「你會下哪幾行指令」，這正是面試與日常最常被考的部分。

## Q1. Git 物件模型：Git 到底存了什麼？

- Git 的三種核心物件：**blob**、**tree**、**commit** 分別存什麼？彼此如何組成一次快照（snapshot）？
- 常聽到「Git 存的是快照，不是差異（diff）」，這句話是什麼意思？跟舊式版本控制（如 SVN 存 delta）有何不同？
- **HEAD** 是什麼？它平常指向誰？什麼情況會進入 **detached HEAD**？
- 「branch 只是一個指標（pointer）」這句話怎麼理解？為什麼在 Git 建立/切換分支這麼快？
- `git commit` 時，parent 是誰？merge commit 有幾個 parent？

## Q2. 分支策略與 PR 流程

- 請比較 **Git Flow**、**GitHub Flow**、**Trunk-Based Development** 三種策略：核心分支長什麼樣、適合什麼團隊/發佈節奏。
- Git Flow 的 `main` / `develop` / `feature` / `release` / `hotfix` 各自的角色是什麼？它的缺點是什麼？
- 什麼是「feature branch → Pull Request → code review → merge」的標準流程？PR 在其中扮演什麼角色？
- PR 合併時常見的三種方式：**Merge commit**、**Squash and merge**、**Rebase and merge** 差在哪？各自的優缺點？
- 為什麼很多團隊改用短命分支（short-lived branch）＋ trunk-based？跟 CI/CD、feature flag 有什麼關係？

## Q3. merge vs rebase

- `git merge` 與 `git rebase` 在「提交歷史長相」上最大的差別是什麼？
- 各自適合什麼情境？（提示：整合他人變更、整理自己還沒推送的提交）
- **rebase 的黃金守則**是什麼？為什麼「不要 rebase 已經推送/共享的公開分支」？違反會發生什麼事？
- `git merge --ff`（fast-forward）、`--no-ff`、`--ff-only` 分別是什麼行為？什麼時候會/不會產生 merge commit？
- 情境：你在 `feature` 開發時，`main` 已經被別人推進了好幾個 commit。你想讓 `feature` 基於最新的 `main`，**你會下什麼指令**？merge 進來與 rebase 上去，歷史各會長怎樣？

## Q4. 互動式 rebase（`rebase -i`）整理提交

- `git rebase -i` 可以做哪些事？請說明 **pick / reword / edit / squash / fixup / drop** 各自的用途。
- **squash** 與 **fixup** 差在哪？（提示：commit 訊息要不要保留）
- 情境：你的 feature 分支有 5 個雜亂的 commit（含 `wip`、`fix typo`、`oops`），想在開 PR 前整理成 2 個乾淨的原子提交。**你會怎麼做？**
- 什麼是 `--autosquash` 搭配 `git commit --fixup=<hash>` 的工作流？它解決了什麼麻煩？
- rebase 過程中卡住了（衝突或想放棄），有哪些控制指令？（`--continue` / `--abort` / `--skip`）

## Q5. cherry-pick、revert vs reset

- `git cherry-pick <hash>` 做什麼？舉一個實際會用到的情境。
- `git revert` 與 `git reset` 的**本質差別**是什麼？哪一個安全、哪一個會改寫歷史？
- 為什麼「要撤銷一個**已經推上遠端/共享分支**的提交」時應該用 `revert` 而不是 `reset`？
- `git reset` 的三種模式 **`--soft` / `--mixed`（預設）/ `--hard`** 分別對「HEAD、暫存區（index）、工作目錄」做了什麼？哪個最危險、為什麼？
- 情境：你剛 `commit` 了，但發現漏了一個檔案、且想改 commit 訊息，**你會下什麼指令**？（不要新增一個多餘的 commit）

## Q6. 衝突解決流程與心法

- merge/rebase 出現 conflict 時，Git 在檔案裡放的 `<<<<<<<` / `=======` / `>>>>>>>` 標記分別代表什麼？
- 描述一個完整的衝突解決 SOP：從看到衝突 → 找出衝突檔 → 解決 → 標記已解決 → 完成。**你會下哪些指令？**
- rebase 時的衝突，`<<<<<<< HEAD` 這一側到底是「我的」還是「對方的」？（提示：rebase 時方向會反過來，容易搞混）
- 有哪些工具/設定能幫忙解衝突？（`git mergetool`、`git checkout --ours/--theirs`、`rerere`）
- 心法：如何**減少**衝突的發生頻率與痛苦程度？

## Q7. 救援工具：stash、reflog、bisect

- `git stash` 解決什麼問題？`stash pop` 與 `stash apply` 差在哪？怎麼帶 untracked 檔案一起 stash？
- 情境：你手上有一堆未 commit 的修改，但緊急要切去別的分支修 bug，**你會怎麼做？**
- `git reflog` 是什麼？為什麼它是「Git 世界的後悔藥」？
- 情境：你不小心 `git reset --hard` 到很前面、或刪掉了一個分支，commit 好像不見了。**你會怎麼救回來？**
- `git bisect` 解決什麼問題？請描述用它二分搜尋定位「哪個 commit 引入 bug」的完整流程。

## Q8. 好的 commit 訊息與原子提交

- 什麼是**原子提交（atomic commit）**？為什麼「一個 commit 只做一件事」很重要？
- **Conventional Commits** 規範長什麼樣？（`type(scope): subject`）常見的 type 有哪些（feat / fix / docs / refactor / test / chore…）？
- 一則好的 commit 訊息，subject 與 body 各該寫什麼？有哪些常見規則（祈使句、長度、為什麼而非做了什麼）？
- Conventional Commits 搭配自動化能帶來什麼好處？（提示：語意化版本 SemVer、自動產生 CHANGELOG）
- 什麼時候該用 `git add -p`（patch 模式）？它跟原子提交有什麼關係？

## Q9. `.gitignore`、標籤（tag）、`git blame`

- `.gitignore` 的作用是什麼？為什麼「已經被追蹤的檔案」加進 `.gitignore` 沒有用、要怎麼處理？
- **輕量標籤（lightweight tag）** 與 **附註標籤（annotated tag）** 差在哪？發佈版本該用哪一種？tag 預設會被 `git push` 推上去嗎？
- `git blame` 做什麼？在什麼情境下你會用它？它會不會「找戰犯」以外還有正經用途？
- 情境：`git blame` 顯示某行是一次大型格式化（reformat）改的，看不出真正邏輯作者。有什麼辦法略過那次雜訊 commit？

## Q10. 綜合情境題：常見「怎麼救」對照

針對以下每個情境，寫出你會下的指令（可多行）：

- 剛剛把一堆 commit 提交到了 **錯的分支**（本來該在 feature，結果 commit 到了 main，且**尚未 push**）。怎麼把它們搬到正確分支、並讓 main 回到乾淨狀態？
- 想把**最後一次 commit** 整個拆成兩個 commit。
- 想把**分散的兩三個 commit 合併**成一個。
- 誤 push 了含密碼/token 的檔案到遠端，該怎麼處理（觀念層面）？
- 想暫時看看「三個 commit 前」的程式碼長怎樣，看完再回來（不改動任何東西）。
- 想把遠端最新變更抓下來但**不要**產生一堆 merge commit 污染歷史。
