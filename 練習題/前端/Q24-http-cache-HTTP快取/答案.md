# Q24 - HTTP 快取（Cache-Control / ETag）解答

## 兩層快取機制

```
發出請求前
  │
  ▼
強快取還有效？（Cache-Control: max-age / Expires）
  ├── 是 ──▶ 直接用本地副本，不發請求（狀態列顯示 from disk/memory cache）
  └── 否 ──▶ 帶上驗證標頭發請求（協商快取）
                │
                ▼
        伺服器判斷資源有沒有變
          ├── 沒變 ──▶ 304 Not Modified（不傳 body，用本地副本）
          └── 變了 ──▶ 200 OK（傳新的 body + 新的快取標頭）
```

## 強快取（Strong Cache）

命中時**完全不發請求**，直接用本地副本。

```
Cache-Control: max-age=31536000, immutable
```

- `max-age=31536000`：副本在 31536000 秒（一年）內視為新鮮。
- `immutable`：告訴瀏覽器這份資源永不改變，即使使用者按重新整理也不要去協商。
- `Expires: Wed, 21 Oct 2026 07:28:00 GMT`：舊式寫法（絕對時間），優先權低於 `Cache-Control: max-age`。

## 協商快取（Negotiated Cache）

強快取過期後，帶驗證標頭去問伺服器「東西變了沒」。

### 方式一：ETag（內容指紋，較精確）

```
首次回應： ETag: "abc123"
後續請求： If-None-Match: "abc123"
未變更回： 304 Not Modified（無 body）
```

### 方式二：Last-Modified（最後修改時間，精度到秒）

```
首次回應： Last-Modified: Wed, 21 Oct 2026 07:28:00 GMT
後續請求： If-Modified-Since: Wed, 21 Oct 2026 07:28:00 GMT
未變更回： 304 Not Modified（無 body）
```

- 兩者並存時，**ETag 優先於 Last-Modified**（內容指紋比時間戳精確，能避免「一秒內多次修改」「內容沒變但時間戳變」等誤判）。
- `304` 不傳 body，省下頻寬，但仍有一次往返（round-trip）。

## 常見策略

| 資源類型 | 策略 | 原因 |
| --- | --- | --- |
| HTML 入口檔 | `Cache-Control: no-cache` | 每次都協商，確保拿到最新版（`no-cache` 不是不快取，是「用前必須先協商驗證」） |
| 帶 hash 的 JS/CSS（如 `app.4f3a.js`） | `Cache-Control: max-age=31536000, immutable` | 內容變了檔名就變，舊檔名可永久快取，零協商成本 |
| 完全不快取的敏感資源 | `Cache-Control: no-store` | 連存都不存 |

```tsx
// 範例：以 Express 設定回應標頭
app.get("/index.html", (_req, res) => {
  res.set("Cache-Control", "no-cache"); // 每次協商
  res.sendFile("index.html");
});

app.use(
  "/static",
  express.static("dist", {
    setHeaders: (res) => {
      // 檔名帶 hash 的資源，長期強快取
      res.set("Cache-Control", "public, max-age=31536000, immutable");
    },
  })
);
```

## 易混淆釐清

- `no-cache`：**可以**快取，但每次使用前都要向伺服器協商驗證。
- `no-store`：**完全不**快取，每次都重新下載完整資源。
