# Next.js App Router：SSR 與渲染策略 解析

## Q1. Server Components vs Client Components？

- **預設是 Server Component（RSC）**。App Router 內 `app/` 目錄下的元件預設在伺服器執行、只把渲染結果（RSC Payload）送到瀏覽器，不進 client bundle。
- **`'use client'`** 是檔案頂端的指令，宣告「從這個模組開始，往下 import 的模組樹都是 client 端」。它是一道「邊界」，不是「只有這個檔案」——被它 import 的子模組也會進 client bundle。
- **Server Component 不能用**：`useState` / `useEffect` / `useRef` 等 hooks、事件處理（`onClick`）、瀏覽器 API（`window`、`localStorage`）。因為它們在伺服器渲染時根本沒有這些環境，也沒有互動生命週期。反過來，Server Component **可以**直接 `async/await` 抓資料、直接讀檔、讀環境變數與祕密金鑰。
- **序列化邊界**：Server → Client 傳的 props 必須可序列化（能被 RSC 協定編碼），所以可以傳字串、數字、陣列、物件、Date、Promise、甚至 JSX（`ReactNode`），但**不能傳函式、class 實例、Symbol**。要傳「行為」給 client，通常改傳 Server Action（函式參考由框架特別處理）或在 client 端自己定義。
- **組合（composition）**：Client Component 不能直接 `import` 一個 Server Component 來 render，但**可以透過 `children`（或任何 ReactNode prop）把 Server Component 當內容塞進去**。這是最重要的 pattern：

```tsx
// app/page.tsx（Server Component）
import ClientWrapper from './client-wrapper'
import ServerContent from './server-content' // 也是 Server Component

export default function Page() {
  return (
    <ClientWrapper>
      {/* ServerContent 仍在伺服器渲染，只是被當 children 傳入 */}
      <ServerContent />
    </ClientWrapper>
  )
}
```

```tsx
// client-wrapper.tsx
'use client'
export default function ClientWrapper({ children }: { children: React.ReactNode }) {
  const [open, setOpen] = useState(true)
  return <div>{open && children}</div> // children 已是渲染好的 RSC，不進 client bundle
}
```

- **減少 client bundle**：Server Component 的程式碼與它用到的依賴（如 markdown parser、date 函式庫）都留在伺服器，只把渲染結果送出去，瀏覽器不必下載這些 JS。

## Q2. 四種渲染策略：SSR / SSG / ISR / Streaming？

- **靜態 vs 動態的分界**：Next.js 預設盡量做**靜態渲染**（build 時或首次請求後產生、可快取重用）。一旦路由用到**動態 API**就會轉為**動態渲染**（每次請求即時產生）。會觸發動態的行為：`cookies()`、`headers()`、`draftMode()`、讀取 `searchParams`、或使用 `cache: 'no-store'` 的 fetch、或 `export const dynamic = 'force-dynamic'`。
- **SSG + 動態路由**：用 `generateStaticParams` 告訴 Next.js 要為哪些參數在 build 時預先產生頁面：

```tsx
// app/blog/[slug]/page.tsx
export async function generateStaticParams() {
  const posts = await fetch('https://api.example.com/posts').then(r => r.json())
  return posts.map((p: { slug: string }) => ({ slug: p.slug }))
}

export default async function Post({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params // Next.js 15 起 params 是 Promise
  const post = await fetch(`https://api.example.com/posts/${slug}`).then(r => r.json())
  return <article>{post.title}</article>
}
```

- **ISR（`revalidate`）**：屬於 stale-while-revalidate——**使用者不會等**，先回傳快取的舊頁面，Next.js 在背景重新產生，下次請求才拿到新版。設定方式是 route segment config 或 fetch 選項：

```tsx
export const revalidate = 60 // 這個路由每 60 秒有機會在背景重新產生
```

- **Streaming**：把頁面拆成多段，先送出已完成的殼（shell），慢的部分用 `<Suspense>` 包起來、資料好了再串流補上。好處是 **TTFB 更快、感知效能更好**（不必等最慢的資料才顯示整頁）。`loading.js` 是 Next.js 幫整個 route segment 自動包一層 `<Suspense>` 的便捷寫法；更細的粒度用手動 `<Suspense>`：

```tsx
// app/dashboard/page.tsx
import { Suspense } from 'react'

export default function Dashboard() {
  return (
    <section>
      <h1>Dashboard</h1>              {/* 立即送出 */}
      <Suspense fallback={<p>載入報表中…</p>}>
        <SlowReport />                {/* async Server Component，好了再串流 */}
      </Suspense>
    </section>
  )
}
```

- **適用情境**：
  - **SSG**：內容不常變、人人相同——行銷頁、文件、部落格。
  - **ISR**：內容會變但不需即時、量大——電商商品頁、新聞列表。
  - **SSR（動態）**：每次請求都不同、需即時或個人化——會員儀表板、依 cookie 的頁面、搜尋結果。
  - **Streaming**：頁面有快有慢的區塊——上方即時顯示、下方掛 `<Suspense>` 串流慢查詢。

## Q3. App Router 的資料抓取與快取（四層快取）？

- **加工過的 `fetch`**：Next.js 擴充了原生 `fetch`，加入去重與持久化快取的能力。
  - **Next.js 14 預設**：`fetch` 會被快取（等同 `force-cache`），要主動 opt-out。
  - **Next.js 15 預設改為不快取**（`no-store`），要主動 opt-in 才快取。回答時務必點出這個版本差異，這是很常見的考點。
- **定時重新驗證 / 完全不快取**：

```ts
// 定時重新驗證（ISR 等級的資料快取）
await fetch('https://api.example.com/data', { next: { revalidate: 60 } })

// 完全不快取，每次都重抓（動態）
await fetch('https://api.example.com/data', { cache: 'no-store' })

// 強制快取（Next.js 15 想要靜態時要明講）
await fetch('https://api.example.com/data', { cache: 'force-cache' })
```

- **四層快取**：
  1. **Request Memoization**：**單次請求生命週期內**，同一個 URL + 選項的 `fetch` 只會真的打一次，其餘回傳記憶結果。作用範圍是 React render 期間，請求結束就沒了。好處是可以在多個元件各自 `fetch` 同一份資料而不重複打 API。
  2. **Data Cache**：**跨請求、跨部署持久**的伺服器端資料快取，由 `revalidate` / `tags` / `cache` 控制。這才是 ISR 的資料層。
  3. **Full Route Cache**：build 時或首次渲染後，把**整個路由的 HTML + RSC Payload** 快取起來（靜態路由才有）。動態路由不進這層。
  4. **Router Cache（client-side）**：瀏覽器端記憶已造訪過的 route segment 的 RSC Payload，讓前進/後退與 `<Link>` prefetch 導覽瞬間完成。可用 `router.refresh()` 或 revalidate 清掉。
- **`revalidatePath` vs `revalidateTag`**：兩者都用來**主動失效** Data Cache / Full Route Cache（通常在 Server Action 或 Route Handler 內，資料寫入後呼叫）：

```ts
// 幫 fetch 標 tag
await fetch('https://api.example.com/posts', { next: { tags: ['posts'] } })
```

```ts
'use server'
import { revalidateTag, revalidatePath } from 'next/cache'

export async function createPost(formData: FormData) {
  await savePost(formData)
  revalidateTag('posts')        // 失效所有標了 'posts' 的資料
  revalidatePath('/blog')       // 失效 /blog 這個路徑的快取
}
```

  - `revalidatePath` 針對「路徑」；`revalidateTag` 針對「資料標籤」，更精準、可跨多個路由一次失效同類資料。

## Q4. Route Segment Config？

- 在 `layout.tsx` / `page.tsx` / `route.ts` 用 `export const` 匯出設定，控制該 segment 的行為：
  - **`dynamic`**：`'auto'`（預設）/ `'force-dynamic'`（強制每次動態渲染、關掉快取）/ `'force-static'`（強制靜態，動態 API 會回傳空值）/ `'error'`（用到動態 API 就報錯，用來確保靜態）。
  - **`revalidate`**：`false`（永久快取）/ `0`（等同動態）/ 數字（秒，ISR 週期）。
  - **`runtime`**：`'nodejs'`（預設）或 `'edge'`。
  - **`fetchCache`**：微調該 segment 內所有 fetch 的預設快取策略（少用，多半靠個別 fetch 控制）。

```tsx
// app/dashboard/page.tsx
export const dynamic = 'force-dynamic' // 這頁一律即時渲染
export const runtime = 'nodejs'
```

- **`force-dynamic` vs `force-static`**：前者無條件動態、常用於一定要即時的頁；後者無條件靜態，若程式裡用到 `cookies()` 這類動態 API，它們會被當成不可用（回傳空），要小心語意。
- **作用範圍與繼承**：設定會**往下繼承**——`layout` 的設定套用到其下所有 page，但**子 segment 可以覆寫**父層。一般把共用策略放 layout、特例放個別 page。

## Q5. Edge Runtime vs Node.js Runtime？

- **底層差異**：
  - **Node.js Runtime**（預設）：完整 Node.js 環境，能用所有 npm 套件、Node API（`fs`、`crypto` 原生模組等），部署在傳統 serverless / server。
  - **Edge Runtime**：基於 Web 標準 API 的輕量環境（類似 Web Worker / V8 isolate），**冷啟動極快、可部署在全球邊緣節點**貼近使用者，但**限制多**：不能用 Node 原生 API、多數需要 `fs` 或原生模組的套件不能跑、有程式大小限制。
- **選擇取捨**：
  - **選 Edge**：極度要求低延遲且邏輯輕量——地理定位重導、A/B 測試、驗 token、簡單個人化。Middleware 預設就跑在 Edge。
  - **選 Node**：需要重量級套件、資料庫原生驅動、Node API、或計算較重的工作。
- **指定 runtime**：

```ts
// app/api/geo/route.ts
export const runtime = 'edge'

export async function GET(req: Request) {
  return Response.json({ ok: true })
}
```

## Q6. 效能：bundle 拆分與 client bundle 最小化？

- **RSC 的根本差異**：傳統 SSR 是「伺服器產 HTML → 瀏覽器下載**整包** component JS → hydrate」。RSC 則是 Server Component 的程式碼**完全不下載到瀏覽器**，只送渲染結果，client bundle 只包含真正需要互動的 Client Component。這是「從源頭少送 JS」，而非只是延後。
- **`next/dynamic` vs `React.lazy`**：兩者都做 code splitting；`next/dynamic` 是 Next.js 對 `React.lazy` + `Suspense` 的封裝，額外支援 `ssr: false`（在 App Router 只能用在 Client Component 內）與自訂 loading：

```tsx
'use client'
import dynamic from 'next/dynamic'

// 只在需要時才載入重元件，且不做 SSR（例如依賴 window 的圖表）
const HeavyChart = dynamic(() => import('./heavy-chart'), {
  ssr: false,
  loading: () => <p>載入圖表中…</p>,
})

export default function Panel() {
  return <HeavyChart />
}
```

- **重量級第三方套件**：
  - 能放 Server Component 就放 Server 端（例如 markdown 轉換、語法高亮），完全不進 client bundle。
  - 一定要在 client 用（編輯器、圖表）就用 `next/dynamic` 延遲載入 + `ssr: false`，避免拖垮首屏、也避免 SSR 時碰 `window` 報錯。
  - 把互動區塊切成小的 Client Component（「島嶼」），其餘保持 Server Component，縮小 `'use client'` 邊界。

## Q7. 與 Pages Router 對照（給有舊經驗的人銜接）？

- **`getServerSideProps` → 在 async Server Component 內直接抓資料，並標記為動態**：

```tsx
// App Router 的 SSR
export const dynamic = 'force-dynamic' // 或用 cookies()/no-store 觸發動態
export default async function Page() {
  const data = await fetch('https://api.example.com/data', { cache: 'no-store' })
    .then(r => r.json())
  return <main>{data.title}</main>
}
```

- **`getStaticProps` + `getStaticPaths` → async Server Component（預設靜態）+ `generateStaticParams`**：

```tsx
export async function generateStaticParams() { /* 對應 getStaticPaths */ }
export default async function Page({ params }) { /* 對應 getStaticProps */ }
```

- **`getStaticProps` 的 `revalidate`（ISR）→ `export const revalidate = N` 或 `fetch(..., { next: { revalidate: N } })`**。
- **心智模型最大差異**：
  - 資料抓取**從頁面層級的特殊函式，下放到任何 Server Component 內**，可以就近抓、可組合、可平行。
  - 快取變成**宣告式**（透過 fetch 選項與 segment config），而不是靠「選哪個 data-fetching 函式」決定渲染模式。
  - 一個頁面可以**同時**有靜態殼 + 動態串流區塊，不再是「整頁 SSR 或整頁 SSG」的二選一。
