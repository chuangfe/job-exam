# Next.js App Router：SSR 與渲染策略

請依序回答以下 Next.js App Router（14 / 15）核心問題。建議先自己用白話講出來，再對照解析。這是資深前端常見的深入題，重點在「能不能講清楚取捨與背後機制」，而非背 API。

## Q1. Server Components vs Client Components？

- App Router 中元件預設是哪一種？`'use client'` 這個指令實際上宣告了什麼、放在哪裡才有效？
- Server Component 能不能用 `useState` / `useEffect` / `onClick` / `window`？為什麼？
- 「序列化邊界」是什麼意思？從 Server Component 傳 props 給 Client Component 有什麼限制（例如能不能傳函式）？
- Client Component 裡還能不能 render Server Component？要怎麼組合（composition）才對？
- 為什麼把元件放在 Server 端能減少 client bundle？

## Q2. 四種渲染策略：SSR / SSG / ISR / Streaming？

- 靜態渲染（static）與動態渲染（dynamic）的分界是什麼？哪些行為會讓一個路由「被迫變成動態」？
- SSG 要怎麼在動態路由（`[slug]`）上預先產生頁面？`generateStaticParams` 扮演什麼角色？
- ISR 的 `revalidate` 是什麼機制？它是「使用者等新資料」還是「先給舊的再背景更新」（stale-while-revalidate）？
- Streaming 是什麼？`loading.js` 與 `<Suspense>` 分別對應到什麼？為什麼 streaming 對 TTFB / 感知效能有幫助？
- 這四種各自適合什麼情境？請各舉一個實際例子。

## Q3. App Router 的資料抓取與快取（四層快取）？

- App Router 對原生 `fetch()` 做了什麼加工？預設的快取行為是什麼？（Next.js 14 與 15 的預設有差異，請說明。）
- 怎麼讓某個 `fetch` 定時重新驗證？怎麼讓它完全不快取？請寫出對應寫法。
- 請說明四層快取：Request Memoization、Data Cache、Full Route Cache、Router Cache（client-side），各自的作用範圍與存活時間。
- `revalidatePath` 與 `revalidateTag` 差在哪？搭配 `fetch` 的 `next: { tags }` 怎麼用？通常在哪裡呼叫？

## Q4. Route Segment Config？

- `export const dynamic`、`export const revalidate`、`export const runtime`、`export const fetchCache` 各控制什麼？
- `dynamic = 'force-dynamic'` 與 `dynamic = 'force-static'` 分別代表什麼？跟「不小心用到動態 API」有什麼關係？
- 這些設定的作用範圍（layout vs page）與繼承關係是什麼？

## Q5. Edge Runtime vs Node.js Runtime？

- 兩者底層執行環境差在哪？各有什麼限制（API、套件相容性、冷啟動）？
- 什麼情境該選 Edge、什麼情境該選 Node？請各舉例。
- 怎麼指定某個路由跑在哪個 runtime？

## Q6. 效能：bundle 拆分與 client bundle 最小化？

- Server Component 是怎麼「從根本上」減少 client JS 的？和傳統 SSR（送整包 hydrate）差在哪？
- `next/dynamic` 與 `React.lazy` 各自的定位？`ssr: false` 在什麼情況該用？
- 一個很重的第三方套件（例如圖表、編輯器）該怎麼處理，才不會拖垮首屏？
