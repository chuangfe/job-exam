# Storybook：元件文件化與視覺回歸（設計系統的「設計—工程契約」）

請依序回答以下 Storybook（8.x）核心問題。建議先自己用白話講出來，再對照解析。這題重點在「能不能說清楚 Storybook 在設計系統裡的定位，以及怎麼把它變成活文件 + 自動化測試防線」，而非背 API。適用對象：需要維護 UI 元件庫、和設計師協作、或想在履歷上寫「建立設計系統/元件文件」的前端工程師。

## Q1. Storybook 是什麼？為什麼說它是設計系統的「設計—工程契約」與活文件？

- 用一句話說明 Storybook 解決什麼問題。它和「開一個測試頁面把元件都塞進去」差在哪？
- 「隔離開發（isolation）」是什麼意思？為什麼在真實頁面外單獨開發元件有價值？
- 為什麼說 Storybook 是設計師與工程師之間的「契約（contract）」？它讓雙方對「一個元件有哪些狀態、每個狀態長怎樣」達成什麼共識？
- 「活文件（living documentation）」和傳統手寫 Markdown 文件相比，最大的差異是什麼？為什麼前者比較不會過期？

## Q2. CSF3（Component Story Format 3）：`meta`、`Story`、`args`、`argTypes`？

- 什麼是 CSF？CSF3 相對 CSF2 最大的改變是什麼（story 從函式變成什麼）？
- default export（`meta`）負責宣告什麼？`title`、`component`、`args`、`argTypes`、`tags`、`parameters` 各自的角色？
- 具名 export（story 物件）長什麼樣？`args` 在 story 層級和 meta 層級如何合併、誰覆蓋誰？
- `argTypes` 用來做什麼？請舉例：怎麼把某個 prop 在 Controls 面板呈現成下拉選單 / 顏色選擇器 / 布林開關？
- `render` 函式什麼時候需要自己寫（args 無法直接對應 props 時）？

## Q3. Controls / Actions / Docs（autodocs）這三個 addon 的作用？

- Controls addon 讓「非工程師」能做什麼？它和 `argTypes` 的關係是什麼？
- Actions addon 解決什麼？當使用者點按鈕時，怎麼在面板看到事件被觸發（含 args 如何自動被辨識為 callback）？
- autodocs 是什麼？怎麼開啟（`tags: ['autodocs']`）？它會自動根據什麼產生文件頁（props table 從哪來）？
- 什麼時候該用 MDX 自己寫文件，而不是只靠 autodocs？

## Q4. 如何用 story 呈現「變體」與「狀態」？

- 同一個元件的不同變體（例如 Button 的 primary / secondary / danger、size 大中小），慣例上怎麼組織成多個 story？
- 載入中（loading）、錯誤（error）、空狀態（empty）這類「狀態」為什麼一定要各開一個 story？對設計師、對 QA、對視覺回歸各有什麼好處？
- 當一個 story 需要外部環境（如 Router、Theme、React Query、i18n）時，用什麼機制注入？（提示：decorator）
- 怎麼 mock 一個「正在載入」或「請求失敗」的資料狀態，讓它穩定重現、不依賴真實後端？

## Q5. a11y（可及性）addon 的角色？

- `@storybook/addon-a11y` 底層用什麼引擎做檢測？它能自動抓出哪類問題（舉三個例子）？
- 它「能」與「不能」取代什麼？為什麼自動檢測無法保證 100% 可及？
- 怎麼讓 a11y 檢查在 CI 失敗時擋下 PR，而不只是面板裡的一個警告？

## Q6. Interaction tests：`play` function + `@storybook/test`？

- `play` function 是什麼？它在 story render 之後做什麼？
- `@storybook/test` 提供哪些工具（`within`、`userEvent`、`expect`、`fn`、`waitFor`）？它們大致對應到哪個既有測試生態系的 API？

## Q7. 視覺回歸測試（Visual Regression）在 CI 的角色？

- 什麼是視覺回歸測試？它抓的是 unit test / interaction test 抓不到的哪一類 bug？
- Chromatic 的運作模型是什麼（對每個 story 截圖 → 和基準比對 → 有差異就等人 review approve）？為什麼「以 story 為單位截圖」很適合設計系統？
- 用 Playwright 自己做 screenshot 測試（`toHaveScreenshot`）相比 Chromatic，取捨在哪（成本、跨瀏覽器、基準管理、flaky）？
- 視覺回歸測試最常見的假警報來源是什麼（字型、動畫、時間、隨機資料）？怎麼降低 flaky？
