# Storybook：元件文件化與視覺回歸（設計系統的「設計—工程契約」）解析

> 以下以 **Storybook 8.x** 為準。7→8 的主要差異：import 從 `@storybook/testing-library` 改為官方整合的 **`@storybook/test`**（一次帶齊 `userEvent` / `expect` / `fn` / `within`）；CSF3 為預設寫法；`autodocs` 用 `tags` 開啟。

## Q1. Storybook 是什麼？為何是「設計—工程契約」與活文件？

- **一句話**：Storybook 是一個把 UI 元件「隔離」出來，在獨立環境中開發、展示、測試、記錄的工具。它替每個元件的每個狀態建立一個可互動的「story」，集合起來就是整個元件庫的目錄。
- **和「塞一個測試頁面」的差別**：測試頁面是臨時、無結構、常常和真實資料/路由耦合、開發完就砍掉的。Storybook 是**有結構、可長期維護、每個狀態都能被定址（有 URL）、能被自動化測試與截圖**的正式產物。
- **隔離開發（isolation）**：把元件抽離真實頁面，不需要先登入、不需要湊出某個 API 回應、不需要點五層才走到那個畫面，就能直接把元件的「錯誤狀態」「空狀態」「超長文字」叫出來開發。邊界案例（edge case）因此變得容易重現，開發回饋迴圈更短。
- **設計—工程契約（contract）**：設計師在 Figma 定義「Button 有 primary/secondary/danger、每種有 hover/disabled/loading」；工程師把這些狀態一一實作成 story。於是雙方對「這個元件到底有哪些狀態、每個狀態長怎樣」有一份**可並排比對、可互動、雙方都看得懂**的共同事實。設計驗收時直接開 Storybook 對，不用等元件被組進頁面。
- **活文件（living documentation）**：autodocs 的 props table、範例、變體都是**從真實原始碼與型別自動生成**的。元件改了 prop，文件跟著更新；story 壞了，CI 會紅。手寫 Markdown 文件則是「寫完那一刻最準，之後每天都在過期」，沒有任何機制強迫它跟程式碼同步。

## Q2. CSF3：`meta` / `Story` / `args` / `argTypes`

- **CSF（Component Story Format）** 是用標準 ES module 寫 story 的格式：**default export 描述元件（meta），每個 named export 是一個 story**。CSF3 相對 CSF2 最大的改變是 **story 從「函式」變成「物件」**——你宣告 `args`、`play` 等欄位，由 Storybook 幫你 render，樣板程式更少、更容易被工具靜態分析。
- **default export（`meta`）** 宣告這個檔案裡所有 story 共用的預設：
  - `title`：在側邊欄的階層路徑（如 `'Components/Button'`）。省略時可由檔案路徑自動推導。
  - `component`：指向元件本身，autodocs 靠它抽 props/型別。
  - `args` / `argTypes`：所有 story 的預設 args 與控制項設定。
  - `tags`：如 `['autodocs']` 開啟自動文件。
  - `parameters`：addon 設定（如 `layout: 'centered'`、`backgrounds`、`a11y`）。
- **named export（story 物件）** 只需覆寫要變動的部分。**args 合併規則：story 層級的 `args` 會覆蓋（淺層合併）meta 層級的 `args`**，未指定的沿用 meta 預設。
- **`argTypes`** 控制每個 arg 在 Controls 面板的呈現方式與文件描述。範例：

```tsx
argTypes: {
  variant: { control: 'select', options: ['primary', 'secondary', 'danger'] }, // 下拉
  size:    { control: 'radio',  options: ['sm', 'md', 'lg'] },                 // 單選鈕
  color:   { control: 'color' },                                              // 顏色選擇器
  disabled:{ control: 'boolean' },                                            // 布林開關
  onClick: { action: 'clicked' },                                            // 觸發 Actions
}
```

- **`render`**：多數情況 args 能直接對應成 props，Storybook 自動 render，不用寫 `render`。但當你需要「用 args 組出比較複雜的 JSX」（例如把 args 傳給 context、或包一層 layout、或一個 story 要同時呈現多顆按鈕）時，就自己寫 `render: (args) => <.../>`。

## Q3. Controls / Actions / Docs（autodocs）

- **Controls**：把 `argTypes` 定義的參數變成 UI 面板上的表單控制項，**非工程師（設計師、PM、QA）可以即時拖拉、切換 props 看元件變化**，不用改任何程式碼。`argTypes` 是「宣告」，Controls 面板是「這份宣告的互動介面」；沒寫 `argTypes` 時 Storybook 也會依 args 型別自動推論控制項。
- **Actions**：把事件 callback 的呼叫記錄到 Actions 面板。當你在 `argTypes` 用 `action: 'clicked'`，或（推薦）在 `args` 裡用 `fn()` 指派 callback，點按鈕時就能在面板看到事件名稱與帶進來的參數。Storybook 8 也會**自動把符合 `on[A-Z]...` 命名的 props 當成 action** 記錄。
- **autodocs**：在 meta 加 `tags: ['autodocs']`，Storybook 會為該元件自動產生一頁文件：**props table 從 `component` 的 TypeScript 型別 / PropTypes / JSDoc 抽出**，加上各 story 當成範例、Controls 可即時玩。等於「零額外成本」的元件文件。
- **何時改用 MDX**：需要大段敘述、使用指南、Do/Don't、設計原則、多個元件混排、或想在文件中穿插自訂排版時，用 `.mdx` 檔自己寫（可 import story 進來嵌入）。autodocs 適合「規格型」文件，MDX 適合「敘事型」文件。

## Q4. 呈現變體與狀態

- **變體（variants）**：每個變體開一個具名 story，命名語意化。慣例是「一個 story = 一種你想被看到 / 被截圖 / 被驗收的樣貌」：

```tsx
export const Primary: Story = { args: { variant: 'primary', children: '送出' } }
export const Danger:  Story = { args: { variant: 'danger',  children: '刪除' } }
export const Small:   Story = { args: { size: 'sm', children: '小按鈕' } }
```

- **狀態（states）**：loading / error / empty **一定各開一個 story**。好處：設計師能一次驗收所有狀態、QA 有明確清單、**視覺回歸能對每個狀態各截一張圖**（否則錯誤狀態的樣式壞了根本沒人截得到）。這正是「把邊界案例變成一等公民」。
- **注入外部環境**：用 **decorator**（story 或 meta 的 `decorators`）包一層 Provider——Router、ThemeProvider、QueryClientProvider、i18n 都用這招。可放在單一 story、meta，或 `.storybook/preview.tsx` 的全域 decorator。
- **mock 資料狀態**：不要打真後端。做法：(1) 把「呈現」與「資料抓取」拆開，story 直接餵 props；(2) 用 `parameters` + MSW（`msw-storybook-addon`）攔截 API 回傳指定 loading/error 回應；(3) 用 decorator 提供一個假的 QueryClient / context。目標是**穩定重現、可截圖、不 flaky**。

## Q5. a11y addon

- **引擎**：`@storybook/addon-a11y` 底層跑 **axe-core**（和 Lighthouse、jest-axe 同一顆引擎）。它能自動抓出：對比度不足（color contrast）、缺 alt 的圖片、表單控制項沒有對應 label、ARIA 屬性用錯、heading 層級跳號、互動元素缺可及名稱等。
- **能與不能**：自動檢測只能覆蓋「可被程式判定」的規則（約佔 WCAG 的一部分）。**不能**驗證：鍵盤操作邏輯是否合理、焦點順序是否符合閱讀順序、螢幕報讀器唸出來是否通順、動態內容更新有沒有正確 announce。這些仍需人工 + 真實輔具測試。所以它是「地板」不是「天花板」。
- **在 CI 擋 PR**：搭配 Storybook **test-runner**，設定其在每個 story 執行 axe 檢查（test-runner 的 `postVisit` hook 裡跑 `@storybook/test-runner` 提供的 axe 整合），有 violation 就讓該 story 測試失敗，PR 因此變紅。這樣 a11y 就不只是面板裡「看得到但沒人管」的警告。

## Q6. Interaction tests：`play` + `@storybook/test`

- **`play` function**：定義在 story 上、**在該 story render 完成後自動執行**的一段腳本。它模擬使用者操作並做斷言，等於把「開瀏覽器手動點」自動化，且結果會顯示在 Interactions 面板（可逐步 debug、倒帶）。
- **`@storybook/test`** 提供：
  - `within(canvasElement)`：限定查詢範圍（對應 Testing Library 的 queries）。
  - `userEvent`：模擬點擊、輸入、鍵盤等（對應 `@testing-library/user-event`）。
  - `expect`：斷言（對應 Vitest/Jest 的 expect + jest-dom matcher）。
  - `fn()`：建立 spy/mock，驗證 callback 呼叫（對應 `vi.fn()`）。
  - `waitFor`：等待非同步條件成立。
  - 整體對應 **Testing Library + Vitest** 生態系的 API，只是整合進 Storybook。
- **「填表單 → 送出 → 斷言成功訊息」步驟**：
  1. `const canvas = within(canvasElement)`。
  2. `await userEvent.type(canvas.getByLabelText('Email'), 'a@b.com')` 填欄位。
  3. `await userEvent.click(canvas.getByRole('button', { name: '送出' }))`。
  4. `await expect(await canvas.findByText('送出成功')).toBeInTheDocument()`。
- **用 `fn()` 驗證 callback**：在 `args` 裡 `onSubmit: fn()`，play 中點送出後 `await expect(args.onSubmit).toHaveBeenCalledWith({ email: 'a@b.com' })`。（`play` 的參數可解構出 `args`、`canvasElement`。）
- **CI 一次跑完**：用 `@storybook/test-runner`（底層 Playwright + Jest）開一顆 headless 瀏覽器，**把每個 story render 一遍、跑它的 `play` 與 a11y**，全部一次跑完並回報通過/失敗。這比「人工在瀏覽器點」的差別是：可重現、可 gate PR、跑得快、不會漏測。

## Q7. 視覺回歸測試（Visual Regression）

- **是什麼**：對畫面截圖，和「已核可的基準圖」逐像素比對，**有差異就標記出來等人 review**。它抓的是 unit/interaction test 幾乎抓不到的一類 bug——**「畫面看起來變了」**：改了某個共用 CSS 變數導致別的元件間距跑掉、字型 fallback、z-index、破版、被覆寫的樣式。這類 bug 邏輯測試全綠但畫面已壞。
- **Chromatic 模型**：Chromatic（Storybook 官方團隊出的雲服務）會**對每個 story 自動截圖 → 和上一版核可的基準比對 → 有差異就在 PR 出現 UI diff 讓你 approve 或打回**。「以 story 為單位截圖」很適合設計系統，因為 story 本來就是「每個狀態一個定址單位」，等於**免費把整個元件庫的每個狀態都納入視覺守護**，還能跨瀏覽器截、記錄 diff 歷史。
- **Playwright 自幹（`toHaveScreenshot`）取捨**：
  - 優點：免費、基準圖進 git 自己掌控、和既有 e2e 共用一套。
  - 缺點：**基準圖管理、跨 OS/瀏覽器渲染差異、flaky 都要自己扛**；不同機器字型算繪不同會整批假 diff，CI 常需固定成同一個 docker 影像。Chromatic 幫你處理雲端一致環境、diff review 流程與並行，代價是付費 + 資料上雲。
  - 選擇：元件庫/設計系統、團隊願意付費求省事 → Chromatic；已有 Playwright e2e、想零成本、可控環境 → 自建 screenshot。
- **假警報來源與降 flaky**：自訂字型未載入完成、CSS 動畫/過場、依當下時間或亂數產生的內容、非同步資料尚未 settle。對策：截圖前 `waitFor` 字型與資料就緒、停用動畫（全域注入 `* { animation: none !important; transition: none !important; }`）、把日期/亂數 mock 成固定值、必要時設 diff 門檻。

## Q8. Figma 對接與 design token

- **`@storybook/addon-designs`**：讓你在 story 的 `parameters.design` 貼上 Figma frame 連結，**Storybook 文件頁就能並排顯示「設計稿」與「實作」**。這讓「契約」變得可即時對照——設計驗收時一眼看出實作和稿子哪裡不一致。
- **design token**：把設計決策（顏色、間距、字級、圓角、陰影…）抽成有名字的變數（如 `color.primary.500`、`space.4`）。它應是**設計與工程的單一事實來源**：設計和工程都引用同一組 token，而不是各自寫死 `#3B82F6`。改一次，所有引用處一起變。
- **在 Storybook 展示 token**：做一頁（用 MDX 或專門 story）把 token 迭代出來——色票色塊、間距刻度、字級樣本一覽。搭配 `@storybook/addon-themes` 的 `withThemeByClassName` 可一鍵切換 light/dark 主題檢視。
- **鏈路**：**Figma 變數 → design token（如用 Style Dictionary / Tokens Studio 匯出）→ CSS 變數 / 主題設定 → 元件消費 → Storybook 展示與截圖**。因為每一環都引用上游同一份 token，設計師在 Figma 改一個色值，透過 token pipeline 就能一路流到程式碼與 Storybook，並被視覺回歸捕捉到差異——這就是設計系統「設計改動自動流到工程」的價值。

## 完整 CSF3 Story 範例（含 argTypes、decorator、play、design、a11y）

```tsx
// Button.stories.tsx
import type { Meta, StoryObj } from '@storybook/react'
import { within, userEvent, expect, fn } from '@storybook/test'
import { Button } from './Button'

const meta = {
  title: 'Components/Button',
  component: Button,
  // 開啟 autodocs：自動從 Button 的型別產生 props table 與文件頁
  tags: ['autodocs'],
  parameters: {
    layout: 'centered',
    // 在 Docs 頁並排顯示 Figma 設計稿（addon-designs）
    design: { type: 'figma', url: 'https://www.figma.com/file/xxxx/Design-System?node-id=1-2' },
    // 針對此元件的 a11y 規則設定（addon-a11y，底層 axe-core）
    a11y: { config: { rules: [{ id: 'color-contrast', enabled: true }] } },
  },
  // 所有 story 共用的預設 args
  args: {
    children: '按鈕',
    variant: 'primary',
    size: 'md',
    disabled: false,
    onClick: fn(), // spy：可在 Actions 面板看到、也可在 play 中斷言
  },
  // Controls 面板呈現方式
  argTypes: {
    variant: { control: 'select', options: ['primary', 'secondary', 'danger'] },
    size: { control: 'radio', options: ['sm', 'md', 'lg'] },
    disabled: { control: 'boolean' },
  },
  // 全元件共用環境（此處示意注入主題）
  decorators: [
    (Story) => (
      <div data-theme="light">
        <Story />
      </div>
    ),
  ],
} satisfies Meta<typeof Button>

export default meta
type Story = StoryObj<typeof meta>

// 變體：只覆寫要變動的 args
export const Primary: Story = { args: { variant: 'primary', children: '送出' } }
export const Danger: Story = { args: { variant: 'danger', children: '刪除' } }
export const Disabled: Story = { args: { disabled: true, children: '無法點擊' } }

// 狀態：載入中（各狀態獨立 story，方便設計驗收與視覺回歸截圖）
export const Loading: Story = {
  args: { children: '處理中…', disabled: true },
}

// Interaction test：點擊後驗證 onClick 有被呼叫
export const ClickInteraction: Story = {
  args: { children: '點我' },
  play: async ({ args, canvasElement }) => {
    const canvas = within(canvasElement)
    const button = canvas.getByRole('button', { name: '點我' })
    await userEvent.click(button)
    await expect(args.onClick).toHaveBeenCalledTimes(1)
  },
}
```

> 上例一個檔案就同時是：**文件**（autodocs + Figma 對照）、**設計契約展示**（各變體/狀態）、**互動測試**（play）、**可及性檢測**（a11y），再交給 Chromatic / test-runner 在 CI 補上視覺回歸與批次執行——這就是 Storybook 作為「設計—工程契約 + 活文件 + 測試防線」的完整樣貌。
