# Q52 - CI/CD 與工程品質門檻（前端）解析

整體心智圖：一次改動從「本機 → PR → 合併 → 上線 → 監控」，每一關都有自動化把關。

```
開發者本機
  │  (git push / 開 PR)
  ▼
GitHub Actions CI  ── install → lint → typecheck → test → build ──▶ 全綠燈？
  │                                                                    │否 → 擋住，不能 merge
  │是（required checks 通過 + review 通過）                            
  ▼
merge 進 main
  │
  ▼
CD 部署（Vercel 自動部署 / Docker 映像檔推到 registry → pm2 或 K8s 起服務）
  │
  ▼
線上監控（RUM Web Vitals、錯誤率、bundle size…）── 回饋 ──▶ 下一輪改進
```

原則：**越早、越自動地發現問題，修復成本越低**。品質門檻把「人為疏忽」變成「機器強制」。

---

## Q1. CI 與 CD 的定義與價值

| 名詞 | 定義 | 到哪一步 |
| --- | --- | --- |
| CI（持續整合） | 頻繁把小改動合進主幹，每次合併都自動 build + 測試 | 驗證程式碼可整合、沒壞 |
| CD（持續交付 Delivery） | 在 CI 之上，讓每個通過的版本**隨時可一鍵上線**（人工按下 approve） | 產出「可部署的產物」，上線需人工點頭 |
| CD（持續部署 Deployment） | 更進一步，通過檢查就**自動上線**，全程無人工介入 | 自動送到 production |

- **Delivery vs Deployment 的關鍵差異**：Delivery 留了一個「人工核准」的閘門（適合金融、法規敏感或需要排程上線的場景）；Deployment 連這個閘門都自動化（適合高信任、測試覆蓋足夠、灰度/回滾機制完善的團隊）。
- **CI 為什麼降低整合風險**：改動小 → 衝突小、出錯範圍好定位；每次都驗證 → 問題當天就浮現。反之「大 branch 三個月才 merge」會累積大量衝突與隱性 bug，merge 當下爆炸且難以追查是哪一次改動造成的（俗稱 merge hell / integration hell）。
- **品質門檻（quality gate）**：一組「必須通過才能往下走」的自動檢查。前端典型四關：
  1. **lint**（ESLint / Prettier）：程式風格與明顯錯誤。
  2. **type check**（`tsc --noEmit`）：型別安全，抓出 runtime 才會炸的型別錯誤。
  3. **test**（單元 / 整合 / E2E）：功能正確性。
  4. **build**（`next build` / `vite build`）：確保真的打包得出來（很多錯誤只有 production build 才會出現）。
  規定「全綠才能合併」，是把品質標準從「靠自律」變成「機器強制」，讓 main 永遠保持可部署狀態。
- **fail fast**：讓便宜又快的檢查先跑（lint、typecheck 幾秒），慢又貴的（E2E、build）後跑。前面一掛就立刻停、快速回饋，不浪費 runner 時間與開發者等待。

## Q2. GitHub Actions 基本組成

```
workflow（一個 .github/workflows/*.yml 檔）
  ├─ trigger（on: push / pull_request …）
  └─ jobs
       ├─ job A（跑在一台全新的 runner 上）
       │    ├─ step 1（uses: 某個 action）
       │    ├─ step 2（run: 某個指令）
       │    └─ …
       └─ job B（另一台 runner，預設與 A 平行）
```

- **workflow**：一個自動化流程，放在 repo 的 `.github/workflows/` 底下，一個 YAML 檔一個 workflow。
- **job**：workflow 底下的工作單位，每個 job 跑在一台**全新、乾淨的虛擬機（runner）** 上。
- **step**：job 裡的步驟，兩種：`uses`（引用現成 action，如 `actions/checkout@v4`）或 `run`（執行 shell 指令）。
- **action**：可重用的封裝單元（別人寫好的積木），透過 `uses` 引用。
- **trigger**：
  - `push`：有 commit 推到指定 branch 時觸發（常用於 main 合併後的部署）。
  - `pull_request`：開/更新 PR 時觸發（PR 檢查用這個，因為它針對「合併後的結果」做驗證，且對 fork PR 有安全限制）。
  - `workflow_dispatch`：手動按鈕觸發（方便重跑或手動部署）。
  - `schedule`：cron 排程觸發（如每晚跑一次完整 E2E、相依套件安全掃描）。
- **`runs-on`**：指定 job 跑在哪種 runner，如 `ubuntu-latest`（GitHub 託管的 Ubuntu VM，最常用也最便宜）。
- **job 平行 vs 序列**：預設**平行**。用 `needs: [job_a]` 宣告依賴，讓某 job 等前一個成功才跑（例如 `deploy` needs `test`）。
- **檔案系統共用**：同一 job 的 step 共用同一台 runner 的工作目錄與檔案系統（前一步 `npm ci` 裝的東西後一步用得到）。**不同 job 各自一台全新 runner，檔案系統不共用**——要傳檔案得靠 artifacts（見 Q3）。

## Q3. 依賴快取、matrix、secrets、artifacts

### 依賴快取

- **為什麼要快取**：每次 CI 都在乾淨 runner 上跑，`npm ci` 重新下載所有相依很慢。快取 `~/.npm`（或 pnpm store）能把安裝時間從數分鐘降到數秒。
- 兩種做法：
  - **`actions/setup-node` 的內建 `cache`**（推薦，最省事）：`cache: 'npm'` 會自動用 lockfile 雜湊當 key 快取 npm 全域快取目錄。
  - **手動 `actions/cache`**：需要更精細控制（快取非標準路徑、多個目錄、如 Next.js 的 `.next/cache`）時使用。
- **cache key 綁 lockfile 雜湊**：`key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}`。lockfile 沒變 → key 相同 → **cache hit**，直接還原快取、跳過下載；lockfile 一改（升版套件）→ 雜湊變 → key 不同 → **cache miss**，重新安裝並存一份新快取。這樣既快又不會用到過期的依賴。

### matrix（矩陣）

解決「同一套流程要跑多種組合」的問題，避免複製貼上多個 job：

```yaml
strategy:
  matrix:
    node: [18, 20, 22]
```

上例會自動展開成 3 個平行 job，分別在 Node 18/20/22 跑相同步驟——常用於確認函式庫在多版本 Node、多瀏覽器、多 OS 都正常。

### secrets

- **是什麼**：加密儲存的機密值（API token、部署金鑰、資料庫密碼），設定在 repo/organization/environment 的 Settings → Secrets。
- **為什麼不能寫進 YAML**：YAML 進版控 = 全世界（或全公司）看得到，等於洩漏。
- **取用**：`${{ secrets.VERCEL_TOKEN }}`。GitHub 會在 log 裡自動把 secret 值遮罩成 `***`。
- **fork PR 拿不到 secrets**：來自 fork 的 PR 預設**不會**注入 secrets，避免惡意 PR 印出或竊取機密（安全設計）。所以「部署」這類需要 secret 的步驟通常只在 `push` 到 main 或受信任環境才跑。

### artifacts（產物）

- **是什麼**：workflow 執行過程產生、想保留或跨 job 傳遞的檔案。
- **典型用途**：
  - 保留 build 產物（`dist`/`.next`）供後續 job 部署。
  - 保留測試報告、coverage、E2E 截圖/錄影供事後檢查。
  - 跨 job 傳檔（因為不同 job 檔案系統不共用）。
- 用 `actions/upload-artifact` 上傳、`actions/download-artifact` 在另一個 job 下載。

## Q4. 必要檢查與分支保護

- **branch protection rule（分支保護規則）**：對特定 branch（如 `main`）設限，防止未經檢查的程式碼直接進主幹。常見設定：
  - 禁止直接 push（一律走 PR）。
  - Require pull request review（至少 N 位 reviewer approve）。
  - Require status checks to pass（必要檢查）。
  - Require branches to be up to date before merging。
  - Require linear history / 禁止 force push。
- **required status checks（必要檢查）**：把 CI 裡特定 job（如 `lint`、`test`、`build`）設為「必須成功」，只要有一個沒過或還在跑，merge 按鈕就是灰的。這就是「CI 沒過就不能 merge」的實際機制。
- **require branches up to date**：要求 PR 先 rebase/merge 到最新的 main 再合，避免「各自對舊 main 都綠燈、合在一起卻壞掉」的語意衝突（semantic conflict）。
- **為什麼要搭配**：光有 CI 只是「跑給你看」，沒設必要檢查的話，即使 CI 紅燈，還是點得下 merge。**必要檢查才是真正的閘門**；兩者一起設定，門檻才有強制力。

## Q5. 完整的前端 CI workflow

`.github/workflows/ci.yml`：

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

# 同一分支若有新 push，取消還在跑的舊流程，省 runner 時間
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      # 1. 把 repo 拉到 runner 上
      - name: Checkout
        uses: actions/checkout@v4

      # 2. 安裝 Node，並開啟 npm 依賴快取（key 自動綁 package-lock.json）
      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      # 3. 依 lockfile 精確、可重現地安裝（見下方說明）
      - name: Install dependencies
        run: npm ci

      # 4. 風格與明顯錯誤
      - name: Lint
        run: npm run lint

      # 5. 型別檢查（package.json 內 "typecheck": "tsc --noEmit"）
      - name: Type check
        run: npm run typecheck

      # 6. 測試（CI 環境務必用非互動模式）
      - name: Test
        run: npm test -- --run

      # 7. 產線建置（很多錯誤只有這步才會現形）
      - name: Build
        run: npm run build

      # 8. 保留 build 產物，供後續部署 job 或除錯用
      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist        # Next.js 改成 .next
          retention-days: 7
```

- **為什麼用 `npm ci` 而不是 `npm install`**：
  - `npm ci` **嚴格照 `package-lock.json` 安裝**，版本完全可重現，適合 CI（保證「這次跑的依賴」和「lockfile 記錄的」一模一樣）。
  - 它會先**清空 `node_modules`** 再裝，不會被殘留狀態污染。
  - lockfile 與 `package.json` 不一致時它會**直接報錯**，而不是像 `npm install` 那樣默默改動 lockfile。
  - 通常也更快。
- 各 step 用意如上註解：checkout 取碼 → setup-node 建環境+快取 → 依序跑四道品質門檻 → 上傳產物。

## Q6. Dockerfile：前端多階段建置

- **multi-stage build**：一個 Dockerfile 裡分多個 `FROM` 階段，前面的階段負責「建置」（需要 Node、原始碼、node_modules），最終映像檔**只複製建置產物**、丟掉建置環境。對前端而言，final image 只需要靜態檔 + 一個輕量 web server，映像檔可以從數百 MB 降到十幾 MB。

```dockerfile
# ---------- build stage ----------
FROM node:20-alpine AS builder
WORKDIR /app

# 先只複製依賴清單，善用 layer 快取：程式碼改動但依賴沒變時，這層不用重跑
COPY package*.json ./
RUN npm ci

# 再複製其餘原始碼並建置
COPY . .
RUN npm run build          # 產出 /app/dist（Vite）或 /app/build（CRA）

# ---------- runtime stage ----------
FROM nginx:1.27-alpine AS runner

# 只把「靜態產物」搬進 nginx，不含 node_modules 與原始碼
COPY --from=builder /app/dist /usr/share/nginx/html

# SPA 路由 fallback：讓 /任意路徑 都回 index.html（見下方 nginx 設定）
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

搭配的 `nginx.conf`（SPA 需要把所有路徑導回 `index.html`，否則重新整理子路由會 404）：

```nginx
server {
  listen 80;
  server_name _;
  root /usr/share/nginx/html;

  location / {
    try_files $uri $uri/ /index.html;   # 前端路由 fallback
  }

  # 帶 hash 的靜態資源可長快取
  location /assets/ {
    expires 1y;
    add_header Cache-Control "public, immutable";
  }
}
```

- **runtime image 不含 node_modules / 原始碼的好處**：
  - **映像檔更小** → 拉取/部署更快、佔用更少。
  - **攻擊面更小** → 沒有原始碼、沒有一堆含漏洞的 dev 依賴、沒有 build 工具，資安風險大幅降低。
- **`.dockerignore`**：把 `node_modules`、`.git`、`dist`、`.env` 等排除在 build context 之外，避免把一堆無關/機密檔案送進 Docker daemon，加快 build 也更安全。
- **layer 快取與指令順序**：Docker 逐層快取，某層的輸入沒變就重用。**先 `COPY package*.json` + `npm ci`、再 `COPY . .`** 的順序，能讓「只改程式碼、沒改依賴」時跳過昂貴的 `npm ci` 那層——若把 `COPY . .` 放前面，任何一行程式碼改動都會讓依賴安裝層失效、重裝一次，白白變慢。

## Q7. pm2：Node 應用 process 管理

- **解決什麼問題**：直接 `node server.js` 一旦程式崩潰就整個掛掉、伺服器重開後不會自動起、也沒有日誌與監控。pm2 是 Node 的 production process manager，補上：
  - **自動重啟**（crash 後拉起）。
  - **開機自啟**（`pm2 startup` + `pm2 save`）。
  - **集中日誌**（`pm2 logs`）與**監控**（`pm2 monit`）。
  - **cluster mode** 多核心擴展與**零停機 reload**。
- **cluster mode**：Node 單執行緒、單一 process 只用得到一顆 CPU。cluster 模式用 Node 的 cluster module fork 出多個 worker，共用同一個 port（由 master 做負載平衡），把多核心都吃滿。`instances: 'max'` 代表「開和 CPU 核心數一樣多的 worker」。

`ecosystem.config.js`：

```js
module.exports = {
  apps: [
    {
      name: 'my-api',
      script: './server.js',
      exec_mode: 'cluster',        // cluster 模式
      instances: 'max',            // 依 CPU 核心數開 worker
      max_memory_restart: '512M',  // 單一 worker 超過 512MB 自動重啟（防記憶體洩漏拖垮機器）
      env: {
        NODE_ENV: 'production',
        PORT: 3000,
      },
      env_development: {
        NODE_ENV: 'development',
        PORT: 3001,
      },
      out_file: './logs/out.log',
      error_file: './logs/error.log',
      merge_logs: true,
    },
  ],
};
```

常用指令：

```bash
pm2 start ecosystem.config.js --env production
pm2 status          # 看各 process 狀態、CPU、記憶體
pm2 logs my-api     # 即時看日誌
pm2 monit           # 互動式監控面板
pm2 reload my-api   # 零停機：逐一換掉 worker，服務不中斷
pm2 restart my-api  # 直接全部殺掉重起（會有短暫中斷）
pm2 save            # 儲存目前清單，配合 pm2 startup 開機自啟
```

- **`reload` vs `restart`**：`reload` 在 cluster 模式下**逐一**重啟 worker（rolling restart），舊 worker 還在服務新請求，達到**零停機**；`restart` 是直接停掉再開，會有短暫 downtime。程式更新上線優先用 `reload`。

## Q8. 部署：Vercel 與環境變數管理

- **Vercel × Next.js 開箱即用**：Vercel 是 Next.js 的原廠（同一家公司），推 code 上去它會自動偵測框架、跑 build、把靜態資源上 **全球 CDN**、把 API Route / Server Component / middleware 佈到 **Serverless / Edge Functions**，還內建映像最佳化、ISR 等。幾乎零設定就能上線。
- **預覽部署（Preview Deployment）**：每開一個 PR（或每次 push 到非正式分支），Vercel 都會**自動產生一個獨立網址**部署該版本。價值：
  - Reviewer / PM / 設計師可以直接點連結看實際成果，而不是在本機拉分支跑。
  - 每個 PR 環境隔離、互不干擾。
  - **Production 部署**通常由「合併/push 到正式分支（main）」觸發。
- **環境變數作用域**：Vercel 把環境變數分三種環境——
  - **Production**：正式網域使用。
  - **Preview**：PR 預覽部署使用（可指向測試用後端）。
  - **Development**：`vercel dev` 本機開發使用。
  同一個 key 可以在不同環境設不同值（例如 Preview 指向 staging API，Production 指向正式 API）。
- **前端環境變數的安全界線（重要）**：
  - 帶 `NEXT_PUBLIC_` 前綴（Vite 是 `VITE_`）的變數會在 **build 時被 inline 進 client bundle**，任何人打開瀏覽器都看得到 → **只能放非機密的公開設定**（如公開 API base URL、GA 追蹤 ID）。
  - **機密**（資料庫連線字串、第三方 API secret、金鑰）**絕不能加公開前綴**；它們只在 **server 端**（Server Component、Route Handler、`getServerSideProps`、API function）用 `process.env.XXX` 讀取，不會進到瀏覽器。
  - 心法：**「會被打包進前端的東西，等於公開」**。判斷一個值能不能放進 client，就問「我願不願意把它印在網頁原始碼上」。

## Q9. Web Vitals / 監控在 CI 與線上的角色

監控分兩個層次，資料來源與目的不同：

| | CI pre-deploy 檢查 | 線上 RUM（真實使用者監控） |
| --- | --- | --- |
| 工具 | Lighthouse CI、bundlesize | `web-vitals` 套件 + 分析後端、Sentry |
| 資料來源 | **實驗室數據**（固定環境模擬一次） | **真實使用者**的各種裝置/網路 |
| 目的 | 上線前擋住明顯退步（回歸防護） | 掌握真實體驗、抓長尾與偶發問題 |
| 時機 | 每個 PR / 合併前 | 持續、上線後 |

- **CI 端**：可用 Lighthouse CI 對每個 PR 量 LCP / CLS / TBT，設門檻不達標就讓檢查紅燈，防止效能無聲退步。
- **performance budget（效能預算）**：預先訂好「LCP 不得超過 2.5s」「bundle 不得超過 200KB」等上限，設成 CI 必要檢查。
  - 好處：把效能退步當成 bug 一樣被 CI 擋下。
  - 風險：實驗室數據會抖動（runner 忙碌造成偶發超標 → flaky check），需設合理容忍區間或取多次中位數，避免動不動就誤擋。
- **線上收集真實 Web Vitals**：用官方 `web-vitals` 套件把指標回報到分析後端——

```js
import { onLCP, onINP, onCLS } from 'web-vitals';

function sendToAnalytics(metric) {
  const body = JSON.stringify(metric);
  // 用 sendBeacon 確保頁面關閉時也送得出去
  navigator.sendBeacon?.('/analytics', body) ||
    fetch('/analytics', { body, method: 'POST', keepalive: true });
}

onLCP(sendToAnalytics);
onINP(sendToAnalytics);
onCLS(sendToAnalytics);
```

  - **為什麼實驗室 + 真實都要**：實驗室數據穩定、可重現、適合做回歸門檻，但它是「單一環境的一次快照」，量不到「用低階手機、弱網路的真實使用者」的長尾體驗。RUM 補上真實世界的分布（p75、p95），兩者相輔相成。（呼應 Q29：INP/LCP/CLS 的 p75 才是 Google 用來評分的依據。）
