# Q52 - CI/CD 與工程品質門檻（前端）

「在我電腦上明明會動」是每個團隊都想根除的一句話。**CI/CD（持續整合 / 持續交付與部署）** 就是把「合併程式碼」「檢查品質」「打包」「上線」這幾件事自動化、標準化，讓每一次改動都經過同一套流水線、每一次上線都可重現。這題請你把前端的 CI/CD 心智圖串起來：從品質門檻（lint / type check / test / build 都要綠燈才能合併），到 GitHub Actions 的組成與撰寫，再到 Docker 多階段建置、pm2 process 管理、Vercel 部署與環境變數管理，最後帶到線上監控（Web Vitals）在流水線中的角色。

> 本題與 Q29（核心網頁指標 Core Web Vitals）互補：Q29 講「要量什麼指標、怎麼優化」，本題講「怎麼把這些指標放進 CI 門檻與線上監控」，請對照閱讀。

## Q1. CI 與 CD 的定義與價值

- 用自己的話說明 **CI（Continuous Integration，持續整合）**、**CD（Continuous Delivery，持續交付）** 與 **CD（Continuous Deployment，持續部署）** 三者的差別。Delivery 與 Deployment 的關鍵差異是什麼？
- CI 的核心精神「頻繁地把小改動合進主幹、每次合併都自動驗證」為什麼能降低整合風險？跟「開一條大 branch 三個月才 merge」相比好在哪？
- 什麼是**品質門檻（quality gate）**？為什麼要規定 lint / type check / test / build 全部綠燈才能合併？
- 「fail fast」是什麼意思？流水線各步驟的排列順序（快又便宜的先跑）為什麼重要？

## Q2. GitHub Actions 基本組成

- 說明 GitHub Actions 的層級結構：**workflow / job / step / action**，各自是什麼、彼此關係為何？
- **trigger（觸發事件）**：`push`、`pull_request` 差在哪？為什麼 PR 檢查通常用 `pull_request`？`workflow_dispatch`、`schedule` 又是做什麼的？
- `runs-on` 是做什麼的？常見的 runner（如 `ubuntu-latest`）代表什麼？
- job 之間預設是平行還是序列？`needs` 關鍵字的作用是什麼？
- 同一個 job 裡的 step 共用什麼（工作目錄、檔案系統）？不同 job 之間為什麼**不**共用檔案系統？

## Q3. 依賴快取、matrix、secrets、artifacts

- 為什麼 CI 要**快取依賴**？請說明 `actions/setup-node` 的 `cache: 'npm'`（或 pnpm/yarn）與手動用 `actions/cache` 兩種做法的差別與時機。
- 快取的 **key** 為什麼通常綁 lockfile 的雜湊（如 `hashFiles('**/package-lock.json')`）？命中（cache hit）與未命中（cache miss）分別會發生什麼？
- **matrix（矩陣）** 策略解決什麼問題？舉例：同一套測試跑在 Node 18 / 20 / 22。
- **secrets** 是什麼？為什麼 API token、部署金鑰要放 repository/environment secrets 而不是寫進 YAML？`${{ secrets.XXX }}` 怎麼取用？為什麼 secrets 預設**不會**傳給來自 fork 的 PR？
- **artifacts（產物）** 是什麼？什麼情況會用 `actions/upload-artifact` / `download-artifact`（例如把 build 產物、測試報告、覆蓋率留存或跨 job 傳遞）？

## Q4. 必要檢查與分支保護

- 什麼是 **branch protection rule（分支保護規則）**？在主幹（main）上通常會開哪些保護？
- 什麼是 **required status checks（必要檢查）**？它如何確保「CI 沒過就不能 merge」？
- 「require PR review」「require branches to be up to date before merging」分別解決什麼問題？
- 為什麼「必要檢查」要搭配 CI 一起設定才有意義？只有 CI、沒設必要檢查會怎樣？

## Q5. 寫一份完整的前端 CI workflow

- 請寫出一份 GitHub Actions workflow，在 `push` 到 main 與對 main 開 `pull_request` 時觸發，依序執行 **install → lint → type check → test → build**。
- 要求：使用 `actions/checkout`、`actions/setup-node`（含依賴快取）、用 `npm ci` 安裝、把 build 產物上傳為 artifact。
- 說明每個 step 的用意，以及為什麼用 `npm ci` 而不是 `npm install`。

## Q6. Dockerfile：前端多階段建置

- 什麼是 **multi-stage build（多階段建置）**？對前端「先 build 出靜態檔、再用輕量 web server 提供」這個情境，多階段能省下什麼？
- 請寫一份前端多階段 Dockerfile：**build stage**（Node 環境 `npm ci && npm run build`）+ **runtime stage**（用 nginx 提供 `dist`/`build` 靜態檔）。
- 為什麼 runtime image 不該包含 node_modules 與原始碼？這對映像檔大小與資安有什麼好處？
- `.dockerignore` 的用途是什麼？層（layer）快取與指令順序（先 copy package.json 再 copy 其餘）為什麼影響 build 速度？

## Q7. pm2：Node 應用 process 管理

- pm2 解決什麼問題？跟直接 `node server.js` 相比多了什麼（自動重啟、開機自啟、日誌、監控）？
- **cluster mode** 是什麼？為什麼能利用多核心 CPU？`instances: 'max'` 代表什麼？
- 請寫一份 `ecosystem.config.js`，設定 app 名稱、cluster 模式、記憶體上限自動重啟（`max_memory_restart`）、環境變數。
- pm2 的日誌與監控怎麼看（`pm2 logs`、`pm2 monit`、`pm2 status`）？`pm2 reload`（零停機）跟 `pm2 restart` 差在哪？

## Q8. 部署：Vercel 與環境變數管理

- Vercel 與 Next.js 的整合有哪些「開箱即用」的好處（自動偵測框架、build、CDN、Serverless/Edge Functions）？
- 什麼是**預覽部署（Preview Deployment）**？每個 PR 產生一個獨立預覽網址對協作、Review 有什麼價值？Production 部署又是怎麼觸發的？
- 環境變數管理：Vercel 的 **Production / Preview / Development** 三種環境變數作用域是什麼？
- 前端環境變數的**安全界線**：為什麼 `NEXT_PUBLIC_` 前綴（或 Vite 的 `VITE_`）的變數會被打包進 client bundle、絕不能放機密？機密（DB 連線、API secret）該放哪、怎麼在 server 端用？

## Q9. Web Vitals / 監控在 CI 與線上的角色

- 監控可以放在兩個地方：**CI 的 pre-deploy 檢查**（如 Lighthouse CI 對每個 PR 量 LCP/CLS/TBT 並設門檻）與**線上 RUM（Real User Monitoring，真實使用者監控）**。兩者的目的與資料來源差在哪？
- 什麼是 **performance budget（效能預算）**？把它設成 CI 的必要檢查有什麼好處與風險（例如偶發抖動造成 flaky）？
- 線上怎麼收集真實 Web Vitals？（提示：`web-vitals` 套件把 LCP/INP/CLS 回報到分析後端）為什麼「實驗室數據」與「真實使用者數據」都需要？（呼應 Q29）
- 除了效能，CI/線上還常監控哪些工程健康度指標（bundle size、錯誤率 / Sentry、部署頻率、變更失敗率）？
