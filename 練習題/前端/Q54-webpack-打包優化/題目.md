# Q54 - Webpack 與打包優化（可規模化前端架構）

現代前端專案動輒上百個模組，瀏覽器不可能一支一支載入原始碼；打包器（bundler）負責把散落的模組、樣式、圖片編譯打包成瀏覽器能吃、又要跑得快的產物。這題把 Webpack 的核心觀念（entry/output/loader/plugin）到規模化優化（Code Splitting、Tree Shaking、長期快取、bundle 分析）串起來，最後對照 Vite/Turbopack 等現代替代方案，並帶到 Module Federation 微前端。

> 本題與 Q22（關鍵渲染路徑）、Q28（圖片懶載入）、Q37（Next.js 渲染策略）互補：本題聚焦「打包器怎麼把程式碼變成最佳化產物」，請對照閱讀。

## Q1. 打包器要解決什麼 & 核心概念

- 沒有打包器的年代，前端會遇到哪些問題？（全域變數污染、`<script>` 順序依賴、HTTP 請求過多、無法用 npm 生態、無法用尚未被瀏覽器支援的語法）打包器如何解決這些？
- 說明 Webpack 五個核心概念的角色：**entry**、**output**、**loader**、**plugin**、**mode**。
- `mode: 'development'` 與 `mode: 'production'` 各自預設開了什麼（例如壓縮、tree shaking、`process.env.NODE_ENV`）？
- 寫一份最小可用的 `webpack.config.js`，包含 entry、output（帶 hash 檔名）、一個 loader、一個 plugin。

## Q2. 模組化、依賴圖，以及 loader vs plugin

- Webpack 如何從 entry 出發建立「依賴圖（dependency graph）」？它怎麼知道要打包哪些檔案？
- **loader** 與 **plugin** 的本質差異是什麼？各自在打包流程的哪個階段介入？
- 舉例說明常見 loader 的用途：`babel-loader`、`ts-loader`、`css-loader` + `style-loader`、`asset modules`（取代舊的 `file-loader`/`url-loader`）。
- 舉例說明常見 plugin 的用途：`HtmlWebpackPlugin`、`MiniCssExtractPlugin`、`DefinePlugin`。
- loader 的執行順序是由左到右還是由右到左？為什麼 `['style-loader', 'css-loader']` 這樣寫？

## Q3. Code Splitting（程式碼分割）

- 為什麼要做 code splitting？只出一支 `bundle.js` 有什麼問題？
- 動態 `import()` 如何觸發分割？它回傳什麼？搭配 React `lazy` + `Suspense` 如何做路由層級分割？
- `optimization.splitChunks` 在做什麼？如何把 `node_modules` 拆成 vendor chunk？拆 vendor 的好處是什麼（與快取的關係）？
- 「路由層級分割」與「元件層級分割」的取捨？分割過細會有什麼副作用？

## Q4. Tree Shaking（搖樹優化）

- Tree Shaking 的原理是什麼？為什麼它**依賴 ESM（`import`/`export`）** 才能做，而 CommonJS（`require`）很難搖？
- `package.json` 的 `"sideEffects"` 欄位是做什麼的？把 CSS import 誤標成無副作用會發生什麼事？
- 為什麼「有做 tree shaking」不代表「一定會被搖掉」？哪些寫法會讓 tree shaking 失效（例如整包 `import * as _ from 'lodash'`、有副作用的模組頂層程式碼）？
- 實務上如何確認某個沒用到的匯出真的被移除了？

## Q5. 快取與檔名 hash（長期快取）

- `[hash]`、`[chunkhash]`、`[contenthash]` 三者差在哪？做長期快取（long-term caching）該用哪一個？為什麼？
- 為什麼要把 webpack runtime 抽成獨立的 `runtime` chunk（`optimization.runtimeChunk`）？不抽會發生什麼？
- 一個典型的長期快取設定長怎樣？（vendor 用 contenthash、HTML 不快取、靜態資源 `immutable`）
- 只改了一支業務程式碼，為什麼有時 vendor chunk 的 hash 也跟著變？如何避免？

## Q6. 效能優化總整理

- 如何用 `webpack-bundle-analyzer` 找出「誰把 bundle 撐大了」？看到某個第三方套件很肥，有哪些處置手段？
- 列出減少 bundle 體積的常見手段：按需引入（`lodash-es`/`import` 具名）、替換肥套件（moment → dayjs）、externals/CDN、polyfill 按需（browserslist）、移除重複依賴。
- production 的壓縮由誰負責？（`TerserPlugin` 壓 JS、`CssMinimizerPlugin` 壓 CSS）Gzip/Brotli 壓縮該在哪一層做？
- 圖片與字型優化：asset modules 的 `asset`/`asset/resource`/`asset/inline` 差異、字型 subset、`font-display`。
- 建置速度優化：`cache: { type: 'filesystem' }`、縮小 loader 的 `include`/`exclude`、`thread-loader`。

## Q7. 現代替代方案與取捨

- Vite 為什麼在 **dev** 環境比 Webpack 快很多？（原生 ESM + esbuild 預打包依賴，dev 不打包、按需編譯）它 **build** 時實際上用什麼？（Rollup）
- Turbopack、esbuild、Rollup、Webpack 各自的定位為何？
- Next.js 內建打包（Webpack / Turbopack）與自行維護 Webpack config 的取捨？
- 什麼情況下「還是該用 / 留在 Webpack」？（複雜自訂 loader/plugin 生態、Module Federation、既有大型專案遷移成本）
