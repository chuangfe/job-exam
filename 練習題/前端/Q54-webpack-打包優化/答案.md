# Q54 - Webpack 與打包優化（可規模化前端架構）解析

打包器的整體心智圖：

```
原始碼（散落的模組）
  entry ──▶ Webpack 從進入點出發，沿著 import/require 建立「依賴圖」
              │
              │ 每個模組經過 loader 轉譯（ts→js、scss→css、圖片→URL…）
              ▼
        依賴圖建構完成
              │ plugin 在各個 hook 介入（產 HTML、抽 CSS、壓縮、注入變數…）
              │ optimization 做 splitChunks / tree shaking / minify
              ▼
        output ──▶ 產出帶 contenthash 的 chunk（app.4f3a.js / vendor.b21c.js …）
```

核心原則：**dev 求快（省略壓縮、開 HMR）、prod 求小求穩（壓縮、tree shaking、長期快取）**。以下逐題解析。

---

## Q1. 打包器要解決什麼 & 核心概念

### 沒有打包器的痛點

- **全域污染 / 順序依賴**：所有 `<script>` 共用 `window`，載入順序錯就壞。
- **請求過多**：幾百支檔案 = 幾百個 HTTP 請求，拖慢首屏。
- **無法用生態**：npm 上的模組多是 CommonJS/ESM，瀏覽器早期不能直接吃。
- **無法用新語法**：JSX、TS、尚未落地的 ES 提案，瀏覽器看不懂。

打包器把這些模組**編譯 + 合併 + 最佳化**成瀏覽器能跑、又能被拆分/快取的產物。

### 五個核心概念

| 概念 | 角色 |
| --- | --- |
| **entry** | 依賴圖的進入點，Webpack 從這裡開始爬 import。可多個。 |
| **output** | 產物輸出設定：資料夾、檔名（可帶 hash）、`publicPath`。 |
| **loader** | **轉譯器**：把非 JS（TS/CSS/圖片）或新語法轉成 Webpack 能處理的模組。作用在「單一檔案」層級。 |
| **plugin** | **擴充器**：介入整個打包生命週期做更廣的事（產 HTML、抽 CSS、壓縮、注入環境變數）。 |
| **mode** | `development` / `production` / `none`，一鍵切換一組預設最佳化。 |

### mode 的預設差異

| | `development` | `production` |
| --- | --- | --- |
| 壓縮（minify） | 否 | 是（Terser） |
| Tree Shaking | 否（`usedExports` 關） | 是 |
| `process.env.NODE_ENV` | `'development'` | `'production'` |
| source map / 建置速度 | 快、好除錯 | 慢、產物最小 |
| HMR 常搭配 | 是 | 否 |

### 最小可用 config

```js
// webpack.config.js
const path = require('path');
const HtmlWebpackPlugin = require('html-webpack-plugin');

module.exports = {
  mode: 'production',
  entry: './src/index.js',
  output: {
    path: path.resolve(__dirname, 'dist'),
    filename: '[name].[contenthash].js', // 帶內容 hash，利於長期快取
    clean: true,                          // 每次 build 先清 dist
  },
  module: {
    rules: [
      {
        test: /\.jsx?$/,
        exclude: /node_modules/,
        use: 'babel-loader', // loader：把 JSX/ESNext 轉成相容 JS
      },
    ],
  },
  plugins: [
    new HtmlWebpackPlugin({ template: './src/index.html' }), // plugin：自動產 HTML 並注入 script
  ],
};
```

## Q2. 模組化、依賴圖，以及 loader vs plugin

### 依賴圖

Webpack 從 `entry` 開始，解析每個檔案裡的 `import` / `require` / `import()`，遞迴地把所有被依賴的模組串成一張**有向圖（dependency graph）**。沒有被任何路徑引用到的模組就不會進包（這也是 tree shaking 的基礎）。

### loader vs plugin（面試常考）

| | loader | plugin |
| --- | --- | --- |
| 本質 | 一個**函式**，輸入檔案原始內容、輸出轉換後內容 | 一個帶 `apply(compiler)` 的**類別**，訂閱 compiler/compilation 的 hook |
| 作用範圍 | 單一檔案的「轉譯」 | 整個打包流程的「加工」 |
| 何時介入 | 模組被載入、加入依賴圖時 | 打包生命週期的各個階段（emit、optimize…） |
| 例子 | 把 `.ts` 轉 `.js`、把 `.scss` 轉 CSS 字串 | 產 HTML、把 CSS 抽成獨立檔、壓縮、注入變數 |

一句話：**loader 管「一個檔案怎麼被讀進來」，plugin 管「整包產物怎麼被加工出去」。**

### 常見 loader

```js
module: {
  rules: [
    { test: /\.tsx?$/, use: 'babel-loader', exclude: /node_modules/ },
    // CSS：css-loader 解析 @import/url()，style-loader 把 CSS 注入 <style>
    { test: /\.css$/, use: ['style-loader', 'css-loader'] },
    // 圖片/字型：Webpack 5 內建 asset modules，不需再裝 file-loader/url-loader
    { test: /\.(png|jpe?g|svg|woff2?)$/, type: 'asset' },
  ],
}
```

- **loader 執行順序是「由右到左（由下到上）」**：`['style-loader', 'css-loader']` 會**先** `css-loader`（把 CSS 變成 JS 模組）**再** `style-loader`（把它插進 DOM）。順序寫反會壞。

### 常見 plugin

```js
const HtmlWebpackPlugin = require('html-webpack-plugin');
const MiniCssExtractPlugin = require('mini-css-extract-plugin');
const webpack = require('webpack');

plugins: [
  new HtmlWebpackPlugin({ template: './src/index.html' }),        // 產 HTML + 自動注入帶 hash 的 script/css
  new MiniCssExtractPlugin({ filename: '[name].[contenthash].css' }), // 把 CSS 抽成獨立檔（prod 用它取代 style-loader）
  new webpack.DefinePlugin({                                       // 編譯期把常數字面替換進程式碼
    'process.env.API_BASE': JSON.stringify(process.env.API_BASE),
  }),
]
```

> prod 環境 CSS 用 `MiniCssExtractPlugin.loader` 取代 `style-loader`，才能抽成可被獨立快取的 `.css` 檔。

## Q3. Code Splitting

### 為什麼要分割

單一 `bundle.js` 的問題：**首屏要載入全站程式碼**（包含使用者這次根本用不到的頁面）、任何一行改動都讓整包 hash 變、無法平行下載。分割後可以**按需載入 + 分層快取**。

### 動態 import() + 路由分割

`import()` 回傳一個 **Promise**，Webpack 會把它指向的模組單獨切成一個 chunk，執行到那行才去下載。搭配 React：

```js
import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

// 每個頁面被切成獨立 chunk，進到該路由才下載
const Home = lazy(() => import('./pages/Home'));
const Dashboard = lazy(() => import('./pages/Dashboard'));

function App() {
  return (
    <Suspense fallback={<div>載入中…</div>}>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/dashboard" element={<Dashboard />} />
      </Routes>
    </Suspense>
  );
}
```

### splitChunks 與 vendor 拆分

`optimization.splitChunks` 會把「被多個 chunk 共用的模組」或「node_modules 的第三方套件」抽出來共用，避免重複打包：

```js
optimization: {
  splitChunks: {
    chunks: 'all',
    cacheGroups: {
      vendor: {
        test: /[\\/]node_modules[\\/]/,
        name: 'vendors',
        priority: 10,
      },
    },
  },
}
```

- **拆 vendor 的核心好處是快取**：第三方套件很少變，抽成獨立 chunk 後，只要它們沒升級，vendor chunk 的 contenthash 就不變，使用者更新版本時**只需重新下載變動的業務 chunk**，vendor 直接吃瀏覽器快取。

### 分割的取捨

- **路由層級**是投報率最高的分割（首屏只載當前頁）。
- **元件層級**（如很重的圖表、富文本編輯器、Modal 內容）適合對「大又不一定會用到」的元件做 `lazy`。
- **過細的副作用**：chunk 太多 → HTTP 請求數暴增、每個 chunk 都有 runtime 開銷、瀑布式載入反而更慢。實務上用 `minSize`/`maxSize` 控制粒度，抓平衡。

## Q4. Tree Shaking

### 原理與為何依賴 ESM

Tree Shaking = 靜態分析後移除「沒被用到的匯出」。它依賴 **ESM 的靜態結構**：`import`/`export` 在**編譯期**就能確定「誰匯出什麼、誰用了什麼」，因為 import 只能寫在頂層、名稱是靜態的。

CommonJS 難搖的原因：`require` 是**執行期**的函式呼叫，`module.exports` 可以動態賦值：

```js
// CommonJS：這些在編譯期無法靜態分析
const name = cond ? 'a' : 'b';
const mod = require('./' + name);   // 路徑執行期才知道
module.exports[dynamicKey] = fn;    // 匯出的 key 執行期才決定
```

打包器無法安全判斷哪些匯出沒被用到，只能整包保留。**所以優先用支援 ESM 版本的套件**（如 `lodash-es` 而非 `lodash`）。

### sideEffects

`package.json` 的 `"sideEffects"` 告訴 Webpack「這個套件的模組，除了匯出之外有沒有副作用」：

```jsonc
// 完全沒副作用 → Webpack 可放心把沒被 import 的模組整個移除
{ "sideEffects": false }

// 有副作用的檔案要列出來，避免被誤刪
{ "sideEffects": ["*.css", "./src/polyfill.js"] }
```

- **陷阱**：若專案標了 `"sideEffects": false` 卻用 `import './styles.css'`（CSS import 是靠副作用把樣式注入的），tree shaking 會認為「這個 import 沒用到匯出值」而**把 CSS 整個搖掉**，畫面就沒樣式了。所以 CSS/polyfill 一定要列進 `sideEffects` 白名單。

### 為什麼「有做」不代表「會被搖掉」

- 整包引入：`import * as _ from 'lodash'` 或 `import _ from 'lodash'`（CJS 版）會把整包拉進來。改用具名 + ESM：`import debounce from 'lodash-es/debounce'`。
- 模組頂層有副作用（改全域、註冊、`console`），Webpack 不敢刪。
- 被 Babel 轉成 CommonJS：確認 `@babel/preset-env` 的 `modules: false`，保留 ESM 讓 Webpack 自己處理。

### 如何確認真的被移除

用 `webpack-bundle-analyzer` 或搜尋產物字串；也可在原始碼放一個明顯不會用到的具名匯出，build 後 grep 產物看它在不在。

## Q5. 快取與檔名 hash（長期快取）

### 三種 hash

| placeholder | 何時變 | 適用 |
| --- | --- | --- |
| `[hash]` | **整包**任一檔案變，所有檔名都變 | 幾乎不用（改一行全站快取失效） |
| `[chunkhash]` | 該 **chunk** 內容變才變 | 可用，但 CSS 從 JS 抽出時仍會連動 |
| `[contenthash]` | 依**單一檔案內容**計算 | **長期快取首選**，JS/CSS 各自獨立 |

長期快取要用 `[contenthash]`：內容沒變 → 檔名沒變 → 瀏覽器直接命中快取。

### runtimeChunk

Webpack 會產生一小段 **runtime**（記錄「哪個 chunk 對應哪個 hash 檔名」的 manifest）。若這段 runtime 內嵌在 vendor 裡，那麼**每次業務 chunk 換 hash，manifest 內容就變 → 連帶讓 vendor 的 hash 也變**，長期快取失效。抽出來就能隔離：

```js
optimization: {
  runtimeChunk: 'single', // 抽出獨立 runtime.js，隔離 manifest 變動
  moduleIds: 'deterministic', // 讓模組 id 穩定，避免無關改動導致 hash 漂移
}
```

### 典型長期快取設定

```js
output: {
  filename: '[name].[contenthash].js',
  chunkFilename: '[name].[contenthash].js',
},
// 搭配 CSS
new MiniCssExtractPlugin({ filename: '[name].[contenthash].css' }),
```

伺服器端配合：
- 帶 hash 的 JS/CSS/圖片 → `Cache-Control: public, max-age=31536000, immutable`
- `index.html` → `no-cache`（每次協商，才能拿到新的檔名引用）

### 為何改一行業務碼 vendor 也變

多半是 **module id 不穩定**（早期用遞增數字 id，插入模組會讓後面全體位移）或 **runtime 內嵌在 vendor**。解法就是上面的 `moduleIds: 'deterministic'` + `runtimeChunk: 'single'`。

## Q6. 效能優化總整理

### bundle 分析

```js
const { BundleAnalyzerPlugin } = require('webpack-bundle-analyzer');
plugins: [ new BundleAnalyzerPlugin() ]; // build 後開互動式 treemap，一眼看出誰最肥
```

看到某套件很肥的處置：換輕量替代、只按需引入、丟到 CDN（externals）、或用動態 import 延後載入。

### 減少 bundle 體積

| 手段 | 說明 |
| --- | --- |
| 按需 / 具名引入 | `import debounce from 'lodash-es/debounce'`，別整包 import |
| 換肥套件 | `moment`（含全語系，肥）→ `dayjs`；大型 UI 庫確認支援 tree shaking |
| externals + CDN | 把 React 等交給 CDN，`externals: { react: 'React' }`，不進 bundle |
| polyfill 按需 | 用 `browserslist` + `@babel/preset-env` 的 `useBuiltIns: 'usage'`，只補目標瀏覽器缺的 |
| 去重複依賴 | 檢查同一套件多版本（`npm dedupe` / analyzer 看重複塊） |
| 動態載入 | 大又不常用的功能改 `import()` 延後 |

### 壓縮由誰負責

```js
const TerserPlugin = require('terser-webpack-plugin');
const CssMinimizerPlugin = require('css-minimizer-webpack-plugin');

optimization: {
  minimize: true, // production 預設就開
  minimizer: [ new TerserPlugin(), new CssMinimizerPlugin() ],
}
```

- Terser 壓 JS、CssMinimizer 壓 CSS（production 下 Terser 預設已啟用）。
- **Gzip / Brotli 屬於「傳輸壓縮」，正解是在伺服器 / CDN 層做**（多數 CDN 自動 Brotli）。也可用 `CompressionWebpackPlugin` 預先產 `.br`/`.gz` 讓伺服器直接吐。

### 圖片與字型

- **asset modules**：`asset/resource`（輸出成獨立檔，回傳 URL）、`asset/inline`（轉成 base64 內嵌，省請求但撐大體積）、`asset`（依 `maxSize` 自動二選一，小圖 inline、大圖 resource）。
- 字型：做 **subset**（只留用到的字元，中文字型尤其重要）、`font-display: swap` 避免 FOIT、優先 `woff2`。
- 圖片：優先 WebP/AVIF、響應式尺寸，大圖走 lazy load（見 Q28）。

### 建置速度

```js
cache: { type: 'filesystem' }, // 二次 build 用磁碟快取，大幅加速
module: {
  rules: [
    { test: /\.jsx?$/, include: path.resolve(__dirname, 'src'), use: 'babel-loader' }, // 縮小掃描範圍
  ],
},
// 重活可加 thread-loader 開多執行緒（小專案反而變慢，斟酌）
```

## Q7. 現代替代方案與取捨

### Vite 為何 dev 快

- **dev 不打包**：Vite 直接用瀏覽器原生 ESM，頁面 import 什麼就**按需**編譯什麼那一支，啟動幾乎瞬間（不像 Webpack 要先把整張依賴圖打包才給你看第一屏）。
- 依賴預打包用 **esbuild**（Go 寫的，比 JS-based 的 Webpack 快數十倍）。
- **build 時實際用 Rollup**（產物穩定、tree shaking 好），所以 dev 與 prod 是兩套引擎——這點要注意「dev 正常、build 出錯」的差異。

### 各工具定位

| 工具 | 定位 |
| --- | --- |
| **esbuild** | 極快的轉譯 + 打包底層引擎，常被別的工具當作依賴使用 |
| **Rollup** | 函式庫打包首選，ESM/tree shaking 乾淨，Vite build 底層 |
| **Vite** | 開發體驗導向的應用打包工具（dev=原生 ESM+esbuild，build=Rollup） |
| **Turbopack** | Vercel 出的 Rust 打包器，主打增量、大型專案，整合進 Next.js |
| **Webpack** | 生態最成熟、可配置性最高，複雜自訂與 Module Federation 的老大哥 |

### Next.js 內建 vs 自維護 config

- Next.js 幫你封裝好打包（Webpack，新版可切 Turbopack），**多數情況不該自己碰**，改動 config 容易失去官方最佳化與升級保障。
- 只有需要特殊 loader/plugin 時，才透過 `next.config.js` 的 `webpack` 欄位微調。

### 何時該留在 Webpack

- 依賴大量成熟的自訂 loader/plugin、需要 **Module Federation**（Webpack 生態最完整）、既有大型專案遷移成本 > 收益、或需要極細緻的打包控制。新專案若無這些包袱，多半優先選 Vite。

## Q8. Module Federation（微前端）

### 解決什麼

讓**多個獨立部署的應用，在執行期直接共享彼此的模組**，不需重新發版或把對方當 npm 依賴打進自己包裡。傳統 iframe 隔離重、體驗差；npm 共用又得對方每次改都重發、你重新打包。Module Federation 讓 `host` **在瀏覽器執行期**動態載入 `remote` 暴露的模組，各團隊可獨立開發部署。

### host / remote / 三個設定

```js
// remote（提供端）webpack.config.js
const { ModuleFederationPlugin } = require('webpack').container;

new ModuleFederationPlugin({
  name: 'cart',
  filename: 'remoteEntry.js',
  exposes: {                       // 我對外暴露哪些模組
    './CartWidget': './src/CartWidget',
  },
  shared: { react: { singleton: true }, 'react-dom': { singleton: true } },
});
```

```js
// host（消費端）webpack.config.js
new ModuleFederationPlugin({
  name: 'shell',
  remotes: {                       // 我要從哪些 remote 拿模組
    cart: 'cart@https://cart.example.com/remoteEntry.js',
  },
  shared: { react: { singleton: true }, 'react-dom': { singleton: true } },
});
```

```js
// host 中執行期動態載入 remote 元件
const CartWidget = React.lazy(() => import('cart/CartWidget'));
```

| 設定 | 作用 |
| --- | --- |
| `exposes` | remote 對外開放的模組清單 |
| `remotes` | host 要消費的遠端應用位址 |
| `shared` | 宣告共用依賴（React 等），避免各自打一份、並協調版本 |

### 為何 shared + singleton 對 React 重要

React 要求整棵樹**共用同一個 React 實例**（hooks、context 才能正常運作）。若 host 與 remote 各自打包一份 React，會出現「Invalid hook call」「兩份 context 互相看不到」等錯誤。`singleton: true` 強制全域只用一份；版本不一致時會退回其一並警告，需靠 `requiredVersion` 協調。

### 風險與代價

- **版本協調**：shared 依賴的版本需跨團隊對齊，否則執行期衝突。
- **型別共享**：TS 型別跨 remote 不會自動同步，需額外方案（如發佈型別包）。
- **執行期耦合 + 可用性**：remote 掛了會拖累 host，必須包 **錯誤邊界 + fallback**。
- **心智與除錯成本**上升，小團隊/單體應用通常不划算——它是給「多團隊、需獨立部署」的大型組織的解法。

**核心心法**：Webpack 的優化都圍繞兩件事——**讓使用者這次少載一點（分割、tree shaking、壓縮、按需）**，以及**讓使用者下次能吃快取（contenthash + vendor 拆分 + runtime 隔離）**。現代工具（Vite/Turbopack）改善的主要是開發體驗與速度，但「打包優化」的觀念是共通的。
