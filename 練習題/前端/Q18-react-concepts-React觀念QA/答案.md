# React 觀念 Q&A 解析

## Q1. 虛擬 DOM 與 reconciliation（協調）？
- 虛擬 DOM 是描述 UI 的 JS 物件樹。狀態變更時產生新 VDOM，與舊樹做 diff，只把差異套用到真實 DOM，降低昂貴的 DOM 操作。
- React diff 採啟發式 O(n)：
  - 不同型別節點直接整顆重建；
  - 同型別比對 props；
  - 同層 children 用 key 比對。
- Fiber 架構讓 reconciliation 可中斷、分片（time slicing），支援並行渲染與優先級排程。

## Q2. key 的作用？為何不要用 index 當 key？
- key 讓 React 在列表 diff 時辨識哪些項目新增/移動/刪除，重用對應的 Fiber 與 DOM。
- 用 index 當 key 時，若列表會「插入/刪除/排序」，index 與內容對不上，導致狀態錯位（如輸入框值跑到別的列）、非必要重渲染。穩定且唯一的 id 才安全。

## Q3. useState / useEffect 如何運作？
- Hooks 依「呼叫順序」存於 Fiber 的 hook 鏈結串列，故不可在條件/迴圈中呼叫。
- useState 回傳當前值與 setter；setter 觸發重渲染，函式式更新 `setX(prev => ...)` 可避免讀到過期閉包值。
- useEffect 在「渲染並 commit 到畫面後」非同步執行；useLayoutEffect 在 commit 後、瀏覽器繪製前同步執行（量測 DOM 用）。

## Q4. useEffect 依賴陷阱與清理？
- 依賴陣列要列出 effect 內用到的所有外部值，否則會讀到過期值（stale closure）。
- 空陣列 `[]` 只在掛載執行一次；省略陣列每次渲染都執行。
- 回傳清理函式在「下次 effect 執行前」與「卸載時」呼叫，用來取消訂閱、清 timer、abort fetch。

## Q5. useMemo / useCallback / React.memo 何時用？
- **React.memo**：以 props 淺比較跳過子元件重渲染。
- **useCallback**：記憶函式參考，避免每次渲染產生新函式導致 memo 子元件失效或 effect 反覆觸發。
- **useMemo**：記憶昂貴計算結果或穩定物件/陣列參考。
- 注意：本身有成本，不要無腦包；用在「真的昂貴」或「需要穩定參考傳給 memo 子元件/依賴陣列」時才有意義。

## Q6. Context 與重渲染問題？
- Context value 變更時，所有 consumer 都會重渲染（不論用到的是哪部分）。
- 優化：
  - 拆分多個 Context（狀態與 dispatch 分開）；
  - value 用 useMemo 穩定；
  - 把不依賴 context 的子樹用 children 傳入避免重渲染；
  - 或改用外部 store（Zustand/Redux）做選擇性訂閱。

## Q7. 受控 vs 非受控元件？
- **受控**：值由 state 驅動（`value` + `onChange`），單一資料來源、易驗證。
- **非受控**：值由 DOM 自己管，用 ref 讀取（`defaultValue`）。表單簡單或整合非 React 程式時用。

## Q8. 批次更新（batching）？
- React 18 起，所有情境（含 setTimeout、Promise、原生事件）預設自動批次合併多次 setState 成一次渲染。
- 需要強制同步更新可用 `flushSync`（少用，會犧牲效能）。
