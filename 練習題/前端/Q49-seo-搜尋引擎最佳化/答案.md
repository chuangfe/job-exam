# 前端 SEO（搜尋引擎最佳化）解析

## Q1. 基本 meta 與語意化 HTML？

- **`<title>`**：頁面標題，是**最重要的 on-page SEO 訊號之一**。它會出現在 SERP 的藍色標題連結、瀏覽器分頁、以及被分享時的預設標題。建議把重點關鍵字放前面，長度控制在約 **50–60 字元**（過長會被 Google 截斷或改寫）。每頁 title 應**獨一無二**。
- **`<meta name="description">`**：頁面摘要。**它本身不是排名因素**，但會（可能）成為 SERP 上標題底下那段說明文字，直接影響**點擊率（CTR）**。建議約 **120–160 字元**，寫得像廣告文案、包含關鍵字。Google 有時會無視你寫的、自己從內文擷取。
- **`<meta name="viewport">`**：`width=device-width, initial-scale=1`。讓頁面在行動裝置正確縮放。Google 採 **mobile-first indexing**（用行動版內容做索引），沒設 viewport 的頁面會被判定為「非行動裝置友善」，影響行動排名。

```html
<meta name="viewport" content="width=device-width, initial-scale=1" />
```

- **語意化 HTML**：用有語意的標籤（`<header>`、`<nav>`、`<main>`、`<article>`、`<section>`、`<aside>`、`<footer>`）而不是一片 `<div>`。好處：爬蟲更容易理解「哪塊是主要內容、哪塊是導覽/側欄」，也讓螢幕閱讀器與可及性（a11y）更好——**a11y 與 SEO 高度重疊**。
- **標題階層**：`<h1>` 是頁面主標題，`<h2>`~`<h6>` 是層層小節，形成大綱（outline）。實務建議**一頁一個 `<h1>`**（技術上 HTML5 允許多個，但單一 h1 最清楚），且**不要跳階層**（h1 直接跳 h3 會讓大綱斷層，害爬蟲與螢幕閱讀器誤判結構）。
- **圖片 `alt`**：描述圖片內容的替代文字。對**視障使用者（螢幕閱讀器）**與**圖片搜尋 SEO** 都有用；圖片載入失敗時也會顯示 alt。裝飾性圖片可用空 `alt=""` 讓螢幕閱讀器略過。

## Q2. Open Graph 與 Twitter Card？

- **Open Graph（OG）**：由 Facebook 提出的協定（`property` 屬性、`og:` 前綴），解決的是**社群分享時的預覽卡片**——當網址被貼到 Facebook、LINE、Slack、Discord 等平台時，要顯示什麼標題、描述、縮圖。**它跟 Google 排名基本無關**，但決定了分享出去「好不好看、點不點得下去」，實務上非常重要。
- **關鍵標籤**：
  - `og:title`：分享卡標題（可與 `<title>` 不同）。
  - `og:description`：分享卡描述。
  - `og:image`：預覽縮圖（**最關鍵**，圖好不好看直接影響點擊）。
  - `og:type`：內容類型（`website`、`article`、`product`…）。
  - `og:url`：此內容的正式網址（通常等同 canonical）。
- **`og:image` 實務要點**：建議 **1200×630（1.91:1）**、用**絕對網址**（`https://...`，不能是相對路徑，否則爬蟲抓不到）、檔案不要過大（一般建議 < 5MB，越小越好以免抓取逾時）、格式用 JPG/PNG。
- **Twitter Card**：Twitter/X 專用的一組 `twitter:` 標籤（用 `name` 屬性）。`twitter:card` 常見值 `summary`（小圖）與 `summary_large_image`（大圖）。**若沒寫 twitter 標籤，Twitter 會退回（fallback）讀 Open Graph**，所以最省事的做法是把 OG 寫齊，Twitter 再補一個 `twitter:card` 指定卡片樣式即可。

```html
<!-- Open Graph -->
<meta property="og:title" content="用 React 打造高效能表單" />
<meta property="og:description" content="從受控元件到虛擬化，一步步優化大型表單。" />
<meta property="og:image" content="https://example.com/og/form-guide.png" />
<meta property="og:type" content="article" />
<meta property="og:url" content="https://example.com/blog/react-forms" />

<!-- Twitter Card（其餘欄位會 fallback 到 OG） -->
<meta name="twitter:card" content="summary_large_image" />
<meta name="twitter:title" content="用 React 打造高效能表單" />
<meta name="twitter:image" content="https://example.com/og/form-guide.png" />
```

- **Next.js App Router** 用 Metadata API 的 `openGraph` / `twitter` 欄位產生（見 Q6 範例）。

## Q3. Canonical URL？

- **重複內容問題**：同一份內容可透過多個網址存取，例如：
  - `https://example.com/shop?color=red&sort=price`（各種 query 排列組合）
  - `www.example.com` vs `example.com`、`http://` vs `https://`
  - 分頁、追蹤參數（`?utm_source=...`）、大小寫、結尾斜線差異。
  Google 看到多個近乎相同的頁面時，會**分散權重、且不確定該索引哪一個**。
- **`<link rel="canonical">`**：告訴搜尋引擎「這一組重複頁面中，**這個**才是正式版本，請把權重集中到它」。注意它是**強烈建議（hint），不是命令**——Google 可能在有把握時忽略你的指定，但正確設定仍是最佳實務。
- **自我參照 canonical**：每頁都指向「自己乾淨的正式網址」。好處是即使頁面被加上 UTM 等參數存取，canonical 仍指回乾淨版，避免參數版被當成獨立頁面索引。
- **用絕對路徑**：canonical **必須是完整絕對網址**（含協定與網域），不要用相對路徑。

```html
<link rel="canonical" href="https://example.com/blog/react-forms" />
```

## Q4. 結構化資料 JSON-LD？

- **結構化資料**：用機器可讀的格式，把頁面內容「標註」成搜尋引擎懂的實體與屬性（這是文章、作者是誰、發布時間、這是商品、價格多少…）。**Schema.org** 是 Google/Bing/Yahoo 共同維護的**詞彙表（vocabulary）**，定義有哪些型別與屬性；**JSON-LD** 則是**表達格式**（用一段 JSON）。Google 官方**推薦 JSON-LD**，因為它獨立於 HTML 版面（一整包 `<script>`，不必散落在 DOM 各處，最好維護），相較之下 microdata/RDFa 要把屬性塞進標籤裡、容易破。
- **好處**：讓頁面有機會拿到 **rich results（複合式搜尋結果）**——例如食譜的星等、商品的價格與庫存、麵包屑導覽、FAQ 可直接在 SERP 展開。這能**提升版面佔比與點擊率**（但不保證一定顯示，也不直接等於排名上升）。
- **常見型別**：
  - **`Article`**：部落格文章、新聞。標註 headline、作者、發布/更新時間、圖片。
  - **`Product`**：商品頁。標註名稱、圖片、價格、幣別、庫存、評價（`AggregateRating` / `Review` 可拿到星等）。
  - **`BreadcrumbList`**：麵包屑，讓 SERP 顯示「首頁 › 分類 › 文章」路徑。
  - **`FAQPage`**：常見問答頁，符合規範時 SERP 可折疊展開問答。
- **放哪裡 / 一致性**：JSON-LD 用 `<script type="application/ld+json">` 放在 `<head>` 或 `<body>` 皆可。**內容必須與使用者實際看得到的頁面內容一致**——例如你標了 FAQ，但頁面上根本沒那段問答，屬於**結構化資料濫用（spammy）**，可能被 Google 施以人工處罰。
- **驗證工具**：Google 的 [Rich Results Test]、[Schema Markup Validator]，以及 Search Console 的「複合式搜尋結果」報告。

```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "用 React 打造高效能表單",
  "image": ["https://example.com/og/form-guide.png"],
  "datePublished": "2026-07-20T09:00:00+08:00",
  "dateModified": "2026-07-20T09:00:00+08:00",
  "author": { "@type": "Person", "name": "Ian Chuang" }
}
```

```json
{
  "@context": "https://schema.org",
  "@type": "BreadcrumbList",
  "itemListElement": [
    { "@type": "ListItem", "position": 1, "name": "首頁", "item": "https://example.com" },
    { "@type": "ListItem", "position": 2, "name": "部落格", "item": "https://example.com/blog" },
    { "@type": "ListItem", "position": 3, "name": "用 React 打造高效能表單" }
  ]
}
```

```json
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "受控元件會不會拖慢表單？",
      "acceptedAnswer": { "@type": "Answer", "text": "大型表單可用非受控或欄位切分來降低重繪成本。" }
    }
  ]
}
```

## Q5. `sitemap.xml` 與 `robots.txt`？

- **`sitemap.xml`**：列出網站「希望被收錄」的網址清單，**幫助搜尋引擎更快、更完整地發現頁面**（尤其是內部連結少、或新站/大站）。它是**輔助發現**，**不保證收錄**。每個 `<url>` 常見欄位：`<loc>`（網址）、`<lastmod>`（最後修改時間，幫爬蟲判斷要不要重抓）；`changefreq`、`priority` 現今 Google 幾乎忽略。
- **`robots.txt`**：放在網站根目錄，控制爬蟲**能不能「爬取（crawl）」**某些路徑。重點觀念：
  - `Disallow` 是「**不要來爬這個路徑**」，**不是「不要索引」**。
  - **陷阱**：被 `Disallow` 的頁面**仍可能因為外部連結被索引**（顯示在 SERP，只是沒有描述、寫著「因 robots.txt 限制無法提供說明」）。因為 Google 沒被允許進去讀內容，也就讀不到你頁面裡的 `noindex`。
- **正確地「不想被索引」**：用 **`<meta name="robots" content="noindex">`**（或 HTTP header `X-Robots-Tag: noindex`），並且**允許**該頁被爬取（否則爬蟲讀不到 noindex 指令，反而失效）。一句話：
  - 要「別索引這頁」→ 用 `noindex`（且不要在 robots.txt 擋它）。
  - 要「別浪費爬取預算在某些路徑」（如後台、API）→ 用 robots.txt `Disallow`。

```txt
# robots.txt
User-agent: *
Allow: /
Disallow: /admin/
Disallow: /api/

Sitemap: https://example.com/sitemap.xml
```

- **產生方式**：小站可手寫；大站用建置工具/框架自動產生（見 Q6 的 `sitemap.ts` / `robots.ts`）。

## Q6. 渲染方式與 SEO？

- **CSR（純前端渲染）的風險**：傳統 SPA 首包 HTML 幾乎是 `<div id="root"></div>` 空殼，內容全靠 JS 在瀏覽器抓資料、渲染。若爬蟲**不執行 JS 或執行失敗/逾時**，看到的就是空頁——**內容等於不存在**。
- **Google 會執行 JS，但仍建議 SSR/SSG**：Google 採**兩波索引（two-wave indexing）**——先抓 HTML，之後把需要 JS 的頁面排進「渲染佇列」再跑 JS。問題是：(1) 第二波可能**延遲數天**；(2) 消耗**爬取預算（crawl budget）**；(3) **其他爬蟲多半不跑 JS**——Bing 較弱、而 **Facebook/LINE/Twitter 等社群分享爬蟲完全不執行 JS**，所以純 CSR 頁面分享出去往往抓不到 OG 標籤、預覽是空的。
- **SSR / SSG 有利的原因**：伺服器直接把**完整 HTML（含內容與 meta）**吐給爬蟲，第一波就拿得到、meta 也就位，索引更快更穩，社群預覽也正常。
- **hydration**：SSR/SSG 送出的是「靜態 HTML」，React 再在瀏覽器把事件與狀態「接上」讓它可互動，這個過程叫 hydration。對 **SEO**：內容已在 HTML 裡，爬蟲不受影響（好事）。對**使用者體驗**：畫面雖已顯示，但在 hydration 完成前**點了沒反應**；若 JS bundle 太大會拖長這段「看得到卻不能用」的時間，並可能傷害 **INP**（見 Q7）。
- **Next.js App Router Metadata API**：
  - **靜態 `metadata`**：內容固定的頁面直接匯出物件。
  - **動態 `generateMetadata`**：title/description/OG 需依資料（如文章標題）決定時使用；它是 async，可抓資料。Next.js 會對同一路由的 `fetch` 做請求記憶化（memoization），所以 `generateMetadata` 與頁面元件各自 fetch 同一支 API 不會真的打兩次。

```tsx
// app/blog/[slug]/page.tsx — 動態 metadata
import type { Metadata } from 'next'

type Props = { params: Promise<{ slug: string }> }

export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { slug } = await params
  const post = await fetch(`https://api.example.com/posts/${slug}`).then(r => r.json())
  const url = `https://example.com/blog/${slug}`
  return {
    title: post.title,
    description: post.excerpt,
    alternates: { canonical: url },            // canonical
    openGraph: {
      title: post.title,
      description: post.excerpt,
      type: 'article',
      url,
      images: [{ url: post.ogImage, width: 1200, height: 630 }],
    },
    twitter: { card: 'summary_large_image', title: post.title, images: [post.ogImage] },
  }
}

export default async function Post({ params }: Props) {
  const { slug } = await params
  const post = await fetch(`https://api.example.com/posts/${slug}`).then(r => r.json())
  const jsonLd = {
    '@context': 'https://schema.org',
    '@type': 'Article',
    headline: post.title,
    datePublished: post.publishedAt,
    author: { '@type': 'Person', name: post.author },
  }
  return (
    <article>
      {/* JSON-LD：官方建議直接內嵌 script */}
      <script
        type="application/ld+json"
        dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
      />
      <h1>{post.title}</h1>
      {/* ...內文... */}
    </article>
  )
}
```

```tsx
// app/layout.tsx — 靜態 metadata（含全站預設與 metadataBase）
import type { Metadata } from 'next'

export const metadata: Metadata = {
  metadataBase: new URL('https://example.com'), // 讓相對圖片路徑自動補成絕對網址
  title: { default: 'Example 站台', template: '%s | Example' }, // 子頁 title 會套模板
  description: '前端工程筆記與教學。',
}
```

```ts
// app/sitemap.ts — 自動產生 /sitemap.xml
import type { MetadataRoute } from 'next'

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const posts = await fetch('https://api.example.com/posts').then(r => r.json())
  const postUrls = posts.map((p: { slug: string; updatedAt: string }) => ({
    url: `https://example.com/blog/${p.slug}`,
    lastModified: p.updatedAt,
  }))
  return [
    { url: 'https://example.com', lastModified: new Date() },
    ...postUrls,
  ]
}
```

```ts
// app/robots.ts — 自動產生 /robots.txt
import type { MetadataRoute } from 'next'

export default function robots(): MetadataRoute.Robots {
  return {
    rules: { userAgent: '*', allow: '/', disallow: ['/admin/', '/api/'] },
    sitemap: 'https://example.com/sitemap.xml',
  }
}
```

## Q7. 效能、Core Web Vitals 與多語系？

- **效能為何是 SEO 議題**：Google 明確把 **Core Web Vitals（LCP 最大內容繪製、INP 互動到下次繪製、CLS 版面位移）** 納入「頁面體驗」排名訊號。它通常不是決勝因素（**內容相關性仍是主角**），但在其他條件相近時，體驗好的頁面較有優勢；差的 CWV 也會提高跳出率、間接影響。詳見 **Q29**。
- **同時傷 CWV 與 SEO 的常見前端問題**：
  - 巨大且未優化的圖片、沒設 `width`/`height` → 拖慢 **LCP**、造成 **CLS**。
  - 過大的 JS bundle、太多第三方腳本 → 拖長 hydration、傷 **INP**。
  - 字型 FOIT/FOUT、廣告/嵌入內容擠動版面 → **CLS**。
  - 純 CSR 導致內容延遲出現（見 Q6）→ LCP 差、索引風險。
- **多語系 `hreflang`**：告訴 Google「同一內容有其他語言/地區版本」，讓它對不同語系使用者顯示對應版本、避免被當重複內容。每個語言版本都要**互相列出所有版本（含自己）**，語言碼用 ISO（`zh-Hant`、`zh-Hant-TW`、`en`、`ja`…）。`x-default` 指「不符合任何指定語系時的預設頁」。

```html
<link rel="alternate" hreflang="zh-Hant" href="https://example.com/zh/page" />
<link rel="alternate" hreflang="en" href="https://example.com/en/page" />
<link rel="alternate" hreflang="x-default" href="https://example.com/en/page" />
```

在 Next.js 可用 Metadata 的 `alternates.languages` 產生：

```tsx
export const metadata: Metadata = {
  alternates: {
    canonical: 'https://example.com/zh/page',
    languages: {
      'zh-Hant': 'https://example.com/zh/page',
      en: 'https://example.com/en/page',
      'x-default': 'https://example.com/en/page',
    },
  },
}
```

- **SEO 友善前端 checklist**：
  - 每頁**獨一無二**的 `<title>` 與 `description`；正確 `viewport`。
  - **語意化 HTML**、單一 `<h1>`、正確標題階層、圖片有 `alt`。
  - **canonical** 自我參照、用絕對網址。
  - **Open Graph + Twitter Card**（`og:image` 用絕對網址、1200×630）。
  - 適當的 **JSON-LD 結構化資料**，且與頁面內容一致。
  - `sitemap.xml` 與 `robots.txt` 就位；不想索引的頁用 `noindex`（不是 robots disallow）。
  - 內容用 **SSR/SSG** 讓首包 HTML 就含內容與 meta。
  - **Core Web Vitals** 達標（圖片/字型/bundle 優化）。
  - 多語系設定 **hreflang + x-default**。
  - 上線後用 **Google Search Console** 監控索引狀態與複合式結果。
