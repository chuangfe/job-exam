# Q61 可規模化的前端架構（資深 / 技術主導）

當團隊從 3 人長到 30 人、程式碼從 1 萬行長到 30 萬行、頁面從 5 頁長到 200 頁時，早期「怎麼方便怎麼寫」的做法會開始崩壞：資料夾找不到東西、改一個共用元件炸掉三個功能、全域狀態變成誰都能改的泥球、建置越來越慢、新人上手要兩週。**可規模化的前端架構**要解決的不是「今天怎麼跑起來」，而是「一年後、十倍規模下，團隊還能不能穩定且快速地交付」。

這是一道資深 / 技術主導（tech lead）取向的題目，重點不在單一 API 的用法，而在**取捨與決策框架**。請以繁體中文（台灣用語）作答，並在需要處佐以 React + TypeScript 生態的思路。沒有唯一正解，但要能說清楚「為什麼這樣選、代價是什麼、什麼情況下不該這樣做」。

---

## Q1. 專案結構：feature-based vs layer-based、colocation 與模組邊界

- 說明 **layer-based（依技術切：`components/`、`hooks/`、`services/`、`utils/` 全站集中）** 與 **feature-based（依功能切：`features/checkout/`、`features/auth/` 各自完整）** 兩種結構的差異，以及各自在規模變大後的痛點。
- 什麼是 **colocation（就近擺放）**？「把測試、樣式、型別、子元件放在使用它的地方旁邊」帶來什麼好處？
- 什麼是**模組邊界（module boundary）**與**依賴方向（dependency direction）**？為什麼「feature 之間不應互相直接 import、共用要往下沉到 `shared/`」是重要規則？
- 如何用工具**強制**這些邊界（而不只是靠 code review 的自律）？

## Q2. Monorepo：Turborepo / Nx 的價值與 polyrepo 取捨

- 什麼是 monorepo？什麼是 workspace（npm / pnpm / yarn workspaces）？
- Turborepo 與 Nx 各自解決什麼問題？請至少說明三項價值：**共用套件（design system / util）**、**建置快取（含遠端快取）**、**任務編排（task pipeline / 只重建受影響的專案）**。
- Turborepo 與 Nx 的定位差異概觀為何？
- monorepo vs polyrepo（多個獨立 repo）的取捨：各自的優缺點與適用時機？什麼團隊規模 / 情境下 monorepo 才划算？

## Q3. Design System 作為獨立套件（版本化、發佈、消費）

- 為什麼要把 design system 抽成**獨立套件**，而不是散在各 app 裡？
- 說明它的三個生命週期：**版本化（versioning，SemVer）**、**發佈（publish，npm registry / 私有 registry / monorepo 內部套件）**、**消費（consume，多個 app 如何相依與升級）**。
- monorepo 內部套件（`workspace:*`）與發佈到 registry 的外部套件，在版本管理與升級節奏上有何不同？
- 發佈自動化可以用什麼工具（例：Changesets）？破壞性變更（breaking change）該如何溝通與遷移？

## Q4. 狀態架構分層：server / client(UI) / URL state

- 請區分三類狀態並各舉例：**server state（伺服器狀態）**、**client / UI state（本地狀態）**、**URL state（網址狀態）**。
- 為什麼 **server state 該交給 React Query（TanStack Query）/ SWR**，而不是塞進 Redux / Zustand？（快取、失效、重新請求、載入 / 錯誤狀態…）
- client state 用 **Zustand / Jotai / Context** 的取捨為何？Context 什麼時候夠用、什麼時候會出現效能問題？
- 哪些狀態**應該放進 URL**（query string / route params）？放進 URL 有什麼好處？
- 「避免全域狀態濫用」：什麼是「把所有東西都丟進全域 store」的反模式？如何判斷一個狀態該住在哪一層？

## Q5. 分層與抽象：API 層、資料模型、UI 元件分層

- 為什麼要有一個獨立的 **API 層 / data-access 層**（把 `fetch` / axios、端點、錯誤處理、DTO 轉換收斂起來），而不是在元件裡到處 `fetch`？
- 什麼是 **DTO（後端回傳格式）與 domain model（前端使用格式）的分離**？為什麼要做這層轉換（anti-corruption layer）？
- 說明 UI 元件分層：**presentational（呈現型）vs container（容器型）** 的分工，以及**原子設計（Atomic Design：atoms / molecules / organisms / templates / pages）** 的概念。
- 這些分層在現代 React（hooks、Server Components）下有什麼演變？「容器 / 呈現」的界線是否還那麼絕對？

## Q6. 效能預算、程式碼分割與可維護性防護網

- 什麼是**效能預算（performance budget）**？可以對哪些指標設上限（bundle 大小、Core Web Vitals：LCP / INP / CLS）？如何在 CI 自動守門（例：bundlesize、Lighthouse CI）？
- 程式碼分割（code splitting）策略：**route-based（按路由）**、**component-based（按元件 / `React.lazy`）**、**vendor 拆分**，各自適用時機？
- 列出讓大型專案「可維護」的防護網：**型別（TypeScript strict）**、**測試（單元 / 元件 / E2E 的比例，測試金字塔）**、**Lint / 格式化**、**Code Owners**、**CI 檢查**。這些各自防住什麼樣的退化？

## Q7. 決策框架：速度 vs 可擴展性 vs 工程成本、漸進式演進 vs 過度設計、ADR

- 提出一套**技術決策框架**：面對「要不要現在就上 monorepo / 微前端 / 自建 design system」這類選擇時，如何在**交付速度、可擴展性、工程成本（維護負擔）** 三者間權衡？
- 什麼是**過度設計（over-engineering）**？什麼是**漸進式演進（evolutionary architecture）**？請說明「為今天的問題設計、為明天保留改動空間」與「為想像中的未來預先建一堆抽象」的差別。
- 舉例：一個 5 人新創與一個 50 人成熟團隊，對同一個架構問題（例如要不要拆 monorepo）合理的答案為什麼會不同？
- 什麼是 **ADR（Architecture Decision Record，架構決策紀錄）**？它記錄哪些欄位（脈絡 / 決策 / 後果 / 替代方案）？為什麼「把決策與當時的理由寫下來」對長期團隊很重要？
