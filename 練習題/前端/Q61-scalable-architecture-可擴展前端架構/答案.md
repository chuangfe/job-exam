# Q61 可規模化的前端架構 解析

核心觀念：**架構是「為了規模與時間」而付出的成本，不是越多越好**。好的前端架構讓「加人、加功能、加流量」時的邊際成本維持在低檔——找得到東西、改動有邊界、建置夠快、退化被 CI 擋下。但每一道抽象、每一層邊界、每一套工具都有維護成本，資深工程師的價值就在於**判斷「現在值不值得付這個成本」**：為今天的問題設計、為明天保留改動空間，而不是為想像中的未來預先蓋一座城堡。

## Q1. 專案結構：feature-based vs layer-based、colocation 與模組邊界

**layer-based（依技術切）**——按「這是什麼技術」分：

```
src/
  components/    # 全站所有元件
  hooks/         # 全站所有 hook
  services/      # 全站所有 API
  utils/
  types/
```

- 小專案很直覺，但規模一大就痛：改一個「結帳」功能要跳好幾個資料夾（`components/Checkout*`、`hooks/useCheckout*`、`services/checkout*`）；`components/` 塞了 200 個檔案；無法一眼看出「哪些檔案屬於同一個功能」，也很難整包刪除或搬移一個功能。

**feature-based（依功能切）**——按「這屬於哪個業務功能」分：

```
src/
  features/
    checkout/
      components/
      hooks/
      api/
      types.ts
      index.ts        # 對外的公開介面（barrel）
    auth/
    product/
  shared/             # 跨 feature 共用的東西才放這
    ui/               # design system 基礎元件
    lib/              # 純函式 util
    api/              # base client、攔截器
  app/                # 路由、providers、進入點
```

- 優點：**高內聚**，一個功能的所有東西住在一起；新人接手某功能只需看一個資料夾；刪 / 搬功能很乾淨；不同人改不同 feature 較少衝突。
- 這是中大型 React 專案的主流建議（Bulletproof React 等實務指南皆採此結構）。痛點：需要紀律決定「什麼夠共用才下沉到 `shared/`」，否則 `shared/` 會變成新的垃圾場。

**colocation（就近擺放）**：把測試、樣式、型別、只被某元件用到的子元件，放在使用它的檔案旁邊，而不是集中到全站 `__tests__/`、`styles/`。

```
features/checkout/components/
  CartSummary/
    CartSummary.tsx
    CartSummary.test.tsx
    CartSummary.module.css
    useCartTotal.ts        # 只有這個元件用 → 就放旁邊
```

- 好處：改動時相關檔案都在眼前、刪除元件時附屬檔一起走（不留孤兒）、降低「這個 util 到底誰在用」的心智負擔。原則：**東西應該住在離它被使用的地方越近越好**，等到真的有第二個使用者，才往上提升到更共用的層。

**模組邊界與依賴方向**：把每個 feature 當成一個有「公開介面」的模組——只透過它的 `index.ts` 對外暴露必要的東西，內部檔案視為私有。關鍵規則：

- **feature 之間不要互相直接 import**（`features/checkout` 不該 `import from features/product/內部檔案`）。橫向相依會讓兩個功能纏在一起，改 A 壞 B。
- **共用要「往下沉」**：兩個 feature 都需要的東西，抽到 `shared/`。依賴方向永遠是**單向、指向更穩定 / 更底層**的方向：`app → features → shared`，反向禁止（`shared` 不可 import `features`）。這其實就是**穩定依賴原則**：越被依賴的東西越該穩定。

**如何強制邊界**（靠工具，不靠自律）：

- ESLint `no-restricted-imports` / `eslint-plugin-import` 的 `import/no-restricted-paths`：禁止跨 feature 深層 import、禁止 `shared` 反向依賴。
- Nx 的 **module boundary rules**（`@nx/enforce-module-boundaries`）：用 tag 標記每個專案 / 資料夾，宣告「誰可以依賴誰」，違反在 lint 階段就報錯。
- 用 `index.ts`（barrel）只導出公開 API，內部路徑不對外，讓「越界 import」在視覺與工具上都困難。

## Q2. Monorepo：Turborepo / Nx 的價值與 polyrepo 取捨

**monorepo**：一個 git repo 內放多個可獨立建置 / 部署的專案（多個 app + 多個 package），共享一份設定與依賴管理。**不等於**把所有東西寫成一坨（那是 monolith）——monorepo 內部仍有清楚的套件邊界。

**workspace**：套件管理器（pnpm / npm / yarn）的功能，讓一個 repo 內的多個 package 互相以本地連結相依。

```
package.json          # root，宣告 workspaces
pnpm-workspace.yaml
apps/
  web/                # Next.js 主站
  admin/             # 後台
packages/
  ui/                 # design system（@acme/ui）
  utils/              # 共用工具（@acme/utils）
  eslint-config/      # 共用設定
  tsconfig/
```

`apps/web/package.json` 裡寫 `"@acme/ui": "workspace:*"`，就會連到本地的 `packages/ui`，改 design system 立刻反映到 app，不用先發佈。

**Turborepo / Nx 的價值**：

1. **共用套件**：design system、util、設定（eslint / tsconfig）只寫一次，多個 app 共用；改一次全站受益，版本天然一致。
2. **建置快取（含遠端快取）**：對每個任務的輸入（原始碼、依賴、環境）算 hash，輸出快取起來。輸入沒變就直接回放上次結果（cache hit），不重跑。**遠端快取**讓 CI 與同事之間共享——同事或 CI 已經 build 過的東西，你本地直接拉快取，秒級完成。
3. **任務編排（task pipeline）**：宣告任務相依（`build` 要在 `test` 前、`ui` 要先 build 才能 build `web`），工具據此**平行化**並排出正確順序。搭配「只重建受影響的專案（affected）」：只有這次改動波及的 package 才重跑 test / build，PR 檢查大幅加速。

**Turborepo vs Nx 定位**（概觀）：

- **Turborepo**（Vercel）：輕量、聚焦「任務編排 + 快取」，設定少（`turbo.json`），不強加專案結構，漸進導入容易。適合「我已經有 monorepo，只想讓建置變快」。
- **Nx**：功能更全的整合式平台——除了快取 / affected，還有 code generators、module boundary 規則、豐富的 plugin 與依賴圖視覺化。功能強但概念多、較有既定作法。適合「想要一整套治理與工具鏈」的較大團隊。

**monorepo vs polyrepo**：

| 面向 | monorepo | polyrepo |
|------|----------|----------|
| 共用程式碼 | 極方便，原子式跨專案改動（一個 PR 同時改 lib + 用它的 app） | 要發版、跨 repo 對齊，容易版本地獄 |
| 版本一致性 | 天然一致 | 各 repo 各自漂移 |
| 權限 / 隔離 | 較粗（整個 repo） | 天然按 repo 隔離、CI 獨立 |
| 工具需求 | **必須**有 Turborepo / Nx，否則建置全跑很慢 | 各 repo 單純 |
| 上手 / clone 成本 | repo 大、需要工具鏈 | 小而獨立 |

- **適用**：monorepo 在「多個 app 共用大量程式碼、團隊要頻繁跨專案改動、有能力維護工具鏈」時划算——通常是**中大型團隊 / 多產品線**。5 人團隊、單一 app，polyrepo 或單 repo 就好，硬上 monorepo + Nx 反而是工程成本的浪費（見 Q7）。

## Q3. Design System 作為獨立套件

**為何抽成獨立套件**：把按鈕、輸入框、色彩 / 間距 token、主題散在各 app，會導致「同一顆按鈕三種樣子」、規範改了 app 沒跟上。抽成獨立套件（`@acme/ui`）後，它是**單一事實來源**：設計規範以程式碼形式集中、版本化、被多個 app 一致消費；並可搭配 Storybook 作為活文件與視覺測試。

**三個生命週期**：

- **版本化（SemVer）**：`MAJOR.MINOR.PATCH`。`PATCH`=修 bug 不改介面；`MINOR`=新增功能、向後相容；`MAJOR`=破壞性變更（改了 prop、刪了元件、換了行為）。消費端據此判斷升級風險。
- **發佈（publish）**：
  - **外部 registry**：發到 npm 公開或私有 registry（GitHub Packages / Verdaccio / 公司私有）。適合「跨組織、跨 repo」共用。
  - **monorepo 內部套件**：不發佈，直接用 `workspace:*` 本地連結，隨主 repo 一起版控。
- **消費（consume）**：app 在 `package.json` 宣告相依並鎖版本（`^`、精確版本或 `workspace:*`）；升級時檢視 changelog、評估 MAJOR 的遷移成本。

**內部套件 vs 發佈套件的差異**：

- 內部（`workspace:*`）：**沒有版本落差**——改了 design system，所有 app 立刻拿到最新，適合快速迭代；代價是破壞性變更會**同時**影響所有 app，得在同一個 PR 一起修（原子式改動，也是 monorepo 的優點）。
- 發佈到 registry：各 app **自己決定何時升級**，解耦、可漸進遷移；代價是版本會漂移、可能有人卡在舊版，需要主動治理。

**發佈自動化與破壞性變更**：

- **Changesets**：開發者在 PR 附一則 changeset（宣告這次是 patch/minor/major + 說明），合併後自動彙整 changelog、bump 版本、發佈。讓「版本 + changelog」變成流程的一部分而非事後補。
- **破壞性變更的溝通與遷移**：升 MAJOR + 在 changelog / migration guide 寫清楚「改了什麼、怎麼改」；能用 **codemod（如 jscodeshift）** 就提供自動遷移腳本；重要 API 先標 `@deprecated` 給一個過渡期，再於下個 MAJOR 移除，別直接砍。

## Q4. 狀態架構分層：server / client(UI) / URL state

**大多數「狀態管理很痛」的專案，根因是把不同種類的狀態混在同一個全域 store 裡管。** 先分類，才知道每種該用什麼工具。

| 類型 | 定義 | 例子 | 建議工具 |
|------|------|------|---------|
| **server state** | 資料的真正擁有者在後端，前端只是**快取的副本** | 商品列表、使用者資料、訂單 | **React Query / SWR** |
| **client / UI state** | 只存在前端、後端不關心 | 側欄開合、目前選的分頁、表單暫存、主題 | **useState / Zustand / Jotai / Context** |
| **URL state** | 該被網址表達、可分享 / 可書籤的狀態 | 目前頁碼、搜尋關鍵字、篩選條件、tab | **router（query string / route params）** |

**server state 為何交給 React Query，而不是 Redux / Zustand**：伺服器資料有一堆「快取」的內建難題——快取、過期失效（stale）、背景重新請求（refetch）、去重（dedupe）、載入 / 錯誤狀態、重試、分頁 / 無限捲動。若塞進 Redux，你得**手刻**這一切（action、reducer、loading flag、失效邏輯），寫一堆樣板還容易錯。React Query 把這些變成宣告式的 `useQuery` / `useMutation`，快取與同步是它的本職。**Redux / Zustand 是「客戶端狀態容器」，不是「伺服器快取」，別拿來當快取用。**

```ts
// server state：宣告「要什麼資料」，快取 / 失效 / 重試交給它
const { data, isLoading, error } = useQuery({
  queryKey: ['products', filters],
  queryFn: () => productApi.list(filters),
});
```

**client state 的取捨**：

- **Context**：適合「低頻變動、範圍明確」的全域值（主題、語系、當前使用者）。陷阱：Context value 一變，**所有 consumer 都會 re-render**，若拿來裝高頻變動或大物件，會造成效能問題。要拆多個小 Context 或改用專門的 store。
- **Zustand**：極簡的全域 store，可**選擇性訂閱**（只訂閱你要的欄位，避免無關 re-render），沒有 Provider 包裹地獄。適合中量的跨元件 client state。
- **Jotai**：atom 為單位的細粒度狀態，由下而上組合，很適合「很多彼此獨立的小狀態」。

**哪些放進 URL**：任何「重新整理後該保留、想分享連結、想用瀏覽器上一頁」的狀態——頁碼、搜尋詞、篩選、排序、目前 tab、Modal 開關（進階）。好處：**可分享、可書籤、可用瀏覽器前進 / 後退、重新整理不遺失、SSR 友善**。把篩選條件放進 query string，使用者把連結貼給同事就能看到同一份結果。

**避免全域狀態濫用**：反模式是「反正之後可能有人要用，先丟進全域 store」，結果 store 變成誰都能改、改了不知道影響誰的泥球。判斷一個狀態該住哪：

1. 資料的擁有者是後端嗎？→ **server state（React Query）**。
2. 該被網址表達 / 分享嗎？→ **URL state**。
3. 只有一個元件與其子孫要用嗎？→ **local useState + 往下傳 / 往上提升到最近共同祖先**。
4. 真的多個不相干的元件要共享、且非上述三種？→ 才用**全域 client store**。

原則：**狀態應該住在「需要它的元件們的最近共同祖先」，能局部就別全域**。

## Q5. 分層與抽象：API 層、資料模型、UI 元件分層

**獨立 API 層 / data-access 層**：不要在元件裡到處 `fetch`。把「base client（baseURL、headers、auth token、攔截器）、端點定義、錯誤處理、重試、DTO↔model 轉換」收斂到一層。

```
shared/api/
  client.ts          # axios 實例、攔截器、統一錯誤處理
features/product/api/
  productApi.ts       # list / get / create，回傳 domain model
```

- 好處：元件不碰 HTTP 細節、端點改一處全站生效、auth / 錯誤處理統一、方便測試時 mock 這一層。React Query 的 `queryFn` 就呼叫這層，各司其職。

**DTO vs domain model（防腐層 anti-corruption layer）**：後端回傳的格式（DTO）不該直接滲透到整個前端。做一層轉換：

```ts
// DTO：後端的樣子（snake_case、巢狀、多餘欄位）
type ProductDTO = { product_id: number; display_name: string; price_cents: number };
// domain model：前端好用的樣子
type Product = { id: string; name: string; price: number };

const toProduct = (d: ProductDTO): Product => ({
  id: String(d.product_id),
  name: d.display_name,
  price: d.price_cents / 100,
});
```

- 為何值得：後端改欄位命名 / 結構時，**只需改轉換函式一處**，UI 不受牽動（隔離外部變化）；前端可用自己語意清楚的模型（camelCase、正確型別、正確單位）。這是一種以小成本換取「不被外部格式綁架」的保險——不必每個專案都做到極致，但在後端易變 / 多來源時很划算。

**UI 元件分層**：

- **presentational（呈現型）**：只管「長怎樣」，靠 props 拿資料、靠 callback 往上通知，沒有資料請求與業務邏輯 → 好重用、好測、好放 Storybook。
- **container（容器型）**：只管「資料與行為」，接 hook / API、算邏輯，把結果餵給呈現型元件。
- **原子設計（Atomic Design）**：atoms（Button、Input）→ molecules（SearchField = Input+Button）→ organisms（Header、ProductCard）→ templates（版面骨架）→ pages（灌真實資料）。它給 design system 一套**由小到大的組合語彙**，讓元件庫有層次、可組合。

**現代 React 的演變**：自從 hooks 出現，「用 hook 把邏輯抽離」大多取代了經典的 container/presentational 拆法——邏輯裝進 custom hook（`useProductList()`），元件自己呼叫，不必硬拆兩個檔案。到了 **RSC（React Server Components）/ Next.js App Router**，界線變成 **Server Component（在伺服器抓資料、零 JS 送到瀏覽器）vs Client Component（`"use client"`，負責互動）**——這其實是「資料 / 互動」關注點分離的新形態。所以「容器 / 呈現」的**精神（分離資料與呈現）仍在**，但不再是死板的二分檔案結構；Atomic Design 的分層語彙則在 design system 中依然實用。

## Q6. 效能預算、程式碼分割與可維護性防護網

**效能預算（performance budget）**：預先為效能指標設上限，超過就讓 CI **紅燈**，把效能當成像測試一樣的驗收條件，避免「一點一點胖回去」。可設的預算：

- **資源大小**：JS bundle（如首屏 JS < 170KB gzip）、單一 chunk 上限、圖片大小。工具：`bundlesize`、`size-limit`、webpack/next 的 bundle analyzer。
- **Core Web Vitals**：**LCP**（最大內容繪製，良好 < 2.5s）、**INP**（互動到下次繪製，取代舊 FID，良好 < 200ms）、**CLS**（版面位移，良好 < 0.1）。工具：**Lighthouse CI**（在 PR 跑、對指標設 assertion）、真實使用者監測（RUM / web-vitals 上報）。

**程式碼分割策略**：

- **route-based（按路由）**：最高 CP 值，換頁才載入該頁的程式碼。Next.js 按頁天然分割；純 React 用 `React.lazy` + `Suspense` 包路由元件。
- **component-based（按元件）**：把「重、且非首屏必要」的元件延後載入——大型圖表、富文字編輯器、Modal 內容、地圖。`const Chart = React.lazy(() => import('./Chart'))`。
- **vendor 拆分**：把變動少的第三方庫拆成獨立 chunk，利用長期快取（app 改版時 vendor chunk 的 hash 不變，使用者不用重載）。
- 取捨：切太碎會有太多小請求 / 瀑布，反而變慢；原則是「切在使用者感知得到、又不常一起用到的邊界」。

**可維護性防護網**（每一道防住不同退化）：

| 防護網 | 防住什麼 |
|--------|---------|
| **TypeScript strict** | 型別錯誤、null / undefined、重構時漏改的呼叫端；讓 IDE 能安全重構 |
| **測試金字塔** | 大量快的**單元 / 元件測試**（Vitest/Jest + Testing Library）+ 少量關鍵路徑 **E2E**（Playwright / Cypress）。防邏輯退化、防「改 A 壞 B」 |
| **Lint / 格式化** | ESLint 抓反模式與規則違反、Prettier 統一格式免除風格爭論；配合 pre-commit hook（husky + lint-staged）擋在提交前 |
| **CODEOWNERS** | 指定某目錄 / 套件的 PR 一定要對應負責人審核（design system 改動要 UI owner 過目），把知識與把關落到人 |
| **CI 檢查** | 把上述全部 + 效能預算變成 merge 的必要條件（required checks），不靠自律靠流程 |

測試比例遵循**測試金字塔**：底層多、頂層少——單元測試快又穩，E2E 慢又脆，只留在最關鍵的使用者流程（登入、結帳）。

## Q7. 決策框架：速度 vs 可擴展性 vs 工程成本、演進 vs 過度設計、ADR

**技術決策框架**（面對「要不要現在上 monorepo / 微前端 / 自建 design system」）：

1. **現在真正的問題是什麼？** 有具體痛點（建置 10 分鐘、共用元件對不齊）才動手，不要為假想問題動工。
2. **三軸權衡**——
   - **交付速度**：這個決策讓我們這季更快還是更慢出貨？
   - **可擴展性**：一年後 / 十倍規模下還撐得住嗎？
   - **工程成本**：導入 + 長期維護的負擔（誰維護工具鏈？新人學習曲線？）划得來嗎？
3. **可逆性**：這個決策好不好回頭？**單向門（難反轉，如選框架、拆微前端）要慎重多方評估；雙向門（好回退）就快速試、錯了再改。**
4. **最小可行架構**：能用最簡單的方案解決今天的問題就先用它，並確保**未來要升級時不會被卡死**（保留清楚邊界、避免不可逆綁定）。

**過度設計 vs 漸進式演進**：

- **過度設計（over-engineering）**：為想像中的未來預先蓋一堆抽象——「以後可能會有 5 個 app 所以現在先上 Nx」「可能會換後端所以每個欄位都做三層 adapter」。代價是今天就付出複雜度、拖慢交付，而那個未來常常沒來，或來的時候長得跟你猜的不一樣（YAGNI：You Aren't Gonna Need It）。
- **漸進式演進（evolutionary architecture）**：**為今天的問題設計，為明天保留改動空間**。先用單 repo + feature 結構，把邊界劃清楚；當建置真的變慢、共用真的變多，**再**演進成 monorepo。清楚的模組邊界本身就是「保留改動空間」——之後要抽套件、拆服務都容易。關鍵差別：**保留選項（低成本、可逆）≠ 預先實作（高成本、賭未來）。**

**同一問題、不同團隊、不同答案**（要不要拆 monorepo）：

- **5 人新創、單一產品**：合理答案通常是**先不要**。人少、共用不多、沒人維護工具鏈，monorepo + Nx 的治理成本大於收益；把時間花在找 PMF 與出貨。單 repo + 乾淨的 feature 結構足矣。
- **50 人、多產品線、共用大量元件**：monorepo 很可能**划算**——原子式跨專案改動、共用 design system、遠端快取省下大量 CI 時間、統一治理。此時「不拆」的協調成本才是真痛點。
- 這說明**沒有放諸四海皆準的最佳架構，只有「對這個團隊、這個階段、這些限制」的最適解**。資深工程師的判斷力正在於此。

**ADR（Architecture Decision Record，架構決策紀錄）**：把「重要架構決策 + 當時的理由」寫成一份短文件，隨程式碼版控（通常放 `docs/adr/0001-xxx.md`）。典型欄位：

```
# ADR-0007: 採用 feature-based 專案結構

狀態（Status）：已接受 / 已取代 / 已棄用
脈絡（Context）：為什麼需要決定？當時面對的問題與限制。
決策（Decision）：我們決定做什麼。
後果（Consequences）：帶來的好處與代價、要承擔什麼。
替代方案（Alternatives）：考慮過哪些、為何沒選。
```

- 為何重要：架構決策的**理由**若不記錄，半年後（或新人來時）沒人記得「當初為什麼這樣選」，於是要嘛盲目維持、要嘛盲目推翻。ADR 讓決策**可追溯、可檢討**——當脈絡改變（規模到了、限制沒了），可以有依據地說「當初的前提已不成立，該演進了」，並寫下新的 ADR 取代舊的。它把架構變成**有紀錄的演進史**，而非一堆無人知曉出處的既成事實。
