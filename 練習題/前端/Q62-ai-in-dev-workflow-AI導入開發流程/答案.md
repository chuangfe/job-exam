# 將 AI 導入開發流程（含使用邊界）解析

核心心法：**AI 是「加速器」與「結對夥伴」，不是「決策者」與「責任承擔者」。** 你按下合併鍵，程式碼就是你的責任，不是 AI 的。以下逐題解析。

## Q1. AI 在開發流程有哪些應用場景？

| 場景 | 怎麼用 | 擅長度 |
|------|--------|--------|
| **程式碼生成** | 產樣板（CRUD、表單、設定檔）、把需求變成初版、寫正規表達式 | 高（樣板類最省時） |
| **測試生成** | 依實作補單元測試、想邊界案例、產測試假資料 | 高，但要人補「真正該驗的商業規則」 |
| **PR 審查輔助** | 摘要 diff、標出可疑段、提醒漏掉的邊界 | 中，當「第二雙眼睛」而非最終裁決 |
| **文件生成** | JSDoc / docstring、README、API 說明、更新既有文件 | 高 |
| **規範／風格檢查** | 解釋 lint 規則、統一命名、指出不一致 | 中高 |
| **重構** | 抽函式、消除巢狀、換型別、改寫法 | 中高，需搭配測試驗證行為不變 |
| **debug 協助** | 解讀錯誤訊息與堆疊、提假設、縮小範圍 | 中，最終仍要自己驗證假設 |
| **commit / PR 說明** | 依 diff 產生訊息、整理變更重點 | 高 |

- **特別擅長**：重複、樣板化、有明確規則、範圍小的工作（boilerplate、格式轉換、寫測試、寫文件、解釋程式碼）。這些「查得到、規則明確」的事 AI 又快又準。
- **要格外小心**：牽涉**商業邏輯正確性、安全性、架構決策、效能取捨、與外部系統的整合細節**。這些需要領域知識與判斷，AI 容易「看起來對但實際錯」。
- 一句話：**AI 幫你打字、幫你想，但不幫你負責。**

## Q2. 工具生態與 CI 整合？

- **GitHub Copilot**：IDE 內的**即時補全**。強在「順手」——邊打邊補下一行/整個函式。適合寫熟悉的樣板、重複結構。互動成本最低，但視野只在游標附近。
- **Claude Code / Cursor**：**agent 型**工具，能讀整個 repo、跨多檔修改、跑指令、跑測試、自我修正。適合「較大的任務」：實作一個功能、跨檔重構、修一個要追很多檔的 bug。威力大，也因此**更要盯產出**。
- **Code review bots**（例如 PR 上自動留言的審查機器人）：在 PR 開啟時自動摘要變更、標出潛在問題並留言。定位是**輔助、非阻擋**——幫 reviewer 節省第一輪的力氣。
- **CI/CD 整合原則**：把 AI 審查當成**非阻擋性（non-blocking）的參考意見**。真正決定能不能合併的 **gate 仍是可客觀驗證的關卡**：lint、type-check、單元／整合測試、安全掃描（SAST／依賴弱點掃描）、人工核准。**不要讓「AI 說 OK」變成放行條件**，因為它會有偽陰／偽陽。
- 三種互動模式的適用：
  - **即時補全** → 寫程式當下、樣板與重複結構。
  - **對話式 agent** → 較大、跨檔、需要來回試的任務。
  - **PR 自動審查** → 合併前的第二意見與變更摘要。

## Q3. 使用邊界——安全、隱私、授權、保密（重點）

這一題是本題的核心，也是職缺最在意的部分。

- **機密資料不外流**：
  - **絕對不要**把以下貼給外部雲端 AI：完整專有原始碼、客戶／使用者個資、金鑰／密碼／API token／憑證、內部架構與資安文件、尚未公開的商業資訊。
  - 後果：資料可能被**留存、被用於訓練、經由日誌外洩**；金鑰外洩可直接被盜用；個資外流違反 GDPR／個資法，公司要負法律責任；營業秘密一旦外流極難挽回。
  - 心法：**假設你貼進去的東西，都可能離開你的控制。** 貼之前先問「這東西外流我扛不扛得起？」
- **授權與著作權**：
  - AI 是用大量公開程式碼訓練的，**有機會吐出與受授權程式碼高度相似的片段**（例如 GPL 這類 copyleft 授權）。若原封不動放進商用專案，可能觸發授權義務（如被要求開源）或侵權。
  - AI 產出的**著作權歸屬是灰色地帶**：多數見解認為「純機器生成、無人類創作性」的產出難以主張著作權，各地法規與平台條款不一。企業使用要看該工具的服務條款怎麼約定產出權利與賠償。
  - 實務：大段、可辨識的產出要警覺；能用開啟「過濾公開程式碼比對」功能的工具就開；有疑慮的片段自己重寫。
- **公司政策與資料留存**（使用前先確認）：
  - 這個服務**會不會拿我的輸入去訓練模型**？（很多有「企業版／API 不用於訓練」的條款）
  - **資料留存多久**、存在哪、誰能存取？
  - 公司有沒有**核可工具清單**？只用被核可的。
  - 有沒有簽 DPA／企業版合約、有沒有零留存（zero-retention）選項。
- **地端 vs 雲端模型**：
  - **地端（self-hosted / on-prem）**：資料不離開公司網路，隱私與合規最強；適合金融、醫療、政府、有嚴格資安要求的團隊。代價是**模型能力通常較弱、要自己養硬體與維運、成本高**。
  - **雲端**：能力最強、最省事，但資料要送出去，得靠合約與企業版控管風險。
  - 取捨：**資料敏感度愈高，愈往地端靠**；一般開放專案用雲端企業版即可。
- **PII 遮罩**（非用不可又碰到個資時）：
  - **去識別化／匿名化**：把姓名、身分證、電話、Email 換成假值再貼。
  - **用假資料重現問題**：debug 時用最小、無真實個資的重現案例。
  - **只貼必要片段**：不要整包貼，剪到剛好能問問題就好。
  - 有條件的話用會自動做 PII 偵測／遮罩的 DLP 工具當守門員。

## Q4. 品質把關——為什麼 AI 產出一定要人審查？

- 常見的坑：
  - **幻覺**：捏造不存在的 API、參數、套件名、設定選項——講得斬釘截鐵但根本沒這東西。
  - **過時**：訓練資料有時間點，會給出已棄用（deprecated）的寫法、舊版 API、過時的最佳實務。
  - **細微 bug**：主線正確，但**邊界條件錯**（空陣列、null、off-by-one、時區、浮點、併發），最難抓。
  - **似是而非的安全問題**：漏掉輸入驗證、字串拼接 SQL、把祕密寫死、用了弱加密、CORS 開太大。
- **「編譯過／測試綠燈」≠「正確」**：測試只驗到你（或 AI）想到的案例；沒有測試涵蓋的錯誤照樣綠燈。AI 有時還會**順手把測試也改成配合它的錯誤實作**，於是「自己驗自己」形成假象。
- 驗證手段：
  1. **讀懂每一行**——看不懂就不要合併（見可解釋性）。
  2. **跑既有測試**，並**自己補邊界測試**（尤其 null／空／極值／錯誤路徑）。
  3. **靜態分析／type-check／安全掃描**。
  4. **小步提交**，一次審一小塊，出錯好回退。
  5. 對關鍵邏輯，**手動走一遍或加 log 實際觀察**。
- **可解釋性**：你要能對 reviewer 與未來的自己解釋「這段為什麼這樣寫」。**看不懂的程式碼不要合併**——要嘛請 AI 逐行解釋直到你懂、要嘛自己重寫成你懂的版本。責任在你，不在 AI。

## Q5. 怎樣算「有效使用」AI？

- **好提示 vs 壞提示**：
  - 壞：「幫我寫個表單。」→ 太模糊，AI 只能亂猜。
  - 好：「用 React + TypeScript + react-hook-form 寫一個登入表單，欄位 email／密碼，email 要驗格式、密碼至少 8 碼，錯誤訊息用繁體中文，樣式用我們既有的 `<TextField>` 元件（附型別）。只回程式碼。」
  - 要素：**明確目標 + 技術限制 + 既有慣例／範例 + 預期輸出格式**。
- **提供上下文是關鍵**：貼相關檔案、型別定義、專案慣例、**錯誤訊息全文與堆疊**。上下文愈完整，幻覺愈少、產出愈貼合你的專案（agent 型工具能自己讀 repo，但你仍要指引它看對地方）。
- **小步驗證**：把大任務拆小，每一小步都**產出→審查→測試→提交**再往下。理由：一次產一大坨很難審、錯誤會互相掩蓋、要回退時整包都得丟。小步走，錯誤好定位、好回退。
- **當結對夥伴而非取代判斷**：
  - AI 提方案，**你決定要不要採納**；把它當一個「知識豐富但會唬爛、需要被審的資淺同事」。
  - **不該外包給 AI 的判斷**：架構與技術選型、安全與隱私決策、商業邏輯正確性的最終確認、要不要合併／上線。這些要有人（你）負責。

## Q6. 團隊怎麼導入 AI？

- **規範（policy）**：
  - **核可工具清單**：只准用哪些工具、哪個版本／方案（企業版 vs 個人版）。
  - **紅線資料清單**：哪些資料絕對不能貼（原始碼？客戶資料？金鑰？）明文寫出。
  - **產出審查流程**：AI 產出一律走正常 Code Review，PR 可標注「含 AI 協助」。
  - **授權注意**：開啟公開程式碼過濾、可疑片段自行重寫。
- **教育訓練**：不只教「怎麼寫好提示、怎麼用工具」，更要教**「什麼不能做」**——資料邊界、授權風險、幻覺與把關。讓「會用」和「守界線」一起長。
- **量測生產力**：
  - 可看：週期時間（lead time）、PR 合併速度、bug 率／回退率、開發者滿意度。
  - **陷阱**：**只看「產出行數／接受率」會誤導**——AI 很會產「量」，但量多不等於價值高，甚至可能製造更多要維護與審查的負債。要看**結果品質與交付速度**，而非碼量。
- **風險控管**：
  - 用**企業版 + 零留存 / 不用於訓練**條款。
  - **DLP**（資料外洩防護）攔截敏感資料外送。
  - **稽核日誌**：誰用了什麼、送了什麼。
  - 敏感專案改用**地端模型**。
  - CI 上跑**安全掃描與依賴弱點掃描**，不假設 AI 產出安全。

## Q7. 綜合——好 vs 壞 AI 使用情境對照 + 團隊守則範例

### 好 vs 壞做法對照

| 情境 | ✅ 好做法 | ❌ 壞做法 |
|------|----------|----------|
| 貼資料 | 貼**去識別化的最小重現片段** | 把整包含客戶個資／金鑰的檔案貼給雲端 AI |
| 產出處理 | **逐行讀懂再合併**、補測試驗證 | 沒看懂就整段貼上、綠燈就合併 |
| 授權 | 開啟公開程式碼過濾、可疑片段重寫 | 大段照抄不明授權的產出進商用專案 |
| 提示 | 給目標＋限制＋慣例＋輸出格式 | 「幫我寫個功能」然後全盤接受 |
| 任務拆解 | 小步：產→審→測→提交 | 一次讓 agent 改 30 個檔再一次收 |
| 工具選擇 | 用公司核可的**企業版**（不留存） | 私自用個人版把公司碼餵進去 |
| debug | 貼**完整錯誤訊息**、自己驗證 AI 的假設 | 盲信 AI 給的「原因」直接改 |
| 責任 | 「我用 AI 協助，但結果我負責」 | 「這是 AI 寫的，錯不在我」 |
| 測試 | 自己補邊界測試 | 讓 AI 把測試改成配合它的錯誤實作 |
| 敏感專案 | 依敏感度改用地端模型 | 不分場合一律丟雲端 |

### 團隊 AI 使用守則範例（可直接貼 wiki）

1. **只用公司核可的 AI 工具與方案**（企業版、契約載明不留存／不用於訓練）；清單見 [連結]。
2. **紅線資料絕不外送**：客戶／使用者個資、金鑰密碼 token 憑證、未公開原始碼與架構文件。非貼不可時先**去識別化並剪到最小片段**。
3. **AI 產出一律走 Code Review**，與人寫的程式碼同標準；PR 標注是否含 AI 協助。
4. **看不懂的程式碼不得合併**——要能對團隊解釋每一行。
5. **AI 產出要用測試驗證**，並自行補關鍵邊界測試；不得讓 AI 竄改測試來遷就實作。
6. **注意授權**：開啟公開程式碼過濾，疑似受授權的大段產出須自行重寫。
7. **CI 的 gate 是 lint／type-check／test／安全掃描與人工核准**；AI 審查僅供參考、不作為放行條件。
8. **敏感專案**依資料敏感度改用地端／私有部署模型。
9. **責任歸屬**：使用 AI 的工程師對其產出負全責，「AI 寫的」不是免責理由。
10. 有**疑慮先問**資安／法務，不要先貼了再說。

---

**一句總結**：把 AI 當成一個很有生產力、但需要被審查、且不能看機密的資淺結對夥伴——用它加速，靠自己把關。
